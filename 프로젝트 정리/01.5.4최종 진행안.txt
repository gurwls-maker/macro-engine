🔥 5.4 전체 목적 (확정)

👉 Records를 기준으로 Today를 안정적으로 연결

한 줄로:

Today(계산) ↔ Record(저장) 흐름을 하나로 만든다
🧠 현재 상태 (문제 구조)

지금은 이렇게 분리돼 있음:

calculate() → todayBalance / todayAdherence
recordsState → meals / weight / adherencePercent

👉 문제:

계산값과 저장값이 “느슨하게 연결됨”
언제 저장되는지 불명확
기준이 섞일 가능성 있음
🎯 5.4에서 반드시 정해야 할 3가지
1️⃣ “Today의 기준”을 정의

👉 제일 중요

선택지는 2개:

A안 (권장)
Today = 항상 calculate() 기준
Record = 결과 저장용
B안
Today = Record 기반

👉 우리는 A안으로 가야 함

이유:

계산 로직이 이미 완성됨
Record는 snapshot 역할이 맞음
2️⃣ “언제 저장할지” 정의

👉 지금 애매한 부분

저장 타이밍 규칙 (확정 필요)
1. 식사 추가/수정/삭제 시 → 즉시 저장
2. calculate() 이후 → adherencePercent 갱신 저장
3. 체중 입력 시 → 즉시 저장

👉 핵심:

👉 TodaySnapshot → Record에 반영되는 시점 명확화

3️⃣ “무엇을 저장할지” 확정

현재 구조:

record = {
  meals,
  weight,
  adherencePercent,
  goalSnapshot
}

👉 유지하면서 규칙만 정리

🔥 리스크 포인트 (미리 경고)

→ localStorage 성능 + 데이터 꼬임

2️⃣ 잘못된 시점 저장

→ 과거 record 덮어쓰기

3️⃣ snapshot 없이 계산 재사용

→ 불일치 발생

✅ 5.4 완료 기준

이 상태 되면 성공:

- Today 계산과 Record 저장이 항상 일치
- 날짜 바꿔도 데이터 꼬임 없음
- Weekly가 안정적으로 record만 읽음
- snapshot 기반으로 확장 가능
🚀 한 줄 요약

👉 5.4는 “계산 → 저장 → 조회 흐름을 하나로 묶는 단계”


보정사항

- 오늘/과거 record 구분
