## Actual fresh-state audit

An isolated Chromium origin was opened with localStorage cleared.

Observed first render:

```text
target kcal: 4,013kcal
protein: 128g
carbs: 595g
fat: 125g
validation warning: none
body source chip: 출처: 기본값
```

Values immediately written by `render() -> saveState()`:

```text
weight: 75kg
height: 173cm
age: 32
gender: male
goal: lean_bulk
activityLevel: moderate
workType: heavy_field
exerciseManagementMode: exercise
exerciseProfile: bodybuilding
routine: PUSH
weightDuration: 60
cardioDuration: 30
```

This is not a harmless visual demo. The values enter the real target-calorie and macro pipeline and are persisted as if the user had chosen them.

### Code-path findings

1. `DEFAULT_STATE` contains a fully calculable example person and exercise day.
2. `loadState()` falls back to that object when no stored values exist.
3. `getSettingsCalculationDefaults()` falls back again to those values.
4. `validateCalculationInputs()` therefore passes on a fresh install.
5. `render()` displays a complete target and then calls `saveState()`, making raw key existence an unreliable first-run detector.
6. `materializeTodayBodyStatusDefaults()` labels the body source as default but does not prevent the personalized-looking result.
7. Full backup stores Settings and Today values; records-only import intentionally does not restore Settings.
8. There is no onboarding completion marker or current first-run ownership contract.

## Superseded historical direction

The archived v8.2 note proposed asking only whether the user exercises. That was reasonable before the current TDEE and macro policy became production behavior, but it is no longer sufficient.

The current engine also depends on:

- sex, age, height, and weight for the energy model;
- goal for target direction;
- activity level, work type, sleep/work/lifestyle hours for time-ownership TDEE;
- exercise mode and, when enabled, exercise context for macro policy.

Therefore this decision supersedes the old `exercise 여부만 묻는 최소 v1` direction. The old note remains historical evidence under `docs/archive/onboarding/` and is not an implementation instruction.

## Chosen product flow

Chosen model: `truthful setup gate + progressive Today`.

This is a short setup flow, not a tutorial carousel and not a marketing screen.

### Entry screen

- Primary action: `내 정보 설정하기`.
- Secondary returning-user action: `백업 불러오기`.
- No permanent `나중에` action that unlocks fake target numbers.
- The rest of the app may remain inspectable, but Today target/macro numbers stay placeholder until required setup is complete.

### Step 1: 내 정보

- gender
- age
- height
- weight
- goal

All five are required and validated before continuing.

### Step 2: 평소 하루

- activity level
- work type
- sleep hours
- work hours
- lifestyle hours

Convenient starting selections are allowed, but the user must review and submit the step. Requested time can exceed 24h only under the existing explicit warning/clamp contract; onboarding must not hide that condition.

### Step 3: 운동

- exercise / general choice is required.
- If general: finish without exercise-profile questions.
- If exercise: require exercise profile and planned weekly training days.
- Do not ask routine plan, today's session, today's duration, or cardio here.

### Completion and Today handoff

- Save long-term values to Settings.
- Do not create a Today draft merely to duplicate Settings.
- Do not assume today's PUSH, 60-minute weights, or 30-minute cardio.
- Today asks for date-owned exercise differences through its existing quick-edit flow.
- Body composition remains optional; if missing, use the existing low-authority/no-body-composition path rather than fabricated 15%/35kg values.

## Calculation truthfulness gate

Before onboarding completion:

- target kcal, protein, carbs, and fat must not show personalized numbers;
- no adherence score should be produced from the fictional default profile;
- `render()` must not persist the historical example profile as user-owned data;
- the screen should show a compact setup action, not a wall of explanation.

After completion:

- all Settings-owned required values have explicit user ownership;
- Today may calculate using Settings baseline plus date-owned overrides;
- optional body-composition absence is visible through source/reliability behavior, not silently replaced with the example body composition.

## Four-state migration contract

Introduce:

```text
ONBOARDING_VERSION = 1
onboardingCompletedVersion

fresh
completed
existing_profile_confirmed
existing_data_profile_unconfirmed
```

The raw storage classifier must run before the first new-version `render() -> saveState()` and use the following four states:

| State | Evidence | Product action |
|---|---|---|
| `fresh` | no completion marker, no user data, and no trusted profile evidence; exact historical example Settings alone also belong here | show full setup before personalized target numbers |
| `completed` | `onboardingCompletedVersion >= ONBOARDING_VERSION` | enter the normal app |
| `existing_profile_confirmed` | no marker, but explicitly customized required profile Settings provide trusted pre-onboarding profile evidence | preserve data, write `onboardingCompletedVersion = 1` once, and enter the normal app |
| `existing_data_profile_unconfirmed` | Records, InBody, Today drafts, presets, templates, or non-profile Settings exist, but the required profile is still the exact historical example or otherwise unconfirmed | preserve all data and require compact profile confirmation before personalized targets |

Mere presence of auto-saved default keys is not completion evidence. Records, InBody records, Today drafts, presets, and templates prove that the browser contains user data; they do not prove that 75kg, 173cm, age 32, male, and lean_bulk are the user's profile. In particular, exact historical defaults plus Records or InBody must classify as `existing_data_profile_unconfirmed`, not `existing_profile_confirmed`.

This correction narrows automatic migration to trusted profile evidence. It does not delete or rewrite data in the unconfirmed state, and backup import remains reachable from the setup entry surface.

### Body-composition authority

- Track body-composition ownership separately from onboarding completion, for example with `bodyCompositionConfirmed`.
- Historical 15% body fat and 35kg skeletal muscle values are not trusted merely because they exist in storage.
- Missing or unconfirmed body composition must use the existing null/low-authority path and an appropriate non-body-composition BMR path; it must not silently feed 15%/35kg into protein or physiology context.
- InBody records alone do not confirm age, height, gender, goal, or the Settings profile.

### Full backup / restore

- Full backup includes the optional completion version.
- New full backups with a valid completion version restore as `completed`.
- Old full backups without the marker use the same four-state classifier. Customized required profile Settings may migrate as `existing_profile_confirmed`; exact historical Settings plus Records/InBody remain `existing_data_profile_unconfirmed`.
- The new optional field is backward-compatible and does not by itself require `FULL_BACKUP_VERSION` bump, but implementation tests must prove old and new restore paths.

### Records-only import

- Records-only import does not complete onboarding because it intentionally contains no Settings profile.
- Importing records on a fresh browser must not unlock a target calculated from the historical example profile.

### Reset and re-entry

- A true full reset clears the completion marker and returns to onboarding.
- `초기 설정 다시 하기` may reopen the flow from Settings without deleting Records; cancel keeps the previous completed profile.
- Reopening setup is not a migration and must not rewrite Today drafts until the user submits.

## Skip policy

- Required steps cannot be permanently skipped while still showing personalized target numbers.
- Body composition, InBody, routine plan, weekday schedule, protein level, adaptive setting, and advanced settings can be skipped.
- A user may leave the setup screen and browse/restore data, but Today remains in setup-required placeholder state.

## UI boundary

- Desktop: centered setup dialog or dedicated setup surface with clear step progress.
- Mobile 390px: one column, stable footer actions, no horizontal overflow, no long tutorial paragraphs.
- Use normal labels and inputs. Do not explain TDEE equations, macro curves, or carb/fat exchange arithmetic in onboarding.
- Keep each step to one decision group; avoid nested cards and decorative onboarding pages.
- Returning-user backup access must remain reachable without completing a fake profile.

## Implementation acceptance tests

1. Clean localStorage shows setup-required state and no 4,013kcal demo target.
2. Historical exact-default-only storage still shows setup.
3. Existing customized required profile Settings classify as `existing_profile_confirmed`, bypass setup once, and receive the completion marker.
4. Exact historical Settings plus Records/InBody/Today drafts classify as `existing_data_profile_unconfirmed`; all data is preserved while profile confirmation remains required.
5. Required core inputs reject empty, nonnumeric, and out-of-range values.
6. General mode completes without exercise-profile fields.
7. Exercise mode requires exercise profile and planned weekly days.
8. First completed Today does not assume PUSH, 60-minute weights, or 30-minute cardio.
9. Missing body composition does not invent 15% body fat or 35kg skeletal muscle.
10. `proteinTargetLevel` remains medium and is not asked in onboarding.
11. adaptive targets remain ON and are not asked in onboarding.
12. Full backup round-trips completion state.
13. Old full backup without marker follows the same four-state profile-evidence classifier; data alone does not complete setup.
14. Records-only import does not complete setup.
15. Confirmed existing profiles are not trapped after upgrade, while profile-unconfirmed users retain backup access and a short path to confirmation.
16. Settings re-entry can be canceled without data loss.
17. Full reset reopens setup.
18. Desktop and mobile layout have no overflow; keyboard focus and step navigation work.
19. Current score, macro-policy, DailyCoach, Records, InBody, backup, and mobile suites remain green.

## Stop conditions

- Stop if implementation treats localStorage key existence alone as existing-user evidence.
- Stop if the setup can be dismissed while fictional target numbers remain visible.
- Stop if body composition is called optional but 15%/35kg is still silently inserted.
- Stop if exercise mode is selected but the code still assumes bodybuilding/PUSH without user or date evidence.
- Stop if records-only import is used as a substitute for Settings ownership.
- Stop if onboarding becomes a tutorial/copy project rather than a calculation-truthfulness gate.
- Stop if implementation changes score curves, nutrition anchors, or old record scores to make setup easier.

## Decision

Accepted:

- onboarding is a real product gap;
- hybrid truthful setup gate + progressive Today;
- explicit ownership for all values that materially drive first target calculation;
- optional body composition without fabricated numbers;
- backward-compatible completion marker and existing-user migration;
- implementation as one complete feature, not a UI-only shell.

Rejected:

- current fictional defaults;
- exercise-only onboarding;
- 4-6 field minimal checklist that leaves TDEE assumptions hidden;
- permanent skip plus personalized output;
- all-settings wizard;
- onboarding as tooltip/copy-only work;
- immediate score or macro-policy changes.
