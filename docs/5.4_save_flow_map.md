# 5.4 Save Flow Map

기준 파일: [macro_engine_5.3.6.html](</N:/개인/000.개인/02.식단,부스터,루틴/00.린매스업엔진/macro_engine_5.3.6.html>)

주의:
- 이 문서는 `5.3.6` 기준 저장 흐름을 읽어서 정리한 지도 문서다.
- 추정이 섞인 부분은 `추정`으로 표시했다.
- 테스트 전용 함수/샌드박스 경로는 별도로 구분했다.

## 1. `saveState()` 호출 지점 전체 목록

### 저장 함수 본체
- `saveState()` [3701]  
현재 `state` 입력값, `uiState.activeTab`, `recordsState.items`를 `localStorage`에 저장한다.  
`recordsState.items`는 `storageKey("records")`로 직렬화된다.

### Production 호출 지점
- `render()` [6681-6790]  
렌더 마지막에서 항상 `saveState()`를 호출한다.  
분류: UI 상태 + 현재 입력값 + 현재 records 전체 저장  
중복 가능성: 높음. `render()`를 부르는 거의 모든 입력 이벤트가 저장까지 동반한다.

- `saveMealEntry()` [5763-5802]  
식사 저장 후 `saveState()` 호출.  
분류: `recordsState` 변경 직후 저장  
중복 가능성: 높음. 함수 마지막에 `render()`도 호출하므로 `saveState()`가 연속 2회 발생한다.

- `saveRecord()` [5805-5825]  
체중/메모/manual adherence 저장 후 `saveState()` 호출.  
분류: `recordsState` 변경 직후 저장  
중복 가능성: 높음. 이후 `render()`가 다시 `saveState()`를 호출한다.

- `deleteRecord()` [5828-5833]  
record 삭제 후 `saveState()` 호출.  
분류: `recordsState` 변경 직후 저장  
중복 가능성: 높음. 이후 `render()`가 다시 저장한다.

- `deleteMealEntry()` [5836-5848]  
meal 삭제 + auto adherence 재계산 후 `saveState()` 호출.  
분류: `recordsState` 변경 직후 저장  
중복 가능성: 높음. 이후 `render()`가 다시 저장한다.

- `importRecordsJson()` [5874-5883]  
JSON import 후 `recordsState.items` 대입, `saveState()` 호출.  
분류: `recordsState` 변경 직후 저장  
중복 가능성: 높음. 이후 `render()`가 다시 저장한다.

- `recordWeight` / `recordAdherence` / `recordNote` 입력 리스너 [6798-6800]  
`input`/`change`에서 직접 `saveState()` 호출.  
분류: 단순 UI/폼 draft 저장  
중복 가능성: 보통. 이 리스너는 `render()`를 직접 부르지 않지만, 다른 조작과 겹치면 연쇄 저장 가능.

- 탭 클릭 리스너 `tabToday`, `tabRecords`, `tabWeekly`, `tabSettings` [6857-6872]  
활성 탭 전환 후 즉시 `saveState()` 호출.  
분류: UI 상태 저장  
중복 가능성: 낮음. 별도 `render()` 없이 active tab만 저장한다.

### `saveState()` 미호출이지만 저장성 있는 경로
- `exportRecordsJson()` [5851-5865]  
`lastBackupAt`만 `localStorage`에 직접 쓴다. `saveState()`는 호출하지 않는다.

### 중복 저장 후보 요약
- `saveMealEntry()` -> `saveState()` -> `render()` -> `saveState()`
- `saveRecord()` -> `saveState()` -> `render()` -> `saveState()`
- `deleteRecord()` -> `saveState()` -> `render()` -> `saveState()`
- `deleteMealEntry()` -> `saveState()` -> `render()` -> `saveState()`
- `importRecordsJson()` -> `saveState()` -> `render()` -> `saveState()`
- `render()` 자체가 광범위 입력 이벤트의 종착점이라 저장 트리거가 매우 넓다.

## 2. `recordsState` 변경 지점 전체 목록

### Production 변경 지점
- `loadState()` [3749-3810]  
`localStorage("records")`를 읽어 `recordsState.items`에 대입 [3800-3805].  
분류: 복원  
오늘/과거 날짜 구분: 없음. 저장된 전체 records를 그대로 복원.

- `resetState()` [3812-3819]  
`recordsState.items = []`로 초기화 [3818].  
분류: 전체 초기화  
오늘/과거 날짜 구분: 없음.

- `upsertRecord(rawRecord)` [5136-5155]  
실질적인 공통 write 지점.  
`existing` record를 기준으로 병합하고, `recordsState.items`를 날짜 기준 교체한다 [5139-5155].  
분류: 공통 삽입/갱신  
오늘/과거 날짜 구분: 호출 측이 날짜를 결정한다. 함수 자체는 구분하지 않는다.

- `deleteRecord(date)` [5828-5833]  
`recordsState.items.filter(...)`로 해당 날짜 record 제거 [5829].  
분류: 직접 삭제  
오늘/과거 날짜 구분: 호출 인자 기준. 함수 자체는 구분하지 않는다.

- `importRecordsJson()` [5874-5883]  
`parseImportedRecordsPayload()` 결과를 `recordsState.items`에 직접 대입 [5879].  
분류: 일괄 치환  
오늘/과거 날짜 구분: 없음.

### 간접 변경 경로
- `saveRecord()` [5805-5825]  
`upsertRecord(record)` 호출 [5821].  
분류: record 메타(weight/adherence/note) 저장.

- `saveMealEntry()` [5763-5802]  
`upsertRecord({ date, meals: nextMeals })` 호출 [5789].  
분류: meal 배열 갱신.

- `deleteMealEntry()` [5836-5848]  
`upsertRecord({ date, meals: nextMeals })` 호출 [5840].  
분류: meal 배열 갱신.

- `persistAutoAdherenceForDate()` [5721-5755]  
`upsertRecord(...)`를 통해 `adherencePercent`, `adherenceSource`, `goalSnapshot`, `snapshotSource`, `adherenceLockedAt`를 갱신 [5738-5754].  
분류: auto adherence + snapshot 저장.

### 읽기 전용 / 비저장 경로
- `ensureRecordByDate(date)` [5130-5134]  
record가 없으면 빈 normalized record를 반환하지만 저장하지는 않는다.  
주의: 이름상 생성처럼 보이지만 write는 없다.

### 테스트 전용 변경 지점
- `withRecordTestSandbox()` [6917-6960]  
샌드박스 종료 시 `recordsState.items` 복원 [6960].  
- `run*Tests()` 다수 [7048 이후]  
테스트에서 `recordsState.items = normalizeRecordsCollection(...)`를 반복 사용한다.  
분류: 테스트 전용, production 경로 아님.

## 3. `adherencePercent` 저장 흐름

### 계산/저장 진입점
- `persistAutoAdherenceForDate(date, currentResult = calculate(), options = {})` [5721-5755]  
핵심 저장 함수다.

### 내부 흐름
1. 날짜 정규화 [5722-5723]
2. 기존 record 조회 `ensureRecordByDate()` [5724]
3. 기존 `goalSnapshot` 존재 여부 확인 [5725]
4. snapshot source 결정 [5726-5731]
5. snapshot 확보: `existing.goalSnapshot || buildGoalSnapshotFromCurrentState(...)` [5731]
6. auto 분석 OFF 또는 snapshot 없음  
   `upsertRecord(... adherencePercent:null, adherenceSource:"manual")` [5738-5745]
7. auto 분석 ON + snapshot 있음  
   `getDailyAdherenceScore(normalizedDate, currentResult, { goalSnapshot: snapshot })` [5748]  
   `upsertRecord(... adherencePercent:score.percent, adherenceSource:"auto")` [5749-5754]

### 호출 경로
- `saveMealEntry()` [5789-5793]  
meal 저장 직후 호출. Today/Records 입력이 실제 record score 저장으로 이어지는 핵심 경로.

- `deleteMealEntry()` [5842-5845]  
meal 삭제 후 score 재평가.

- 테스트 경로  
`runAdherenceScoringTests()` / `runWeeklyAdherenceTests()` 내부에서 직접 호출 [7388, 7408, 7424, 7431, 7451, 7458, 7472, 7577, 7578 등].

### 수동 입력값과 자동 계산값 충돌 가능성
- `getStoredAdherenceEntry(record)` [5546-5555]  
`adherencePercent + adherenceSource === "auto"`가 있으면 auto를 우선 사용 [5548-5549].  
없으면 manual `adherence`를 90/75/60으로 환산 [5551-5554].

- `saveRecord()` [5805-5825]  
auto 분석 ON이면 `adherence`는 기존 값을 유지하고 `adherenceSource`도 기존 값을 유지 [5812-5814].  
즉 manual dropdown은 비활성화되고, auto score 필드는 `persistAutoAdherenceForDate()` 경로에서 관리된다.

- 충돌 추정  
manual `adherence` 텍스트 값은 남아 있어도, `adherencePercent + adherenceSource:"auto"`가 존재하면 weekly/today 저장 해석은 auto 우선으로 읽는다.  
즉 데이터 공존은 가능하지만 읽기 우선순위가 명확하다.  
표면상 충돌은 낮지만, record 내부에 manual/adherence와 auto/adherencePercent가 동시에 남는 구조는 유지된다.

## 4. `goalSnapshot` 저장 흐름

### 생성
- `buildGoalSnapshotFromCurrentState(currentResult = calculate(), snapshotSource = "saved_at_entry")` [5209-5236]  
현재 입력 상태와 계산 결과를 snapshot으로 만든다.  
`calculate()` 결과가 유효하지 않으면 `null` 반환 [5210].

### 저장
- `persistAutoAdherenceForDate()` [5721-5755]  
기존 snapshot이 없으면 새로 생성 [5731].  
`baseUpdate.goalSnapshot` / `snapshotSource`로 `upsertRecord()`에 넘긴다 [5732-5735].

### 유지/재사용
- 기존 record에 `goalSnapshot`이 있으면 우선 재사용 [5731].  
- `saveMealEntry()`는 기존 `record.goalSnapshot`이 있으면 `snapshotSource`를 유지하고, 없을 때만 `forceLateSnapshot:true`로 늦은 snapshot 생성을 허용 [5790-5793].
- `deleteMealEntry()`도 `record.snapshotSource || "saved_at_entry"`를 넘기지만, 실제 snapshot 재생성 여부는 `persistAutoAdherenceForDate()`가 기존 snapshot 존재 여부로 결정한다 [5842-5845].

### Today snapshot과의 관계
- `buildTodayDataSnapshot(currentResult)` [5530 근처]  
Today UI용 read-only bridge다.  
실제 저장은 하지 않고, 현재 today balance/adherence/signals를 묶어 UI 렌더로 넘긴다.  
record에 이미 저장된 `goalSnapshot`은 `getTodayMacroBalance()` -> `getMacroBalanceForDate()`에서 목표 기준으로 참조될 수 있다 [5251-5293].

### 덮어쓰기 가능성
- `upsertRecord()` 자체는 `goalSnapshot` 필드가 rawRecord에 있으면 그 값으로 교체한다 [5146-5149].  
- 다만 production 경로에서 `persistAutoAdherenceForDate()`는 기존 snapshot이 있으면 그것을 계속 재사용하므로, 의도치 않은 덮어쓰기 가능성은 현재 구조상 낮다.  
- 추정: 새로운 위험은 `upsertRecord()`를 다른 경로에서 직접 호출하며 `goalSnapshot`를 넘길 때 발생할 수 있다.

## 5. Today -> Record 방향 흐름

### Today 계산 결과가 Record에 반영되는 경로
- `render()` [6681-6790]  
`calculate()` -> `getTodayMacroBalance()` / `getTodayAdherenceScore()` / `buildTodayDataSnapshot()`로 Today UI를 렌더한다 [6771-6788].

- `saveMealEntry()` [5763-5802]  
Records 탭에서 meal 저장 -> `upsertRecord(meals)` [5789] -> `persistAutoAdherenceForDate(date, calculate(), ...)` [5790-5793]  
즉 현재 Today 계산 기준(`calculate()`)이 특정 record 날짜의 snapshot/adherence 저장으로 들어간다.

- `saveRecord()` [5805-5825]  
체중/메모/manual adherence를 record에 저장.  
직접 `calculate()` 결과를 저장하지는 않지만, Today와 동일한 state를 쓰는 시점이라 간접 연동된다.

### Record -> Today 역방향 읽기 흐름
- `getTodayRecord()` [5296-5298]  
오늘 날짜 record를 조회한다.

- `getTodayMacroBalance(currentResult)` 내부 경로 추정  
`getMacroBalanceForDate(getTodayDateString(), currentResult, ...)`를 통해 오늘 record의 meals와 goalSnapshot을 읽는다.  
Today panel은 record에 저장된 meals를 읽어 consumption/remaining을 만든다.

- `getTodayAdherenceScore(currentResult)` 경로 추정  
오늘 record의 meals/snapshot/auto setting을 읽어 Today adherence를 만든다.

- `buildTodayDataSnapshot(currentResult)` [5530 근처]  
Today용 signals/adherence bridge를 만들며 저장은 하지 않는다.

정리:
- 쓰기 방향의 핵심은 `saveMealEntry()` / `persistAutoAdherenceForDate()`
- 읽기 방향의 핵심은 `getTodayMacroBalance()` / `getTodayAdherenceScore()` / `buildTodayDataSnapshot()`

## 부록: 빠른 요약

- write 진입점의 중심은 `saveMealEntry()`, `saveRecord()`, `deleteMealEntry()`, `deleteRecord()`, `importRecordsJson()`
- 실제 공통 record 갱신기는 `upsertRecord()`
- auto score + snapshot 저장 핵심은 `persistAutoAdherenceForDate()`
- Today UI는 live `calculate()`와 stored record(meals/snapshot)를 함께 읽는다
- Weekly는 stored record의 `adherencePercent/adherenceSource` 또는 manual `adherence`를 읽는다
- 5.4의 첫 번째 실질 리스크는 `saveState()` 중복 호출과 snapshot/adherence 저장 타이밍의 엇갈림이다
