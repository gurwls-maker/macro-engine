## Plain explanation

새 브라우저에서는 예시 프로필로 개인 목표를 먼저 계산하지 않는다. 사용자가 계산에 필요한 기본정보를 확인하기 전에는 목표 kcal·탄단지·점수를 placeholder로 두고, 3단계 설정을 끝낸 뒤에만 계산한다. 예전 데이터가 있어도 그 사실만으로 예시 Settings가 본인 정보라고 간주하지 않는다.

## State model

`ONBOARDING_VERSION = 1` and `onboardingCompletedVersion` are implemented with four classifications:

| Classification | Raw evidence | Result |
|---|---|---|
| `fresh` | no completion marker and no user evidence; exact historical defaults alone also count as fresh | show full setup and calculation placeholders |
| `completed` | completion version is current | load the normal app |
| `existing_profile_confirmed` | customized required profile Settings provide trusted pre-onboarding profile evidence | preserve state and write the completion marker once |
| `existing_data_profile_unconfirmed` | Records/InBody/Today draft/presets/templates or non-profile Settings exist, but required profile ownership is unconfirmed | preserve data, keep placeholders, and require profile confirmation |

Classification runs from raw storage before the first render/save cycle. `hasMeaningfulPreOnboardingUserState` can report either profile or data evidence, but only trusted profile evidence may auto-complete setup.

Exact historical defaults plus Records or InBody are `existing_data_profile_unconfirmed`. Records-only import does not complete setup. This closes the gap where user data could otherwise legitimize the fictional 75kg/173cm/32-year-old male lean-bulk profile.

## Implemented flow

- Added a full-screen setup gate with an entry surface, backup restore access, a three-step form, and a completion-only handoff to the app.
- Step 1 collects gender, age, height, weight, and goal.
- Step 2 collects activity, work type, sleep hours, work hours, and lifestyle hours.
- Step 3 requires general/exercise mode; exercise mode additionally requires profile and planned weekly days.
- Required values reject empty, invalid, and out-of-range input. A requested time total above 24 hours keeps the existing explicit warning/automatic-adjustment contract visible.
- `proteinTargetLevel` remains `medium`; onboarding does not ask for it.
- `adaptiveMacroTargetsEnabled` remains ON; onboarding does not ask for it.
- Settings now exposes `초기 설정 다시 하기`. Re-entry is prefilled, cancelable, and does not mutate profile or Records until submit.
- The Settings header keeps `초기 설정 다시 하기` and `입력 초기화` on equal, stable action tracks: 148x42px on desktop and equal-width halves on the narrow layout.
- Full reset clears completion and returns to setup.
- Focus is trapped inside the open setup surface; desktop uses a centered dialog and the 390px layout uses a full-height one-column surface with a stable footer.

## Calculation and persistence gate

- Pending setup renders neutral placeholders for target kcal, protein, carbs, fat, and score-facing personalized output.
- Pending render returns before `calculate()` and before Settings persistence.
- Records remain writable/importable while setup is pending, but their presence does not unlock personalized output.
- `saveState()` continues to preserve Records and refuses to persist the fictional Settings snapshot until onboarding is complete.
- Completion writes user-owned Settings and starts Today at `REST`, zero weight-training minutes, and zero cardio minutes.
- Completion does not create a Today draft merely to copy Settings.
- Existing users with a genuinely customized required profile are migrated once. Historical PUSH/60-minute weights/30-minute cardio are neutralized only in the specific untouched historical Today context.

An attempted global change to make every Settings-to-Today handoff neutral caused eight established smoke regressions. That change was too broad and was reverted. Today neutrality is limited to first-run completion and the exact historical migration case, preserving existing normal ownership behavior.

## Body-composition authority

- Added `bodyCompositionConfirmed` as a separate authority signal.
- Body fat and skeletal muscle are optional; required numeric validation no longer forces them.
- Fresh completion leaves both values empty unless trusted user-owned body composition already exists.
- Historical 15% body fat and 35kg skeletal muscle values do not become trusted merely because they were stored by the old default flow.
- Missing/unconfirmed body composition uses the no-body-composition reliability path and Mifflin-St Jeor BMR instead of silently entering body-composition protein/physiology adjustments.
- Explicit Settings body composition, a user source, or an applied InBody source can establish body-composition authority; InBody records alone do not confirm the whole Settings profile.
- Source text now distinguishes confirmed Settings values from `체성분 입력 없음`.

## Backup and migration

- New full backups include optional `onboardingCompletedVersion` without changing `FULL_BACKUP_VERSION`.
- A new backup with the completion marker restores as `completed`.
- An old backup with customized required profile Settings may auto-migrate as `existing_profile_confirmed`.
- An old backup with exact historical Settings plus Records/InBody restores data as `existing_data_profile_unconfirmed` and keeps target placeholders.
- Records-only import preserves imported Records and does not complete onboarding.
- Current backup regression fixtures explicitly model the new completion field; the old-backup fixture intentionally omits it.

## Additional requested product audit: not implemented

The following two ideas were inspected against the current code and copy policy, but no related UI, score, Coach, tooltip, storage, or onboarding behavior was implemented in this branch.

### 1. DailyCoach, help ownership, and selectable tone

Observed:

- The current Coach already separates selected decisions into `state`, `reason`, and `action`, and keeps `copyIntent`/goal context in the output path.
- The actual wording is still spread across large goal/decision maps, so adding three complete copies of every branch would multiply drift and contradictory advice.
- The same Coach items currently feed both `오늘의 코치` details and the score card's `추가 확인`, which creates duplicated ownership.
- Existing copy truth already says compact cards own numbers, DailyCoach owns interpretation/action, and tooltip/glossary needs a small spec. The new request is consistent with that direction and makes the routing boundary more concrete.

Recommended surface contract:

| Information type | Surface |
|---|---|
| stable concept, persistent setting, or definition the user may choose to read | context-local `?` tooltip or Settings help |
| today's detected state, priority, reason, and next action | `오늘의 코치` |
| blocking validation, destructive warning, or immediate safety condition | inline beside the control or in the confirm dialog; never hide it in Coach/tooltip |
| score/source limitation that changes how the visible number can be trusted | compact note beside the score, with optional help for the stable definition |

Do not keep explanatory prose permanently visible across the app merely because it is globally true. Put the short fact where the user encounters the concept and move the fuller explanation behind help. Conversely, do not move a day-specific warning into a static tooltip; it belongs in today's Coach.

Selectable Coach tone is a valid product feature, but it is not a copy-only toggle:

- Implement it as a semantic-message layer: decision priority, facts, numbers, safety strength, and action must remain identical; only phrasing changes.
- Generate canonical message intent first, then apply a tone renderer at the final copy boundary. Do not fork the decision engine or maintain three unrelated copies of every rule.
- A persistent `coachTone`-style setting would require Settings, save/load, full backup/restore, defaults, tests, and copy fixtures.
- In onboarding it must be optional and non-blocking, placed after or alongside the final step with a clear default. It must never become part of the calculation-truthfulness gate.
- Tooltip definitions, units, warnings, and destructive confirmations stay canonical. The selected tone may affect a short conversational wrapper, but cannot rewrite the factual core.
- `열정형` and `냉철형` are clear prototypes. `AI형` is fun as an experiment, but it conflicts with the project's explicit effort to remove AI-like copy; compare it against a `차분형`/`기본형` candidate before fixing product labels.

### 2. Diet score labels, component display, and extra checks

Observed current behavior:

- Overall score bands are `매우 우수` (90+), `우수` (80+), `보통` (70+), `보완 필요` (50+), and `많이 부족`.
- A high clean score then adds `안정적` below `매우 우수`, which usually repeats the same conclusion without adding an action or exception.
- The four compact cards expose production contribution values as `감점 없음` or a negative point value. That is an implementation-facing view of a 100-minus-penalties engine, not the easiest user-facing view.
- The production score has kcal, protein, fat, carb/exchange, and alcohol-physiology contributions, while the compact grid permanently shows only kcal/protein/carbs/fat. Therefore four apparent 100-point component cards cannot be presented as an average or complete decomposition of the final score.
- `추가 확인` exists both under the score and under Coach, often from the same Coach items. Some entries are actions/context, while source reliability is genuinely score-specific.

Recommended score-surface decision:

- Keep one score-band label if it helps scan the number. Remove the generic second line when it only says `안정적`/`대체로 안정` again.
- Reuse that second-line space only for a high-signal exception, such as a high score with mixed over/under items or a non-card factor that materially explains the total. Otherwise render no line and keep the card shorter.
- Prefer 0-100 display for the four compact components. The least distorting candidate is a display-only retained score such as `clamp(100 + componentContribution, 0, 100)`, with legacy component scores normalized to the same scale. It must not feed calculation, storage, Records regrading, or score tuning.
- Before implementation, prove that kcal's combined target/TDEE axis and carbs' combined exercise/exchange axis remain understandable, and explicitly handle the alcohol contribution that is not one of the four permanent cards.
- A small static help text may say that component values show each area separately and the total reflects them together; it should not expose penalty tables or imply that the four numbers are averaged.
- Remove the generic score-card `추가 확인` disclosure by default. Move day-specific action/context to Coach, keep only compact score-validity/source notes beside the score, and delete low-value entries rather than moving everything wholesale.
- Do not add a permanent fifth card merely to explain rare conditions; use the meaningful secondary line or Coach when such a condition is present so the panel does not grow vertically.

### Recommended order

1. Finish and merge the current onboarding implementation without either idea mixed into it.
2. Run one joint docs-only `Today score and guidance surface ownership inventory / decision`. Inventory every score band/secondary state, every final contribution axis, all `추가 확인` producers, tooltip candidates, and representative Coach intents/persona samples.
3. Implement the narrow score-card semantic cleanup first: 0-100 component display, redundant secondary-line removal, and score-vs-Coach extra ownership. Keep the production score formula unchanged.
4. Then implement `DailyCoach v2 + selectable tone` as a separate complete feature: semantic intent boundary, natural Coach rewrite, tone setting/storage/backup, optional onboarding choice, tooltip tone boundary, and desktop/mobile/copy regression coverage.
5. Only after those two implementations should broad tooltip/glossary or app-wide copy batch 2 be reconsidered. Do not start with a broad tooltip rollout or a global string rewrite.

Reason for this order: the score cleanup first establishes which facts remain beside the score and which actions belong to Coach. The larger Coach/persona implementation can then work against one clear ownership model instead of preserving today's duplicate `추가 확인` surfaces.
