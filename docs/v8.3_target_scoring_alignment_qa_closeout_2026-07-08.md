# v8.3 target/scoring alignment QA closeout

작성일: 2026-07-08

## 결론

`8fdd36e v8.3 목표-점수 정합성 구현`은 방향상 유지한다.
다만 QA 중 하나는 즉시 보강했다.

```text
유지:
- target/scoring/card display shared policy
- Today adaptive target
- high-volume flat permission 금지
- Records goalSnapshot selected calculation target 보존

즉시 수정:
- 3일/7일 trend limiter의 count bucket 계산

수정 후:
- continuous recency-weighted excess pressure 모델로 변경
```

이번 closeout은 단순 문서 승인이 아니다. 실제 코드에서 `getV83RecentMacroSkewLimiter`를
다시 보았고, 초기 구현의 `highCarb3 >= 2`, `highCarb7 >= 4` 같은 count bucket은
v8.3의 연속 곡선 방향과 맞지 않는다고 판단했다.

## 왜 즉시 수정했는가

사용자 의도는 “먹은 대로 목표가 무한히 따라가는 면제 구조”도 아니고,
“특정 날짜 수를 넘으면 갑자기 제한하는 계단식 구조”도 아니다.

따라서 limiter는 다음처럼 바뀌어야 한다.

```text
나쁜 구조:
- 최근 3일 중 2일 이상이면 갑자기 0.45
- 최근 7일 중 2일 이상이면 갑자기 0.70

수정 구조:
- 최근 기록이 목표를 얼마나 넘었는지 본다.
- 오늘에 가까운 기록일수록 더 크게 본다.
- pressure가 커질수록 자동 조정폭이 부드럽게 줄어든다.
```

구현된 모델:

```text
pressure = ((actual / target) - 1) / 0.35 * recencyWeight
recencyWeight = (8 - distanceDays) / 7
multiplier = 1 / (1 + combinedPressure * 0.85)
clamp: 0..1
```

이 수치들은 외부근거 hard cutoff가 아니라 앱 내부 overfitting guard의 완만한 shape다.
별도 floor를 두지 않는다. 극단적인 반복 고탄수/고지방 이력이 있으면 자동 조정폭은
연속적으로 0에 가까워지고, 오늘 목표는 selected calculation target 쪽으로 돌아온다.
정책 의미는 “최근에 반복적으로 한쪽으로 치우칠수록 오늘 목표가 그 방향으로 따라가는 폭을 줄인다”이다.

## target 그대로 섭취 invariant

유지해야 하는 핵심 invariant:

```text
valid full-day / no alcohol / no outlier / sufficient source 조건에서
앱이 제시한 target macros를 그대로 섭취하면 낮은 점수로 떨어지면 안 된다.
```

테스트는 `runMacroRangeContinuousScoringTests` 안의
`target macros align with scoring ranges and card ranges`가 담당한다.

고정한 것:

```text
- target macros 그대로 섭취 시 score >= 95
- target kcal은 targetEnergyDeviationPenalty를 만들지 않음
- target kcal == totalBurn이면 tdeeOverloadPenalty를 만들지 않음
- target carbs는 scoring carb range 안에 있음
- target fat은 scoring fat range 안에 있음
- macro card range는 scoring boundary를 사용함
- high-volume carb anchor는 10.5g/kg flat no-penalty display가 아님
```

## adaptive target overfitting risk

보강 테스트는 다음을 고정한다.

```text
ON:
- carb-leading intake면 carb target은 올라갈 수 있음
- 같은 kcal 안에서 fat target은 내려감

OFF:
- adaptive target은 null
- fixed Today target 유지

과식:
- adaptive target이 target kcal 초과를 숨기지 않음
- adaptive target이 TDEE overload를 숨기지 않음

Records:
- buildGoalSnapshotFromCurrentState는 adaptive live target이 아니라 selected calculation target 저장
- snapshot이 있는 Records scoring은 live 운동 context보다 snapshot basis를 우선
- Records display는 current-version stored auto score를 `stored_auto_snapshot`으로 표시
```

즉 사용자가 탄수를 더 먹었다고 해서 앱이 점수 방어용으로 무한히 target을 따라가지는 않는다.
같은 kcal 안의 탄수/지방 교환만 설명하고, 총량 과식은 target/TDEE 축에서 계속 나빠진다.

## 3일/7일 limiter QA

초기 구현의 계단식 count bucket은 폐기했다.

현재 테스트가 고정하는 것:

```text
- trendLimiter.pressureModel === "continuous_recency_weighted_excess"
- mild repeated high-carb도 non-zero pressure를 만든다.
- mild pressure는 strong repeated high-carb보다 덜 제한한다.
- extreme repeated high-carb는 strong repeated high-carb보다 더 제한한다.
- adaptive carb target은 no trend와 strong trend 사이에서 연속적으로 이동한다.
- limiter는 hard bucket 한 값으로 고정되지 않는다.
```

남은 의도:

```text
최근 고탄수/고지방 기록이 쌓이면 앱이 계속 목표를 올려주는 흐름을 막는다.
하지만 특정 날짜 수를 넘었다고 갑자기 다른 정책으로 뛰지 않는다.
```

## high-volume carb no-permission QA

high-volume context는 여전히 carb curve context다.
운동일이라는 이유로 final score bonus를 주지 않고, 8~12g/kg anchor를 무감점 허가증으로 쓰지 않는다.

추가로 QA 중 자동 운동 context의 label threshold도 다시 확인했다.
초기 구현처럼 `normal_resistance`와 `high_volume_or_long_session` label이 바뀌는 순간
loadFactor나 carb boundary가 같이 뛰면 v8.3의 continuous 방향과 충돌한다.

따라서 자동 계산 경로는 `continuous_training_load_interpolation` 모델을 쓴다.
descriptive label은 남길 수 있지만 실제 carb context는 운동 시간, 주당 운동일,
운동 kcal 신호에서 나온 loadFactor로 rest → normal → high anchor를 보간한다.
명시적으로 test fixture가 `trainingContext`를 넘긴 경우에는 fixture 의미를 보존한다.

고정한 것:

```text
- high-volume scoringContext.carbRange.max < 700 in target alignment fixture
- high-volume context는 TDEE overload를 덮지 않음
- alcohol risk는 training context 때문에 할인되지 않음
- 자동 운동 context는 label threshold를 넘어도 loadFactor와 carb boundary가 튀지 않음
```

## Records snapshot stability

QA 중 가장 중요한 보정은 이 부분이다.

Today 화면과 Today score는 adaptive live target을 사용할 수 있다.
하지만 Records `goalSnapshot`은 식사 기록에 반응해 움직인 target을 저장하지 않는다.

이유:

```text
식사 내용이 목표값을 바꿈
→ 그 바뀐 목표값이 다시 record snapshot으로 저장됨
→ 이후 기록 해석 기준이 식사에 의해 자기 수정됨
```

이 순환 구조는 피해야 한다.
따라서 snapshot은 selected calculation target을 보존한다.

## macro card display QA

macro card range는 scoringContext의 boundary를 본다.
UI-only display cap으로 보기 좋게 숨기는 방식은 쓰지 않는다.

상단 목표 요약 카드의 탄수화물/단백질/지방 목표 g 값과 kcal 비율도
`todayBalance.target` 기준으로 맞춘다.

즉 같은 Today 화면에서 다음이 서로 다른 기준을 보지 않는다.

```text
- 상단 목표 카드
- 오늘 섭취 카드
- macro card range
- Today score
```
