## 4. 1순위: macro range rebalance audit

상태:

```text
현재 구현 지시서 아님
v8.1 copy/help 범위 아님
구현 금지
별도 사전 감사 필요
```

이 후보는 단순 문구 개선이 아니다.

영향 가능 영역:

- 매크로 목표 계산.
- 단백질 목표 / 탄수 범위 / 지방 범위 정책.
- Today KPI와 남은 양 표시.
- 식단 점수와 evidence.
- Coach action / reason / fallback.
- Recent 3일 / 7일 흐름 해석.
- Settings 단백질 기준 옵션.
- Records `goalSnapshot` 해석.
- backup / restore 호환성.

감사에서 먼저 확인할 것:

1. 현재 고정 macro g 목표가 어디서 계산되고 어디에 저장되는지.
2. Today 남은 양 표시가 지방 부족 / 탄수 초과를 어떻게 사용자에게 보여주는지.
3. 점수가 지방 부족, 탄수 초과, 단백질 부족을 어떻게 감점하는지.
4. `goalSnapshot`이 과거 Records에서 고정 목표와 점수를 어떻게 보존하는지.
5. 범위 기반 구조를 도입할 경우 기존 Records를 재해석할지, 저장 당시 기준으로 둘지.
6. backup / restore schema 변경이 필요한지.
7. `targetCal` authoritative, diet `targetCal <= totalBurn`, macro kcal sum guard가 유지되는지.

다음 작업 후보명:

```text
v8.2 macro range rebalance feasibility audit
```

이 audit은 코드 읽기와 재현 중심이어야 하며, 산식 / 점수 / schema를 바로 수정하지 않는다.

## 5. 2순위: onboarding start flow audit

상태:

```text
현재 구현 지시서 아님
v8.1 copy/help 범위 아님
구현 금지
별도 UX 설계 감사 필요
```

온보딩은 오래된 후보지만, 지금까지 밀린 이유가 있다.

- 첫 실행 UX와 Settings 기본값 저장 흐름을 결정한다.
- Today 첫 계산 진입과 기존 사용자 재노출 조건을 결정한다.
- InBody 보유 여부와 체성분 입력 UX에 영향을 준다.
- macro range rebalance가 열리면 온보딩에서 묻는 단백질 / 매크로 기준 질문이 달라질 수 있다.

감사에서 먼저 확인할 것:

1. 신규 사용자와 기존 사용자를 구분하는 조건.
2. 온보딩 완료 값을 Settings에만 저장할지, 첫 Today draft에도 채울지.
3. 운동 여부만 묻는 최소 v1로 시작할지, 목표 / 키 / 성별 / 나이 / 체중까지 받을지.
4. 건너뛰기 후 현재 Settings 흐름이 그대로 유지되는지.
5. backup / restore 이후 온보딩이 다시 뜨지 않아야 하는 조건.
6. mobile 390px에서 첫 실행 흐름이 과하게 길어지지 않는지.

다음 작업 후보명:

```text
v8.2 onboarding start flow UX audit
```

macro range 정책을 먼저 열 경우, 온보딩 audit은 그 결과를 보고 질문 범위를 다시 잡는다.

## 6. 3순위: 남은 저위험 copy 후보

아래 후보는 남아 있지만 필수 작업이 아니다.

| 후보 | 판단 |
| --- | --- |
| Today detail summary | 열 필요 낮음. 열면 단독 pass로 둔다. |
| InBody detail intro | source/apply separation을 흔들지 않는 범위에서만 단독 pass 가능. |
| Settings routine note | 긴 문구지만 의미 보호가 필요하다. 우선순위 낮음. |

이 후보들은 v8.1 copy/help를 다시 여는 사유가 되기 어렵다.

열어야 한다면:

- 한 pass에 한 문구 또는 한 영역만 다룬다.
- `index.html`과 `docs/97_v8.1_copy_help_inventory_2026-07-02.md`만 변경한다.
- 산식, 저장 구조, backup, Records 흐름을 섞지 않는다.

## 7. 4순위: 위험 copy 재오픈 조건

다음은 기본적으로 재오픈하지 않는다.

| 영역 | 기본 판단 |
| --- | --- |
| Records basis edit | 현재 유지. 당시 계산 기준 수정의 재계산 의미가 핵심이다. |
| Records delete confirm | 현재 유지. 전체 삭제와 draft 식사 행 삭제의 반영 시점이 다르다. |
| Backup/Restore warning | 현재 유지. full restore, records-only, smart restore conflict의 반영 범위가 다르다. |
| Coach/score dynamic copy | 현재 유지. action / reason / evidence / record shortage가 얽혀 있다. |

재오픈 조건:

1. 실제 사용자 혼란이나 오조작이 재현된다.
2. 기존 문구가 데이터 반영 시점을 잘못 말한다.
3. 모바일 390px에서 핵심 위험 문구가 읽히지 않는다.
4. 테스트가 현재 copy의 의미를 보호하지 못한다.

재오픈할 경우:

- 한 번에 한 영역만 연다.
- 구현 전 audit을 먼저 한다.
- confirm/modal 반영 시점, 취소 시 원본 유지, 저장 시점, mobile 390px을 같이 확인한다.
