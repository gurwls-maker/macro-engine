# v8.2 macro range contract storage design

상태:

```text
문서 성격: known macroRangeContract storage design 후보 정리
작업 범위: docs-only
구현 변경: 없음
index.html 변경: 없음
테스트 코드 추가: 없음
goalSnapshot 저장 field 추가: 없음
macroRangeContract 저장 구현: 없음
macroRangeBasisSignature 저장 구현: 없음
normalizeGoalSnapshot 변경: 없음
normalizeMacroRangeContractForSnapshot production 구현: 없음
GOAL_SNAPSHOT_SIGNATURE_KEYS 변경: 없음
backup / restore schema 변경: 없음
Recent aggregation 변경: 없음
getDailyAdherenceScore 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
Today / Records / Recent UI 변경: 없음
production macro range 구현: 금지 유지
```

이 문서는 `v8.2_macro_range_storage_compatibility_checkpoint_2026-07-03.md` 이후 단계다.

목적:

```text
macroRangeContract를 저장 가능한 known contract로 올릴 수 있는지 판단하기 전에
후보 shape, storage decision, normalizer / signature / backup / Recent / score 영향표를 정리한다.
```

중요:

```text
이번 문서는 production storage 구현 지시서가 아니다.
이번 문서는 goalSnapshot schema 변경 승인 문서가 아니다.
이번 문서는 macroRangeContract 저장을 승인하지 않는다.
이번 문서는 range score 또는 production macro range 구현을 승인하지 않는다.
```

현재 결론:

```text
known macroRangeContract storage 설계 후보는 문서로 정리 가능하다.
하지만 storage 구현은 아직 금지다.
구현을 열려면 normalizer, signature, backup/restore, Recent, score/Coach 정책이 먼저 닫혀야 한다.
```

## 1. 진입 기준

직전 checkpoint에서 통과한 것:

```text
report-only helper 통과
score candidate report-only 통과
snapshot current contract no-leak 통과
explicit normalizer simulation 통과
signature decision test-local comparison 통과
backup / restore no-leak fixture 통과
Recent mixed-basis test-local detector 통과
```

직전 checkpoint에서 아직 미통과인 것:

```text
macroRangeContract 저장 가능성
macroRangeBasisSignature 저장 가능성
range-aware score / Recent / backup / restore production policy
production normalizer 구현
production macro range 구현
```

따라서 이번 단계가 다루는 것은 storage 구현이 아니라 storage 후보의 조건표다.

## 2. 현재 source of truth

현재 production source of truth:

```text
fixed target goalSnapshot
targetCal
targetProtein
targetCarbs
targetFat
ADHERENCE_SCORING_VERSION = v6.1_alcohol_penalty_v2
GOAL_SNAPSHOT_SIGNATURE_KEYS fixed-only
Recent fixed component counts
backup / restore fixed snapshot normalize path
```

현재 저장 계약이 아닌 것:

```text
macroRangeContract
macroRangeBasisSignature
rangeScoreCandidate
candidateScorePreview
scoreDeltaPreview
normalizedContract
range basis debug output
```

현재 정상 동작:

```text
unknown macroRangeContract는 normalizeGoalSnapshot allowlist에서 prune된다.
candidateScorePreview는 저장 / backup / restore / Recent / UI에 남지 않는다.
Recent는 range candidate를 fixed component count로 조용히 확장하지 않는다.
ADHERENCE_SCORING_VERSION은 유지된다.
```

이 동작은 결함이 아니라 아직 range storage schema를 열지 않은 현재 계약이다.

이번 문서에서 하지 않는 것:

```text

normalizeMacroRangeContractForSnapshot production 함수 구현
macroRangeBasisSignature 저장 field 추가

backup payload에 range field 추가
restore conflict preview 변경
Recent aggregation range-aware 변경
score version 변경
Today UI range 표시
DailyCoach range action 변경
old fixed Records range 재해석
```

이번 문서에서 하는 것:

```text
known macroRangeContract 후보 shape 정리
저장 가능 조건과 차단 조건 분리
normalizer / signature / backup / Recent / score 영향표 작성
future test gate 후보 정리
구현 전 남은 decision 목록 정리
```

## 4. storage 후보 계층

storage 후보는 세 계층으로 나눈다.

### 4-1. fixed_v8

의미:

```text
현재 production 계약
goalSnapshot에는 fixed macro target만 저장
macroRangeContract 없음
range basis signature 없음
```

status:

```text
production source of truth
유지
```

### 4-2. range_contract_candidate

의미:

```text
report-only / test-local에서 valid shape로 검증 가능한 후보
아직 goalSnapshot 저장 field가 아님
backup payload field도 아님
Recent basis도 아님
```

status:

```text
문서 / test-local 후보
저장 금지
```

### 4-3. range_contract_storable

의미:

```text
future explicit normalizer, signature, backup/restore, Recent, score policy가 모두 닫힌 뒤에만 가능한 저장 후보
```

status:

```text
아직 미승인
이번 문서에서 구현하지 않음
```

## 5. macroRangeContract 후보 shape

아래 shape는 storage design 후보이며 production schema 확정이 아니다.

```js
{
  version: "macroRangePolicy.v1",
  basisSource: {
    generatedBy: "macro_range_policy",
    targetCalSource: "authoritative_target_cal",
    fixedTargetSnapshotVersion: "fixed_v8",
    policyVersion: "macro_range_policy_v1"
  },
  targetCal: number,
  protein: {
    targetG: number,
    minAcceptableG: number,
    upperReviewG: number,
    mode: "light" | "standard" | "high",
    basis: "body_weight" | "ffm" | "adjusted_weight",
    evidenceTier: string,
    appPolicyKey: string
  },
  fat: {
    physiologicalFloorG: number,
    actionFloorG: number,
    recommendedMinG: number,
    recommendedMaxG: number,
    flexibleMaxG: number,
    scoreLowThresholdG: number,
    scoreHighThresholdG: number,
    coachWarnLowThresholdG: number,
    coachWarnHighThresholdG: number,
    statusSource: string
  },
  carbs: {
    trainingSupportMinG: number,
    recommendedMinG: number,
    recommendedMaxG: number,
    flexibleMaxG: number,
    derivedFromFatRange: true
  },
  dataQuality: {
    basisConfidence: "high" | "medium" | "low" | "unknown",
    bodyCompositionBasis: "measured" | "estimated" | "unknown",
    trainingContextBasis: "explicit" | "defaulted" | "unknown"
  },
  scoring: {
    candidateVersion: string,
    appliesToSavedRecord: false
  }
}
```

주의:

```text
"macroRangePolicy.v1"은 후보 이름이다.
현재 test-local simulation의 "macroRangePolicy.v0.reportOnly"와 production schema 확정을 혼동하지 않는다.
```

## 6. 저장 금지 field

아래 field는 macroRangeContract 저장 후보에 포함하지 않는다.

```text
candidateScorePreview
scoreDeltaPreview
rangeScoreCandidate
reportOnly
uiVisible
persisted
productionScoreChanged
scoreVersionChanged
debugReason
explanation
rawExternalEvidence
uiCopy
warnings
normalizedContract
```

이유:

```text
score preview는 user-facing score로 오해될 수 있다.
debug / evidence / copy field는 signature와 backup diff를 흔들 수 있다.
report-only meta는 저장 계약이 아니다.
```

## 7. valid 조건 후보

known macroRangeContract를 저장 후보로 보려면 최소한 아래를 만족해야 한다.

```text
contract는 plain object다.
version은 supported known storage version이다.
targetCal은 finite number다.
contract.targetCal은 fixed goalSnapshot.targetCal과 0.01 이내 일치한다.
protein targetG / minAcceptableG / upperReviewG는 finite non-negative다.
protein minAcceptableG <= targetG <= upperReviewG다.
fat physiologicalFloorG <= actionFloorG <= recommendedMinG <= recommendedMaxG <= flexibleMaxG다.
fat scoreLowThresholdG <= scoreHighThresholdG다.
fat coachWarnLowThresholdG <= coachWarnHighThresholdG다.
carbs recommendedMinG <= recommendedMaxG <= flexibleMaxG다.
carbs trainingSupportMinG <= flexibleMaxG다.
carbs.derivedFromFatRange === true다.
scoring.appliesToSavedRecord === false다.
forbidden score/debug/copy/evidence field가 없다.
```

valid여도 이번 단계에서는 저장하지 않는다.

## 8. invalid / unsupported 처리 후보

invalid:

```text
targetCal mismatch
required section missing
required number missing
NaN / Infinity / negative number
protein order violation
fat order violation
carb order violation
candidateScorePreview present
scoreDeltaPreview present
scoring.appliesToSavedRecord === true
```

unsupported:

```text
unknown version
unknown policyVersion
unknown protein basis
unknown dataQuality enum
future contract shape not recognized by current app
```

처리 원칙:

```text
invalid / unsupported macroRangeContract는 fixed snapshot을 깨면 안 된다.
invalid / unsupported contract는 조용히 저장하면 안 된다.
unsupported version pass-through 여부는 backup/restore policy와 함께 별도 결정한다.
```

현재 판단:

```text
production 저장이 열리기 전까지 invalid / unsupported contract는 저장하지 않는다.
old fixed snapshot survival이 우선이다.
```

## 9. normalizer 영향표

| 항목 | 현재 | future 후보 | 이번 문서 판단 |
| --- | --- | --- | --- |
| `normalizeGoalSnapshot` | fixed allowlist만 보존 | macroRangeContract optional 처리 가능성 | 변경 금지 |
| `normalizeMacroRangeContractForSnapshot` | 없음 | explicit production normalizer 후보 | 구현 금지 |
| invalid contract | unknown field라 prune | fixed snapshot survival + explicit reason | 설계 후보 |
| unsupported contract | 현재 prune | 보류/경고/prune 중 결정 필요 | 미결정 |
| unknown nested field | 현재 저장 안 됨 | prune + warning 또는 preserve 결정 필요 | 미결정 |

future production normalizer가 생기더라도 다음 원칙은 지켜야 한다.

```text
fixed snapshot normalize와 range contract normalize를 분리한다.
range contract invalid가 fixed snapshot null로 이어지면 안 된다.
range contract warning은 저장 계약 field가 아니다.
```

## 10. signature 영향표

현재:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS fixed-only 유지
macroRangeContract는 signature 대상 아님
macroRangeBasisSignature 저장 안 함
```

future 후보:

```text
A. current fixed signature에서 계속 제외
B. current signature에 직접 포함
C. separate macroRangeBasisSignature로 분리
```

현재 판단:

```text
B는 아직 위험하다.
C가 future 후보로 가장 안전하지만 저장은 아직 금지다.
이번 문서에서는 GOAL_SNAPSHOT_SIGNATURE_KEYS를 바꾸지 않는다.
```

separate basis signature를 열려면 필요한 것:

```text
field allowlist
stable field order
numeric serialization policy
forbidden field rejection
old fixed absent handling
invalid / unsupported handling
backup roundtrip assertion
Records 무변경 저장 영향 검증
```

## 11. backup / restore 영향표

| 영역 | 현재 | future 후보 | 이번 문서 판단 |
| --- | --- | --- | --- |
| full backup | fixed snapshot payload | valid macroRangeContract roundtrip 후보 | schema 변경 금지 |
| full restore | unknown range field prune | known contract normalize 후보 | policy 미결정 |
| records-only import | fixed records 중심 | old/new basis co-existence 후보 | policy 미결정 |
| smart restore keep | local basis 유지 | local range basis 유지 후보 | 변경 금지 |
| smart restore replace | backup fixed basis 적용 | backup range basis 적용 후보 | future decision |
| smart restore meals-only | local snapshot basis 유지 | basis conflict warning 후보 | 변경 금지 |
| conflict preview | range basis 모름 | basisConflict 후보 | UI/preview 변경 금지 |

필수 원칙:

```text
old backup은 계속 정상 import되어야 한다.
meals-only는 식사 병합이지 snapshot basis 교체가 아니다.
candidateScorePreview는 future에도 backup payload에 저장하지 않는다.
```

## 12. Recent 영향표

현재:

```text
Recent는 fixed targetProtein / targetCarbs / targetFat 기준 component counts를 만든다.
range 후보는 Recent output에 들어가지 않는다.
```

future 후보:

```text
fixedBasisCounts
rangeBasisCounts
mixedBasisSummary
invalidRangeBasisDays
unsupportedRangeBasisDays
```

현재 금지:

```text
getRecentContextRecordSummary 변경
getRecentContextComponentCounts 변경
old fixed Records range 재해석
range record를 fixed fatLowDays에 조용히 합치기
Recent UI basis warning 추가
DailyCoach mixed-basis action 변경
```

storage 후보가 열리더라도 Recent는 가장 늦게 열어야 한다.

이유:

```text
Recent는 여러 날짜와 여러 basis를 섞는다.
storage만 먼저 열면 집계 의미가 흔들릴 수 있다.
```

## 13. score / Coach 영향표

현재:

```text
fixed score가 source of truth다.
ADHERENCE_SCORING_VERSION = v6.1_alcohol_penalty_v2
candidateScorePreview와 scoreDeltaPreview는 null 또는 absent다.
DailyCoach production action은 range basis를 보지 않는다.
```

future 후보:

```text
range-aware score candidate numeric preview
range score production version
DailyCoach range basis action
score evidence reason id 분리
```

현재 금지:

```text
candidateScorePreview 숫자화
scoreDeltaPreview 숫자화
ADHERENCE_SCORING_VERSION 변경
getDailyAdherenceScore 변경
DailyCoach action 변경
Records old score 재해석
```

저장 계약과 score 계약은 분리해서 닫아야 한다.

주의:

```text
macroRangeContract를 저장한다고 해서 range score가 production이 되는 것은 아니다.
range score가 production이 되려면 score version과 old/new record policy가 따로 필요하다.
```

## 14. storage approval gate 후보

storage 구현을 열기 전 필요한 최소 gate:

```text
1. macroRangeContract supported version / shape 확정
2. production normalizer 설계와 tests 확정
3. invalid / unsupported contract 처리 확정
4. signature policy 확정
5. backup/restore roundtrip policy 확정
6. smart restore basis conflict policy 확정
7. Recent fixed/range handling policy 확정
8. score/Coach와 storage의 분리 여부 확정
9. old fixed Records no-migration 원칙 확정
10. user-facing UI 노출 여부 별도 결정
```

하나라도 빠지면 production storage 구현을 열지 않는다.

## 16. false positive 방지

정상으로 볼 것:

```text
이번 단계에서 index.html이 바뀌지 않음
macroRangeContract 저장이 아직 안 됨
unknown range field가 현재 normalize에서 prune됨
backup payload가 range field를 보존하지 않음
Recent가 fixed basis로만 집계됨
score version이 유지됨
```

blocker로 볼 것:

```text
storage design이라는 이름으로 production schema를 추가함
normalizeGoalSnapshot allowlist를 바꿈
GOAL_SNAPSHOT_SIGNATURE_KEYS를 바꿈
backup payload에 macroRangeContract를 넣음
candidateScorePreview를 숫자화하거나 저장함
Recent aggregation을 바꿈
DailyCoach action을 바꿈
old fixed Records를 range 기준으로 재해석함
```

## 17. 아직 결정하지 않은 것

미결정:

```text
1. macroRangeContract version 이름.
2. basisSource section을 저장할지.
3. dataQuality section을 저장할지.
4. unsupported future version을 prune할지, 보류할지, warning으로 둘지.
5. macroRangeBasisSignature를 저장할지.
6. fixed signature와 range basis signature를 어떻게 함께 보여줄지.
7. backup conflict preview에 basisConflict를 보여줄지.
8. records-only import에서 range contract를 보존할지.
9. smart restore meals-only의 basis conflict를 user-facing으로 보여줄지.
10. range score가 production이 되기 전에도 range contract 저장을 허용할지.
```

현재 보수적 권장:

```text
range score production gate가 닫히기 전에는 macroRangeContract 저장도 열지 않는다.
저장만 먼저 열면 Records / Recent / backup / score 의미가 갈라질 수 있다.
```
