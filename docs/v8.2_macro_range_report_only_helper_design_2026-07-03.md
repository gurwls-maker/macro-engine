# v8.2 macro range report-only helper design

작성일: 2026-07-03

## 1. 상태

```text
작업 성격: report-only helper 설계
구현 변경: 없음
index.html 변경: 없음
production 산식 변경: 없음
점수 계산 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
goalSnapshot schema 변경: 없음
backup / restore 변경: 없음
Recent 변경: 없음
Today UI range 표시: 없음
저장: 없음
UI 노출: 없음
```

이 문서는 `docs/v8.2_macro_range_policy_design_2026-07-03.md`의 Phase 1을 구체화한다.

후속 구현 기록:

```text
docs/v8.2_macro_range_report_only_helper_implementation_2026-07-03.md
```

후속 구현은 report-only helper skeleton과 guard tests만 추가하며, production 산식 / score / 저장 / UI / Recent / backup에는 연결하지 않는다.

목표:

```text
current fixed target 결과와 macro range candidate 결과를 같은 입력으로 나란히 비교한다.
production result를 바꾸지 않는다.
score / schema / UI 적용 전 검증용 debug/report-only 구조만 설계한다.
```

이번 문서는 구현 지시서가 아니다. 다음 단계에서 helper 구현을 열더라도 production 산식, UI, 저장, score version, `goalSnapshot`, backup, Recent는 계속 변경 금지다.

## 2. helper의 역할

report-only helper는 다음 질문에 답하기 위한 비교 도구다.

```text
현재 fixed target 계약은 어떤 판단을 내리는가?
같은 입력을 range candidate로 보면 어떤 상태가 되는가?
두 판단이 다르면 왜 다른가?
그 차이가 사용자 문제를 줄이는가, 아니면 새 혼란을 만드는가?
불가능한 macro 조합은 어디에서 발생하는가?
```

helper가 하지 않는 일:

- `calculate()` 결과를 바꾸지 않는다.
- `calculateMacroTargets()`의 production output을 바꾸지 않는다.
- `buildExternalMacroActiveProductionApplication()`의 production output을 바꾸지 않는다.
- `getDailyAdherenceScore()`의 production score를 바꾸지 않는다.
- Today / Records / Recent / Coach UI에 노출하지 않는다.
- Records에 저장하지 않는다.
- backup / restore payload에 포함하지 않는다.

helper의 결과는 transient debug object다.

```text
메모리 안에서 테스트와 감사가 읽는 후보 출력
저장 schema 아님
Records snapshot 아님
backup field 아님
user-facing contract 아님
```

## 3. 위치와 호출 원칙 후보

실제 구현 단계에서 허용될 수 있는 변경은 제한적이다.

허용 후보:

```text
index.html 안에 report-only helper 함수 추가
내부 테스트에서 helper 호출
debug/report-only report object 생성
```

계속 금지:

```text
production result에 helper output 병합
state 저장
Records record에 저장
goalSnapshot normalize/signature에 연결
backup / restore normalize에 연결
Recent context에 연결
Today render에 연결
DailyCoach decision에 연결
```

권장 위치 후보:

```text
calculate() 이후의 currentResult를 입력으로 받는 별도 pure helper
또는 internal test/report builder 전용 helper
```

주의:

```text
helper가 calculate() 안에서 production path처럼 호출되면 위험하다.
처음 구현은 internal tests/report builder가 명시적으로 호출하는 형태가 더 안전하다.
```

## 4. input contract 후보

helper input은 fixed target과 range candidate를 같은 조건에서 비교할 수 있어야 한다.

후보 shape:

```text
macroRangeReportOnlyInput: {
  targetCal: number,
  fixedTarget: {
    proteinG: number,
    carbsG: number,
    fatG: number,
    source: "currentResult" | "goalSnapshot"
  },
  consumed: {
    kcal: number,
    proteinG: number,
    carbsG: number,
    fatG: number,
    mealCount: number,
    hasFullDaySignal: boolean
  },
  goalContext: {
    goal: string,
    mode: string | null,
    exerciseManagementMode: string | null
  },
  bodyProfile: {
    weightKg: number,
    heightCm: number | null,
    gender: string | null,
    age: number | null,
    bodyFatPercent: number | null,
    skeletalMuscleKg: number | null,
    basisWeightKind: "body_weight" | "ffm" | "adjusted_weight" | "unknown"
  },
  activityContext: {
    activityLevel: string | null,
    workType: string | null,
    sleepHours: number | null,
    isWeightTrainingDay: boolean,
    cardioKcal: number,
    totalBurn: number | null
  },
  fixedScoreContext: {
    scoringVersion: string | null,
    score: number | null,
    label: string | null,
    componentScores: object | null,
    reasons: string[]
  },
  dataQuality: {
    oneMealPending: boolean,
    insufficientRecord: boolean,
    snapshotless: boolean,
    sourceDate: string | null
  },
  oldGoalSnapshot: object | null
}
```

input 원칙:

- `targetCal`은 이미 production 계산이 만든 값을 사용한다.
- helper가 `targetCal`을 새로 산출하지 않는다.
- fixed macro target은 current result 또는 old goalSnapshot에서 읽는다.
- old goalSnapshot이 있을 경우 old fixed basis로 표시한다.
- consumed macro는 실제 기록 또는 fixture가 제공한 값을 읽는다.

## 6. protein anchor candidate

protein candidate는 숫자를 확정하지 않는다.

report-only helper가 할 수 있는 일:

```text
현재 fixed protein target을 anchor로 재사용
policy design의 protein mode 후보를 label로만 연결
BW / FFM / adjusted weight 중 어떤 basis가 필요한지 flag로 표시
```

금지:

- 목표별 단백질 계수를 새로 확정하지 않는다.
- 단백질 preset 이름을 production copy처럼 확정하지 않는다.
- protein target을 production result에 덮어쓰지 않는다.

후보 output:

```text
proteinAnchor: {
  targetG: number,
  source: "fixed_target_reused" | "candidate_policy_unresolved",
  basisWeightKind: string,
  modeCandidate: "unresolved",
  needsDecision: [
    "protein_mode_names",
    "bw_vs_ffm_vs_adjusted_weight"
  ]
}
```

## 7. fat range candidate

fat range candidate는 지방 의미를 여러 층으로 나눈다.

후보 output:

```text
fatRange: {
  source: "report_only_candidate",
  physiologicalFloorG: number | null,
  actionFloorG: number | null,
  recommendedMinG: number | null,
  recommendedMaxG: number | null,
  flexibleMaxG: number | null,
  scoreLowThresholdG: number | null,
  scoreHighThresholdG: number | null,
  coachWarnLowThresholdG: number | null,
  coachWarnHighThresholdG: number | null,
  status: "below_action_floor" | "inside_recommended" | "above_recommended" | "above_flexible" | "unresolved",
  needsDecision: string[]
}
```

원칙:

- `physiologicalFloorG`, `actionFloorG`, `recommendedMinG`, `scoreLowThresholdG`, `coachWarnLowThresholdG`는 같은 값이라고 가정하지 않는다.
- report-only 단계에서는 확정 숫자가 없으면 `null`과 `needsDecision`으로 남긴다.
- 기존 `fatTargetRange`를 그대로 사용자 계약으로 승격하지 않는다.

## 8. carb range candidate

carb range는 protein anchor와 fat range를 기준으로 targetCal 안에서 산출되는 후보다.

후보 output:

```text
carbRange: {
  source: "derived_from_targetCal_protein_fat_range",
  trainingSupportMinG: number | null,
  recommendedMinG: number | null,
  recommendedMaxG: number | null,
  flexibleMaxG: number | null,
  currentStatus: "below_training_support" | "inside_candidate_range" | "above_candidate_range" | "unresolved",
  derivedFromFatRange: true,
  needsDecision: string[]
}
```

후보 계산 개념:

```text
energyAfterProtein = targetCal - proteinAnchorG * 4
carbMaxIfFatAtActionFloor = (energyAfterProtein - fatActionFloorG * 9) / 4
carbMinIfFatAtRecommendedMax = (energyAfterProtein - fatRecommendedMaxG * 9) / 4
```

주의:

- 이 공식은 helper candidate의 debug calculation이다.
- production macro target을 바꾸지 않는다.
- `trainingSupportMinG`가 정해지지 않았으면 carb status를 확정하지 않는다.

## 9. exchange validity

range candidate의 핵심은 탄수/지방 교환이 합리적인지 보는 것이다.

후보 output:

```text
exchangeValidity: {
  kcalMatched: boolean,
  proteinSatisfied: boolean,
  fatFloorSatisfied: boolean | null,
  fatWithinRecommended: boolean | null,
  carbsWithinCandidateRange: boolean | null,
  fixedCarbOverExplainedByFatLower: boolean | null,
  fixedFatLowExplainedByCarbHigher: boolean | null,
  shouldPenalizeCarbsHigh: "yes" | "no" | "unresolved",
  shouldPenalizeFatLow: "yes" | "no" | "unresolved",
  explanationIds: string[]
}
```

해석 예시:

```text
총칼로리와 단백질이 맞고, 지방 action floor를 넘으며, 탄수가 range 안이면
fixed target 기준 탄수 초과라도 range candidate에서는 정상일 수 있다.
```

반대로:

```text
지방이 action floor 아래이고, 단백질도 부족하며, 기록이 한 끼뿐이면
range candidate도 강한 결론 대신 pending / insufficient로 남겨야 한다.
```

## 10. impossible state 후보

helper는 불가능한 조합을 숫자로 억지 해결하지 않는다.

후보 상태:

```text
impossibleState: {
  code: string,
  severity: "info" | "warning" | "blocker",
  explanation: string,
  affectedFields: string[],
  suggestedNextStep: string
}
```

후보 code:

| code | 의미 |
| --- | --- |
| `protein_anchor_consumes_targetCal` | protein anchor만으로 targetCal이 거의 소진된다. |
| `fat_floor_breaks_carb_training_minimum` | fat action floor를 넣으면 carb training minimum을 만족할 수 없다. |
| `fat_recommended_max_makes_carb_min_negative` | fat recommended max를 넣으면 carbMin이 음수가 된다. |
| `targetCal_macro_range_infeasible` | targetCal 자체가 goal/activity와 맞지 않아 macro range를 닫을 수 없다. |
| `fixed_target_conflicts_with_range_candidate` | fixed target과 range candidate의 사용자 의미가 충돌한다. |
| `insufficient_policy_numbers` | 필요한 앱 정책값이 없어 candidate status를 확정할 수 없다. |
| `insufficient_data_quality` | 한 끼/기록 부족 때문에 하루 판단을 확정할 수 없다. |

중요:

```text
impossibleState는 production warning이 아니다.
report-only 설계/테스트가 읽는 debug 상태다.
```

## 11. fixed vs range 비교 항목

## 14. macroRangeContract 취급 원칙

`macroRangeContract`는 아직 저장 schema가 아니다.

이번 단계의 의미:

```text
report-only output 후보
debug comparison shape
future snapshot compatibility discussion input
```

아직 아닌 것:

```text
goalSnapshot field
backup field
Records field
Recent basis field
UI data contract
production scoring basis
```

금지:

- `normalizeGoalSnapshot()`에 연결하지 않는다.
- `GOAL_SNAPSHOT_SIGNATURE_KEYS`에 추가하지 않는다.
- full backup / records-only backup에 넣지 않는다.
- smart restore conflict preview에 넣지 않는다.
- Records detail edit roundtrip에 넣지 않는다.

별도 필요 문서:

```text
v8.2 macro range snapshot compatibility design
```

이 문서가 작성되기 전까지 `macroRangeContract`는 저장되지 않는다.

## 17. 최종 결론

```text
report-only helper design: 진행 가능
production macro range 구현: 아직 금지
숫자 정책 확정: 아직 금지
macroRangeContract: 저장 schema가 아니라 report-only 후보 계약
```

helper의 첫 목적은 정답을 내는 것이 아니다.

첫 목적은 다음을 안전하게 관찰하는 것이다.

```text
fixed target 계약이 어떤 사용자 오해를 만드는지
range candidate가 그 오해를 줄이는지
range candidate가 새 모순을 만드는지
어떤 숫자 정책이 아직 비어 있는지
어떤 경우에는 macro range 자체가 닫히지 않는지
```

따라서 다음 구현 후보도 production 적용이 아니라 `report-only helper`까지만 열 수 있다.
