# v8.3 target/scoring alignment implementation

작성일: 2026-07-08
문서 성격: implementation log / release-blocker fix

## 배경

`v8.3_target_scoring_alignment_incident_decision_2026-07-08.md`는 target generator와 scoring range가 서로 다른 기준을 보는 문제를 release blocker로 분류했다.

대표 재현은 다음 상태였다.

```text
target carbs: 594.9g
target fat: 124.8g
card range / scoring boundary:
- carbs 210g~525g
- fat 53g~94g
target macros 그대로 섭취 시 score: 83.712
```

이 상태는 v8.3의 range-aware score가 아니라, 앱이 제시한 목표값을 점수기가 다시 벌하는 구조다.

이 implementation은 이 release blocker를 닫는다.

## 세션 원문에서 추가 반영한 판단

최신 세션 원문에서 사용자는 단순히 카드에 range chip을 붙이는 것보다, 오늘 카드의 목표 g 값 자체가 남은 칼로리 안에서 움직이는 방식이 더 자연스럽다고 판단했다.

반영한 핵심은 다음과 같다.

```text
수용:
- 표시 범위와 scoring boundary는 다르면 안 됨.
- 목표값 자체가 Today에서 탄수/지방 교환에 따라 움직일 수 있어야 함.
- 사용자가 원하면 고정 목표값으로 되돌릴 수 있어야 함.
- 반복 고탄수 / 반복 고지방 흐름은 3일/7일 맥락으로 자동 조정폭을 줄여야 함.

폐기:
- UI만 좁게 cap해서 보기 좋게 만드는 방식.
- high-volume이면 10.5g/kg까지 flat no-penalty로 허용하는 방식.
- 운동일이라는 이유로 final score bonus를 주는 방식.
- 기존 테스트 문구에 맞추려고 제품적으로 어색한 문구나 산식을 유지하는 방식.
```

## 구현 요약

### 1. shared target-aligned macro range helper

`index.html`에 target/scoring/card display가 함께 사용할 helper를 추가했다.
상단 목표 요약 카드의 탄수화물/단백질/지방 목표 g 값과 kcal 비율도 같은
`todayBalance.target` 기준을 사용한다. 즉 Today 화면 안에서 상단 목표 숫자,
오늘 섭취 카드, 점수 산식이 서로 다른 macro target을 보지 않는다.

```text
getV83TargetAlignedMacroRanges(balance, currentResult, options)
```

이 helper는 다음 범위를 한 번에 만든다.

```text
- proteinRange
- carbRange
- fatRange
- target-aligned trainingContext
- trainingLoadFactor
```

핵심은 macro card range와 scoring penalty boundary가 같은 source를 보게 한 것이다.

### 2. carb exercise context 재정렬

기존 `trainingCarbContexts`의 `highGkg` 값은 계속 참고 anchor로 남긴다.

하지만 이 값을 그대로 “여기까지 무조건 무감점”으로 쓰지 않는다.

```text
이전 위험:
high_volume_or_long_session highGkg 10.5
→ 체중 75kg이면 787.5g까지 no-penalty처럼 보일 수 있음

현재:
target carbs 중심의 연속 boundary를 만들고
trainingLoadFactor는 그 boundary를 약간 이동시키는 context로만 사용
```

새 테스트는 high-volume day에서도 target 594.9g fixture의 carb range max가 700g 미만으로 남는 것을 고정한다.

### 3. fat target/range mismatch 해결

`fatRange`도 target fat을 밖으로 밀어내지 않도록 target-aligned로 만든다.

이것은 fat 1000g 같은 extreme을 봐주기 위한 것이 아니다.

```text
목표 fat 자체:
- range 안에 들어와야 함

extreme fat:
- fatRangePenalty + target/TDEE energy axis로 계속 악화됨
```

### 4. Today 목표 자동 조정

새 설정을 추가했다.

```text
state.adaptiveMacroTargetsEnabled = true
Settings label: 오늘 목표 자동 조정
기본값: ON
```

ON이면 Today의 live balance에서 탄수화물과 지방 목표가 같은 kcal 안에서 교환된다.
이 교환 target은 Today 화면과 Today score의 live interpretation이다. Records `goalSnapshot`에는
식사 기록으로 반응해 움직인 target을 저장하지 않고, 사용자가 선택한 원 계산 기준의
`targetCal/protein/carbs/fat`를 저장한다. 식사 내용이 자기 저장 기준을 다시 쓰는 순환 구조를 막기 위해서다.

예:

```text
탄수 진행률이 지방 진행률보다 높음
→ 오늘 탄수 target은 조금 올라감
→ 오늘 지방 target은 같은 kcal 안에서 내려감

지방 진행률이 탄수 진행률보다 높음
→ 오늘 지방 target은 조금 올라감
→ 오늘 탄수 target은 같은 kcal 안에서 내려감
```

단백질은 exchange target이 아니라 anchor로 유지한다.

OFF이면 기존처럼 고정 target을 쓴다.

### 5. 3일/7일 trend limiter

반복 고탄수 또는 반복 고지방 기록이 있으면 자동 조정폭을 줄인다.

```text
최근 3일 중 2일 이상 high-carb
또는 최근 7일 중 4일 이상 high-carb
→ carb target upward relaxation 축소

최근 3일 중 2일 이상 high-fat
또는 최근 7일 중 4일 이상 high-fat
→ fat target upward relaxation 축소
```

이 장치는 “계속 고탄수로 먹으면 앱이 계속 목표를 올려 주는 문제”를 막기 위한 것이다.

### 6. carb/fat exchange double punishment 보정

총칼로리와 단백질이 target에 맞고, 지방 target에서 빠진 kcal만큼 탄수화물이 올라간 경우에는 탄수 초과 penalty를 다시 크게 주지 않는다.

이것은 탄수 초과를 무시하는 것이 아니다.

```text
허용:
- fat target deficit kcal 안에서 설명되는 carb increase
- protein target 또는 protein floor가 충분함
- target/TDEE energy axis가 과하게 높지 않음

계속 반영:
- total kcal / TDEE overload
- target energy deviation
- fat range penalty
- explainable exchange room을 넘는 carb excess
```

## storage / schema 판단

이번 작업에서 Records preview field, scoreDeltaPreview, candidateScorePreview, pointsPreview, UI-only schema field는 열지 않았다.
Records `goalSnapshot`도 adaptive live target으로 덮어쓰지 않는다. snapshot은 selected calculation target을 보존한다.

추가된 것은 Settings boolean이다.

```text
adaptiveMacroTargetsEnabled
```

이 값은 사용자가 Today 목표 자동 조정을 켜고 끄기 위한 설정이다.

old records migration, old records recompute, scoreDeltaPreview, storage/schema expansion은 계속 별도 gate다.

## 테스트 보강

`runMacroRangeContinuousScoringTests`에 다음 guard를 추가했다.

```text
- target macros 그대로 섭취하면 score >= 95
- target kcal == totalBurn이면 target/TDEE penalty가 0
- target carbs/fat이 scoring range 안에 있음
- macro card range가 scoring boundary와 일치
- high-volume carb anchor가 flat 10.5g/kg permission으로 열리지 않음
- adaptive Today target이 carb/fat을 같은 kcal 안에서 교환
- repeated high-carb history가 further carb relaxation을 제한
- setting OFF이면 fixed target 유지
```

기존 `runMacroRangeProductionScoreImplementationDecisionTests`의 carb/fat exchange guard도 계속 통과하도록 산식을 보정했다.
