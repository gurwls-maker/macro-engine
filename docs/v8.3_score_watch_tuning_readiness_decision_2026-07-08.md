# v8.3 score WATCH tuning readiness decision

작성일: 2026-07-08
문서 성격: docs-only WATCH 재검토 / tuning readiness decision

## 목적

이 문서는 `docs/v8.3_score_distribution_tuning_audit_2026-07-07.md`와 `docs/v8.3_score_distribution_tuning_decision_2026-07-07.md`에서 남긴 WATCH 후보를 다시 본다.

여기서 WATCH는 "실패는 아니지만 계속 지켜봐야 하는 후보"라는 뜻이다.

이번 판단은 다음 셋을 분리한다.

1. 실제 산식 튜닝이 지금 필요한 후보.
2. 산식 변경 전 회귀 테스트로 먼저 잠가야 하는 후보.
3. 산식이 아니라 문구, source visibility, product tone으로 다룰 후보.

이번 문서는 코드 구현 문서가 아니다.

## 현재 상태

현재 앱은 v8.3 continuous scoring 구현 상태다.

- production scoring version: `v8.3_anchor_continuous_macro_score_v1`
- formula name: `anchor_continuous_macro_score_v1`
- app display label: `v8.3`
- source visibility: Today live score에서 missing total burn / targetProteinFallback warning 표시
- v8.2 fixed table: production body 아님

현재 산식은 상태별 고정 감점표가 아니라 앵커 기반 곡선이다.

- TDEE overload: `tdeeOverloadAnchors`
- target energy deviation: `targetEnergyOverAnchors`, `targetEnergyUnderAnchors`
- fat high/low: `fatHighAnchors`, `fatLowAnchors`
- alcohol physiology: `alcoholDrinkAnchors`
- carb exercise context: `trainingCarbContexts` + carb high/low anchors

즉 지금 판단할 일은 "고정 감점표로 돌아갈지"가 아니다.
지금 판단할 일은 "현재 곡선에서 남은 WATCH 후보가 즉시 튜닝 대상인지"다.

## 전체 결론

지금은 score tuning implementation으로 바로 가지 않는다.

내 판단은 아래와 같다.

```text
즉시 anchor 튜닝: no
즉시 score formula 변경: no
즉시 UI/storage/schema 변경: no
즉시 scoreDeltaPreview 작업: no

다음 본류:
v8.3 WATCH 후보 회귀 테스트 보강
```

이유는 단순히 보수적으로 미루기 위함이 아니다.

현재 WATCH 후보들은 대부분 다음 중 하나다.

- 이미 source visibility 구현으로 닫힌 문제.
- final display가 0으로 clamp되지만 rawScore는 계속 악화되는 문제.
- 숫자보다 사용자 문구와 product tone이 더 중요한 문제.
- 튜닝 가능성은 있지만, 먼저 "지금 기대하는 관계"를 테스트로 고정해야 하는 문제.

따라서 지금 anchor 값을 바꾸면 근거 기반 튜닝이 아니라 두 번째 관성 변경이 된다.
먼저 WATCH 후보를 regression fixture로 잠그고, 그 결과로 실제 조정이 필요한지 봐야 한다.

## W01 판단 상세

TDEE 2.0은 "조금 벗어남"이 아니라 총소비량의 2배 수준이다.
따라서 final score가 0으로 내려가는 것 자체는 제품적으로 이상하지 않다.

다만 중요한 점은 final display의 0이 중간 score cap처럼 작동하면 안 된다는 것이다.

현재 감사표에서는 다음 관계가 이미 보인다.

```text
TDEE 1.5: final 39.19
TDEE 2.0: final 0, rawScore -16
TDEE 3.2: final 0, rawScore -122.41
```

즉 final display는 0으로 같아 보여도 rawScore는 계속 나빠진다.
이건 "더 먹어도 똑같다"가 아니라 "사용자 표시 범위는 0 아래로 내려가지 않지만 내부 심각도는 계속 커진다"는 구조다.

따라서 W01은 지금 anchor 튜닝 대상이 아니다.
다음 회귀 테스트에서 rawScore relation을 더 분명히 잠그면 된다.

## W02 판단 상세

fat upper 3.0x가 final 0 근처인 것은 볼만한 WATCH 후보다.
하지만 여기서 바로 fat penalty를 완화하면 반대로 지방 초과를 너무 가볍게 볼 수 있다.

현재 중요한 제품 요구는 이것이다.

```text
fat upper +1g: 거의 감점 없음
fat upper 2.0x: 의미 있는 감점
fat upper 3.0x: 매우 큰 감점 가능
fat 1000g: 자연스럽게 collapse
```

이 방향 자체는 맞다.

따라서 W02는 "지금 완화"가 아니라 "2.0x, 3.0x, 1000g 관계를 테스트로 고정"하는 쪽이 맞다.
테스트로 잠근 뒤 실제 사용자 샘플에서 3.0x가 지나치게 빨리 무너진다는 근거가 생기면 그때 `fatHighAnchors`와 energy 합산 강도를 같이 조정한다.

## W03/W05 판단 상세

알코올은 이번 v8.3에서 이미 두 축으로 들어가 있다.

1. 술의 칼로리: total kcal / target energy / TDEE overload에 반영.
2. 술 자체의 생리학적 risk: `alcoholPhysiologyPenalty`에 반영.

따라서 "술을 판단하지 않는다"가 아니다.
오히려 v6.1처럼 마지막에 별도 alcoholImpactPenalty를 한 번 더 빼는 방식으로 돌아가지 않고, v8.3에서는 kcal 축과 physiology 축을 분리해서 본다.

W03과 W05의 남은 문제는 숫자보다 제품 톤이다.

- 112g/140g alcohol에서 사용자가 "점수만 보면 충분히 위험하게 느끼는가?"
- 운동일 high carb 맥락이 alcohol risk를 덮어버리는 말투가 되지 않는가?
- DailyCoach가 술을 단순 칼로리 초과로만 설명하지 않는가?

따라서 다음 immediate score tuning 대상은 아니다.
다만 DailyCoach/copy tone pass가 열릴 때 최우선 후보로 유지한다.

## W04 판단 상세

W04가 이번 WATCH 중 가장 실제 튜닝 가능성이 큰 후보라고 본다.

현재 code policy는 다음이다.

```text
rest highGkg: 4.2
unknown highGkg: 5.5
normal_resistance highGkg: 7.0
high_volume_or_long_session highGkg: 10.5
```

unknown은 rest와 normal 사이에 있다.
따라서 "운동 여부를 모르면 무조건 rest로 본다"도 아니고, "운동한 날처럼 본다"도 아니다.

하지만 사용자가 운동을 입력하지 않았거나 운동 맥락이 불확실한 상태에서 450g 탄수가 너무 쉽게 높은 점수를 받는다면, 앱이 운동 맥락을 과하게 추정하는 인상을 줄 수 있다.

이 후보는 감으로 바꾸면 안 된다.
다음 작업에서 아래 관계를 테스트로 먼저 고정해야 한다.

```text
같은 탄수 섭취량일 때:
rest <= unknown <= normal_resistance <= high_volume_or_long_session

하지만:
unknown은 high_volume_or_long_session처럼 작동하면 안 됨.
high_volume_or_long_session이어도 TDEE overload를 지우면 안 됨.
```

이 테스트를 먼저 둔 다음, 실제로 unknown이 너무 관대하다고 판단되면 `unknown.highGkg`나 carb high curve를 조정한다.

## W06 판단 상세

W06은 source visibility implementation으로 이미 닫힌 쪽이다.

full macro day에서 TDEE가 없으면 앱은 TDEE overload penalty를 만들지 않는다.
이건 맞다. 없는 총소비량 값을 억지로 추정해 penalty를 만들면 더 위험하다.

대신 현재 구현은 사용자에게 총소비량 기준이 없다는 warning을 보여준다.
그리고 stored current-version auto snapshot을 hidden recompute하지 않는다.

따라서 W06은 지금 score tuning 대상이 아니다.
이미 source visibility guard로 처리된 후보로 분류한다.

## 회귀 테스트 보강이 필요한 이유

이번 판단에서 "바로 튜닝하지 않는다"는 말은 "나중에 기억나면 하자"가 아니다.

다음 gate를 별도로 둬야 한다.

```text
v8.3 WATCH 후보 회귀 테스트 보강
```

이 작업은 숫자를 바꾸지 않고 아래 관계를 테스트로 고정한다.

| fixture group | test direction |
| --- | --- |
| TDEE overload | 1.5x > 2.0x > 3.2x 순서로 final/raw severity가 악화되어야 함 |
| fat excess | upper +1g는 near-zero, 2.0x와 3.0x는 악화, 1000g는 더 심하게 collapse |
| alcohol amount | 80g < 112g < 140g로 physiology penalty가 커지고, v6.1 후처리 감산은 돌아오지 않음 |
| carb context | rest, unknown, normal, high-volume 순서의 context 차이를 보되, 운동 맥락이 TDEE overload를 지우지 않음 |
| missing TDEE | score는 가능하지만 source warning이 보이고 TDEE penalty를 발명하지 않음 |

이 회귀 테스트 보강이 먼저 닫혀야 다음 tuning implementation을 열 수 있다.
