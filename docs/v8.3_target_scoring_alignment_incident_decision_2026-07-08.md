## 1. 목적

이번 문서는 `v8.3_macro_card_range_display_implementation_2026-07-08.md` 이후 드러난 목표값과 점수 범위의 정합성 문제를 release blocker로 재분류한다.

직전 macro card 작업은 Today card의 표시 범위를 실제 v8.3 scoring boundary와 맞췄다. 이 정정 자체는 맞다. 사용자에게 보이는 `오늘 범위`가 실제 점수 산식과 다르면 더 큰 혼선이 생기기 때문이다.

하지만 그 결과 더 근본적인 문제가 드러났다.

```text
앱이 제시한 target macros를 그대로 섭취해도
v8.3 score가 낮게 떨어질 수 있다.
```

이것은 UI 표시 문제가 아니다. target macro generator, v8.3 scoring range, exercise carb context, fat range, macro card display가 같은 기준을 보지 못한 incident다.

## 2. incident summary

현재 v8.3은 anchor-based continuous macro scoring을 production score path로 구현했다.

유지해야 할 원칙:

```text
- 외부근거는 hard cutoff가 아니라 curve anchor다.
- 운동은 final score bonus가 아니라 carb curve context다.
- UI range는 scoring boundary와 달라지면 안 된다.
- 테스트 통과는 생리학적/제품 정합성 통과가 아니다.
```

이번 incident의 핵심 실패:

```text
target macro generator가 제시한 목표 탄수/지방이
scoring neutral/display range 밖에 있다.
```

따라서 이전의 `stabilization/tag readiness checkpoint update` 흐름은 보류한다. 다음 본류는 tag update가 아니라 target/scoring alignment gate다.

## 4. source analysis

### 4.1 target macro generator

관련 함수:

```text
calculateMacroTargets
applyPlausibilityLayer
getFatTargetContext
```

현재 target macro는 대략 아래 흐름이다.

```text
protein = goal protein kg * basis weight
fat = fatTargetPreferredG / fat practical ceiling 기준
carbs = (targetCal - protein*4 - fat*9) / 4
plausibility layer에서 protein/carbs/fat cap 및 unresolved kcal 재배분
```

즉 target carbs는 많은 경우 남은 칼로리를 받는다.

### 4.2 v8.3 scoring range

관련 함수:

```text

getV83TrainingContext

getCarbExerciseContextPenalty
```

현재 carb scoring range:

```text

unknown: 1.8~5.5 g/kg
normal_resistance: 2.8~7.0 g/kg
high_volume_or_long_session: 4.0~10.5 g/kg
```

현재 training context는 아래 조건으로 계단식 분류된다.

```text
high_volume_or_long_session:
- weightDuration >= 90
- exerciseKcal >= 750

- weightDuration > 0
- cardioKcal >= 150
```

이 구조는 운동 aware가 아예 없는 것은 아니지만, 연속적인 운동량/강도 반영이 아니라 coarse bucket이다.

## 6. carb exercise context decision

현재 구조:

```text
rest: 1.3~4.2 g/kg
unknown: 1.8~5.5 g/kg
normal_resistance: 2.8~7.0 g/kg
high_volume_or_long_session: 4.0~10.5 g/kg
```

이 값들은 외부근거 anchor로는 쓸 수 있다. 그러나 현재 구현에서는 사실상 scoring neutral band처럼 쓰인다.

이것이 문제다.

ISSN nutrient timing / carbohydrate 근거에서 8~12g/kg/day는 glycogen storage/recovery 또는 높은 훈련량 맥락에서 언급되는 값이다. 이 값을 high-volume bucket의 flat no-penalty upper처럼 쓰면 외부근거를 너무 단순화한다.

결정:

```text
current bucket-only carb context는 release-ready가 아니다.
bucket label은 설명/분류에 남길 수 있지만,
scoring range는 continuous trainingLoadFactor 또는 shared policy helper에서 산출해야 한다.
```

우선 검토할 구현 방향:

```text
C. bucket은 label로만 유지하고, scoring range는 continuous factor로 산출
E. target generator와 scoring range를 shared policy helper로 통합
```

`C`와 `E`는 배타적이지 않다. 최종 구현에서는 `shared macro target/scoring policy helper`가 target range와 scoring neutral range를 함께 산출하고, training label은 사용자 설명과 테스트 grouping에만 남기는 쪽이 가장 안전하다.

## 7. fat target/range decision

현재 재현 문제:

```text
target fat: 124.8g
scoring/display fat range: 53g~94g
fatRangePenalty: 5.15
```

이는 target generator와 scoring fat range가 같은 기준을 보지 못한다는 뜻이다.

결정:

```text
target fat이 scoring neutral range 밖에 있는 상태를 release-ready로 볼 수 없다.
```

우선 검토할 구현 방향:

```text
C. target fat은 center/preferred로 두고,
   scoring fat range는 target-centered tolerance + AMDR/bodyweight/external anchor를 함께 써서 산출
```

단순히 UI range만 target 중심으로 넓히는 것은 금지한다. scoring range와 target range를 같은 helper에서 정렬해야 한다.

## 8. UI range principle

이번 incident 이후에도 UI 원칙은 바뀌지 않는다.

```text
유지:
- macro card 오늘 범위는 실제 scoring boundary와 일치해야 한다.
- UI-only display cap은 폐기 유지.
- 사용자가 보는 range와 점수 산식 range가 다르면 안 된다.

폐기:
- 보기에 넓으니 UI만 좁히기.
- target/scoring mismatch를 copy로만 설명하기.
- "운동했으니 괜찮음" 같은 면죄부 copy.
```

따라서 다음 구현은 UI display cap이 아니라 target/scoring shared policy를 고쳐야 한다.
