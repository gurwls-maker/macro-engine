05 required result log format - current truth
작성일: 2026-07-11
문서 성격: 저장소 공통 결과로그 계약 / 비개발자 우선 설명

목적
- 사용자가 코드를 몰라도 무엇이 달라졌고 큰 정책 판단이 맞는지 검토할 수 있게 한다.
- 개발 검증에 필요한 기술 세부사항도 버리지 않고, 사용자 설명 아래에서 별도로 확인할 수 있게 한다.
- 특정 Codex 대화의 기억에 의존하지 않고 새 로컬 Codex도 같은 순서로 보고하게 한다.

적용 범위
- 구현, 의사결정, 감사, QA, 문서 정책, 병합, 배포와 같은 모든 실질적 작업의 저장 결과로그에 적용한다.
- 같은 작업의 최종 사용자 보고에도 적용한다.
- 단순 상태 확인, 명령 출력, 링크 수정, 무변경 점검은 축약형을 쓸 수 있다. 그래도 사용자 영향과 확인 결과를 기술 세부사항보다 먼저 말한다.
- 이 형식 때문에 결과 문서를 새로 만들 필요는 없다. 원래 결과로그가 필요한 작업에 적용하고, 짧은 작업은 최종 사용자 보고에서 축약한다.

작성 순서
1. v8.3 mandatory pre-read처럼 기계 검증에 필요한 머리말이 있으면 그 블록만 먼저 둔다.
2. 첫 본문 섹션은 반드시 `비개발자용 설명`으로 둔다.
3. 개발자용 파일, 함수, 데이터 shape, 커밋, 테스트 명령은 그 아래 `기술 검증`에 둔다.

비개발자용 설명 필수 항목
1. 무엇을 바꿨는지
2. 왜 바꿨는지
3. 실제 사용자 화면이나 계산 결과가 어떻게 달라지는지
4. 정책, 산식, 데이터 해석이 바뀌었는지
5. 바꾸지 않은 범위와 보류한 내용
6. 사용자가 큰 틀에서 확인해야 할 판단점
7. 테스트에서 어떤 사용자 동작이나 계산 결과를 검증했는지

작성 원칙
- 코드명, 함수명, 파일명만 나열해서 설명을 대신하지 않는다.
- 정책이나 계산 구조는 가능한 한 실제 숫자 예시로 풀어쓴다.
- 예를 들어 같은 총칼로리 안에서 탄수화물 25g을 줄이고 지방 약 11g을 늘리는 구조라면, 내부 helper 이름보다 먼저 이 교환이 총칼로리를 늘리지 않는다는 뜻을 설명한다.
- 정책, 산식, 저장 데이터 해석이 바뀌지 않았으면 생략하지 말고 `변경 없음`이라고 명시한다.
- 구현한 사실과 검토 후보, 보류, 추후 제안을 섞지 않는다.
- 테스트는 스크립트 이름만 적지 말고 무엇을 검증했는지 함께 적는다. 예: `모바일 390px에서도 도움말 버튼이 보이고 키보드로 열리는지 확인`.
- 실제 화면이나 계산에 변화가 없으면 그렇게 적는다. 문서 변경을 제품 변경처럼 과장하지 않는다.
- 사용자에게 판단을 요청할 내용이 없으면 `추가 판단 필요 없음`이라고 적는다.
- 설명 길이는 작업 위험에 맞춘다. 작은 작업을 긴 형식으로 부풀리지 않는다.

이 형식의 금지선
- 비개발자용 설명보다 기술 세부사항을 먼저 두지 않는다. 단, mandatory pre-read 같은 기계 검증 머리말은 예외다.
- 함수명이나 테스트 통과 수치만으로 사용자 영향을 설명했다고 간주하지 않는다.
- 테스트가 통과했다는 이유만으로 정책이나 산식 판단이 맞다고 단정하지 않는다.
- 반대로 문구 테스트가 실패했다는 이유만으로 어색한 기존 문구를 되살리지 않는다. 의미 계약과 사용자 자연스러움을 함께 검토한다.
- 이 결과로그 계약을 이유로 앱 기능, 정책, 산식, storage/schema 범위를 새로 열지 않는다.
