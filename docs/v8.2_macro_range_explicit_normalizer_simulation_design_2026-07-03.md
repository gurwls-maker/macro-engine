# v8.2 macro range explicit normalizer simulation design

상태:

```text
문서 성격: explicit normalizer simulation design
작업 범위: docs-only
구현 변경: 없음
index.html 변경: 없음
테스트 코드 추가: 없음
normalizeGoalSnapshot 변경: 없음
normalizeMacroRangeContractForSnapshot 구현: 없음
GOAL_SNAPSHOT_SIGNATURE_KEYS 변경: 없음
backup / restore schema 변경: 없음
Recent range-aware logic 변경: 없음
getDailyAdherenceScore 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
UI 노출: 없음
production macro range 구현: 금지
```

이 문서는 `v8.2_macro_range_snapshot_compatibility_design_tests_implementation_2026-07-03.md`의 다음 단계다.

중요:

```text
이번 문서는 production normalizer 구현 지시서가 아니다.
이번 문서는 macroRangeContract 저장 schema 확정 문서가 아니다.
이번 문서는 test-local simulation 설계 문서다.
```

## 1. 왜 이 단계가 필요한가

직전 단계에서 확인한 것:

```text
old fixed snapshot은 정상 normalize된다.
unknown macroRangeContract / rangeScoreCandidate는 현재 normalize에서 prune된다.
invalid unknown range field는 fixed snapshot 전체를 깨지 않는다.
backup / records-only / smart restore / Recent / score version에 range field가 새지 않는다.
```

하지만 아직 증명하지 않은 것:

```text
known macroRangeContract shape
explicit normalizer의 valid / invalid 처리
invalid known contract가 fixed snapshot을 살리는 방식
signature include / exclude 판단 기준
known range field backup roundtrip
records-only old/new basis roundtrip
smart restore range basis conflict handling
Recent fixed/range mixed-basis detection
```

따라서 다음은 production normalizer가 아니라, test-local simulation으로 `known macroRangeContract` 후보를 어떻게 검증할지 먼저 정해야 한다.

## 2. simulation의 역할

후보 이름:

```text
simulateNormalizeMacroRangeContractForSnapshot(rawContract, fixedSnapshot, options)
```

또는 테스트 suite 내부 local helper:

```text
simulateMacroRangeContractNormalization(rawContract, fixedSnapshot, options)
```

역할:

```text
future explicit normalizer가 검증해야 할 최소 조건을 테스트 안에서만 흉내 낸다.
fixed snapshot 생존 조건과 range contract 유효성을 분리한다.
known range field의 allowlist / invalid / warning 기준을 문서화한다.
```

비역할:

```text
normalizeGoalSnapshot에 연결하지 않는다.
window export하지 않는다.
goalSnapshot에 macroRangeContract를 저장하지 않는다.
backup / restore / Recent / score / UI에 연결하지 않는다.
```

## 3. 입력 후보

simulation 입력:

```text
rawContract
fixedSnapshot
options
```

`fixedSnapshot`은 이미 현재 `normalizeGoalSnapshot()`으로 살아남는 old fixed target snapshot이어야 한다.

필수 fixed fields:

```text
targetCal
protein
carbs
fat
goal
exerciseManagementMode
```

`rawContract` 후보:

```text
version
targetCal
protein
fat
carbs
scoring
meta
```

`options` 후보:

```text
mode: "simulation_only"
allowReportOnlyCandidate: true
allowPersistedContract: false
targetCalTolerance: 0.01
unknownFieldPolicy: "prune_with_warning"
```

주의:

```text
targetCalTolerance는 큰 tolerance가 아니다.
fixed snapshot과 contract targetCal이 같은 값인지 확인하기 위한 floating serialization 보정 한도다.
정책 차이를 숨기는 데 쓰지 않는다.
```

## 4. output 후보

simulation output:

```js
{
  status: "absent" | "valid" | "invalid" | "unsupported",
  normalizedContract: object | null,
  reasons: string[],
  warnings: string[],
  fixedSnapshotImpact: "none",
  persisted: false,
  uiVisible: false,
  appliesToSavedRecord: false
}
```

output 원칙:

```text
status가 invalid여도 fixedSnapshotImpact는 none이어야 한다.
invalid contract는 null이 될 수 있지만 fixed snapshot은 null이 되면 안 된다.
warnings는 저장하지 않는다.
normalizedContract는 simulation output일 뿐 goalSnapshot field가 아니다.
```

금지 output:

```text
score
adherenceScore
finalScore
candidateScorePreview
scoreDeltaPreview
backupPayloadFragment
recentBasisSummary
uiCopy
```

## 5. allowlist 후보

아래 shape는 simulation 후보이며 production schema 확정이 아니다.

```js
{
  version: "macroRangePolicy.v0.reportOnly",
  targetCal: number,
  protein: {
    targetG: number,
    minAcceptableG: number,
    upperReviewG: number,
    mode: string,
    basis: string,
    evidenceTier: string,
    appPolicyKey: string
  },
  fat: {
    physiologicalFloorG: number,
    actionFloorG: number,
    recommendedMinG: number,
    recommendedMaxG: number,
    flexibleMaxG: number,
    scoreLowThresholdG: number,
    scoreHighThresholdG: number,
    coachWarnLowThresholdG: number,
    coachWarnHighThresholdG: number,
    statusSource: string
  },
  carbs: {
    trainingSupportMinG: number,
    recommendedMinG: number,
    recommendedMaxG: number,
    flexibleMaxG: number,
    derivedFromFatRange: true
  },
  scoring: {
    candidateVersion: string,
    appliesToSavedRecord: false
  }
}
```

절대 저장하지 않는 field:

```text
candidateScorePreview
scoreDeltaPreview
reportOnly
uiVisible
persisted
debugReason
explanation
rawExternalEvidence
```

이 값들은 report-only helper output이나 debug 설명에 남을 수는 있지만, snapshot contract 후보에는 넣지 않는다.

## 6. valid 조건 후보

simulation에서 valid로 볼 최소 조건:

```text
rawContract는 object
version은 지원 후보 allowlist 안에 있음
targetCal은 finite number
targetCal은 fixedSnapshot.targetCal과 0.01 이내 일치
protein targetG / minAcceptableG / upperReviewG는 finite non-negative
protein minAcceptableG <= targetG <= upperReviewG
fat numeric fields는 finite non-negative
fat physiologicalFloorG <= actionFloorG <= recommendedMinG <= recommendedMaxG <= flexibleMaxG
fat scoreLowThresholdG <= scoreHighThresholdG
fat coachWarnLowThresholdG <= coachWarnHighThresholdG
carbs numeric fields는 finite non-negative
carbs recommendedMinG <= recommendedMaxG <= flexibleMaxG
carbs derivedFromFatRange === true
scoring.appliesToSavedRecord === false
candidateScorePreview 없음
scoreDeltaPreview 없음
```

valid output이라도 아직 저장하지 않는다.

## 7. invalid / unsupported 조건 후보

invalid:

```text
targetCal mismatch
required section missing
required number missing
NaN / Infinity / negative number
fat range order broken
carb range order broken
protein min/target/upper order broken
candidateScorePreview present
scoreDeltaPreview present
scoring.appliesToSavedRecord === true
```

unsupported:

```text
version unknown
policy key unknown
protein basis unknown
contract shape가 old design과 충돌
```

invalid / unsupported 처리:

```text
normalizedContract: null
fixedSnapshotImpact: "none"
reason code를 남김
fixed target snapshot은 계속 사용 가능
```

금지:

```text
invalid contract 때문에 normalizeGoalSnapshot 전체를 null로 만들기
invalid contract를 unknown field처럼 조용히 저장하기
unsupported version을 pass-through하기
```

## 8. unknown field policy

simulation 후보:

```text
unknown top-level field: prune + warning
unknown nested field: prune + warning
unknown scoring field: prune + warning
unknown evidence/debug field: prune + warning
```

주의:

```text
unknown field pruning은 forward compatibility의 한 방식일 뿐이다.
known contract를 저장하기 전에 backup/restore roundtrip 요구가 생기면 unknown 보존 정책을 다시 검토해야 한다.
```

현재 단계의 판단:

```text
simulation에서는 unknown field를 저장하지 않는다.
unknown field가 있더라도 allowlist known fields가 valid하면 status는 valid + warning 가능하다.
단, candidateScorePreview / scoreDeltaPreview는 unknown이 아니라 forbidden field로 본다.
```

## 9. fixed snapshot survival rule

가장 중요한 규칙:

```text
macroRangeContract는 optional이다.
macroRangeContract가 없거나 invalid여도 fixed snapshot은 살아야 한다.
```

simulation test가 반드시 증명할 것:

```text
absent contract -> fixed snapshot valid
valid contract -> fixed snapshot unchanged
invalid contract -> fixed snapshot unchanged
unsupported contract -> fixed snapshot unchanged
forbidden score preview field -> fixed snapshot unchanged
unknown field prune -> fixed snapshot unchanged
```

실패 시 의미:

```text
old Records가 future range field 때문에 깨질 수 있음
```

## 10. signature와의 관계

이번 simulation design에서는 signature 정책을 결정하지 않는다.

다만 다음 단계의 판단 재료를 만든다.

simulation이 남길 수 있는 비교 항목:

```text
fixedSignatureBefore
fixedSignatureAfterContractIgnored
contractStableHashCandidate
signatureIncludeCandidate
signatureExcludeCandidate
```

금지:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS 변경
production signature에 macroRangeContract 추가
Records 무변경 저장 판정을 range contract로 바꾸기
```

판단 후보:

```text
exclude mode:
  range contract 변화가 fixed snapshot signature를 바꾸지 않음

include mode:
  range contract 변화가 별도 explicit basis signature를 가짐
```

이번 문서의 결론:

```text
include/exclude는 아직 결정하지 않는다.
다음 별도 signature decision design에서 다룬다.
```

## 11. backup / restore와의 관계

이번 simulation design은 backup/restore를 바꾸지 않는다.

다만 future 테스트가 분리해야 할 케이스:

```text
old fixed snapshot backup roundtrip
valid simulated range contract backup roundtrip candidate
invalid simulated range contract fallback
unsupported version fallback
candidateScorePreview no-persist
records-only import old/new basis
smart restore keep local basis
smart restore replace backup basis
smart restore meals-only no snapshot replacement
```

주의:

```text
valid simulation output이 있다고 해서 backup roundtrip을 열어도 된다는 뜻이 아니다.
backup/restore old-new fixture design이 별도로 필요하다.
```

## 12. Recent와의 관계

이번 simulation design은 Recent를 바꾸지 않는다.

future mixed basis 판단에 필요한 최소 signal 후보:

```text
snapshotMacroBasis: "fixed" | "range_candidate" | "unknown"
rangeContractStatus: "absent" | "valid" | "invalid" | "unsupported"
scoringBasis: "fixed_v6_1" | "range_candidate_report_only"
```

아직 금지:

```text
getRecentContextRecordSummary 변경
getRecentContextComponentCounts 변경
fatLowDays를 range floor로 대체
Recent UI에 basis warning 표시
```

## 13. 테스트 후보

다음 구현이 열린다면 suite 후보:

```text
runMacroRangeExplicitNormalizerSimulationDesignTests
```

테스트 후보:

```text
1. absent contract returns absent and fixed snapshot survives
2. valid minimal report-only contract returns valid normalized simulation output
3. valid contract prunes unknown non-forbidden fields with warning
4. targetCal mismatch returns invalid and fixed snapshot survives
5. fat order violation returns invalid and fixed snapshot survives
6. carb order violation returns invalid and fixed snapshot survives
7. protein order violation returns invalid and fixed snapshot survives
8. unsupported version returns unsupported and fixed snapshot survives
9. candidateScorePreview returns invalid and is not persisted
10. appliesToSavedRecord true returns invalid in report-only simulation
11. simulation does not mutate raw contract or fixed snapshot
12. simulation output is not connected to normalizeGoalSnapshot / backup / Recent / score / UI
```

필수 병행 테스트:

```text
runMacroRangeSnapshotCompatibilityDesignTests

npm run test:core
```

## 14. false positive 방지

정상으로 봐야 하는 것:

```text
production normalizeGoalSnapshot에 normalizeMacroRangeContractForSnapshot이 없음
unknown macroRangeContract가 현재 prune됨
simulation output이 backup에 들어가지 않음
simulation output이 Recent에 들어가지 않음
candidateScorePreview가 null 또는 absent
ADHERENCE_SCORING_VERSION unchanged
```

blocker로 봐야 하는 것:

```text
simulation이라는 이름으로 production normalizer를 추가함
simulation output을 goalSnapshot에 저장함
GOAL_SNAPSHOT_SIGNATURE_KEYS를 바꿈
backup payload에 normalizedContract가 들어감
Recent component counts가 바뀜
getDailyAdherenceScore output이 바뀜
candidateScorePreview가 숫자로 저장됨
UI text가 바뀜
```

## 16. 최종 판단

explicit normalizer simulation design은 진행 가능하다.

하지만 아직:

```text
macroRangeContract 저장 schema 확정 아님
normalizer production 구현 아님
signature 정책 확정 아님
backup/restore roundtrip 구현 아님
Recent mixed basis 구현 아님
production macro range 구현 아님
```

최종 결론:

```text
known macroRangeContract 후보를 다루려면 explicit normalizer simulation이 먼저 필요하다.
simulation은 fixed snapshot survival과 range contract validity를 분리해야 한다.
valid simulation output도 저장하지 않는다.
```
