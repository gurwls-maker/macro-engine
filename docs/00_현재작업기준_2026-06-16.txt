탄단지 다이어리 현재 작업 기준
작성일: 2026-06-16

이 문서는 Codex가 docs 폴더를 읽을 때 가장 먼저 봐야 하는 현재 기준이다.
다른 문서는 근거, 과거 판단, 세부 설계를 확인하기 위한 참고자료다.
이 문서와 기존 문서가 충돌하면 이 문서, 최신 대화 의도, 실제 앱 상태를 우선한다.
특히 대화에서 실제로 확정된 방향은 `docs/02_대화의도_근거표_2026-06-16.txt`로 확인한다.

0. 이 문서의 역할

- 이 문서는 "정답지"가 아니라 현재 작업 방향을 잃지 않기 위한 나침반이다.
- 문서에 적힌 과거 단계명을 기계적으로 이어가면 안 된다.
- 실제 앱 화면, 실제 계산 결과, 사용자의 최신 의도와 충돌하면 문서를 고치는 쪽이 맞다.
- 기준문서 때문에 앱의 완성도가 떨어지는 선택을 하면 안 된다.
- 개발 편의성이나 안전한 회피보다 앱의 최종 완성도를 우선한다.
- 다만 감사 단계에서 사용자 결정이 필요한 선택지가 생기면, Codex가 혼자 확정하지 말고 코드 모르는 사용자도 이해할 수 있게 질문한다.

0-1. 진행 번호 1~5 규칙

사용자가 말한 1~5 넘버링은 독립된 작업 묶음이나 별도 루프가 아니다.
기존 개발 흐름은 그대로 유지하고, 진행 보고 번호만 1부터 5까지 순환해서 붙이는 운용 규칙이다.

- 원래 개발이 1~10단계라면 그 1~10단계를 유지한다.
- 다만 진행 번호가 5에 해당하는 시점에는 기존 문서를 갱신한다.
- 문서 갱신 때문에 원래 하던 개발 순서를 새 1~5 묶음으로 갈아엎지 않는다.
- 5 다음에는 번호만 다시 1로 돌아가고, 실제 작업은 원래 다음 개발 단계로 이어진다.
- 이 규칙을 핑계로 의미 없는 단계를 늘리면 안 된다.
- 단계 수정이 필요하면 무엇을 삭제/추가/수정했는지와 이유를 사용자에게 설명한다.

1. 최상위 대화 의도

현재 앱의 방향은 단순한 탄단지 계산기가 아니다.
사용자의 몸 상태, 운동 여부, 운동 프로필, 활동, 업무, 식사 기록, 체성분 변화를 함께 보고 오늘의 식단 기준을 잡는 영양 코치 앱이다.

따라서 다음 원칙을 따른다.

- 현재 앱과 최신 대화 의도가 모든 문서보다 우선한다.
- "일단 문서대로"가 아니라 "이 선택이 앱 의도에 맞는가"를 계속 판단한다.
- 대표 케이스 몇 개만 맞는 산식보다, 다양한 사용자 유형에서 말이 되는 산식을 우선한다.
- 산식은 생리학적으로 타당해야 하며, 가능하면 신뢰도 높은 외부근거를 따른다.
- 사용자에게 입력을 과하게 요구하지 않되, 의미 있는 차이가 실제 계산에 반영되어야 한다.
- UI는 기능을 숨기기 위해 단순화하는 것이 아니라, 사용자가 자연스럽게 이해하도록 단순화한다.

2. 현재 가장 중요한 방향: 운동 여부 상위 모드

가이드 기준/활동량 기준을 최상위 선택지로 보는 흐름은 현재 의도와 맞지 않는다.
더 상위 개념은 "이 사용자가 운동을 하는 사람인가, 운동을 하지 않는 사람인가"다.

현재 결정:

- 온보딩이 구현되기 전에는 Settings 탭에서 운동 여부를 바꿀 수 있게 한다.
- 추후 온보딩에서는 운동 여부를 간단히 묻는다.
- 운동하는 사람은 운동 프로필, 운동 수준, 루틴, 세션, 웨이트, 유산소, 체성분 중심 계산을 사용할 수 있다.
- 운동하지 않는 사람은 운동 관련 입력과 해석을 화면에서 숨긴다.
- 운동하지 않는 사람은 일반 식단 관리와 체중 관리 중심으로 계산한다.
- 두 모드는 같은 백업/내보내기 형식을 쓴다.
- 한쪽 모드에서 읽지 않는 값은 계산과 렌더에서 0 또는 fallback으로 처리한다.
- 단, 모드를 바꿨다는 이유로 과거의 유의미한 운동 데이터가 0으로 덮어써져 사라지면 안 된다.
- 반대로 운동하지 않던 기간에 의미 있는 운동 데이터가 갑자기 생긴 것처럼 보여도 안 된다.

3. 가이드 기준 / 활동량 기준의 위치

중요 정정:
기존 문서에는 두 기준을 사용자 선택 토글로 병렬 유지한다는 식의 표현이 강하게 남아 있었지만,
그 문장은 최신 대화 의도를 정확히 반영하지 못한다.
최신 대화에서 확정된 방향은 "사용자가 가이드/활동량 토글을 고르는 구조"가 아니라
"운동 여부를 먼저 고르고, 그 여부에 따라 앱이 맞는 기준을 적용하는 구조"다.

현재 결정:

- 최상위 모드: 운동 여부
- 사용자가 먼저 고르는 것은 "가이드 기준 / 활동량 기준"이 아니라 "운동을 관리 기준에 포함할지"다.
- 운동하는 사람 모드에서는 운동생리학 기준으로 계산하고, 체성분, 운동 프로필, 루틴/세션, 웨이트, 유산소를 사용할 수 있다.
- 운동하지 않는 사람 모드에서는 일반적인 기준으로 계산하고, 운동 관련 입력과 해석을 숨긴 일반 식단/체중 관리 중심 화면을 사용한다.
- 가이드 기준/활동량 기준은 사용자가 직접 고르는 Today mini toggle로 유지하지 않는다.
- 기존 가이드 기준 개념은 운동하는 사람 모드의 운동생리학 기준으로 흡수한다.
- 기존 활동량 기준 개념은 운동하지 않는 사람 모드의 일반 기준으로 흡수한다.
- Today 카드 안의 가이드/활동량 mini toggle은 삭제하거나, 구현상 필요하면 사용자에게 보이지 않는 내부 기준으로만 둔다.
- 운동하지 않는 사람 모드에서는 운동 관련 기준과 문구를 숨기거나 일반 식단 관리에 맞게 축소한다.
- 이 결정을 Codex가 다시 "토글 유지"로 되돌리면 안 된다.

대화 근거:

- 사용자는 처음에 가이드 기준은 운동생리학 기준, 활동량 기준은 일반 기준에 가까웠다고 설명했다.
- 이후 사용자는 이 둘을 계속 유지할지, 앱 성격을 운동 중심으로 확실히 잡을지 애매하다고 문제를 제기했다.
- 이후 사용자는 더 상위 개념으로 운동 여부를 먼저 확인하고, 운동을 하면 운동 기준으로 계산하고 운동 UI를 보여주며, 운동을 안 하면 운동 관련 내용을 숨기고 일반 식단 기준으로 보자는 방향을 제안했다.
- 이후 사용자는 온보딩에서는 운동 여부만 간단히 묻고, 온보딩 전에는 Settings에서 바꾸면 된다고 정리했다.
- 따라서 기존 "가이드/활동량" 사용자 토글은 운동 여부 상위 모드에 흡수하는 것으로 본다.

4. TDEE와 활동 반영 철학

사용자는 일반적인 BMR * 활동계수 방식보다 더 정확한 계산을 원한다.
다만 매일 수면, 업무, 생활활동, 운동을 너무 세세하게 입력시키면 앱을 쓰기 어려워진다.

현재 철학:

- BMR은 몸이 기본적으로 쓰는 에너지다.
- 운동, 업무, 생활활동은 실제 소비가 늘어난다면 목표 칼로리에 의미 있게 반영되어야 한다.
- 업무나 활동을 무조건 작은 보정으로 격하하지 않는다.
- 다만 같은 시간을 두 번 계산하면 안 되므로 중복 계산을 막는 구조가 필요하다.
- 생활활동은 "업무, 운동, 수면을 뺀 나머지 시간의 평균 활동량"으로 보는 방향이 자연스럽다.
- 별도 추가활동 카드는 기본 UI에 바로 넣지 않는다.
- 이삿짐, 대청소, 긴 야외활동 같은 예외적인 날을 더 정확히 처리하는 고급/예외 UI는 후속 후보로 둔다.
- 생활활동의 높음/매우 높음은 남은 시간 전체 평균이라는 점을 도움말에서 분명히 알려야 한다.

중요:
시간 소유권 TDEE 후보 검증은 참고 주제다.
최신 우선 작업은 운동 여부 상위 모드 정리이며, 오래된 CH 단계 흐름을 기계적으로 이어가면 안 된다.

5. Coach / 최근 흐름 문구 원칙

Coach와 최근 흐름은 앱 정책을 설명하는 공간이 아니다.
사용자가 지금 무엇을 보면 되고 다음에 무엇을 하면 되는지 말해야 한다.

다음 같은 문구는 본문에 직접 노출하지 않는 방향으로 정리한다.

- "운동 기록은 현재 관리 기준에 포함하지 않습니다."
- "단백질은 일반 권장 범위 안에서 관리합니다."
- "체성분 변화와 식단 기록을 함께 봅니다."
- "목표칼로리는 하나로 유지됩니다."

정책 설명이 필요하면 Settings 도움말, 온보딩, 기준문서에 둔다.
사용자 화면에서는 행동 중심 문구로 바꾼다.

6. 문서 정리 원칙

이번 문서 정리는 기존 문서를 모두 읽지 않고 삭제하는 작업이 아니다.
또 기존 문서에 끌려가서 최신 의도를 잃는 작업도 아니다.

문서 정리 순서:

1. 대화 의도를 먼저 확정한다.
2. 기존 문서 목록을 만든다.
3. 각 문서를 유지, 병합, 삭제 후보로 분류한다.
4. 최신 기준과 충돌하는 문서는 충돌 이유를 적는다.
5. 필요한 내용만 현재 기준 문서나 새 통합 문서로 흡수한다.
6. 흡수 완료된 문서는 삭제한다.
7. archive는 사용자가 명시적으로 원하거나 Git 이력만으로 복구성이 부족한 경우에만 예외적으로 쓴다.
8. 마지막에 유지/병합/삭제/보류 결과표를 남긴다.

삭제 기준:

- 최신 기준과 충돌하고, 이미 흡수된 문서
- 과거 단계 실행 지시처럼 보여 Codex를 잘못된 방향으로 끌고 가는 문서
- 현 앱 상태와 맞지 않고 참고 가치도 낮은 문서

유지 기준:

- 외부근거, 산식 근거, UI 의도, 앱 철학처럼 아직 근거 가치가 있는 문서
- 최신 기준문서가 세부 내용을 참조해야 하는 문서
- 아직 흡수 여부를 판단하지 못한 문서

7. 다음 작업 순서

문서 목록화와 1차 삭제는 `docs/01_문서정리_분류표_2026-06-16.txt`에 기록했다.
단, 분류표도 최종 기준문서가 아니다.
항상 이 문서의 최신 대화 의도와 실제 앱 상태를 기준으로 기존 문서를 심사한다.

현재 정리 결과:

- 과거 stage note와 report-only 설계 원문은 필요한 교훈만 남기고 삭제했다.
- 삭제된 원문이 필요하면 Git 이력에서 복구한다.
- `코덱스-인텔리전스.txt`처럼 앱 기준문서가 아닌 파일은 사용자 결정 전까지 보류한다.
- `archive/20260525_스티치_UI_기능_이식_누락_보고서.txt`는 이번 V8 산식/기준문서 정리의 직접 대상이 아니므로 보류한다.

다음 작업은 문서 정리 자체를 계속 늘리는 것이 아니라, 현재 앱과 최신 의도 기준으로 실제 개발 흐름에 복귀하는 것이다.
새로 충돌하는 문서를 발견하면 먼저 사용자 의도와 실제 앱 상태를 확인한 뒤 정리한다.
