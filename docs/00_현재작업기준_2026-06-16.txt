# 00. 현재 작업 기준

작성일: 2026-06-16
용도: 현재 앱 방향과 다음 작업 판단 기준을 한 곳에 모아 둔다.

## 1. 최우선 원칙

현재 앱과 최신 대화 의도가 기준문서보다 우선한다. 기준문서가 현재 의도와 충돌하면 문서를 고친다.

검증은 좁은 대표 케이스보다 전체 사용 흐름을 먼저 본다. 일부 케이스만 봐도 되는 경우에는 그 이유가 있어야 한다.

산출식은 생리학적으로 설명 가능해야 하며, 가능하면 신뢰도 높은 외부 근거를 사용한다.

사용자 경험은 코드만 보지 않고 실제 화면에서 확인한다.

감사 중 결정이 필요한 항목은 임의로 구현하지 않는다. 코드 모르는 사용자가 판단할 수 있게 쉽게 풀어서 질문한다.

## 2. 현재 핵심 결정

### 운동 여부가 상위 모드다

앞으로 앱은 먼저 사용자가 운동을 하는지 아닌지를 기준으로 나뉜다.

- 운동함: 기존 `가이드 기준`이 맡던 스포츠/운동생리학 기준을 흡수한다.
- 운동 안 함: 기존 `활동량 기준`이 맡던 일반 식단/일반 체중 관리 기준을 흡수한다.

온보딩이 생기기 전까지는 Settings에서 운동 여부를 바꾼다.

### Today의 가이드/활동량 mini toggle은 사용자 기준 선택지가 아니다

Today 화면에는 `가이드 기준 / 활동량 기준` mini toggle을 독립 선택지처럼 보여주지 않는다.

계산 기준은 Settings의 운동 여부에서 결정한다.

- 운동함이면 스포츠/운동생리학 기준으로 계산한다.
- 운동 안 함이면 일반 식단 기준으로 계산한다.

백업/기록 호환을 위해 내부 값이나 과거 기록에는 기존 이름이 남을 수 있다. 단, 사용자에게 “오늘 어떤 기준을 고를지”처럼 보이면 안 된다.

## 3. 데이터 보존 기준

운동함과 운동 안 함은 같은 기록/백업 형식을 사용한다.

현재 모드에서 쓰지 않는 값은 계산과 화면에서 숨기거나 fallback 처리한다. 그러나 원본 기록에 이미 저장된 의미 있는 값은 지우지 않는다.

예시:

- 운동 안 함 모드에서는 운동 UI와 운동 계산을 숨긴다.
- 과거 운동 기록을 삭제하거나 0으로 덮어쓰지 않는다.
- 나중에 운동함으로 바꿨을 때 없는 운동 데이터가 갑자기 생겨서는 안 된다.

## 4. TDEE와 목표 칼로리 방향

업무, 운동, 생활활동이 실제 소비량을 늘렸다면 목표 칼로리도 움직일 수 있어야 한다.

막아야 하는 것은 활동 반영 자체가 아니라 같은 시간을 두 번 계산하는 것이다.

2026-06-21 9단계 검증 기준으로, 시간 소유권 TDEE는 실제 `calculate()` 경로에 적용되어 있다. 이제 이것을 "앞으로 구현할 후보"처럼 말하지 않는다.

- 수면 시간
- 업무 시간
- 운동 시간
- 운동과 업무를 뺀 생활활동 시간
- 남은 휴식 시간

하루 시간 소유권을 나누고, 각 시간의 소비량을 합쳐 TDEE를 만든다.

`v8.0-CC`에서 업무유형을 목표 칼로리에 덜 반영하거나 제외한 것은 임시 보호막이었다. 최종 방향은 아니다.

### 2026-06-21 적용 확인 상태

`v8.0-CP_time_ownership_runtime_formula_application_v0`에서 시간 소유권 산식이 production 목표 칼로리 계산에 연결됐고, `v8.0-CQ_time_ownership_runtime_formula_validation_after_production_application_v0`에서 적용 후 검증이 통과했다.

실제 동작:

- 기본 구조는 `BMR + 훈련기본값 + 업무 순소모 + 남은 생활활동 순소모 + 운동 순소모`다.
- 업무/생활활동/운동 소모가 커지면 목표 칼로리도 움직인다.
- 웨이트, 유산소, EPOC는 한 번의 운동 순소모로 접어서 중복 반영을 막는다.
- 수면+업무+운동 시간이 24시간을 넘으면 시간 소유권 산식을 적용하지 않고 기존 계산으로 fallback한다.
- 아직 UI/input alignment와 문서 정리가 남아 있으므로, full V8 완료로 말하지 않는다.

9단계 검증값:

- 앉아서 일함 + 낮은 생활활동 + 휴식: 목표 1,907kcal, 소비 기준 2,253kcal
- 앉아서 일함 + 보통 생활활동 + 휴식: 목표 2,141kcal, 소비 기준 2,553kcal
- 고강도 현장 + 매우 활동적 + 휴식: 목표 4,088kcal, 소비 기준 4,892kcal
- 수면 6h / 업무 8h / 보통 생활활동: 목표 2,261kcal, 남은 생활활동 10h
- 수면 10h / 업무 8h / 보통 생활활동: 목표 2,021kcal, 남은 생활활동 6h
- 고강도 업무 4h: 목표 2,651kcal, 업무 순소모 900kcal
- 고강도 업무 12h: 목표 3,897kcal, 업무 순소모 2,699kcal
- 웨이트 60분 + 유산소 30분: 목표 2,685kcal, 운동 순소모 353kcal

다음 산식/화면 작업은 아래 조건을 계속 만족해야 한다.

- 운동 여부 상위 모드를 유지한다.
- Today에 예전 `가이드 기준 / 활동량 기준` mini toggle을 되살리지 않는다.
- 업무, 운동, 생활활동, 수면 시간을 중복 계산하지 않는다.
- 코드 편의를 위해 업무나 생활활동을 작은 보정으로 격하하지 않는다.
- 사용자가 입력값을 바꿨을 때 왜 목표 칼로리가 움직이는지 화면에서 자연스럽게 이해되어야 한다.

## 5. 생활활동 입력 방향

생활활동은 `업무·운동·수면을 뺀 나머지 시간의 평균 활동량`이다.

별도 “추가활동” 카드는 기본값으로 두지 않는다. 생활활동 안에서 평균 강도를 조절한다.

도움말은 항상 보이는 설명문이 아니라 `?` 툴팁으로 처리한다.

도움말에는 다음 의미를 넣는다.

- 생활활동은 남은 시간 전체의 평균이다.
- 일부 시간만 아주 힘들었다고 전체를 `매우 높음`으로 두면 과대 계산될 수 있다.
- 정말 여가 시간 대부분이 높은 강도였을 때만 높은 단계를 고른다.
- 별도 운동으로 기록할 수 있는 활동은 운동/유산소 입력을 우선 사용한다.

## 6. 매크로 정책 방향

운동함과 운동 안 함은 탄단지 배분 정책이 다르다.

운동함:

- 운동 프로필, 운동 수준, 체성분 상태, 목표를 반영한다.
- 단백질과 지방은 외부 근거 기반 범위를 우선한다.
- 남는 칼로리는 운동인에게 자연스러운 방식으로 탄수화물에 배분하는 방향을 기본 후보로 둔다.

운동 안 함:

- 일반 체중 관리 기준을 사용한다.
- 운동 기록이나 운동 수행능력 입력을 요구하지 않는다.
- 단백질, 지방, 탄수화물은 일반 권장 범위를 우선한다.
- 운동 소모를 0으로 처리하는 것만으로는 부족하다. 단백질/지방/탄수 정책표도 운동함과 분리해야 한다.
- 2026-06-21 감사에서 현재 앱은 운동 안 함에서도 `diet 1.8g/kg`, `cut 2.2g/kg`, `maintain 1.8g/kg`처럼 운동인 단백질 정책을 공유하는 문제가 확인됐다.
- 운동 안 함은 훈련 탄수 g/kg 하한을 적용하지 않는다. 탄수는 남은 칼로리로 배분하고, 일반 성인 AMDR 범위를 sanity check로 본다.

세부 정책은 `v8_외부근거_매크로_정책표_2026-06-05.txt`와 후속 검증 결과를 따른다.

## 7. 문구 기준

사용자에게 내부 정책을 설명하는 문구를 코치나 흐름 분석에 직접 노출하지 않는다.

피해야 할 문구 예:

- “운동 기록은 현재 관리 기준에 포함하지 않습니다.”
- “단백질은 일반 권장 범위 안에서 관리합니다.”
- “체성분 변화와 식단 기록을 함께 봅니다.”

대신 사용자가 바로 이해할 수 있는 상태와 행동을 말한다.

예시:

- “오늘은 식사 기록이 부족해 판단하지 않습니다.”
- “단백질이 자주 부족했습니다. 다음 식사에 단백질 식품을 먼저 넣어보세요.”
- “목표를 바꾼 기록이 있어 이전 식단 흐름과 직접 비교하기 어렵습니다.”

## 8. 문서 관리 기준

문서는 현재 작업을 돕기 위해 유지한다.

도움이 안 되는 선언형 문장은 실제 체크 규칙으로 바꾼다.

예시:

- 선언문: “예전 방식으로 돌아가지 않는다.”
- 체크 규칙: “Today 화면에 가이드/활동량 mini toggle이 렌더되면 실패다.”

오래된 문서를 아카이브로 남기지 않는다. 현재 판단에 쓰이지 않으면 삭제하고 Git 이력에 맡긴다.

## 9. 현재 작업 순서

1. 운동 여부 상위 모드와 가이드/활동량 기준 흡수 구조를 정리했다.
2. Today/Settings/Records/Recent Flow에서 사용자에게 낡은 기준 선택지가 보이지 않게 했다.
3. 운동 여부 모드가 데이터 보존과 백업 호환을 깨지 않는지 검증했다.
4. 2026-06-21 매크로 감사에서 운동 여부 연결은 동작하지만, 운동 안 함이 아직 운동인 단백질 정책을 공유하는 문제를 확인했다.
5. 위 결과를 기준문서에 반영했다.
6. 코드에서 매크로 정책표를 운동함/운동 안 함으로 분리했다.
7. 전체 목표별 시나리오로 운동함/운동 안 함의 단백질, 지방, 탄수, 목표 칼로리 차이를 검증했다.
8. Today/Records/Coach/Recent에서 사용자에게 낡은 가이드/활동량 토글이나 내부 정책 설명이 새지 않는지 화면 기준으로 확인했다.
9. 업무, 생활활동, 운동, 수면이 목표 칼로리에 중복 없이 반영되는 TDEE 시간 소유권 산식이 실제 production 계산에 적용되어 있음을 검증했다.
10. 9단계 결과를 기준문서에 반영했다.
11. 다음 작업은 수면/업무/생활활동 입력 UI와 실제 화면 자연스러움을 확인하는 것이다.
