```text
작업 범위: docs-only

record schema 변경: 없음
goalSnapshot schema 변경: 없음
backup / Recent / UI / DailyCoach 변경: 없음
scoreDeltaPreview 작업 진행: 없음
```

이 문서는 `v8.2_macro_range_scoring_direction_realignment_decision_2026-07-06.md` 이후 production score transition decision이다.

후속 test design / implementation 기록:

```text
v8.2_macro_range_production_score_transition_test_design_2026-07-06.md
```

목적:

```text
v8.2 macro range의 본류를 fixed target score와의 비교가 아니라
range-aware score를 production adherence score로 전환하는 경로로 닫는다.
```

최종 판단:

```text
production score transition direction: 승인
future ADHERENCE_SCORING_VERSION target: v8.2_macro_range_score_v1

candidateScorePreview formula: production score formula의 source 후보로 승격

scoreDeltaPreview: optional audit-only, transition blocker 아님
old fixed Records: no silent mutation, but permanent preservation requirement 아님
storage/schema: initial transition blocker 아님, 별도 evidence/storage gate로 분리
```

## 1. 이번 decision이 정하는 것

이번 문서가 정하는 것:

```text
production score transition direction
future scoring version name
getDailyAdherenceScore와 range-aware formula의 관계
candidateScorePreview formula의 production 승격 방향
pointsPreview를 저장/노출 field로 둘지 여부
new Records scoring policy
old local Records 처리 방침
minimum QA matrix
scoreDeltaPreview의 non-blocking status
```

이번 문서가 정하지 않는 것:

```text
실제 getDailyAdherenceScore 구현 변경
실제 ADHERENCE_SCORING_VERSION 상수 변경
component formula 코드 이동
record / goalSnapshot / backup schema 변경
Today / Records / Recent UI copy 변경
DailyCoach action threshold 변경
Recent aggregation 변경
test suite 구현/등록
```

따라서 이번 문서 이후에도 로컬 production code 상태는 그대로다.

## 2. production 전환 방향

결정:

```text
v8.2 macro range score는 별도 comparison score가 아니라 production adherence score로 전환한다.
```

의미:

```text
future implementation에서 getDailyAdherenceScore는 fixed target exact macro component score 대신
range-aware component contribution 기반 점수를 primary result로 반환해야 한다.
```

비목표:

```text
fixed score와 range score를 사용자-facing으로 나란히 표시
scoreDeltaPreview를 점수 전환의 필수 검증 field로 사용
candidateScorePreview를 별도 저장 field로 추가
pointsPreview를 Records / backup / UI에 저장 또는 노출
```

제품 판단:

```text
이 앱은 현재 실사용 데이터 보존 부담이 낮다.
따라서 old fixed score와의 장기 병행보다, 새 정책이 맞다면 새 scoring version으로 전환하는 쪽이 우선이다.
```

## 3. future scoring version decision

future implementation target:

```text
ADHERENCE_SCORING_VERSION = "v8.2_macro_range_score_v1"
```

이름 의미:

```text
v8.2: macro range policy line
macro_range_score: range-aware adherence score
v1: 첫 production score version
```

금지:

```text
기존 "v6.1_alcohol_penalty_v2" 이름을 유지한 채 score 의미만 바꾸기
candidate preview version을 ADHERENCE_SCORING_VERSION으로 재사용하기
testLocalCandidateScorePreviewVersion을 production scoring version으로 승격하기
scoreDeltaPreviewVersion을 production adherence scoring version처럼 사용하기
```

이번 문서에서는 상수를 바꾸지 않는다.

## 4. getDailyAdherenceScore relation

future relation:

```text
getDailyAdherenceScore()
  -> range-aware production score primary path
  -> detail.scoringVersion = "v8.2_macro_range_score_v1"
```

전환 원칙:

```text
새 함수를 user-facing score 옆에 따로 붙이는 것이 아니라,
production adherence score 계산 경로 자체를 v8.2 range-aware score로 교체한다.
```

implementation 전제:

```text
component contribution table 확정
candidate total formula 확정
dataQuality blocked/null policy 확정
invalid numeric input rejection 확정
one-meal / insufficient / snapshotless handling 확정
Today / Records / DailyCoach reason alignment QA 통과
```

출력 방향:

```text
percent: range-aware score number 또는 null
label: range-aware score label
source: "auto"
isAvailable: true/false
status: full_day_candidate 또는 blocked/unavailable 상태
detail.scoringVersion: v8.2_macro_range_score_v1
detail.componentContributions: optional internal detail 후보
detail.rangeScoreBasis: optional internal detail 후보
```

이번 decision은 detail field shape를 확정하지 않는다.

## 5. candidateScorePreview formula 승격 방향

기존 candidate formula 후보:

```text
candidateScorePreview = clamp(
  100 + sum(kcal/protein/fat/carbExchange pointsPreview),
  0,
  100
)
```

이번 decision:

```text
이 formula는 future production score formula의 source 후보로 유지한다.
```

단, production implementation에서는 이름을 다시 판단한다.

권장:

```text
production result field: percent
internal formula identity: macro_range_score_v1
candidateScorePreview field: production storage/UI field로 사용하지 않음
```

이유:

```text
production score로 전환하면 더 이상 candidate preview가 아니라 실제 adherence score다.
사용자-facing / Records 저장 경로에서는 기존 adherencePercent + adherenceScoringVersion 계약을 활용하는 편이 단순하다.
```

## 6. pointsPreview 처리

결정:

```text
pointsPreview는 production record / goalSnapshot / backup / UI field로 열지 않는다.
```

future implementation 방향:

```text
component contribution은 getDailyAdherenceScore 내부 계산 또는 detail 내부 evidence 후보로만 둔다.
```

허용 후보:

```text
internal component contribution map
test-only expected value table
debug-only score evidence
future detail.componentContributions
```

계속 금지:

```text
record.pointsPreview
goalSnapshot.pointsPreview
backup.data.records[].pointsPreview
Recent points aggregation
Today / Records / Recent UI points display
DailyCoach action threshold directly from pointsPreview
```

이유:

```text
component contribution은 score 산식 구현에는 필요하지만,
사용자-facing 저장 field로 열면 점수 설명 / UI / backup schema 부담이 커진다.
```

## 7. dataQuality / blocked policy

production score는 full-day candidate에만 numeric score를 낼 수 있다.

numeric allowed:

```text
dataQuality.status === full_day_candidate
targetCal available
protein/fat/carb range basis resolved
detailed macro data sufficient
invalid/impossible state 없음
```

numeric blocked:

```text
one_meal_pending
insufficient_record
snapshotless_debug_only
blocked_by_impossible_state
unsupported range basis
invalid range basis
range policy missing
```

blocked output 원칙:

```text
percent: null
isAvailable: false
status: blocked reason
no sentinel 0
```

중요:

```text
0점은 가능한 최저 numeric score다.
blocked / unavailable / insufficient 상태를 0점으로 표현하지 않는다.
```

## 8. new Records scoring policy

future production implementation 이후 새 auto Records:

```text
adherencePercent: range-aware score number 또는 null
adherenceScoringVersion: "v8.2_macro_range_score_v1" 또는 null
adherenceSource: "auto" 또는 기존 source policy 유지
```

full-day numeric auto record:

```text
adherencePercent = range-aware score
adherenceScoringVersion = "v8.2_macro_range_score_v1"
```

blocked / unavailable auto record:

```text
adherencePercent = null
adherenceScoringVersion = null
```

이유:

```text
numeric score가 없으면 적용된 production scoring version도 저장하지 않는다.
blocked / unavailable 상태를 versioned 0점처럼 보이게 하지 않는다.
```

manual / simple / weight-only record:

```text
기존 manual/simple/weight-only policy 유지
range score로 억지 승격하지 않음
```

저장 schema:

```text
초기 transition은 기존 adherencePercent / adherenceScoringVersion 필드를 활용한다.
candidateScorePreview / pointsPreview / scoreDeltaPreview 저장 field를 추가하지 않는다.
```

## 9. old local Records 처리

이번 decision의 제품 판단:

```text
실사용 데이터 보존 부담이 낮으면 old local Records 보존은 production 전환 blocker가 아니다.
```

그래도 금지할 것:

```text
사용자 모르게 old fixed Records의 saved adherencePercent 의미를 바꾸기
old fixed Records의 adherenceScoringVersion만 새 버전으로 바꾸기
old fixed Records에 candidateScorePreview / pointsPreview / scoreDeltaPreview 주입
old fixed Records를 Recent range trend에 조용히 섞기
```

허용 가능한 transition 선택지:

```text
1. existing old fixed Records는 그대로 두고 새 Records부터 v8.2 score 적용
2. 개발/로컬 데이터는 release 전 clear 또는 재생성
3. 필요하면 명시적 dev migration / recompute 도구를 별도 docs/test gate에서 설계
4. migration impact가 필요할 때만 scoreDeltaPreview류 audit 도구를 별도로 사용
```

이번 decision의 기본값:

```text
new Records forward-only transition
old local Records are non-blocking legacy/dev data
no silent mutation
```

## 10. storage/schema 판단

초기 production score transition의 blocker가 아닌 것:

```text
macroRangeContract 저장
macroRangeBasisSignature 저장
pointsPreview 저장
candidateScorePreview 저장
scoreDeltaPreview 저장
```

이유:

```text
새 production score는 adherencePercent + adherenceScoringVersion으로 저장 가능하다.
score evidence replay / detailed range basis persistence는 별도 storage/evidence gate에서 닫을 수 있다.
```

분리할 후속 gate:

```text
range score evidence detail shape
goalSnapshot range basis storage
backup roundtrip
Records detail explanation replay
Recent range-aware aggregation
DailyCoach action context
```

주의:

```text
storage/schema를 blocker에서 빼는 것은 backup/restore no-leak를 무시한다는 뜻이 아니다.
초기 transition은 새 score 저장에 기존 score metadata를 쓰고,
range basis 저장은 별도 승인 전까지 열지 않는다는 뜻이다.
```

## 12. minimum QA matrix

future implementation 전 최소 QA matrix:

```text
eligible neutral full day -> 100
kcal high + fat observation -> 85
anchor severe protein low + range penalties -> expected exact score
fat low + carb exchange explained + kcal/protein matched -> no carb/fat double punishment
protein low + kcal matched -> anchor penalty 유지
one meal pending -> percent null, no sentinel 0
insufficient record -> percent null
snapshotless old record -> no silent recompute
positive component contribution fixture -> reject
fractional / NaN / Infinity / string numeric fixture -> reject
new auto Record -> adherenceScoringVersion v8.2_macro_range_score_v1
old fixed Record -> no silent mutation
backup payload -> no candidate/points/delta fields
Today / Records / DailyCoach reason alignment -> no contradictory old fixed copy
```

이 QA는 다음 test design에서 suite로 분리한다.
