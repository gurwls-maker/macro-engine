## 0. 이번 단계의 판단

4단계의 핵심은 사용자를 초보/일반/상급 같은 단순 라벨로 나누는 것이 아니다.
앱이 이미 받는 값들, 즉 운동 프로필/루틴/세션, 주간 빈도, 오늘 운동 부하, 유산소 부하, 최근 기록 품질, 체성분 신뢰도, 목표/기준 snapshot을 묶어서 "산식이 어디까지 말할 자격이 있는가"를 정하는 것이다.

따라서 이번 단계는 production targetCal/protein/fat/carbs를 바꾸지 않는다.
이번 단계는 user-level formula 구현도 아니고, candidate 승인도 아니고, performanceLevel selector 추가도 아니다.

이번 단계가 막아야 하는 우회:

- 주간 운동일수만 보고 상급/고부하로 판단
- InBody가 있다는 이유만으로 단백질/칼로리를 올림
- recent28 추세만 보고 targetCal을 자동 변경
- targetReliefRisk가 high라는 이유만으로 targetCal을 자동 완화
- profileSession만 보고 실제 루틴/세션 선택 경로를 건너뜀
- 러닝의 weightDuration=0을 휴식으로 오해
- 기록 품질이 낮은데도 고부하 세션만 보고 production 변경

중요한 해석:

- `profileSession`은 derived policy key다. 사용자 입력 shortcut이 아니다.
- `performanceLevel`은 derived context다. 사용자에게 직접 고르게 하는 selector가 아니다.
- `bodyCompReliability`는 LBM/protein basis 신뢰도를 돕는 축이지 targetCal을 단독 변경하는 축이 아니다.
- `recent28Trend`는 목표/기록/snapshot 품질을 통과해야 의미가 있다. 단독 자동 적용 축이 아니다.
- `selectedMacroBasis`는 기록 기준의 연속성을 확인하는 축이다. 서로 다른 기준을 같은 목표처럼 비교하면 안 된다.

## 3. context axis

BD는 7개 context axis를 고정한다.

1. trainingLoadContext
   - resistanceLoad, cardioLoad, recoveryNeed, fuelNeed, weeklyTrainingDays, weightDuration, intensityOverride를 묶는다.
   - 주간 빈도만으로 수준을 결정하지 않는다.
   - running weightDuration=0을 휴식으로 읽지 않는다.

2. profileSpecificityContext
   - exerciseProfile, routinePlan, routineSession, profileSession의 실제 경로를 요구한다.
   - 1프로필 1기본세션식 우회를 막는다.
   - bodybuilding split, powerbuilding main/accessory, strength heavy/deload, running cardio-owned, mixed hybrid를 유지한다.

3. recentRecordQualityContext
   - recent14/recent28, recordConfidence, windowFit, contextFit, goalSnapshotCoverage를 묶는다.
   - 기록 품질은 산식 권한을 낮출 수 있다.
   - recent28은 자동 target delta가 아니다.

4. bodyCompositionAuthorityContext
   - bodyCompReliability와 snapshot/source continuity를 묶는다.
   - InBody 하나만으로 targetCal이나 protein을 공격적으로 올리지 않는다.

5. targetReliefPressureContext
   - targetReliefRisk, goal, fuelNeed, recent28Trend, cardioLoad를 묶는다.
   - high pressure는 review signal이다. 자동 적용값이 아니다.

6. formulaAuthorityContext
   - context_only, candidate_report_only, candidate_review_ready, scoped_production_existing, production_formula_not_yet을 구분한다.
   - 깨끗한 report-only evidence를 production implementation으로 읽지 않게 막는다.

7. explanationSurfaceContext
   - Today Coach, Score, Records detail, snapshot, backup/import에 설명이 남을 수 있어야 한다.
   - recordConfidence/windowFit/contextFit 같은 내부 용어는 그대로 사용자에게 노출하지 않는다.

## 4. gate

BD의 필수 gate는 7개다.

```text
performance_selector_blocked
weekly_frequency_not_sufficient
body_comp_reliability_not_sole_authority
recent28_trend_not_auto_apply
target_relief_not_auto_apply
record_quality_can_downgrade_authority
profile_training_model_required
```

이 gate들은 전부 통과해야 다음 5단계 "산출식 원칙 설계"로 넘어갈 수 있다.

주의:

이 gate 통과는 production formula 구현 통과가 아니다.
이 gate 통과는 "다음 단계에서 어떤 원칙으로 target/protein/fat/carb를 계산할지 설계해도 되는가"에 대한 context 준비 통과다.

## 5. probe scenario

BD는 7개 probe를 둔다.

1. bodybuilding_simple_training_preserves_legacy_off
   - bodybuilding advanced OFF는 여전히 simple PUSH baseline이다.
   - 이 경로가 bodybuilding split 전체를 대체하면 안 된다.

2. running_tempo_cardio_owned_zero_weight_duration
   - 러닝 tempo/interval은 weightDuration=0이어도 고부하 cardio context일 수 있다.
   - `weightDuration=0 => rest` 해석을 막는다.

3. mixed_strength_cardio_hybrid_context
   - mixed strength+cardio는 resistance와 cardio를 둘 다 보존해야 한다.

4. strength_deload_downgrades_load_context
   - strength 프로필이라도 deload는 recovery context다.
   - strength profile 자체를 high load로 읽지 않는다.

5. high_load_low_record_quality_review_only
   - 고부하 훈련과 낮은 기록 품질이 같이 있으면 자동 적용이 아니라 review-only다.

6. reliable_inbody_sparse_records_still_blocked
   - InBody 신뢰도가 높아도 최근 기록이 부족하면 자동 macro 권한은 막힌다.

7. weekly_high_without_session_specificity_blocked
   - 주간 운동일수가 많아도 실제 프로필/루틴/세션이 없으면 context incomplete다.

## 7. 다음 단계

다음 단계는 5단계 산출식 원칙 설계다.

5단계에서 해야 할 일:

- targetCal, protein, fat, carbohydrate가 어떤 순서로 작동할지 정한다.
- 외부 근거 숫자와 앱 정책 숫자를 분리한다.
- context가 좋은 경우에도 production 자동 적용 조건을 따로 둔다.
- 기록 품질이 낮거나 source가 약한 경우에는 산식이 어떻게 보수적으로 작동할지 정한다.
- Coach/Score/Records snapshot이 설명할 수 없는 산식은 production에 넣지 않는다.

5단계에서 하면 안 되는 일:

- BD gate 통과를 user-level formula 구현 완료로 말하기
- report-only candidate를 production 산식으로 바로 승격하기
- profileSession shortcut이나 performanceLevel selector로 설계를 단순화하기
- 최근 기록 품질이 낮은 사용자를 고부하/고신뢰 산식으로 자동 처리하기
