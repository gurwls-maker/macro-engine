# v8.3.1 scoring tuning protocol decision

Date: 2026-07-09

Scope:
- docs-only protocol decision
- no scoring formula change
- no anchor/penalty number change
- no `index.html` change
- no UI/storage/schema change
- no scoreDeltaPreview product path
- no old records cleanup/reset/fallback implementation
- no v8.3 tag rewrite

## 1. Why this document exists

The post-v8.3 backlog triage correctly reopened score distribution, carb upper, and high-volume carb concerns as v8.3.1 candidates. The risky part is not "tuning later" by itself. The risky part is tuning by feel, or accepting an old prompt that only checks the named item and misses adjacent causes.

This document closes the protocol for any future scoring tuning. It does not tune numbers. It defines what must be proven before numbers can change, which candidates are score-tuning candidates, which are copy/source/records decisions, and which paths are rejected.

Plain-language summary:
- v8.3 already replaced the fixed table with continuous curves.
- Some curve regions may still be too forgiving or too harsh.
- We do not fix that by manually nudging a number because it "feels off".
- A future tuning branch must bring fixtures, external-anchor relation, before/after outputs, invariants, and explicit user confirmation.

## 2. Independent prompt-risk judgment

The attached GPT prompt is directionally useful, but too narrow if followed literally. It names "scoring tuning protocol" correctly, but a narrow execution could still miss four surrounding risks:

1. Tuning could become a hidden score implementation if this protocol does not block number edits.
2. Alcohol and copy issues could be mistaken for score-curve issues.
3. Old records cleanup could be pulled into score tuning even though it is a separate product-data decision.
4. "Later" candidates could be forgotten unless current truth and docs-policy keep the next gate visible.

Decision:
- Accept the scoring tuning protocol direction.
- Reject immediate numeric tuning.
- Split score candidates from copy/source/records candidates.
- Make the protocol itself a docs-policy checked gate.
- Set the next gate to evidence/user-confirmation, not implementation.

## 3. External anchor vs app policy coefficient

Future tuning must separate two layers.

External anchor:
- A physiological or guideline reference that says a zone is reasonable, risky, or context-dependent.
- Examples: carbohydrate needs by training volume, fat intake range, protein sufficiency range, alcohol risk thresholds, TDEE overload meaning.
- External anchors are not hard cutoffs by themselves.

App policy coefficient:
- The app's penalty curve shape, interpolation, steepness, and product stance.
- Examples: how quickly penalty rises after a carb upper anchor, how harshly TDEE overload dominates final score, how much repeated 3-day/7-day skew limits adaptive targets.
- App coefficients require product judgment and user confirmation.

Required rule:
- External anchor can justify where a curve should be anchored.
- App policy coefficient decides how strongly the app penalizes between and beyond anchors.
- A future branch must state which layer it is changing.

## 4. Tunable and non-tunable areas

Tunable only through this protocol:
- continuous penalty curve anchor values
- interpolation/extrapolation steepness
- carb exercise context curve behavior
- fat/carb exchange curve behavior
- alcohol physiology risk curve if copy cannot solve the issue
- adaptive target limiter strength

Not tunable through this protocol:
- kcal range display product UI
- scoreDeltaPreview product path
- old records migration/recompute/reset
- package version policy
- storage/schema expansion
- DailyCoach wording rewrite
- tooltip/glossary implementation
- v8.3 tag movement

Those require their own decision gates.

## 6. Required evidence pack

Any future tuning implementation must attach an evidence pack with:

- exact current code version and scoring version
- fixture names and inputs
- current output table
- proposed output table
- external anchor relation for each changed curve
- app policy coefficient being changed
- expected user-facing meaning
- invariant list
- failure conditions
- user confirmation answers when product judgment is involved

Minimum fixture groups:
- neutral target-aligned full day
- TDEE overload 1.2x / 1.5x / 2.0x / 3.0x / 5.0x
- carb high at rest / unknown / normal / high-volume
- fat high under same kcal exchange vs total kcal excess
- repeated 3-day and 7-day high carb or high fat
- alcohol 4 / 5.7 / 8 / 10 / 12 / 16 drink equivalents
- source missing TDEE and targetProteinFallback states

## 7. Invariants that must survive tuning

Future tuning must keep these invariants:

- target macros 그대로 섭취한 valid full-day/no alcohol/no outlier 케이스는 낮은 점수로 떨어지면 안 된다.
- total kcal/TDEE overload는 adaptive target이나 carb/fat exchange로 가려지면 안 된다.
- high-volume exercise is not a flat high-carb no-penalty permission.
- exercise is carb context, not final score bonus.
- kcal range display product UI remains rejected.
- scoreDeltaPreview product path: REJECT.
- v6.1 alcoholImpactPenalty post-score subtraction remains rejected.
- fixed v8.2 high/severe penalty table remains rejected.
- no hidden recompute / silent mutation / silent delete for saved records.
- old records cleanup/reset/fallback is separate from score tuning.
- DailyCoach/copy wording must not be bent only to satisfy outdated tests if the old test forces AI-like copy.

## 8. User confirmation questions before number tuning

These are not optional niceties. They decide app policy coefficients.

1. When total intake is 2.0x TDEE, should score usually be near failure even if macros are balanced?
2. When total intake is 3.0x TDEE or higher, should score naturally collapse toward 0?
3. For fat high, what feels wrong today: normal flexible high-fat eating, or extreme fat excess not collapsing enough?
4. For high-volume carb, which real scenario should remain acceptable and which should clearly fail?
5. For unknown training context, should the app be conservative until exercise is logged?
6. For alcohol, is the mismatch mainly score severity or user-facing explanation/tone?
7. For repeated high-carb/high-fat days, how strongly should 3-day and 7-day pressure limit adaptive target movement?
8. Should old dev/local records be reset/fallback-cleaned after v8.3.1 planning, or left untouched until explicit manual cleanup?

Score tuning implementation is blocked until a future branch closes:

- evidence pack

- chosen candidate subset
- explicit non-candidate exclusions
- user confirmation answers for policy coefficients
- updated regression tests

Allowed next gate:

## 10. Decision

Decision:
- v8.3.1 scoring tuning protocol is closed.
- Numeric tuning is not opened.
- The first next step is an evidence/user-confirmation gate, not implementation.
- TDEE overload, fat high, carb upper/high-volume, unknown training context, and adaptive limiter are valid candidate areas.
- Alcohol is copy/product-tone first unless evidence shows score curve mismatch.
- Calorie target/source reliability is research/support, not kcal range display.
- Old records are a separate cleanup/reset/fallback decision.

Minimum vs complete distinction:
- Minimum implementation now: none. Do not change code or numbers.
- Complete work now: lock the protocol, matrix, evidence requirements, invariants, and next gate so that later tuning cannot proceed by inertia.
