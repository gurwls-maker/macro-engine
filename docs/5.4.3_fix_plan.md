# 5.4.3 Fix Plan

## 1. 배포 전 전제

- 현재 프로젝트는 아직 배포 전 1인 개발 상태다.
- 기존 사용자 데이터나 legacy record를 우선 호환할 필요가 없다.
- 따라서 5.4.3의 기준은 "기존 애매한 상태를 어떻게 살릴까"가 아니라 "앞으로 애매한 상태가 저장되지 않도록 정책을 단순화할까"에 둔다.
- 목표는 호환성보다 데이터 구조 단순화, 저장 의미 명확화, 후속 유지보수성 확보다.

## 2. 최종 데이터 정책

### 정책 A. auto adherence 저장에는 `goalSnapshot`이 반드시 필요하다

- `adherenceSource === "auto"`인 record에는 항상 유효한 `goalSnapshot`이 있어야 한다.
- `goalSnapshot`이 없으면 `adherencePercent`를 auto 값으로 저장하지 않는다.

### 정책 B. snapshot 없는 record는 manual/fallback 상태로만 둔다

- snapshot이 없는 record는:
  - `adherencePercent: null`
  - `adherenceSource: "manual"`
  상태로만 유지한다.
- auto score를 흉내 내는 중간 상태는 허용하지 않는다.

### 정책 C. 과거 날짜라고 해서 현재 live 입력으로 late snapshot을 자동 생성하지 않는다

- 과거 날짜 meal 추가 시 `goalSnapshot`이 없더라도, 현재 `calculate()` 결과를 사용해 snapshot을 자동 생성하지 않는다.
- 즉 `generated_on_late_meal_add` 같은 late snapshot 정책은 제거 후보로 본다.

### 정책 D. Today live 계산과 stored record 의미를 분리한다

- Today는 계속 현재 live 입력 기준으로 계산한다.
- stored record의 auto adherence는 snapshot 기준으로만 저장한다.
- live 기준과 stored 기준의 혼합 저장은 만들지 않는다.

### 정책 E. 데이터 구조는 가능한 한 단순한 방향으로 정리한다

- auto 상태: `goalSnapshot` + `adherencePercent` + `adherenceSource:"auto"`
- manual/fallback 상태: `goalSnapshot` 없음 또는 auto score 없음 + `adherenceSource:"manual"`

핵심은 "auto score가 있으면 snapshot도 반드시 있다"는 단일 규칙을 만드는 것이다.

## 3. 5.4.3에서 고칠 문제

### 문제 1. `persistAutoAdherenceForDate()`가 snapshot 없이도 저장 정책을 넓게 처리하는 문제

- 현재는 `goalSnapshot`이 없으면 내부에서 snapshot을 새로 만들 수 있다.
- 이 구조가 현재 live 입력을 과거 record 의미에 끼워 넣는 시작점이다.

### 문제 2. `saveMealEntry()`가 과거 날짜에 late snapshot 생성을 허용하는 문제

- 현재 `saveMealEntry()`는 `forceLateSnapshot: !record?.goalSnapshot`를 통해 snapshot 없는 날짜에 auto snapshot 생성을 허용한다.
- 배포 전 기준에서는 이 유연성이 아니라 명확한 저장 규칙이 더 중요하다.

### 문제 3. `generated_on_late_meal_add` 정책 자체

- 현재 snapshot source는 `saved_at_entry`, `generated_on_late_meal_add`, `legacy_import`를 허용한다.
- 이 중 `generated_on_late_meal_add`는 "당시 목표"가 아니라 "나중 입력 시점 목표"를 snapshot으로 남길 수 있어 의미가 불명확하다.

### 문제 4. auto/manual 경계가 저장 구조상 느슨한 문제

- manual `adherence`와 auto `adherencePercent` 공존 자체는 가능하다.
- 다만 5.4.3에서는 적어도 auto score 저장 조건만큼은 엄격히 고정할 필요가 있다.

## 4. 5.4.3에서 고치지 않을 문제

- `calculate()`와 매크로 계산식
- `getDailyAdherenceScore()` 계산 공식
- `render()`와 Today 렌더 순서
- `saveState()` 구조
- Weekly 평균/해석 로직
- Coach 해석 로직
- CSS / 탭 / DOM 구조
- `upsertRecord()` 전면 리팩터링
- import/export 정책 전면 재설계
- legacy 데이터 보정 로직 추가

즉 5.4.3은 저장 정책 정리 단계이지, 계산 엔진이나 표시 로직 수정 단계가 아니다.

## 5. 최소 코드 수정 후보

### 후보 1. `saveMealEntry()`에서 late snapshot 자동 생성을 끊는다

- 현재:
  - `forceLateSnapshot: !record?.goalSnapshot`
- 방향:
  - snapshot이 없는 과거 날짜라고 해서 현재 live 입력으로 snapshot을 만들지 않도록 호출 정책을 단순화

효과:

- 과거 record 의미가 나중 입력으로 바뀌는 경로를 차단할 수 있다.

### 후보 2. `persistAutoAdherenceForDate()`에서 "snapshot 없으면 auto 저장 금지"를 명시적으로 고정한다

- 현재도 결과적으로 snapshot이 없으면 manual로 떨어지지만, 내부에 snapshot 생성 경로가 남아 있다.
- 방향:
  - snapshot 생성 정책과 auto 저장 조건을 분리
  - snapshot이 없으면 무조건 manual/fallback 상태로만 저장

효과:

- `adherencePercent:auto`와 `goalSnapshot`의 결합 관계가 강제된다.

### 후보 3. `generated_on_late_meal_add` source와 관련 분기를 제거 후보로 본다

- 현재는 late snapshot을 허용하면서 source까지 남긴다.
- 배포 전 기준에서는 이 경로 자체를 없애는 편이 구조가 단순하다.

효과:

- snapshot source 의미가 단순해지고, QA 기준도 명확해진다.

### 후보 4. `getStoredAdherenceEntry()`는 이번 단계에서 직접 건드리지 않되, 읽기 우선순위 전제는 유지한다

- auto 우선, manual fallback 규칙은 그대로 둔다.
- 대신 "어떤 값이 auto로 저장될 수 있는가"의 조건만 먼저 정리한다.

효과:

- 저장 정책만 먼저 안정화하고, Weekly/Today 읽기 로직 회귀를 줄일 수 있다.

## 6. 수정 우선순위

1. `saveMealEntry()`의 late snapshot 자동 생성 중단
2. `persistAutoAdherenceForDate()`에서 snapshot 없는 auto 저장 금지 규칙 명시
3. `generated_on_late_meal_add` 정책 제거 또는 사실상 비활성화

우선순위 이유:

- 가장 먼저 끊어야 하는 경로는 "과거 날짜 + snapshot 없음 + 현재 live 목표로 자동 보정"이다.
- 그 다음은 auto 저장 조건을 단일 규칙으로 고정하는 것이다.
- source 정리는 그 정책을 적용한 뒤 자연스럽게 따라오는 작업이다.

## 7. 회귀 위험

### 높은 회귀 위험

- 기존 테스트 중 late snapshot 생성 자체를 기대하는 케이스
- snapshot 없는 과거 날짜 meal 추가 후 자동 준수율이 바로 생기던 흐름

### 중간 회귀 위험

- meal 삭제 후 `adherencePercent` 재계산
- auto analysis OFF 상태에서 manual/fallback 저장
- `saveRecord()` 후 records panel 표시값

### 낮은 회귀 위험

- `saveState()` 타이밍
- 탭 복원
- Settings 관련 로직
- Today live 계산

## 8. QA 체크리스트

- 오늘 날짜에서 snapshot이 존재하는 상태로 meal 저장 시 auto adherence가 정상 저장되는가
- 오늘 날짜에서 meal 수정/삭제 시 기존 snapshot 기준으로 `adherencePercent`만 재계산되는가
- 과거 날짜에 snapshot이 이미 있는 record는 meal 추가/삭제 후 snapshot이 유지되는가
- 과거 날짜에 snapshot이 없는 record는 meal 추가 후에도 auto adherence가 새로 저장되지 않는가
- snapshot 없는 record는 `adherenceSource:"manual"` / `adherencePercent:null` 상태로 남는가
- `generated_on_late_meal_add` 경로가 더 이상 실행되지 않는가
- `saveRecord()`가 weight/note/manual adherence 저장에만 머무는가
- Today는 여전히 live 입력 기준으로 계산되는가
- Weekly는 저장된 auto 값 또는 manual fallback을 기존 우선순위대로 읽는가

## 9. 5.4.4로 넘길 항목

- `upsertRecord()`의 필드별 덮어쓰기 보호 장치 강화
- manual `adherence`와 auto `adherencePercent`의 장기 데이터 모델 정리
- `legacy_import` source 처리 정책 재정의
- import payload의 snapshot/source 정합성 검증
- Today와 stored record 차이를 UI에 더 명확히 드러내는 개선
- Weekly/Coach에 live 기준 vs stored 기준 차이를 설명하는 개선

최종 판단:

- 5.4.3은 호환성 대응이 아니라 저장 의미 정리 단계로 보는 것이 맞다.
- "snapshot 없는 auto record는 없다"를 가장 강한 규칙으로 먼저 확정해야 한다.
- 이를 위해 가장 먼저 건드릴 곳은 `saveMealEntry()`와 `persistAutoAdherenceForDate()`의 late snapshot 자동 생성 정책이다.

## 적용 결과

### 1. 변경한 함수

- `saveMealEntry()`
- `persistAutoAdherenceForDate()`

### 2. 차단된 경로

- `saveMealEntry()`의 `forceLateSnapshot: !record?.goalSnapshot` 경로를 비활성화했다.
- `persistAutoAdherenceForDate()`가 snapshot 없는 상태에서 현재 live 입력으로 새 snapshot을 만드는 경로를 차단했다.
- `generated_on_late_meal_add`가 실행되던 피드백 경로를 제거했다.

### 3. 유지한 동작

- meal 저장 자체는 그대로 유지된다.
- 기존 `goalSnapshot`이 있는 record는 기존처럼 snapshot 기준 auto adherence 계산을 유지한다.
- snapshot 없는 record는 manual/fallback 상태로만 유지된다.
- Today / Weekly / Coach / `calculate()` / `render()` / `saveState()`는 건드리지 않았다.

### 4. QA 필요 항목

- snapshot이 이미 있는 날짜에서 meal 추가/삭제 시 auto adherence가 계속 정상 저장되는지
- snapshot 없는 날짜에서 meal 추가 후 auto adherence가 저장되지 않는지
- snapshot 없는 날짜에서 `adherencePercent:null`, `adherenceSource:"manual"`이 유지되는지
- `generated_on_late_meal_add` 관련 피드백이 더 이상 노출되지 않는지
