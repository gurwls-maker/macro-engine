## 1. 현재 Recent 흐름

현재 Recent 흐름:

```text
buildRecentContext()
  -> buildRecentContextWindow(windowDays)
  -> sourceRecords normalizeRecordsCollection()
  -> getRecentContextRecordSummary(record)
  -> getRecentContextComponentCounts(recordSummaries)
```

현재 record summary:

```text
getRecentContextRecordSummary(record)
  -> normalizeRecord(record)
  -> normalizeGoalSnapshot(raw.goalSnapshot)
  -> targetCal / targetProtein / targetCarbs / targetFat를 fixed snapshot에서 읽음
```

현재 macro component count:

```text
proteinLowDays:
  protein < targetProtein * 0.90

proteinSevereLowDays:
  protein < targetProtein * 0.65

carbLowTrainingDays:
  weight training day이고 carbs < targetCarbs * 0.85

carbLowRestDays:
  rest/non-weight day이고 carbs < targetCarbs * 0.85

fatLowDays:
  fat < targetFat * 0.75

fatHighDays:
  fat > targetFat * 1.25
```

현재 계약:

```text
Recent는 fixed target goalSnapshot 기준으로만 component counts를 만든다.
macroRangeContract / macroRangeBasisSignature / rangeScoreCandidate는 현재 normalize 과정에서 prune된다.
현재 unknown range field가 Recent output에 보이지 않는 것은 정상이다.
```

## 2. 왜 mixed-basis가 위험한가

macro range가 future known contract로 승격되면 같은 숫자라도 의미가 달라진다.

예:

```text
old fixed record:
  targetFat = 70g
  fatLowDays 기준 = fat < 52.5g
  fatHighDays 기준 = fat > 87.5g

future range record:
  fat actionFloor / recommendedMin / recommendedMax / flexibleMax가 따로 존재
  fat status는 fixed target 대비 +/-25%가 아니라 range threshold 기준으로 해석될 수 있음
```

따라서 old fixed record와 future range record를 같은 `fatLowDays`, `fatHighDays`, `carbLowTrainingDays`로 조용히 합치면:

```text
1. 과거 fixed target 기록이 새 range 기준처럼 보일 수 있다.
2. new range record가 old fixed target threshold로 잘못 감점될 수 있다.
3. Recent/Coach가 잘못된 반복 원인을 만들 수 있다.
4. 사용자는 "최근 지방 부족" 같은 문구의 기준이 무엇인지 알 수 없게 된다.
```

## 3. basis 분류 후보

future test-local detector는 record summary를 다음 basis로 분류한다.

```text
fixed_v8:
  현재 normalized goalSnapshot만 있고 macroRangeContract가 없는 기록.
  현재 production Recent의 source of truth.

current_unknown_pruned:
  raw fixture에는 range field가 있었지만 현재 normalize 과정에서 제거된 기록.
  production output에서는 fixed_v8과 구분되지 않는 것이 현재 정상.

range_known_valid_candidate:
  future explicit normalizer와 storage decision 후에만 가능한 known range basis 후보.
  현재 production에는 존재하면 안 된다.

range_known_invalid_candidate:
  future known field는 있으나 range order / targetCal mismatch / forbidden score preview 등으로 invalid인 후보.

range_known_unsupported_candidate:
  future known field의 version이 현재 앱에서 해석 불가능한 후보.

snapshotless_or_component_ineligible:
  goalSnapshot이 없거나 detailed component count 조건을 만족하지 않는 기록.
```

주의:

```text
current_unknown_pruned는 production record에서 직접 감지할 수 없다.
이 단계에서는 fixture/test-local detector의 분류 용어로만 둔다.
```

## 4. handling 선택지

### 선택지 A: current fixed-only 유지

내용:

```text
getRecentContextRecordSummary()
getRecentContextComponentCounts()
buildRecentContextWindow()
```

를 그대로 유지한다.

장점:

```text
old fixed Records 안정성 최상
현재 Recent/Coach/UI 회귀 위험 낮음
schema 확정 전 안전
```

단점:

```text
future known range record를 해석할 수 없음
range basis가 저장되면 fixed threshold로 잘못 해석될 위험
```

현재 단계 판단:

```text
선택지 A 유지
```

### 선택지 B: known range record를 fixed component counts에서 제외

내용:

```text
future known range basis가 있는 record는
fixed component counts에 넣지 않고 별도 excluded/mixed basis count로 둔다.
```

장점:

```text
old/new 기준 혼합을 막기 쉽다.
range-aware scoring이 완성되기 전에도 안전하다.
```

단점:

```text
Recent macroAnalysisDays가 줄어들 수 있다.
사용자에게 "분석 제외"를 어떻게 보여줄지 결정 필요.
```

future 후보:

```text
range-aware aggregation 전 임시 안전모드 후보
```

### 선택지 C: fixed counts와 range counts를 분리

내용:

```text
fixedBasisCounts:
  old fixed records만 집계

rangeBasisCounts:
  future known range records만 집계

mixedBasisSummary:
  같은 window 안에 fixed/range가 섞였는지 표시
```

장점:

```text
old fixed 해석 보존
future range 해석 확장 가능
Recent mixed basis를 명시적으로 감지 가능
```

단점:

```text
Recent/Coach copy, UI, tests, 저장 contract가 모두 필요
window summary 구조가 커짐
```

future 후보:

```text
production range score/version/schema가 닫힌 뒤 우선 검토
```

### 선택지 D: old fixed records를 range basis로 재해석

내용:

```text
old fixed target을 future range 기준으로 변환해 다시 집계
```

판단:

```text
금지
```

이유:

```text
과거 Records의 저장 당시 해석을 조용히 바꾼다.
goalSnapshot 신뢰성과 backup/restore roundtrip 의미를 훼손한다.
```

## 8. fixture 후보

다음 테스트가 열린다면 fixture는 다음을 포함한다.

```text
old fixed record:
  macroRangeContract 없음
  targetProtein / targetCarbs / targetFat fixed basis 사용

current unknown range raw record:
  raw input에는 macroRangeContract가 있지만 normalize 후 production Recent에는 보이지 않음
  current fixed counts unchanged 확인

future known range candidate record:
  test-local detector에서는 rangeBasisDays로 분류
  production Recent summary/counts에는 연결하지 않음

invalid range candidate record:
  test-local detector에서는 invalidRangeBasisDays로 분류
  fixed snapshot survival 확인

unsupported range candidate record:
  test-local detector에서는 unsupportedRangeBasisDays로 분류
  production Recent는 변경하지 않음

snapshotless / one-meal pending / component-ineligible:
  mixed basis detector가 macro component counts로 착각하지 않음
```
