## 비개발자용 설명

### 무엇을 바꿨는지

오늘 훈련을 `휴식`으로 골랐는데 탄수화물 부족 이유가 `오늘 운동량을 고려한 탄수화물 범위보다 부족해요.`라고 나오던 문제를 고쳤다.

문장 하나만 `휴식일`로 바꾼 것이 아니다. 오늘 설정에서 앱이 실제로 알고 있는 운동 상태를 먼저 다음처럼 구분하고, 탄수화물 설명과 별도의 운동 안내가 같은 구분을 사용하도록 정리했다.

```text
운동 관리 끔
실제 휴식
근력운동만 입력
유산소만 입력
러닝 프로필의 유산소
근력운동 + 유산소
선택한 운동의 필수 입력이 덜 끝난 상태
```

또한 독립 감사에서 각각 통과한 두 선행 보정도 먼저 하나의 master 상태로 통합했다.

```text
d2a365d: DailyCoach InBody 행동 방향 정합성 보정
fed26a7: Records 화면용 숫자 비교와 엄격한 점수 권한 비교 분리
```

두 브랜치가 함께 수정한 current-truth 문서는 어느 한쪽을 버리지 않고 양쪽 결론을 모두 보존했다. 통합본 전체 회귀를 통과한 뒤 master에 병합하고 원격에도 반영했다.

### 왜 바꿨는지

기존 탄수화물 이유 문구는 실제 오늘 설정을 읽지 않고 저탄수든 고탄수든 항상 `오늘 운동량`이라는 표현을 사용했다. 그래서 쉬는 날에도 마치 운동량이 있는 것처럼 말했다.

별도의 운동 안내에는 `오늘은 쉬는 날이에요.`가 있었지만 오늘의 코치는 최대 세 항목만 보여 준다. 칼로리·탄수화물·지방 문제가 먼저 세 자리를 차지하면 올바른 휴식 안내는 숨고, 탄수화물 항목의 잘못된 문구만 화면에 남을 수 있었다. 사용자가 본 화면이 정확히 이 경우다.

운동 상태를 판정하는 기존 기준에도 더 넓은 문제가 있었다.

```text
기존 일부 판정
근력운동 시간이 0분이면 휴식으로 취급

문제
- 러닝은 원래 근력운동 시간이 0분이어도 정상 운동이다.
- 유산소만 입력한 날도 휴식이 아니다.
- 복합운동에서 실제로 입력한 근력/유산소 구성만 말해야 한다.
- REST로 바꾼 뒤 남아 있는 과거 근력운동 시간은 다시 운동으로 새면 안 된다.
- 운동 관리를 끈 상태에서는 저장돼 있던 운동 입력을 말하면 안 된다.
```

따라서 문구 치환만 하면 휴식 사례 하나는 고쳐져도 러닝·유산소·복합운동에서 같은 종류의 거짓 설명이 다시 생긴다. 이번에는 앱이 실제로 아는 구성요소를 기준으로 운동 맥락을 한 번 판정하고 모든 코치 문구가 그 결과를 사용하게 했다.

### 독립 감사 후 추가로 바로잡은 것

최초 구현은 휴식일 오류와 기본 운동 구분은 고쳤지만, 선택한 세션 이름과 실제 계산된 운동이 다른 두 경로를 충분히 막지 못했다. 독립 감사에서 다음 반례가 확인됐다.

```text
복합 프로필 + 근력 계산 0 + 정상 유산소 25분
-> 실제 운동은 유산소뿐인데 `근력+유산소 25분` 또는 `근력 중심 25분`처럼 말할 수 있었음

보디빌딩 PUSH + 근력 60분 + 유산소를 입력한 뒤 Settings 프로필을 러닝으로 변경
-> 날짜별 근력 입력은 계산에 남아 있는데 새 러닝 세션 이름을 붙여 `템포 러닝 60분`처럼 말할 수 있었음
```

날짜별 Today 운동 입력은 사용자가 그날 입력한 값이므로 Settings 프로필을 바꿨다는 이유로 조용히 삭제하지 않는다. 대신 오늘의 코치는 새로 선택된 세션 이름을 구성요소 이름으로 복사하지 않고, 실제 계산 결과에서 근력 kcal와 유산소 kcal가 생겼는지를 먼저 본다.

```text
실제 근력 kcal만 있음       -> 근력운동
실제 유산소 kcal만 있음     -> 실제 유산소 종목
둘 다 있음                  -> 근력운동 + 실제 유산소 종목
일부 값만 입력돼 계산 불가  -> 입력 미완료
계산된 운동이 전혀 없음     -> 휴식 또는 선택 세션 입력 미완료
```

러닝 프로필의 `인터벌 러닝`, `템포 러닝` 같은 세션 이름은 날짜별 세션 소유권이 현재 프로필과 호환되고, 실제 유산소 종목도 그 세션의 기본 종목과 일치할 때만 유산소 이름으로 사용한다. 프로필을 바꾸기 전에 입력한 `트레드밀 워킹`이 남아 있다면 새 세션 이름으로 바꾸지 않고 그대로 말한다. 새 프로필이 근력 세션이라고 해도 그날 근력은 0분이고 유산소만 실제 계산됐다면 `미완성 근력운동`이 아니라 유산소만 한 날로 설명한다.

### 실제 사용자 화면이나 계산 결과가 어떻게 달라지는지

탄수화물이 부족하거나 높은 상태 자체는 기존 점수 결과를 따른다. 바뀌는 것은 그 상태를 설명하는 이유다.

```text
오늘 설정: 휴식
이전: 오늘 운동량을 고려한 탄수화물 범위보다 부족해요.
현재: 오늘 탄수화물 권장은 쉬는 날 기준으로 계산했어요.
```

휴식일에도 쉬는 날 기준의 최소 탄수화물 권장보다 적게 먹었다면 `아직 부족해요`라는 상태는 나올 수 있다. 이번 수정은 부족 판정을 없애는 것이 아니라, 부족한 이유를 운동한 날처럼 잘못 말하지 않게 한 것이다.

운동 설정이 바뀌면 같은 탄수화물 항목의 이유도 다시 계산된다.

```text
PUSH 60분
-> PUSH 60분을 오늘 탄수화물 권장에 반영했어요.

인터벌 러닝 35분
-> 인터벌 러닝 35분을 오늘 탄수화물 권장에 반영했어요.

근력운동 45분 + 트레드밀 러닝 25분
-> 두 운동 시간 모두 오늘 탄수화물 권장에 반영했다고 설명

복합 세션 + 근력 0분 + 트레드밀 러닝 25분
-> 트레드밀 러닝 25분만 반영했다고 설명

PUSH 60분 + 트레드밀 러닝 25분 입력 후 러닝 프로필로 변경
-> 근력운동 60분과 실제 유산소 25분을 각각 유지
-> 근력운동을 `템포 러닝 60분`으로 바꾸어 부르지 않음

운동 관리 끔
-> 오늘 식단 설정을 기준으로 계산했어요.
```

저탄수뿐 아니라 고탄수 항목도 같은 운동 맥락을 쓴다. 화면이 다시 그려질 때 이전의 `쉬는 날`, `PUSH`, `러닝`, `복합운동` 문구가 다음 상태에 남지 않는다.

선택한 운동에 필요한 시간이 없거나 유산소 시간은 있는데 속도가 없는 경우에는 휴식이나 정상 운동으로 꾸미지 않는다. 입력된 정보까지만 반영했다고 알리고, 비어 있는 `운동 시간`, `유산소 시간`, `유산소 속도`를 실제 상태에 맞게 말한다.

### 사용자가 큰 틀에서 확인해야 할 판단점

추가 정책 판단은 필요 없다. 확인할 제품 원칙은 하나다.

```text
오늘의 코치는 설정 이름을 기계적으로 반복하지 않고,
앱이 실제로 알고 있는 오늘 운동 구성만 말한다.
```

실사용에서는 휴식에서 PUSH, 러닝, 복합운동, 운동 관리 끔으로 바꿨을 때 탄수화물 이유가 즉시 그 상태에 맞게 바뀌는지만 보면 된다. 휴식일의 부족 상태 자체는 쉬는 날 권장 범위보다 실제 섭취가 적으면 정상적으로 남는다.

### 테스트에서 무엇을 검증했는지

- 사용자가 본 것처럼 칼로리·탄수화물·지방 세 항목이 모두 화면을 차지해 별도 휴식 안내가 숨더라도, 탄수화물 이유는 반드시 `쉬는 날 기준`이라고 말하는지 확인했다.
- 운동 관리 끔, 실제 휴식, 저장된 과거 근력 시간이 남은 휴식, REST + 실제 유산소, 보디빌딩, 파워빌딩, 파워리프팅, 일반 근력운동, 러닝, 유산소 전용, 완전한 복합운동, 근력만 입력한 복합 프로필, 입력 미완료를 각각 검증했다. 앱이 지원하는 운동 프로필 5종이 모두 직접 포함된다.
- 탄수화물 부족과 초과 양방향에서 모든 운동 맥락이 맞게 설명되는지 확인했다.
- REST로 바꾸면 저장돼 있던 이전 근력운동 시간이 문구에 새지 않는지 확인했다.
- 러닝과 유산소 전용 날을 `쉬는 날` 또는 `근력운동`이라고 부르지 않는지 확인했다.
- 복합운동에서는 실제 입력된 구성과 시간만 말하고 없는 유산소를 지어내지 않는지 확인했다.
- 화면을 `휴식 -> PUSH -> 러닝 -> 복합운동 -> 운동 관리 끔` 순서로 다시 그려 이전 문장이 남지 않는지 확인했다.
- Today 빠른 설정에서 `LEGS + 유산소 -> REST -> LEGS`로 바꿨을 때 실제 DailyCoach 판정이 `복합 -> 휴식 -> 근력`으로 바뀌는지 확인했다.
- 앱이 제공하는 복합 프로필의 비휴식 세션 5개를 `근력 0/45분`과 `유산소 없음/정상/시간만/속도만`으로 전부 교차한 40개 조합에서 production `calculate()`를 직접 실행했다. 실제 계산된 구성만 말하고 부분 입력은 미완료로 남기는지 확인했다.
- 실제 Settings 프로필 change 이벤트를 사용해 5개 프로필 사이의 서로 다른 방향 있는 전환 20가지를 전부 확인했다. 여기에 `보디빌딩 -> 러닝`, `복합 -> 러닝`, `러닝 -> 근력`의 종목 일치·불일치·근력 0분 경계 사례 3개를 별도로 확인해, 날짜별로 입력한 근력·유산소 값은 계산에 남되 새 프로필의 세션 이름으로 잘못 바뀌지 않는지 검증했다.
- 기존 테스트가 `회복 유산소`라는 선택 세션 이름을 실제 유산소 이름처럼 요구하던 두 기대값은 production 사실에 맞게 `트레드밀 워킹`으로 바로잡았다. 새 문구를 테스트에 억지로 맞춘 것이 아니라, 감사에서 드러난 잘못된 테스트 계약을 수정한 것이다.
- desktop, mobile, smoke, core, macro-policy와 전체 회귀에서 기존 점수·기록·화면 동작이 깨지지 않는지 확인했다.

### 구현 구조

`getDailyCoachTrainingContext`는 이제 routine 이름 하나가 아니라 실제 입력된 weight/cardio 구성과 profile 소유권을 바탕으로 `activityKind`를 만든다.

```text
general    운동 관리 끔
rest       실제 근력/유산소 입력 없음
resistance 실제 근력운동만 있음
cardio     실제 유산소만 있음
mixed      실제 근력운동과 유산소가 모두 있음
incomplete 선택한 세션에 필요한 입력이 비어 있음
```

독립 감사 후에는 판정 우선순위를 더 정확히 고정했다.

```text
1. calculate()가 실제로 만든 weightKcal / cardio.kcal 구성
2. 시간만 있고 강도 또는 속도가 없는 부분 입력
3. 실제 계산 구성은 없지만 선택한 세션에 필요한 입력도 없는 상태
```

따라서 세션 제목은 실제 계산 구성요소를 덮어쓰지 않는다. Settings 프로필 변경 뒤에도 날짜별 명시 입력은 보존하고, 근력은 실제 근력운동으로, 유산소는 실제 선택 종목으로 각각 이름 붙인다.

`getDailyCoachActivityAmountLabel`은 앱이 알고 있는 운동 이름과 시간만 만든다. `getDailyCoachCarbContextReason`은 이 공통 evidence를 저탄수/고탄수 이유에 사용한다. `buildDailyCoachTrainingContextItem`도 같은 `activityKind`를 사용해 별도 운동 항목과 탄수화물 이유가 모순되지 않게 한다.
