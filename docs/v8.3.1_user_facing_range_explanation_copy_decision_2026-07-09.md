## 3. Original intent preservation

The user's role is not to pick exact copy strings, score coefficients, or test literals.

The user's role is to identify product intent and call out awkward user experience, for example:
- the card should not show one thing while the score means another
- "Today range" should not become a fake UI-only cap
- exercise context should not become a high-carb permission slip
- specialist terms can be useful, but only when placed carefully
- "회복 재료" / "회복재료" reads AI-like and should not be frozen just because a test expects it
- if copy tests fail, tests may be wrong, not necessarily the new wording

Codex's responsibility:
- preserve the original intent from the conversation
- separate product meaning from implementation inertia
- choose a natural surface responsibility model
- design narrow implementation gates when the decision is strong enough
- update tests when tests preserve awkward wording instead of user-facing meaning

## 4. Surface responsibility model

### Today macro card

Role:
- fast numeric scan
- current consumed amount
- today's target value
- compact today range
- short in/out/below/above status

Not its role:
- explain physiology
- explain glycogen
- explain all adaptive target mechanics
- justify alcohol risk
- teach TDEE
- become a long copy surface

Decision:

```text
Keep Today macro card compact.
Do not add explanatory paragraphs inside the card.
```

### DailyCoach

Role:
- interpret the day
- connect range status to action
- mention training/carb context when it affects what to do next
- keep alcohol / total kcal / fat risk visible
- avoid pretending exercise makes every macro choice acceptable

Decision:

```text
DailyCoach owns interpretation and next action.
```

### Tooltip / glossary

Role:
- optional explanation of a small number of technical terms
- help when a term is useful but not obvious
- keep the primary UI clean

Not its role:
- patch unclear primary copy
- expose internal scoring terms
- turn every card into a help manual

Decision:

```text
Do not implement tooltip/glossary yet.
First create a small glossary/copy spec if technical terms are introduced.
```

### Settings

Role:
- explain persistent settings and behavioral switches
- hold "오늘 목표 자동 조정" explanation
- allow ON/OFF control

Decision:

```text
Settings remains the primary surface for explaining the adaptive target setting.
DailyCoach may reflect the effect, but should not read like a settings manual.
```

## 5. U01. Today macro card explanation boundary

- card uses `오늘 범위 안`, `오늘 범위까지 N g`, `오늘 범위보다 N g 높음`
- kcal and alcohol do not show a macro range
- visible range is aligned with scoring boundary rather than a UI-only cap
- a small help tooltip exists around today's target auto adjustment in the target header

Decision:

```text
no-change
```

Details:
- Keep `오늘 범위` as the card term.
- Keep range chip in the header/right-side compact location.
- Keep denominator without repeating `기준` when the layout already communicates target.
- Keep kcal range display rejected.
- Keep alcohol range display rejected.
- Do not add extra explanatory text inside macro cards.
- Keep accessibility text that preserves "오늘 범위" meaning.

Reason:
- The card is now doing the right job: quick scan.
- Adding more prose would make the card worse, not clearer.
- The next visible problem is not card density; it is explanatory copy around what the range means.

## 6. U02. DailyCoach explanation boundary

Current behavior:
- DailyCoach already has training/carb, alcohol, fat, source, and record-caution paths.
- Existing tests protect alcohol severity and prevent training context from becoming alcohol permission.
- Some existing wording and tests still preserve "회복 재료" style language.

Decision:

```text
copy-implementation-needed
```

DailyCoach should:
- explain action, not formula
- say what to do with remaining food
- mention 운동 전후 탄수화물 when training context matters
- keep total kcal and alcohol risks separate from carb range
- avoid saying or implying "운동했으니 괜찮다"
- avoid broad physiology lectures in the primary item

DailyCoach should not:
- explain `adaptiveMacroTargetsEnabled`
- expose TDEE / anchor / curve / penalty
- say "회복 재료" as a primary user-facing phrase
- turn high-volume into a blanket high-carb approval
- soften alcohol risk because the user trained

Implementation implication:
- A narrow DailyCoach/range copy naturalness branch is justified after this decision.
- It should update both copy and tests where tests freeze awkward wording.

## 7. U03. Tooltip / glossary policy

Decision:

```text
tooltip-spec-needed
tooltip-implementation-needed: no, not yet
```

Rationale:
- Tooltip can be useful for terms like glycogen, but adding it directly risks clutter and inconsistent mobile behavior.
- The app already has many help affordances. More `?` buttons should be intentionally placed.
- A glossary/spec should first define which terms deserve tooltips and which should be avoided entirely.

Allowed future direction:
- small glossary spec
- maximum 3-5 nutrition/scoring terms at first
- no tooltip for every macro card label
- no internal scoring terms
- mobile accessibility and tap target review before implementation

Not allowed now:
- broad tooltip implementation
- card-level tooltip proliferation
- exposing `TDEE`, `anchor`, `curve`, `penalty`, `scoringContext` as user-facing glossary terms
