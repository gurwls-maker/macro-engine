# v8.2 macro range storage compatibility checkpoint

상태:

```text
문서 성격: macro range storage/schema 진입 전 중간점검
작업 범위: docs-only
index.html 변경: 없음
테스트 코드 추가: 없음
production 산식 변경: 없음
production score 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
goalSnapshot schema 변경: 없음
macroRangeContract 저장: 없음
macroRangeBasisSignature 저장: 없음
backup / restore schema 변경: 없음
Recent aggregation 변경: 없음
Today / Records / Recent UI 변경: 없음
production macro range 구현: 금지 유지
```

이 문서는 `macroRangeContract`나 range score를 저장 가능한 계약으로 올리기 전, 지금까지 쌓은 report-only / test-local 작업을 한 번 묶어 현재 고정된 계약과 아직 후보인 영역을 분리한다.

핵심 결론:

```text
현재까지 증명한 것:
  fixed target 기반 production 동작은 유지된다.
  range 후보 field는 production 저장 / 복원 / Recent / score / UI에 새지 않는다.
  future range basis를 다룰 때 필요한 위험 지점은 test-local로 관찰 가능하다.

아직 증명하지 않은 것:
  macroRangeContract를 저장해도 된다는 것.
  macroRangeBasisSignature를 저장해도 된다는 것.
  range-aware score / Recent / backup / restore 정책이 확정됐다는 것.
```

따라서 다음 단계도 바로 storage 구현이 아니라, storage 후보를 열지 말지 판단하는 설계 또는 외부 검수여야 한다.

## 1. 이 체크포인트의 목적

이번 macro range 라인은 단순 문구 정리가 아니다.

영향 범위:

```text
target macro 계산
Today 남은 양 표시
식단 점수
DailyCoach action / reason
Records goalSnapshot
Records 무변경 저장 판정
backup / restore
smart restore
Recent component counts
과거 기록 해석
```

따라서 통과 기준은 “테스트가 많다”가 아니라 아래다.

```text
old fixed basis가 깨지지 않았는가
range 후보가 저장 / 복원 / Recent / UI / score에 새지 않는가
현재 계약과 future 후보가 문서와 테스트에서 분리되어 있는가
숫자 정책이나 schema 의미를 임의 확정하지 않았는가
```

## 2. 현재 production 계약

현재 앱에서 실제 source of truth인 계약은 fixed target 계약이다.

```text
goalSnapshot:
  targetCal
  targetProtein
  targetCarbs
  targetFat
  기타 현재 allowlist field

signature:
  GOAL_SNAPSHOT_SIGNATURE_KEYS 기준 fixed-only signature

score:
  getDailyAdherenceScore()
  ADHERENCE_SCORING_VERSION 유지

Recent:
  getRecentContextRecordSummary()
  getRecentContextComponentCounts()
  fixed targetProtein / targetCarbs / targetFat 기준 component count

backup / restore:
  현재 backup payload와 restore normalize 경로는 fixed snapshot 계약 기준
```

현재 `macroRangeContract`, `macroRangeBasisSignature`, `rangeScoreCandidate`, `candidateScorePreview`, `scoreDeltaPreview`는 production 저장 계약이 아니다.

현재 정상으로 보는 동작:

```text
unknown macroRangeContract는 normalizeGoalSnapshot allowlist에서 prune된다.
unknown macroRangeBasisSignature는 저장 / 복원 / signature에 남지 않는다.
candidateScorePreview는 backup / restore / Recent / DailyCoach / UI에 들어가지 않는다.
Recent는 range 후보를 fixed component count로 조용히 확장하지 않는다.
ADHERENCE_SCORING_VERSION은 바뀌지 않는다.
```

이 동작은 “range 지원 실패”가 아니라, 아직 range schema를 열지 않은 현재 계약이다.

## 5. 남은 핵심 결정

storage 후보로 넘어가기 전에 최소한 아래를 별도 문서 또는 외부 검수로 닫아야 한다.

### 5-1. macroRangeContract shape

결정 필요:

```text
version
basis source
protein anchor field
fat action floor / recommended min / recommended max / flexible max field
carb training min / range max field
data quality state
policy unresolved state
invalid / unsupported reason format
```

주의:

```text
현재 simulation shape는 production schema 확정이 아니다.
```

### 5-2. signature policy

결정 필요:

```text
current GOAL_SNAPSHOT_SIGNATURE_KEYS는 계속 fixed-only로 둘지
range basis signature를 별도 field로 저장할지
separate basis signature를 저장한다면 어떤 field만 포함할지
field order / numeric serialization 기준
debug / copy / evidence field exclusion 기준
```

주의:

```text
signature를 잘못 열면 Records 무변경 저장과 Today/Records 갱신 필요 판단이 흔들린다.
```

### 5-3. backup / restore policy

결정 필요:

```text
known macroRangeContract를 full backup payload에 포함할지
records-only import에서 known range basis를 보존할지
smart restore keep / replace / meals-only에서 basis conflict를 어떻게 다룰지
conflict preview에 basisConflict를 표시할지
unsupported range contract를 prune할지 보류할지 경고할지
```

주의:

```text
meals-only에서 snapshot basis까지 바꾸면 사용자 기대와 달라진다.
```

### 5-4. Recent policy

결정 필요:

```text
fixed counts와 range counts를 분리할지
range record를 fixed component counts에서 제외할지
mixed-basis 상태를 Recent UI에 표시할지
DailyCoach가 mixed-basis context를 볼지
old fixed Records를 절대 range 기준으로 재해석하지 않는지
```

주의:

```text
fixed/range를 같은 fatLowDays로 조용히 합치면 과거 기록 의미가 바뀐다.
```

### 5-5. score / Coach policy

결정 필요:

```text
range score candidate를 숫자로 만들지
range score weight / severity / data quality policy
score version bump 여부
DailyCoach action에 range basis를 언제 반영할지
fat low / carb high / protein low를 어떻게 분리해 말할지
```

주의:

```text
candidateScorePreview가 숫자가 되는 순간 사용자-facing score처럼 오해될 수 있다.
```
