Date: 2026-06-03
Branch 기준: `codex/v8-formula-audit`
Latest checked baseline before AS1: `f990208 v8 goal resume playbook audit`

AS1 hotfix note (2026-06-03): Today/Records stale-basis prompts now ignore display/score-insignificant `goalSnapshot` residue such as 1 kcal, 0.1 g, or 0.01~0.02 kg while preserving real basis-change prompts. This is an internal product/UX tolerance, not external exercise-physiology evidence, and it does not close `full_8_2_cartesian_execution` or `full_v8_completion`.

이 문서는 추후 `/GOAL`로 V8 작업을 다시 시작할 때 첫 기준으로 읽어야 한다. 목적은 다음 Codex가 작은 단계 통과, 문서 요약, 파생 필드, 스크린샷 수, bounded shard 실행을 전체 완료로 오해하지 않게 하는 것이다.

## 1. 현재 결론

V8 전체는 아직 완료가 아니다.

완료에 가까운 영역:
- candidate-v2 formula selection / active runtime production targetCal application / post-wiring production visual QA는 현재 runner와 render audit 기준으로 닫힌 상태다.
- Today 상세 공복 체중 snapshot hotfix는 AQ1에서 닫혔다.
- Settings/Today 운동 프로필/루틴/세션 소유권은 AQ에서 바로잡혔고, AR/AS에서 Today 상단 훈련 요약 오해 가능성까지 닫았다.
- evidence boundary audit은 AS 기준 runner에 포함되어 있다.

미완 또는 완료 주장 금지 영역:
- `full_8_2_cartesian_execution`
- `full_v8_completion`
- 모든 profile/routine/session 조합의 사람 눈 UX 검토
- 내부 코드/제품정책 근거를 외부 운동생리학 근거처럼 읽는 문제의 전면 해소
- V8 전체 작업방식이 사용자의 의도대로 매번 다각적으로 검토되고 있다는 보증

## 6. 작업방식 문제에서 나온 강제 원칙

다음 `/GOAL`은 아래 원칙을 먼저 적용해야 한다.

1. 작은 stage 통과를 전체 진척으로 포장하지 않는다.
   - 새 단계가 "증거 경계"인지 "실제 앱 동작 개선"인지 "사용자 UX 개선"인지 먼저 분류한다.

2. 기존 기능에 덧붙이기 전, 기존 원리를 먼저 확인한다.
   - 운동 프로필/세션 문제에서 기존 bodybuilding 루틴/요일계획을 이해하지 않고 profileSession 중심으로 가면 기준통과용 우회가 된다.

3. 사용자 소유 입력 경로를 파생 필드로 대체하지 않는다.
   - `profileSession`은 snapshot/draft/report/candidate contract용 파생 필드다.
   - 사용자 선택의 소유권은 `exerciseProfile + routinePlan + routine`에 있다.

4. 테스트가 통과해도 실제 화면/사용자가 이상하면 실패다.
   - 특히 설정 탭과 Today 탭은 "어디서 기본값을 정하고, 어디서 오늘 값을 바꾸는가"가 실제 사용성의 핵심이다.

5. 문서/runner/render/test가 서로 다르면 작업을 진행하지 말고 먼저 동기화한다.
   - AS는 이 문제 때문에 생긴 stage다.

6. hardcoded copy table로 기준을 맞추지 않는다.
   - 가능하면 실제 source definition, actual formatter, actual DOM, actual runtime metadata를 읽는다.

7. 내부 코드 근거와 외부 운동생리학 근거를 섞지 않는다.
   - 앱 내부 정책/기존 산식/테스트 fixture는 외부 근거가 아니다.
   - 외부 근거로 쓸 때는 source, directness, app-fit, 영향을 받는 산식을 문서화한다.

8. 18개 human-review, pairwise, targeted stress, bounded shard, render capture는 서로 대체물이 아니다.
   - 806억 full Cartesian을 18개나 50개로 "재현"했다고 쓰면 안 된다.

9. "maintain"은 유지 목표 production contract일 뿐, mixed/running profile formula 완료가 아니다.

10. 사용자가 지적한 예시는 그대로 복붙 적용하지 말고 근본 원리를 각 단계에 맞게 적용한다.
    - 예: 운동 프로필 문제의 근본은 "기준 통과용 파생 경로가 실제 사용자 경로를 우회했다"이다.
    - 같은 원리를 full Cartesian, render audit, formula approval, external evidence에도 적용한다.

11. shutdown 예약은 full V8 완료 + git clean + 사용자의 명시 지시가 있을 때만 한다.

## 8. 다음 실행 우선순위 제안

우선순위 1: V8 full completion 정의를 owner-ready checklist로 고정
- `full_8_2_cartesian_execution`을 정말 전수 실행으로 유지할지, 아니면 별도 owner decision으로 대체 gate를 만들지 명확히 해야 한다.
- Codex가 임의로 대체하면 안 된다.

우선순위 2: profile/routine/session human UX review matrix
- profiles: bodybuilding, powerbuilding, strength, running, mixed
- surfaces: Settings, Today quick-open, Today summary, Records detail/basis-open
- cases: 각 profile의 default plan, at least one non-rest session, REST, zero-weight non-rest cardio/recovery, bodybuilding legacy plans
- evidence: screenshot + runtime metadata + 사람이 읽는 UX 판정

우선순위 3: full Cartesian clean campaign strategy
- current clean campaign은 48 unique / contiguous 40이다.
- 다음 exact range는 40~48부터 이어져야 한다.
- 단, 8개 단위 수동 실행은 completion strategy가 아니다. 대량 실행 계획, checkpoint, failure retry, time estimate, source clean policy가 먼저 필요하다.

우선순위 4: basis/evidence source audit
- 신규 산식 조정이나 기준 보정은 반드시 근거 level을 기록한다.
- `external_sports_nutrition_reference`, `external_population_nutrition_reference_guard`, `internal_product_policy`, `internal_formula_coverage_only`를 혼동하지 않는다.

## 9. 지금 바로 완료라고 말하면 안 되는 이유

- runner가 `fullV8Complete=false`를 직접 보고한다.
- runner `notApprovedYet`에 `full_8_2_cartesian_execution`, `full_v8_completion`이 남아 있다.
- full Cartesian clean campaign은 unique 48 / contiguous 40일 뿐이다.
- broad profile/routine/session human UX review가 아직 open이다.
- 사용자가 이미 실제 화면에서 기준통과용 작업물을 발견했다. 따라서 "테스트 통과"는 충분한 완료 근거가 아니다.
