# 5.4.3 Data Flow Analysis

## 1. adherencePercent 흐름

### 계산 위치

- `getDailyAdherenceScore(date, currentResult = calculate(), options = {})`
  - 날짜별 식사 합계와 목표값을 비교해 준수율을 계산하는 실제 계산 진입점이다.
  - 내부에서 `getMacroBalanceForDate()`를 호출해 해당 날짜 섭취량과 목표 기준을 잡는다.
- `getTodayAdherenceScore(currentResult = calculate())`
  - 오늘 날짜 전용 읽기 경로다.
  - `preferSnapshot: false`로 호출하므로 Today 화면은 저장된 `goalSnapshot`보다 현재 live 입력 기준으로 계산한다.

### 저장 위치

- 실제 저장은 `persistAutoAdherenceForDate(date, currentResult, options)`가 맡는다.
- 이 함수는 두 경우로 나뉜다.
  - auto analysis ON + snapshot 존재
    - `getDailyAdherenceScore(..., { goalSnapshot: snapshot })`로 계산
    - `upsertRecord()`에 `adherencePercent`, `adherenceSource: "auto"`, `adherenceLockedAt` 저장
  - auto analysis OFF 또는 snapshot 없음
    - `upsertRecord()`에 `adherencePercent: null`, `adherenceSource: "manual"`, `adherenceLockedAt: null` 저장

### persistAutoAdherenceForDate() 역할

- 날짜 정규화
- 기존 record 조회
- 기존 `goalSnapshot` 존재 여부 확인
- 필요 시 snapshot 생성
- snapshot 기준으로 auto adherence 재계산
- 결과를 `upsertRecord()`로 기록

즉 이 함수는 단순 helper가 아니라:

- snapshot 생성 정책
- auto/manual source 결정
- `adherencePercent` 저장
- lock 시점 기록

을 한 번에 담당하는 핵심 저장 오케스트레이터다.

### 수동 입력 vs 자동 계산 충돌 여부

- `saveRecord()`는 auto analysis ON일 때:
  - `adherence`는 기존 manual 값을 유지
  - `adherenceSource`도 기존 값을 유지
- `getStoredAdherenceEntry(record)`는:
  - `adherencePercent`가 있고 `adherenceSource === "auto"`면 auto를 우선 사용
  - 아니면 manual `adherence`를 90/75/60으로 환산해 사용

정리:

- 데이터 저장 구조상 manual `adherence`와 auto `adherencePercent`는 공존 가능하다.
- 읽기 우선순위는 auto가 더 높다.
- 따라서 "동시 저장"은 허용되지만 "동시 해석"은 아니다.
- 이 구조는 의도된 fallback으로 보이지만, record 내부 값이 서로 다른 의미를 담은 채 함께 남는 점은 구조적 충돌 후보다.

## 2. goalSnapshot 흐름

### 생성 시점

- `buildGoalSnapshotFromCurrentState(currentResult = calculate(), snapshotSource = "saved_at_entry")`
  - 현재 live 입력과 `calculate()` 결과를 snapshot으로 만든다.
  - `targetCal`, `protein`, `carbs`, `fat`뿐 아니라 mode, goal, weight, bodyFat, routine, cardio, workAdj 등 현재 엔진 입력을 함께 묶는다.
  - `currentResult?.isCalculable`가 아니면 `null`을 반환한다.

### 저장 시점

- 실제 저장은 `persistAutoAdherenceForDate()` 안에서만 일어난다.
- 흐름:
  1. 기존 record 조회
  2. 기존 snapshot이 있으면 재사용
  3. 없으면 `buildGoalSnapshotFromCurrentState()`로 새 snapshot 생성
  4. `upsertRecord()`에 `goalSnapshot`, `snapshotSource` 저장

### 기존 record 덮어쓰기 여부

- `upsertRecord()`는 필드 단위 병합이지만, `goalSnapshot` 키가 들어오면 그 값으로 교체한다.
- 다만 production 경로에서 `persistAutoAdherenceForDate()`는 `existing?.goalSnapshot || buildGoalSnapshotFromCurrentState(...)`를 사용하므로 기존 snapshot이 있으면 덮어쓰지 않는다.
- 현재 production 경로 기준으로는 snapshot 재생성이 제한되어 있어 무차별 덮어쓰기는 낮다.
- 하지만 `upsertRecord()` 자체는 보호 장치가 없어서, 다른 경로가 `goalSnapshot`를 넣으면 그대로 교체된다.

### Today 계산과의 관계

- Today 화면의 `getTodayMacroBalance(currentResult)`는 `getMacroBalanceForDate(..., { preferSnapshot: false })`를 사용한다.
- 즉 Today는 저장된 `goalSnapshot`보다 현재 live 입력 기준을 우선한다.
- 반대로 record 날짜 기반 계산이나 persisted auto adherence 계산은 snapshot 기준으로 잠글 수 있다.

정리:

- Today는 live state 중심
- stored record는 locked snapshot 중심

이라는 이중 기준 구조다.

## 3. 덮어쓰기 위험 경로

### 경로 1. `upsertRecord(rawRecord)`

- 날짜를 기준으로 기존 record를 찾아 새 객체를 다시 만든다.
- `adherencePercent`, `goalSnapshot`, `snapshotSource`, `adherenceLockedAt`도 모두 병합 대상이다.
- `rawRecord`에 특정 필드가 있으면 해당 필드가 기존 값 위에 올라간다.

리스크:

- 잘못된 `date` 또는 잘못 구성된 `rawRecord`가 들어오면 해당 날짜 record 전체 의미가 바뀔 수 있다.

### 경로 2. `persistAutoAdherenceForDate(date, ...)`

- snapshot 생성 여부
- auto/manual source 결정
- `adherencePercent` nulling 또는 재계산

을 한 번에 수행한다.

리스크:

- 이 함수 호출 자체가 "그 날짜의 준수율 저장 정책"을 실행하는 행위다.
- auto OFF 상태로 호출하면 기존 auto percent가 null로 지워질 수 있다.
- snapshot이 없는 과거 날짜에 호출되면 현재 입력 기반 late snapshot이 최초 생성될 수 있다.

### 경로 3. `saveMealEntry()`

- `upsertRecord({ date, meals: nextMeals })`
- 이어서 `persistAutoAdherenceForDate(date, calculate(), ...)`

리스크:

- 과거 날짜 meal 추가 시 현재 live 입력으로 snapshot이 최초 생성될 수 있다.
- 코드상 이를 의도적으로 허용하고 있으며, `generated_on_late_meal_add`로 source를 남긴다.

### 경로 4. `deleteMealEntry()`

- meal 배열 수정 후 `persistAutoAdherenceForDate()`를 다시 호출한다.

리스크:

- snapshot 자체는 재사용되지만, `adherencePercent`는 같은 snapshot 기준으로 재계산되어 바뀔 수 있다.
- 즉 snapshot은 고정, percent는 변동되는 구조다.

## 4. 과거 record 변형 가능성

### 가능한 경로

- `saveMealEntry()`로 과거 날짜 meal 추가
  - snapshot이 없으면 현재 입력 기준으로 최초 snapshot 생성 가능
- `deleteMealEntry()`로 과거 날짜 meal 삭제
  - 기존 snapshot 유지한 채 `adherencePercent` 재계산
- `saveRecord()`로 과거 날짜 weight/note/manual adherence 갱신
  - auto score 자체는 직접 건드리지 않지만 same-date record는 계속 병합됨
- `importRecordsJson()`
  - `recordsState.items` 전체를 외부 payload로 교체

### 직접 영향이 없는 경로

- `render()`만으로는 과거 record를 재채점하지 않는다.
- 테스트도 이를 검증한다.
  - past adherence remains locked after input changes
  - weekly snapshot average ignores current input changes

### Record -> Today 역방향 흐름 존재 여부

존재한다.

- Today 소비량은 오늘 날짜 record의 meals를 읽는다.
- Today 균형 계산은 `getTodayMacroBalance()` -> `getMacroBalanceForDate(getTodayDateString(), ...)`
- Today 준수율도 `getTodayAdherenceScore()` -> `getDailyAdherenceScore(getTodayDateString(), ...)`

다만 Today는 `preferSnapshot: false`라서:

- 오늘 record의 meals는 읽지만
- 저장된 `goalSnapshot`은 기본적으로 무시하고
- 현재 live 목표를 기준으로 다시 계산한다.

즉 역방향 흐름은 "기록된 meal 데이터의 반영"은 존재하지만, "저장된 snapshot을 Today에 고정 적용"하는 흐름은 아니다.

## 5. 구조적 리스크 요약

### 리스크 1. auto/manual 이중 저장 구조

- 하나의 record 안에 manual `adherence`와 auto `adherencePercent`가 함께 남을 수 있다.
- 읽기 우선순위는 명확하지만, 데이터 구조는 단일 truth source가 아니다.

### 리스크 2. 과거 날짜 late snapshot 생성

- 과거 날짜에 snapshot이 없으면 meal 추가 시 현재 입력값으로 최초 snapshot이 생성될 수 있다.
- 의도된 fallback이지만, "당시 목표"가 아니라 "추가 입력 시점 목표"가 기록될 수 있다.

### 리스크 3. 중앙 쓰기 경로의 과도한 책임 집중

- `persistAutoAdherenceForDate()`와 `upsertRecord()`가 together로 snapshot 생성, source 전환, percent 저장, lock 기록을 동시에 책임진다.
- 작은 호출 차이도 record 의미를 크게 바꿀 수 있다.

### 리스크 4. Today와 stored snapshot 기준의 이중화

- Today는 live 입력 기준
- persisted adherence는 locked snapshot 기준

이어서, 같은 날짜라도 화면에서 보는 Today 준수율과 저장된 record의 auto adherence가 시점에 따라 다르게 해석될 수 있다.

## 6. 수정 필요 후보 (아직 수정하지 않음)

- `persistAutoAdherenceForDate()` 책임 분리
  - snapshot 생성
  - adherence 계산
  - record 저장
  - source/lock 관리
  를 분리하면 회귀 위험 분석이 쉬워진다.
- `upsertRecord()` 보호 장치 보강
  - `goalSnapshot`나 `adherencePercent`를 덮어쓸 때 더 좁은 조건이 필요할 수 있다.
- late snapshot 정책 재검토
  - 과거 날짜 최초 meal 추가 시 snapshot 생성 기준을 별도 정책으로 분리할 필요가 있다.
- manual/auto 단일 해석 모델 정리
  - `adherence`와 `adherencePercent` 공존 구조를 유지할지, 표시용/저장용 역할을 분리할지 검토가 필요하다.
- Today vs stored snapshot 차이의 UI 명시
  - 현재 값은 live 기준, record 값은 locked 기준이라는 차이를 더 명확히 보여줄 필요가 있다.

## 7. 절대 수정 금지 영역

- `calculate()`와 매크로 계산식 본체
- `render()` 전체 렌더 순서
- `saveState()` 저장 구조
- Weekly 해석 로직
- Coach 해석 로직
- CSS 및 탭/패널 DOM 구조

부가 메모:

- 이번 단계의 핵심은 계산식이 아니라 "언제 어떤 기준값을 snapshot으로 잠그고, 어떤 시점에 percent를 다시 계산해 저장하느냐"의 문제다.
- 따라서 후속 수정도 계산 결과를 바꾸기보다 저장 책임과 데이터 의미를 분리하는 방향이 더 안전하다.
