# v8.2 macro range contract normalizer design

상태:

```text
문서 성격: macroRangeContract production normalizer design 후보 정리
작업 범위: docs-only
구현 변경: 없음
index.html 변경: 없음
테스트 코드 추가: 없음
normalizeGoalSnapshot 변경: 없음
normalizeMacroRangeContractForSnapshot production 구현: 없음
goalSnapshot schema 변경: 없음
macroRangeContract 저장: 없음
macroRangeBasisSignature 저장: 없음
GOAL_SNAPSHOT_SIGNATURE_KEYS 변경: 없음
backup / restore schema 변경: 없음
Recent aggregation 변경: 없음
getDailyAdherenceScore 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
Today / Records / Recent UI 변경: 없음
production macro range 구현: 금지 유지
```

이 문서는 `v8.2_macro_range_phase_gate_closeout_2026-07-03.md` 이후 안전 후보인 contract normalizer production design을 docs-only로 정리한다.

중요:

```text
이번 문서는 production normalizer 구현 지시서가 아니다.
이번 문서는 normalizeGoalSnapshot 변경 승인 문서가 아니다.
이번 문서는 macroRangeContract 저장 승인 문서가 아니다.
이번 문서는 storage/schema gate를 여는 문서가 아니다.
```

목적:

```text
future production normalizer가 필요해질 경우 어떤 입력, 출력, valid/invalid/unsupported 처리,
fixed snapshot survival rule, signature/backup/Recent/score 경계가 필요한지 설계한다.
```

현재 결론:

```text
contract normalizer 설계 후보는 문서화 가능하다.
production normalizer 구현은 아직 금지다.
macroRangeContract 저장도 아직 금지다.
```

## 1. 배경

지금까지 고정한 것:

```text
unknown macroRangeContract는 현재 normalizeGoalSnapshot에서 prune된다.
test-local simulation은 absent / valid / invalid / unsupported contract를 분리할 수 있다.
invalid / unsupported contract가 fixed snapshot을 깨면 안 된다는 survival rule을 테스트로 관찰했다.
known macroRangeContract storage design은 후보 shape와 영향표를 문서화했다.
phase gate closeout은 production storage/schema readiness가 아직 아니라고 닫았다.
```

아직 고정하지 않은 것:

```text
production normalizeMacroRangeContractForSnapshot 함수
normalizeGoalSnapshot 연결 방식
known macroRangeContract 저장 여부
macroRangeBasisSignature 저장 여부
backup/restore known contract roundtrip
Recent range-aware handling
range score production version
```

따라서 이번 단계는 구현이 아니라 normalizer가 열리기 전 필요한 계약을 정리한다.

## 2. normalizer의 역할 후보

future 후보 함수명:

```text
normalizeMacroRangeContractForSnapshot(rawContract, fixedSnapshot, options)
```

역할:

```text
macroRangeContract 후보를 absent / valid / invalid / unsupported로 분류한다.
known allowlist field만 normalizedContract 후보로 만든다.
fixed snapshot survival을 보장한다.
forbidden score/debug/copy/evidence field를 차단한다.
future signature / backup / Recent / score policy가 판단할 status를 제공한다.
```

비역할:

```text
fixed target macro를 재계산하지 않는다.
getDailyAdherenceScore를 호출하지 않는다.
candidateScorePreview를 만들지 않는다.
backup payload fragment를 만들지 않는다.
Recent component counts를 만들지 않는다.
UI copy를 만들지 않는다.
DailyCoach action을 만들지 않는다.
```

## 3. 입력 후보

후보 입력:

```text
rawContract
fixedSnapshot
options
```

`fixedSnapshot` 조건:

```text
현재 normalizeGoalSnapshot fixed contract로 valid해야 한다.
targetCal / protein / carbs / fat을 유지해야 한다.
macroRangeContract가 없더라도 old fixed record로 정상 동작해야 한다.
```

`rawContract` 후보 section:

```text
version
basisSource
targetCal
protein
fat
carbs
dataQuality
scoring
```

`options` 후보:

```text
mode: "storage_design_candidate" | "production_candidate"
allowPersistence: false
targetCalTolerance: 0.01
unknownFieldPolicy: "prune_with_warning" | "reject"
unsupportedVersionPolicy: "reject" | "prune" | "hold_for_future"
```

현재 판단:

```text
allowPersistence는 이번 단계에서 항상 false다.
```

## 4. output 후보

후보 output:

```js
{
  status: "absent" | "valid" | "invalid" | "unsupported",
  normalizedContract: object | null,
  reasons: string[],
  warnings: string[],
  fixedSnapshotImpact: "none",
  persisted: false,
  uiVisible: false,
  appliesToSavedRecord: false,
  storageApproved: false,
  scoreVersionChanged: false
}
```

output 원칙:

```text
status가 invalid여도 fixedSnapshotImpact는 none이다.
normalizedContract는 output 후보일 뿐 goalSnapshot 저장 field가 아니다.
warnings는 저장 계약 field가 아니다.
storageApproved는 이번 단계에서 false다.
```

금지 output:

```text
score
adherenceScore
finalScore
candidateScorePreview
scoreDeltaPreview
backupPayloadFragment
recentBasisSummary
uiCopy
coachAction
```

## 5. valid 조건 후보

valid 후보 조건:

```text
rawContract는 plain object다.
version은 supported known version이다.
targetCal은 finite number다.
targetCal은 fixedSnapshot.targetCal과 0.01 이내 일치한다.
protein.targetG / minAcceptableG / upperReviewG는 finite non-negative다.
protein.minAcceptableG <= protein.targetG <= protein.upperReviewG다.
protein.mode / basis는 allowlist 안에 있다.
fat numeric fields는 finite non-negative다.
fat physiologicalFloorG <= actionFloorG <= recommendedMinG <= recommendedMaxG <= flexibleMaxG다.
fat scoreLowThresholdG <= scoreHighThresholdG다.
fat coachWarnLowThresholdG <= coachWarnHighThresholdG다.
carbs numeric fields는 finite non-negative다.
carbs.trainingSupportMinG <= carbs.flexibleMaxG다.
carbs.recommendedMinG <= carbs.recommendedMaxG <= carbs.flexibleMaxG다.
carbs.derivedFromFatRange === true다.
dataQuality enum은 allowlist 안에 있다.
scoring.appliesToSavedRecord === false다.
forbidden field가 없다.
```

주의:

```text
valid는 "형태가 해석 가능하다"는 뜻이다.
valid는 "저장 승인"이 아니다.
valid는 "range score production 승인"이 아니다.
```

## 6. invalid 조건 후보

invalid:

```text
targetCal mismatch
required section missing
required number missing
NaN / Infinity / negative number
protein order violation
fat order violation
carb order violation
unknown enum under reject policy
candidateScorePreview present
scoreDeltaPreview present
rangeScoreCandidate present
scoring.appliesToSavedRecord === true
debug / copy / evidence field present under forbidden path
```

invalid 처리:

```text
normalizedContract: null
fixedSnapshotImpact: "none"
reason code를 남김
fixed snapshot은 계속 사용 가능
```

금지:

```text
invalid contract 때문에 normalizeGoalSnapshot 전체를 null로 만들기
invalid contract를 unknown field처럼 조용히 저장하기
invalid contract를 backup payload에 pass-through하기
```

## 7. unsupported 조건 후보

unsupported:

```text
version unknown
policyVersion unknown
future section required by unknown version
known app cannot interpret basisSource
future app에서 생성한 contract로 보이나 현재 app이 roundtrip policy를 모름
```

unsupported 처리 선택지:

```text
A. prune
B. reject with reason
C. hold_for_future without production use
```

현재 판단:

```text
production storage가 열리기 전에는 unsupported도 저장하지 않는다.
future backup/restore policy가 닫히기 전 pass-through를 허용하지 않는다.
```

## 8. unknown field policy

unknown field 선택지:

```text
prune_with_warning
reject
preserve_for_roundtrip
```

현재 권장:

```text
normalizer design 단계에서는 prune_with_warning 또는 reject만 검토한다.
preserve_for_roundtrip은 backup/restore schema gate가 닫힌 뒤에만 검토한다.
```

이유:

```text
unknown field 보존은 forward compatibility 장점이 있지만,
signature / backup diff / privacy-like debug leakage / future score preview persistence 위험이 있다.
```

## 9. forbidden field policy

아래는 unknown이 아니라 forbidden으로 본다.

```text
candidateScorePreview
scoreDeltaPreview
rangeScoreCandidate
reportOnly
uiVisible
persisted
productionScoreChanged
scoreVersionChanged
debugReason
explanation
rawExternalEvidence
uiCopy
warnings
normalizedContract
```

처리:

```text
invalid
normalizedContract: null
fixedSnapshotImpact: none
저장 금지
```

이유:

```text
candidate score와 debug/copy/evidence field는 저장 계약이 아니다.
이 값들이 저장되면 user-facing score, backup conflict, signature diff로 오해될 수 있다.
```

## 10. fixed snapshot survival rule

핵심 규칙:

```text
macroRangeContract는 optional이다.
macroRangeContract가 absent / invalid / unsupported여도 fixed snapshot은 살아야 한다.
```

normalizer가 지켜야 할 것:

```text
fixedSnapshot normalize와 contract normalize를 분리한다.
contract normalize 실패가 fixedSnapshot null로 이어지지 않는다.
fixedSnapshot targetCal / protein / carbs / fat은 contract 때문에 바뀌지 않는다.
old fixed records는 contract 없이 정상이다.
```

실패 의미:

```text
old Records와 old backup이 future range field 때문에 깨질 수 있다.
```

## 11. normalizeGoalSnapshot 연결 후보

현재:

```text
normalizeGoalSnapshot은 fixed allowlist만 보존한다.
macroRangeContract는 prune된다.
```

future 연결 후보:

```text
1. normalizeGoalSnapshot에서 fixed snapshot을 먼저 normalize한다.
2. fixed snapshot이 null이면 전체 null.
3. fixed snapshot이 valid이면 raw macroRangeContract를 별도 normalizer에 넘긴다.
4. normalizer status가 valid이고 storage gate가 승인된 경우에만 optional field를 붙인다.
5. invalid / unsupported이면 fixed snapshot만 유지하거나 explicit internal status를 별도 처리한다.
```

현재 판단:

```text
이 연결은 아직 구현하지 않는다.
storage gate와 backup/restore policy 전에는 optional field를 붙이지 않는다.
```

## 12. signature와의 관계

현재:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS fixed-only
range field signature 제외
macroRangeBasisSignature 저장 없음
```

future 후보:

```text
normalizer valid output -> separate basis signature candidate input
invalid / unsupported output -> no basis signature
forbidden field present -> no basis signature
```

아직 금지:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS 변경
getGoalSnapshotSignature 변경
Records 무변경 저장 로직 변경
macroRangeBasisSignature 저장
```

필수 분리:

```text
fixed snapshot signature 안정성
range basis signature 후보 안정성
```

## 13. backup / restore와의 관계

현재:

```text
backup/restore는 fixed snapshot normalize path를 따른다.
unknown range field는 no-leak이다.
smart restore meals-only는 local snapshot basis를 유지한다.
```

future normalizer가 열리면 필요한 결정:

```text
valid contract를 full backup payload에 포함할지
records-only import에서 valid contract를 보존할지
unsupported contract를 prune / reject / hold할지
smart restore replace에서 backup range basis를 적용할지
smart restore meals-only에서 basis conflict를 어떻게 표시할지
conflict preview에 basisConflict를 보여줄지
```

현재 금지:

```text
backup schema 변경
restore policy 변경
conflict preview 변경
macroRangeContract roundtrip 구현
```

## 14. Recent와의 관계

normalizer valid output이 생기면 Recent는 basis 혼합을 고려해야 한다.

필요한 future signal 후보:

```text
rangeContractStatus: absent | valid | invalid | unsupported
snapshotBasis: fixed_v8 | range_contract_candidate | range_contract_storable
basisSignatureStatus: absent | candidate | stored | invalid | unsupported
```

현재 금지:

```text

getRecentContextComponentCounts 변경
fixed/range basis를 같은 fatLowDays로 합치기
Recent UI basis warning 추가

```

원칙:

```text
old fixed records를 range 기준으로 조용히 재해석하지 않는다.
```

## 17. false positive 방지

정상:

```text
이번 문서에서 index.html이 바뀌지 않음
production normalizer가 아직 없음
normalizeGoalSnapshot이 아직 fixed-only임
macroRangeContract가 아직 저장되지 않음
backup payload가 아직 range field를 보존하지 않음
Recent가 아직 fixed basis로 집계됨
score version이 유지됨
```

blocker:

```text
normalizer design이라는 이름으로 production 함수를 추가함
normalizeGoalSnapshot allowlist를 바꿈
macroRangeContract를 goalSnapshot에 저장함
GOAL_SNAPSHOT_SIGNATURE_KEYS를 바꿈
backup payload에 range field를 넣음
candidateScorePreview를 숫자화함
Recent aggregation을 바꿈
DailyCoach action을 바꿈
```

## 18. 아직 결정하지 않은 것

미결정:

```text
1. production normalizer 함수명.
2. supported version 이름.
3. unknown field를 prune할지 reject할지.
4. unsupported version을 prune / reject / hold 중 무엇으로 둘지.
5. dataQuality section을 required로 둘지 optional로 둘지.
6. basisSource section을 required로 둘지 optional로 둘지.
7. valid normalizedContract를 언제 저장 가능하게 볼지.
8. invalid / unsupported reason을 저장하지 않고 어디까지 관찰할지.
9. normalizer output을 signature candidate와 어떻게 연결할지.
10. storage gate 이전에 production normalizer만 먼저 열 수 있는지.
```

현재 보수적 판단:

```text
storage gate 이전에 production normalizer만 먼저 여는 것도 위험하다.
normalizer output이 생기는 순간 저장/backup/Recent/score와 연결될 오해가 생긴다.
```
