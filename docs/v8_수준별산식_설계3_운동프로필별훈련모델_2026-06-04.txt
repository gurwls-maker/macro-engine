## 0. 이번 단계의 판단 원칙

이미지의 3단계는 "운동 프로필별 훈련 모델 설계"다.
하지만 이 문구를 그대로 따라가서 곧바로 production 탄단지 산식을 바꾸면 또 과대완료가 된다.

이번 단계에서 실제로 해야 하는 일은 다음이다.

1. 현재 앱의 실제 운동 프로필 / 루틴 / 세션 정의를 읽는다.
2. 각 세션이 사용자 관점에서 어떤 훈련 맥락인지 정리한다.
3. 그 맥락을 다음 수준별 context 설계가 받을 수 있는 형태로 고정한다.
4. production targetCal / protein / fat / carbs는 아직 바꾸지 않는다.
5. `performanceLevel` shortcut selector를 만들지 않는다.

따라서 이번 단계의 결과는 production formula가 아니라 report-only training model contract다.

## 1. 직접 확인한 현재 코드 원리

### 1-1. 프로필은 5개다

`EXERCISE_PROFILE_OPTIONS` 기준:

- bodybuilding
- powerbuilding
- strength
- running
- mixed

### 1-2. 보디빌딩은 기존 루틴 스킴을 유지한다

`ROUTINE_SCHEMES` 기준:

- PPL-UL
- 2분할
- 3분할
- 4분할
- 5분할

보디빌딩은 이 루틴/세션을 없애고 "기본 보디빌딩 세션 1개"로 통과시키면 안 된다.
보디빌딩의 실제 사용자 의미는 split routine/session 선택에 있다.

### 1-3. 비보디빌딩은 전용 routine plan/session을 가진다

`PROFILE_ROUTINE_PLANS` 기준:

- powerbuilding
  - upper/lower main
  - PPL powerbuilding
- strength
  - heavy/volume
  - technique/deload
- running
  - base running
  - speed/recovery
- mixed
  - balanced
  - circuit/recovery

이 값들은 더 이상 report-only 껍데기가 아니라 현재 앱의 routine/session 경로와 계산 snapshot으로 이어지는 실제 input path다.
다만 이 input path가 존재한다는 것이 "수준별 production macro formula가 완성됐다"는 뜻은 아니다.

### 1-4. 고급설정 OFF / ON 의미

고급설정 OFF:

- 화면상 visible routine/session은 REST / PUSH 단순 흐름이다.
- 보디빌딩 OFF 훈련은 기존 simple training baseline인 xw 0.70을 유지한다.
- 비보디빌딩 OFF도 화면은 REST / PUSH로 단순하지만, 내부 훈련 모델은 해당 프로필의 기본 session을 사용한다.
- 이 구조는 "비보디빌딩도 고급설정 OFF에서 아무 차이가 없다"는 뜻이 아니다.
- 반대로, OFF에서 고급 세션을 모두 노출시키면 고급설정의 의미가 사라진다.

고급설정 ON:

- 프로필별 상세 routine plan/session을 사용자가 직접 고른다.
- session xw, defaultWeightDuration, cardioDefaults가 세션별로 반영된다.
- 러닝과 mixed의 cardio-owned 세션은 weightDuration이 0이어도 휴식으로 해석하면 안 된다.

### 1-5. REST는 공통 휴식 세션이다

이번 구현 중 첫 검증 실패에서 중요한 판단을 바로잡았다.

`BODYBUILDING_PROFILE_SESSION_BY_ROUTINE` 안에 `REST`가 들어있지만, REST는 보디빌딩 fallback이 아니라 모든 프로필의 공통 휴식 세션이다.
따라서 "비보디빌딩이 보디빌딩 토큰을 쓰면 안 된다"는 guard에서 REST는 제외해야 한다.

이 예외를 문서에 남기는 이유:

- 무조건 코드 테이블 이름만 보고 판단하면 REST를 보디빌딩 fallback으로 오해할 수 있다.
- 실제 사용자 의미에서는 REST가 공통 휴식이다.
- 러닝 REST와 보디빌딩 REST는 모두 휴식이지만, 러닝 tempo/interval/long run은 절대 보디빌딩 REST/PUSH로 접으면 안 된다.

## 2. 훈련 모델 dimension

이번 단계에서 `profileTrainingModelDesign`이 고정하는 dimension은 6개다.

1. resistanceLoad
   - 루틴 세션의 xw, weightDuration, 세션 타입이 만드는 저항운동 부하
   - running은 기본적으로 none
   - mixed는 hybrid resistance 또는 supporting resistance가 될 수 있다.

2. cardioLoad
   - cardioDefaults와 Today cardio fields가 만드는 유산소 부하
   - running은 primary cardio
   - mixed는 integrated cardio 또는 primary cardio in hybrid
   - 보디빌딩/파워빌딩/스트렝스는 기본적으로 supplemental 또는 none

3. recoveryNeed
   - 세션 후 회복 압력
   - strength heavy, powerbuilding lower, running interval/long은 high 가능
   - strength deload, running recovery, mixed recovery는 low

4. fuelNeed
   - 세션 수행과 회복을 위해 필요한 연료 맥락
   - 러닝 tempo/interval/long은 high
   - mixed strength+cardio/cardio-focus/circuit은 moderate_high
   - deload/recovery/rest는 low 또는 none

5. targetReliefRisk
   - 목표 칼로리가 너무 낮을 때 target relief 검토가 필요한지
   - 이 값은 자동 targetCal 변경이 아니라 다음 context 설계의 review signal이다.

6. levelSignals
   - weeklyTrainingDays
   - weightDuration
   - intensityOverride
   - cardioDuration
   - cardioSpeed
   - recent14RecordCoverage
   - recent28Trend
   - bodyCompReliability

`levelSignals`는 `performanceLevel`을 계산할 재료다.
이번 단계는 standalone beginner/normal/advanced selector를 승인하지 않는다.

## 3. 프로필별 모델

### 3-1. bodybuilding

훈련 도메인:

- hypertrophy resistance

주요 입력:

- routineSession
- weightDuration
- intensityOverride

고급설정 OFF:

- REST/PUSH 단순 흐름
- PUSH xw 0.70 baseline 유지

고급설정 ON:

- 기존 보디빌딩 split routine/session 전체를 사용
- PPL-UL, 2분할, 3분할, 4분할, 5분할이 모두 의미를 가진다.

금지:

- 보디빌딩을 기본세션 1개로 대체
- `profileSession` shortcut만 넣고 실제 routine/session을 무시

### 3-2. powerbuilding

훈련 도메인:

- heavy resistance plus hypertrophy

주요 입력:

- upper/lower main
- PPL powerbuilding
- accessory volume

해석:

- lower/main lift는 recoveryNeed가 높을 수 있다.
- accessory volume은 main lift와 같은 회복 압력으로 보면 안 된다.

금지:

- 보디빌딩 PUSH/PULL/LEGS fallback으로 처리
- 파워빌딩 모든 날을 "하체/복합" 하나로 처리

### 3-3. strength

훈련 도메인:

- strength heavy / volume / technique / deload

해석:

- heavy upper/lower는 recoveryNeed가 높다.
- volume은 중간 이상 부하로 본다.
- technique/deload는 heavy session과 다르게 fuel/recovery pressure를 낮춰야 한다.

금지:

- heavy와 deload를 같은 세션으로 처리
- 스트렝스를 보디빌딩 lower fallback으로 처리

### 3-4. running

훈련 도메인:

- cardio endurance

주요 입력:

- running session type
- cardio duration
- cardio speed
- cardio incline

해석:

- easy
- tempo
- interval
- long
- recovery

중요:

- running non-rest session은 defaultWeightDuration이 0이어도 휴식이 아니다.
- running load는 cardio-owned다.
- weightDuration을 억지로 넣어 "운동일"임을 증명하려 하면 잘못이다.

금지:

- running tempo/interval/long을 REST로 접기
- running을 bodybuilding UPPER/PUSH fallback으로 처리

### 3-5. mixed

훈련 도메인:

- hybrid resistance + cardio

해석:

- mixed_strength_cardio는 resistanceLoad와 cardioLoad가 모두 있다.
- mixed_strength_focus는 resistance 중심이다.
- mixed_cardio_focus는 cardio 중심이다.
- mixed_circuit은 hybrid/circuit 맥락이다.
- mixed_recovery는 회복 맥락이다.

금지:

- mixed를 순수 보디빌딩 또는 순수 러닝으로 강제
- mixed session들을 "근력+유산소" 한 개로만 통과

## 5. 이번 단계에서 일부러 하지 않은 것

하지 않은 것:

- targetCal 변경
- protein/fat/carbs production 산식 변경
- performanceLevel selector 추가
- InBody reliability만으로 공격적인 산식 적용
- recent trend 자동 target 변경
- Coach/Score/Records snapshot/backup contract 변경
- 80,621,568,000개 full Cartesian 실행 완료 주장
- broad human UX review 완료 주장

이유:

- 이번 단계는 훈련 모델 설계 단계다.
- 숫자 산식과 사용자 설명 흐름은 다음 단계들에서 별도로 닫아야 한다.
- 내부 코드 모델을 외부 운동생리학 근거처럼 읽으면 안 된다.

## 6. 다음 단계로 넘기는 입력

다음 4단계 수준별 context 설계는 아래 입력을 받아야 한다.

- exerciseProfile
- routinePlan
- routineSession
- profileSession
- resistanceLoad
- cardioLoad
- recoveryNeed
- fuelNeed
- targetReliefRisk
- levelSignals
- weeklyTrainingDays
- recent14RecordCoverage
- recent28Trend
- bodyCompReliability

다음 단계에서 특히 조심할 점:

- 초보/일반/상급 같은 라벨을 shortcut selector로 만들지 말 것
- weeklyTrainingDays만으로 수준을 단정하지 말 것
- 기록 품질과 최근 지속성을 같이 볼 것
- 러닝/혼합을 보디빌딩 부하로 환산해서 편하게 처리하지 말 것
- target relief risk와 실제 targetCal 변경을 분리할 것
