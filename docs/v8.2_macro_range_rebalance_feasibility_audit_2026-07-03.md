## 2. 감사 범위

확인한 축:

1. 현재 target macro 계산 경로.
2. Today KPI / 남은 양 / 점수 표시가 고정 g 목표를 어떻게 쓰는지.
3. Records `goalSnapshot`이 target macro를 어떻게 저장하는지.
4. Recent / Coach가 저장 당시 목표와 현재 점수 근거를 어떻게 해석하는지.
5. macro range 후보가 기존 score version / backup / restore와 충돌하는지.

금지한 것:

- `targetCal`, protein, carbs, fat 계산식 변경.
- `ADHERENCE_SCORING_VERSION` 변경.
- `goalSnapshot` 구조 변경.
- backup / smart restore schema 변경.
- Today / Records / Recent / Coach UI 수정.

## 3. 현재 코드 흐름 요약

### 3.1 targetCal은 authoritative 값

`calculate()`는 입력값을 읽은 뒤 BMR, 활동, 업무, 운동, 유산소, time ownership, goal target rate를 거쳐 `targetCal`을 만든다.

중요한 기존 gate:

- diet에서는 `targetCal`이 소비량을 부당하게 넘지 않아야 한다.
- guide/activity basis는 `targetCal`을 다시 덮어쓰지 않는다.
- macro는 `targetCal` 안에서 배분되어야 한다.

이번 macro range 후보도 이 원칙을 유지해야 한다.

### 3.2 1차 macro 계산은 고정 target g을 만든다

핵심 함수:

```text
calculateMacroTargets()
getFatTargetContext()
applyPlausibilityLayer()
buildMacroAllocationInfo()
```

현재 흐름:

1. 단백질은 `GOALS[s.goal].proteinKg * proteinBasisWeight`에서 출발한다.
2. 지방은 `getFatTargetContext()`의 preferred target에서 출발한다.
3. 탄수화물은 `targetCal - protein kcal - fat kcal`의 residual이다.
4. `applyPlausibilityLayer()`가 단백질 / 탄수 / 지방 상한과 macro gap을 처리한다.
5. 결과는 여전히 `protein`, `carbs`, `fat` 단일 g 목표로 반환된다.

내부에는 이미 다음 값이 있다.

```text
fatTargetRange
fatPhysioFloorG
fatActionFloorG
fatPracticalCeilingG
macroSoftRange
macroAllocationInfo
```

하지만 이 값들은 현재 사용자-facing 계약의 중심이 아니다.

### 3.3 dual basis가 guide/activity 범위를 계산한다

핵심 함수:

```text
buildDualBasisCurrentContext()
buildGuideMacroRangeContext()
calculateGuideBasisMacrosWithinTarget()
calculateActivityBasisMacros()
buildDualBasisProduction()
syncDualBasisMacroSubLabelsWithProduction()
```

`buildGuideMacroRangeContext()`는 단백질 / 지방 / 탄수 g/kg 범위를 만든다.

예:

```text
protein: min / mid / max
fat: floor / min / mid / max / flexibleMax
carbs: min / mid / max
```

다만 최종 UI와 점수는 range 자체가 아니라 `production` macro 값으로 동기화된다.

따라서 현재 앱은 이미 range context를 갖고 있지만, 사용자는 최종적으로 다음처럼 본다.

```text
목표 칼로리: 고정 kcal
단백질: 고정 g
탄수화물: 고정 g
지방: 고정 g
```

### 3.4 external macro active application이 최종 target macro를 덮는다

핵심 함수:

```text
buildExternalMacroActiveProductionApplication()
```

이 함수는 `selectedTargetCal`을 유지하면서 외부 매크로 정책 후보에서 protein / fat / carbs를 다시 계산한다.

조건이 맞으면:

```text
externalMacroPolicyApplied: true
productionFormulaChanged: true
activeProductionApplied: true
```

즉 현재 production macro는 단순히 `calculateMacroTargets()` 결과가 아니라:

```text
calculateMacroTargets
-> dualBasis
-> externalMacroActiveProductionApplication
-> final result protein / carbs / fat
```

순서로 봐야 한다.

### 3.5 Today 남은 양과 초과 판단은 고정 target g 기준이다

핵심 함수:

```text
getMacroBalanceForDate()
getTodayMacroBalance()
isMeaningfulTodayOver()
renderTodayConsumptionPanel()
renderTodayRemainingPanel()
```

현재 계약:

```text
remaining.protein = target.protein - consumed.protein
remaining.carbs = target.carbs - consumed.carbs
remaining.fat = target.fat - consumed.fat
exceeded.carbs = consumed.carbs > target.carbs + deadzone
exceeded.fat = consumed.fat > target.fat + deadzone
```

따라서 지방 target이 124.8g이고 현재 7.7g이면, UI/계산 상태는 `지방 117.1g 남음` 식으로 해석될 수 있다.

이것이 `macro_range_rebalance_note`가 지적한 사용자 오해 지점이다.

### 3.6 score는 고정 target 대비 component score다

핵심 함수:

```text
getDailyAdherenceScore()
getMacroComponentScore()
buildMacroReason()
renderAdherenceMacroCards()
```

현재 score weight:

```text
protein: 0.40
kcal: 0.30
carbs: 0.20
fat: 0.10
```

현재 scoring version:

```text
ADHERENCE_SCORING_VERSION = v6.1_alcohol_penalty_v2
```

`buildMacroReason()`은 다음과 같이 말한다.

```text
탄수화물이 많아 오늘 점수에서 깎였습니다.
지방이 낮게 기록됐지만, 하루 단독으로 강한 행동을 정하지 않습니다.
```

즉 지방 낮음은 copy에서 행동을 약하게 처리하지만, 점수의 fat component는 여전히 고정 target 대비 낮게 계산된다.

### 3.7 Records goalSnapshot은 고정 macro만 저장한다

핵심 함수:

```text
buildGoalSnapshotFromCurrentState()
normalizeGoalSnapshot()
GOAL_SNAPSHOT_SIGNATURE_KEYS
getGoalSnapshotTarget()
getMacroBalanceForDate()
```

현재 snapshot 핵심 필드:

```text
targetCal
protein
carbs
fat
mode
exerciseManagementMode
goal
body values
activity / work / sleep / exercise basis
profileMacroCandidate metadata
```

현재 snapshot에는 다음이 없다.

```text
macroTargetVersion
proteinTargetMode
fatRange
carbRange
rangeScoringVersion
rangeInterpretationPolicy
```

따라서 범위 기반 macro를 도입하면 기존 Records를 어떤 기준으로 해석할지 먼저 정해야 한다.

### 3.8 Recent는 저장 당시 고정 target을 비율로 해석한다

핵심 함수:

```text
summarizeRecordForRecentContext()
getRecentContextComponentCounts()
getRecentFlowMacroMetrics()
getRecentFlowReferenceAnalysis()
```

현재 Recent context는 snapshot의 고정 목표를 읽는다.

```text
targetProtein
targetCarbs
targetFat
```

반복 패턴 판단:

```text
proteinLowDays: item.protein < targetProtein * 0.90
carbLowTrainingDays / carbLowRestDays: item.carbs < targetCarbs * 0.85
fatLowDays: item.fat < targetFat * 0.75
fatHighDays: item.fat > targetFat * 1.25
```

range 기반으로 바꾸면 Recent는 `targetFat` 대신 `fatMin/fatMax`, `targetCarbs` 대신 `carbMin/carbMax` 또는 `todayCoachingRange`를 읽어야 한다.

## 4. 재현 관찰

### 4.1 기본 lean_bulk target 예시

현재 기본 상태에서 관찰한 값:

```text
goal: lean_bulk
selectedMacroBasis: guide
targetCal: 4012.7 kcal
protein: 127.5 g
carbs: 594.9 g
fat: 124.8 g
pctProtein: 13
pctCarb: 59
pctFat: 28
externalMacroPolicyApplied: true
```

동시에 내부 guide range는 다음처럼 잡힌다.

```text
fat range: 52.5 ~ 93.75 g
fat flexibleMax: 150 g
fatTargetPreferredG: 124.8 g
```

관찰:

```text
현재 최종 fat target 124.8g은 guide range max 93.75g보다 높다.
이는 fat flexibleMax 안에서 external macro active application이 targetCal을 닫기 위해 지방을 올린 결과로 보인다.
따라서 range를 사용자-facing으로 올리려면 "권장 범위", "실천 상단", "target fill"의 의미를 분리해야 한다.
```

### 4.2 지방 낮음 + 탄수 보충 full-day 예시

설정:

```text
targetCal과 protein은 거의 맞춤
fat은 42 g
남은 kcal은 carbs로 채움
```

관찰:

```text
consumed kcal: 4041.5 / target 4041.5
consumed protein: 127.5 / target 127.5
consumed carbs: 788.4 / target 600.1
consumed fat: 42.0 / target 125.7
exceeded carbs: true
exceeded fat: false
score: 81
label: 우수
```

component score:

```text
protein: 1.000
kcal: 1.000
carbs: 0.415
fat: 0.237
```

score reason:

```text
탄수화물이 많아 오늘 점수에서 깎였습니다.
지방이 낮게 기록됐지만, 하루 단독으로 강한 행동을 정하지 않습니다.
```

판단:

```text
총칼로리와 단백질을 맞춘 날도 탄수/지방 교환을 고정 target 기준으로 감점한다.
따라서 "지방은 반드시 채우는 목표가 아니다"라고 copy만 바꾸면 score와 말이 어긋난다.
```

### 4.3 하루 중간 1끼 예시

설정:

```text
carbs 150.3 g
protein 102.2 g
fat 7.7 g
totalKcal 1079.3 kcal
```

관찰:

```text
remaining protein: 25.3 g
remaining carbs: 449.8 g
remaining fat: 118.0 g
score status: pending
score reason: 기록 부족으로 부족/초과 단정 보류
```

판단:

```text
score는 pending으로 방어하지만, remaining fat은 여전히 고정 target 기준으로 크게 남는다.
이 값이 사용자에게 "지방을 그만큼 더 먹어야 한다"로 읽힐 수 있다.
```

## 5. 핵심 발견

### 발견 1. 현재 앱은 이미 range context를 갖지만 사용자 계약은 고정 g이다

`dualBasis.rangeContext`, `fatTargetRange`, `macroSoftRange`는 존재한다.

하지만 다음 계약은 고정 g이다.

- Today KPI.
- Today remaining.
- score component.
- Coach signal.
- Records `goalSnapshot`.
- Recent component counts.

따라서 range rebalance는 새 데이터 구조를 추가하는 작업이라기보다, 이미 흩어진 range context를 어떤 사용자-facing 계약으로 승격할지 결정하는 작업이다.

### 발견 2. copy-only 보정은 부족하다

copy만 바꿔서 `지방을 억지로 채울 필요는 없습니다`라고 말할 수는 있다.

하지만 현재 점수는 계속:

```text
fat component low
carbs component high
remaining fat large
```

를 만들 수 있다.

copy-only는 단기 완화는 가능하지만 제품 의미를 닫지 못한다.

### 발견 3. score version 변경 없이 구현하면 과거 기록 해석이 섞인다

현재 자동 점수 버전은 `v6.1_alcohol_penalty_v2`이고, 많은 테스트가 이 값을 보호한다.

range 기반 점수는 아래 중 하나가 필요하다.

```text
새 scoring version
old fixed-target score와 new range score 병행 표시
기존 Records는 저장 당시 fixed-target score로 유지
새 Records부터 range scoring 적용
```

기존 version 그대로 점수 의미만 바꾸면 Records / Recent / backup 해석이 흐려진다.

### 발견 4. goalSnapshot schema 확장이 필요할 가능성이 높다

현재 snapshot은 `targetCal`, `protein`, `carbs`, `fat`만 target으로 보존한다.

range 기반으로 가려면 최소 후보는 다음이다.

```text
macroTargetVersion
proteinTargetG
proteinTargetMode
fatRangeMinG
fatRangeMaxG
fatActionFloorG
fatFlexibleMaxG
carbRangeMinG
carbRangeMaxG
macroRangeBasis
scoringVersion
```

다만 schema 확장은 backup / smart restore / Records diff / goalSnapshot signature 테스트를 모두 건드린다.

### 발견 5. Recent 3일/7일 보정은 가장 늦게 열어야 한다

현재 Recent는 고정 target 대비 0.85 / 0.75 / 1.25 같은 비율로 반복 패턴을 본다.

range 기반 Recent는 다음을 구분해야 한다.

```text
기본 생리학적 범위
오늘 코칭 범위
저장 당시 target range
현재 정책으로 재해석한 range
```

이 구분 없이 최근 흐름 보정을 넣으면, 과거 Records가 매번 다른 기준으로 해석되는 것처럼 보일 수 있다.

## 6. 구현 분기 판단

| 분기 | 판단 | 이유 |
| --- | --- | --- |
| A. copy-only 완화 | 비추천 | remaining / score가 고정 g이면 copy와 실제 평가가 어긋난다. |
| B. display-only range 표시 | 제한적 가능 | 기존 `fatTargetRange`를 보여줄 수는 있지만 score와 goalSnapshot이 fixed target이면 혼선이 남는다. |
| C. v8.2 macro-policy 설계 | 추천 | target, display, score, snapshot을 한 계약으로 설계할 수 있다. 후속 설계 문서에서 외부 근거 / 앱 정책값 / 점수 / 저장 계약 분리를 시작했다. |
| D. 바로 구현 | 금지 | scoring version, snapshot, backup, Recent 호환이 닫히지 않았다. |
| E. 추가 외부 근거 조사 | 필요 | 지방 하한/상한, 상한 근처 반복 섭취 경고, 목표별 단백질 옵션은 근거 확인이 필요하다. |

## 7. 권장 다음 단계

다음은 구현이 아니라 설계 문서가 맞다.

```text
v8.2 macro range policy design
```

이 설계에서 먼저 결정할 것:

1. 단백질은 fixed target으로 유지할지, 하한/표준/상한 preset을 둘지.
2. 지방 range의 의미를 `생리학적 하한`, `행동 하한`, `권장 범위`, `실천 상단`으로 나눌지.
3. 탄수 range를 지방 range와 실시간 교환 가능하게 만들지.
4. Today remaining을 `남음`에서 `추천 범위까지` / `현재 가능 범위`로 바꿀지.
5. score를 fixed-target component score에서 range-aware score로 바꿀지.
6. `ADHERENCE_SCORING_VERSION`을 새로 둘지.
7. Records `goalSnapshot`에 range 필드를 추가할지.
8. 기존 Records는 저장 당시 fixed target으로 유지할지.
9. Recent는 old/new scoring basis를 어떻게 섞어 보여줄지.
10. backup / restore conflict preview에서 새 range field를 어떻게 경고할지.

## 8. 최소 안전 구현 순서 후보

아직 확정 구현 지시가 아니다.

가능한 안전 순서:

1. `macroRangePolicyDesign` 문서 작성.
2. 현재 fixed-target score와 range-aware score의 병행 계산 report-only helper 추가.
3. 대표 목표별 sample matrix에서 range score가 실제로 더 나은지 검증.
4. 새 `ADHERENCE_SCORING_VERSION` 후보를 report-only로 둔다.
5. `goalSnapshot` range field 후보를 normalize/backup 영향 없이 report-only로 시뮬레이션한다.
6. Today display prototype을 render-only 후보로 검토한다.
7. 그 후 production 적용 여부를 결정한다.

바로 production target / score / snapshot을 바꾸지 않는다.
