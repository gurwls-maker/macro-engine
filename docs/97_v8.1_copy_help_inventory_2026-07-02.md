# v8.1 설명문 / 도움말 inventory

작성일: 2026-07-02

이 문서는 v8.1 copy/help 작업을 바로 구현하기 전에, 현재 화면에 있는 사용자-facing 문구를 화면별로 분류하기 위한 inventory로 시작했다.

초기 inventory 작성 단계에서는 `index.html`의 문구를 수정하지 않았다. 이후 pass별 구현 결과는 각 화면 섹션과 누적 정리 항목에 반영한다.

## 1. 분류 기준

| 분류 | 의미 |
| --- | --- |
| 유지 | 현재 화면 위치와 문구가 기능 이해에 직접 필요하다. |
| 축약 | 의미는 유지하되 기본 화면에서는 더 짧게 만드는 후보. |
| `?` 도움말 이동 | 항상 보이는 본문보다 hover/focus 도움말이 적합한 후보. |
| 접힘 이동 | 계산 과정, 근거, 위험 설명처럼 필요하지만 기본 화면에는 무거운 후보. |
| 삭제 | 반복, 내부 설명, 이미 UI가 말해주는 내용이라 제거 후보. |
| 테스트 보호 필요 | 변경 시 기존 회귀 테스트 또는 새 copy guard가 필요하다. |

우선순위:

1. 기본 화면에서 사용자가 매번 보는 설명을 줄인다.
2. 저장, 삭제, 복원, 기준 갱신처럼 위험한 action 문구는 기본 화면 또는 modal 안에 남긴다.
3. 산식/기준 설명은 `?` 도움말이나 접힘 영역으로 보내되 의미는 보존한다.
4. Coach와 점수 문구는 기존 copy regression test가 많으므로 구현 전 테스트 보호 범위를 먼저 확인한다.
5. 중요한 기준 설명의 source of truth는 `?` 도움말 또는 `details`로 둔다. `title`은 보조/중복 힌트로만 허용한다.

## 2. 전체 판단

현재 앱은 이미 많은 문구가 `?`, `details`, modal, summary로 이동되어 있다. v8.1에서 할 일은 모든 문구를 짧게 만드는 것이 아니라, 다음 세 부류를 분리하는 것이다.

- 기본 화면에서 반복 노출되는 안내.
- 계산/산식/기준을 설명하는 참고 문구.
- 저장/복원/삭제/기준 갱신처럼 실수 방지에 필요한 문구.

초기 inventory 기준 첫 구현 후보였던 `Today 기본 화면/빠른 설정`, `Settings 기본 프로필/생활 패턴`, `Records 선택 날짜 설명`, `InBody 상단 설명`, `Recent 흐름 설명`은 pass1~9에서 비위험 copy/help 중심으로 처리했다.

## 2.1 pass1~9 누적 정리

2026-07-03 기준 pass1~9는 v8.1 copy/help 개선 범위 안에서 닫힌 작은 단위 작업이다. 산식, 저장 구조, `goalSnapshot`, backup/restore, smart restore, Records 저장 흐름은 변경하지 않았다.

닫힌 축:

- Today 기본 화면과 빠른 설정의 반복 설명 축약.
- Today KPI 기준 설명의 `?` 도움말 보강.
- Today 전체 입력 패널의 InBody/load/save hint 축약.
- Settings 기본값/프로필/생활 패턴 설명 축약과 Today 직접 수정값 우선순위 도움말 보존.
- InBody 상단 설명 축약과 원본 기록 / Today 적용 분리 보호.
- Recent hero, 3일/7일/fallback caption 축약과 설정 자동 변경 방지 copy 유지.
- Records 선택 날짜 비위험 안내 축약.
- Records quality state 중 한 끼 / 간단 기록 / 체중·메모만 있는 날 안내 축약.

아직 보류할 축:

- Records 상세 당시 계산 기준 수정 설명.
- Records 상세 식사 행 삭제 / 기록 삭제 confirm copy.
- Records 저장 / 복원 / 백업 warning / smart restore 문구.
- Backup replace, records-only, smart restore conflict copy.
- Coach/score dynamic copy 전반.
- macro range rebalance는 copy/help 후보가 아니라 제품 정책/산식/점수/저장 해석 후보로 별도 보류한다.
- onboarding start flow는 copy/help 후보가 아니라 첫 실행 UX / Settings 기본값 초기화 / Today 첫 계산 진입 후보로 별도 보류한다.

다음 판단:

- 바로 위험 action 문구로 들어가지 않는다.
- Records/Backup 위험 action을 열려면 별도 pass에서 현재 confirm/modal 흐름과 테스트 보호 범위를 먼저 확인한다.
- 위험 action 진입 전 `runRecordsDeleteActionPolishTests`, `runRecordDetailMealDeleteConfirmTests`, backup/restore 관련 테스트, mobile profile을 필수 보호선으로 둔다.
- `docs/v8.2_macro_range_rebalance_note_2026-07-02.md`는 고정 매크로 목표를 범위 기반 재배분/점수 구조로 바꾸는 미래 설계 메모이므로, copy/help pass와 섞지 않는다. 해당 메모는 별도 산식/점수/저장 해석 감사 단계에서 다루고, 현재 copy/help pass에서는 기존 고정 target macro 해석을 바꾸지 않는다.
- `docs/v8.2_onboarding_start_flow_note_2026-07-03.md`는 첫 실행 UX와 초기 Settings/Today 진입 흐름 후보이므로, copy/help pass와 섞지 않는다. 온보딩은 구현 전 별도 UX 설계 audit에서 다룬다.

현재 화면 구조:

전체 백업 restore 흐름:

- `buildFullBackupPayload()`는 Settings, Records, InBody, 유산소 프리셋, 저장한 식사, Today 계산값, 주간 운동 상태를 포함한다.
- `readDataManagementBackupFile()`은 파일을 읽은 뒤 바로 복원하지 않고 `queueDataManagementImport()`로 보낸다.
- `queueDataManagementImport()`는 `dataManagementState.pendingImport`를 만들고 confirm modal을 연다.
- preview 단계에서는 원본 `recordsState`, Settings, InBody, 유산소 프리셋, Today 계산값이 바뀌지 않는다.
- 사용자가 `전체 복원`을 누르면 `confirmDataManagementImport()`가 `restoreFullBackupData()`를 호출한다.
- `restoreFullBackupData()`는 현재 앱 상태를 백업 파일 기준으로 대체한다.
- 따라서 full restore copy는 `현재 브라우저 데이터를 백업 파일 기준으로 바꿉니다`, `자동으로 합치지 않습니다`, `현재 브라우저 데이터가 백업 기준으로 대체됩니다` 의미를 유지해야 한다.

records-only restore 흐름:

- `importRecordsJson()`도 파일을 바로 적용하지 않고 `queueDataManagementImport()`를 거쳐 confirm modal을 연다.
- records-only preview는 `기록 중심 파일입니다. 설정과 Today 계산값은 바꾸지 않습니다. 아직 적용되지 않았습니다.`라고 설명한다.
- 사용자가 `복원하기`를 눌러야 `confirmDataManagementImport()`가 `applyRecordsOnlyImportData()`를 호출한다.
- records-only import는 Records를 적용하되 Settings, Today 계산값, 유산소 프리셋은 유지한다.
- records-only payload에 InBody가 포함된 경우에만 InBody import를 처리하고, 포함되지 않은 경우 기존 InBody를 유지한다.
- 따라서 records-only copy는 full backup과 다르다는 점, 다른 기기에서 앱 상태 전체를 이어 쓰려면 전체 백업이 필요하다는 점을 유지해야 한다.

smart restore / conflict 흐름:

- full backup preview에서 `추가 기록만 가져오기`를 누르면 `startSmartRestoreImport()`가 preview를 확인한다.
- 같은 날짜 충돌이 없으면 `applySmartRestoreImportData(data, "keep")`로 추가 기록과 새 InBody만 보강한다.
- 같은 날짜 충돌이 있으면 `smartRestoreStep = "conflict"`로 바꾸고 `같은 날짜 기록이 있습니다.` modal을 연다.
- conflict modal은 같은 날짜별 현재/백업 식사 수, 체중 유무, 메모 유무를 보여준다.
- conflict mode는 `현재 기록 유지`, `백업 기록으로 교체`, `식사만 합치기` 세 가지다.
- `replace`는 Records conflict record를 백업 record로 교체하지만 Settings / Today 계산값 / 유산소 프리셋은 유지한다.
- `meals`는 식사만 합치고 완전 중복 식사는 제외한다.
- 따라서 smart restore copy는 full restore처럼 앱 전체를 바꾸는 것이 아니라 Records/InBody 보강 중심이라는 의미를 유지해야 한다.
