# v8.2 macro range backup/restore old-new fixture design

상태:

```text
문서 성격: backup/restore old-new fixture design
작업 범위: docs-only
구현 변경: 없음
index.html 변경: 없음
테스트 코드 추가: 없음
backup schema 변경: 없음
restore policy 변경: 없음
macroRangeContract 저장 schema 구현: 없음
macroRangeBasisSignature 저장: 없음
normalizeGoalSnapshot 변경: 없음
GOAL_SNAPSHOT_SIGNATURE_KEYS 변경: 없음
restore conflict preview 변경: 없음
Recent aggregation 변경: 없음
getDailyAdherenceScore 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
UI 노출: 없음
production macro range 구현: 금지
```

이 문서는 `v8.2_macro_range_signature_decision_tests_implementation_2026-07-03.md`의 다음 단계다.

중요:

```text
이번 문서는 backup schema 변경 지시서가 아니다.
이번 문서는 restore policy 변경 지시서가 아니다.
이번 문서는 macroRangeContract roundtrip을 production으로 열겠다는 결정 문서가 아니다.
```

목적:

```text
macro range 후보가 future known contract로 승격될 때
full backup, records-only import, smart restore keep/replace/meals가
old fixed basis와 new range basis를 어떻게 다뤄야 하는지 fixture와 assertion을 설계한다.
```

## 1. 현재 기준

현재 backup/restore 경로:

```text
full backup:
  buildFullBackupPayload()
  -> records: getSortedRecords()

full restore:
  normalizeFullBackupPayload()
  restoreFullBackupData()
  -> normalizeRecordsCollection()
  -> normalizeRecord()
  -> normalizeGoalSnapshot()

records-only import:
  applyRecordsOnlyImportData()
  -> normalizeRecordsCollection()

smart restore:
  applySmartRestoreImportData()
  -> normalizeRecordsCollection()
  -> mode별 merge / replace / keep
```

현재 range 관련 계약:

```text
unknown macroRangeContract는 normalizeGoalSnapshot에서 prune된다.
backup payload에 range helper output이 들어가지 않는다.
candidateScorePreview는 저장되지 않는다.
smart restore meals-only는 local snapshot basis를 유지한다.
Recent는 current fixed basis로만 component counts를 계산한다.
```

현재 정상으로 봐야 하는 것:

```text
backup이 unknown range field를 보존하지 않음
records-only import가 unknown macroRangeContract를 prune
smart restore meals-only가 backup snapshot basis를 가져오지 않음
restore conflict preview에 range basis 차이가 표시되지 않음
```

## 2. fixture 층위

fixture는 세 층으로 나눠야 한다.

### 2-1. old fixed fixture

의미:

```text
v8.0/v8.1 fixed target record
macroRangeContract 없음
macroRangeBasisSignature 없음
ADHERENCE_SCORING_VERSION = v6.1_alcohol_penalty_v2
```

필수 fields:

```text
goalSnapshot.targetCal
goalSnapshot.protein
goalSnapshot.carbs
goalSnapshot.fat
goalSnapshot.goal
goalSnapshot.exerciseManagementMode
meals
adherencePercent
adherenceScoringVersion
```

기대:

```text
full backup roundtrip 후 fixed snapshot 유지
records-only import 후 fixed snapshot 유지
smart restore keep mode에서 local fixed basis 유지
smart restore replace mode에서 backup fixed basis 적용
smart restore meals-only에서 local fixed basis 유지
```

### 2-2. current unknown range fixture

의미:

```text
현재 production이 알지 못하는 future range field가 raw backup / raw import에 섞인 상태
```

후보:

```js
goalSnapshot: {
  targetCal,
  protein,
  carbs,
  fat,
  macroRangeContract: { version: "macroRangePolicy.v0.reportOnly", ... },
  macroRangeBasisSignature: "candidate-only",
  rangeScoreCandidate: { candidateScorePreview: 92 }
}
```

현재 기대:

```text
normalizeGoalSnapshot 후 unknown field prune
fixed target fields 유지
candidateScorePreview 미저장
backup payload에 macroRangeContract / macroRangeBasisSignature / candidateScorePreview 없음
```

주의:

```text
이 fixture는 range field 보존 실패가 아니다.
현재 allowlist contract를 명확히 하는 fixture다.
```

### 2-3. future known range fixture candidate

의미:

```text
explicit normalizer / schema / signature / backup policy가 future 단계에서 승인된 뒤 테스트할 후보
```

현재 단계에서 기대하지 않는 것:

```text
known macroRangeContract roundtrip
macroRangeBasisSignature persistence
restore conflict preview range basis 표시
Recent range-aware aggregation
```

future 기대 후보:

```text
valid macroRangeContract roundtrip
invalid contract fallback
unsupported contract fallback
basis signature include/exclude/separate policy 명시
old/new mixed basis detection
```

## 3. full backup fixture 설계

### 3-1. old fixed full backup roundtrip

목적:

```text
기존 기록이 backup/restore 후 그대로 읽히는지 확인한다.
```

assertion 후보:

```text
payload records contains old fixed goalSnapshot
payload does not contain macroRangeContract
restore result has fixed targetCal / protein / carbs / fat
getGoalSnapshotSignature restored === getGoalSnapshotSignature normalized original
ADHERENCE_SCORING_VERSION unchanged
```

### 3-2. current unknown range full backup no-leak

목적:

```text
raw record에 unknown range field가 있어도 현재 backup payload에 range field가 새지 않는지 확인한다.
```

assertion 후보:

```text
normalizeRecordsCollection prunes unknown range field
buildFullBackupPayload does not contain macroRangeContract
buildFullBackupPayload does not contain macroRangeBasisSignature
buildFullBackupPayload does not contain candidateScorePreview
restore keeps fixed snapshot valid
```

### 3-3. future known range full backup roundtrip candidate

목적:

```text
future schema가 열릴 경우 필요한 roundtrip 테스트를 미리 정의한다.
```

future assertion 후보:

```text
valid macroRangeContract survives full backup roundtrip only after explicit normalizer
macroRangeBasisSignature survives only if separate basis signature policy is approved
invalid macroRangeContract does not null fixed snapshot
unsupported macroRangeContract status is explicit
candidateScorePreview never persists
```

현재 금지:

```text
이 fixture를 현재 통과시키기 위해 backup schema를 바꾸지 않는다.
```

## 4. records-only import fixture 설계

### 4-1. old fixed records-only import

assertion 후보:

```text
old fixed record imported
fixed goalSnapshot fields retained
adherenceScoringVersion retained or normalized by current contract
no macroRangeContract field introduced
```

### 4-2. current unknown range records-only import

assertion 후보:

```text
raw import record may include unknown macroRangeContract
applyRecordsOnlyImportData prunes unknown range field
fixed snapshot survives
candidateScorePreview not persisted
```

### 4-3. future known range records-only import candidate

future assertion 후보:

```text
valid known range contract survives only after explicit normalizer and backup policy approval
old fixed import and new range import can coexist without changing old basis
invalid range contract does not null fixed snapshot
unsupported version is marked for later handling, not silently accepted
```

현재 금지:

```text
records-only import에 range-aware policy를 추가하지 않는다.
```

## 5. smart restore fixture 설계

smart restore는 가장 위험하다.

이유:

```text
동일 날짜 local record와 backup record가 서로 다른 basis를 가질 수 있다.
mode에 따라 식사만 합치거나 기록을 교체할 수 있다.
meals-only mode에서 snapshot basis까지 바뀌면 사용자 기대와 다르다.
```

### 5-1. keep mode

현재/future 공통 원칙:

```text
local record가 유지된다.
backup record basis는 적용되지 않는다.
```

assertion 후보:

```text
local fixed snapshot retained
backup unknown/range field not introduced
local meals retained unless mode explicitly says otherwise
```

### 5-2. replace mode

현재 원칙:

```text
backup record가 record 전체를 교체한다.
현재 unknown range field는 normalize 과정에서 prune된다.
```

future 후보:

```text
known valid range contract가 승인된 schema라면 backup basis가 의도적으로 적용된다.
replace mode 결과는 backup basis가 적용된 상태임을 internal summary로 알 수 있어야 한다.
```

assertion 후보:

```text
replace mode applies backup fixed basis
unknown range field is pruned under current contract
future valid range basis applies only after explicit schema approval
```

### 5-3. meals-only mode

현재 원칙:

```text
backup meals만 local record에 병합한다.
local snapshot basis는 유지한다.
backup snapshot basis는 적용하지 않는다.
```

필수 assertion 후보:

```text
merged meal count changes
local targetCal / protein / carbs / fat unchanged
local fixed signature unchanged
backup macroRangeContract not introduced
backup macroRangeBasisSignature not introduced
candidateScorePreview not introduced
```

future 판단 필요:

```text
backup meal만 합쳤는데 backup basis가 더 최신이면 표시할지.
range basis conflict를 internal warning으로만 둘지.
user-facing conflict preview를 열지.
```

현재 금지:

```text
meals-only mode에서 snapshot basis conflict UI를 추가하지 않는다.
```

## 6. conflict preview fixture 후보

현재 backup conflict preview는 range basis를 모른다.

현재 정상:

```text
range basis 차이를 표시하지 않음
unknown range field는 prune
existing fixed conflict preview 유지
```

future 후보:

```text
basisConflict: none | fixed_vs_range | range_version_mismatch | invalid_range_contract | unsupported_range_contract
```

주의:

```text
conflict preview에 basis 차이를 보여주려면 user-facing 문구와 restore action 의미까지 설계해야 한다.
이번 단계에서는 절대 구현하지 않는다.
```

## 7. backup payload no-leak 기준

현재 payload에 있으면 실패:

```text
macroRangeContract
macroRangeBasisSignature
rangeScoreCandidate
candidateScorePreview
scoreDeltaPreview
normalizedContract
range basis debug reason
report-only helper output
```

future에도 저장 금지 후보:

```text
candidateScorePreview
scoreDeltaPreview
debugReason
explanation
rawExternalEvidence
uiCopy
```

future에 조건부 저장 후보:

```text
macroRangeContract
macroRangeBasisSignature
macroTargetBasisVersion
```

단, explicit normalizer / signature decision / backup roundtrip tests / restore conflict policy가 모두 닫힌 뒤에만 검토한다.

## 9. false positive 방지

정상:

```text
unknown range field가 backup/restore에서 prune됨
backup payload가 macroRangeContract를 보존하지 않음
records-only import가 candidateScorePreview를 보존하지 않음
smart restore meals-only가 local snapshot basis를 유지함
conflict preview에 range basis 차이가 표시되지 않음
```

blocker:

```text
backup schema를 바꿈
restore policy를 바꿈
macroRangeContract가 normalize 없이 살아남음
macroRangeBasisSignature가 payload에 들어감
candidateScorePreview가 payload나 restored record에 들어감
smart restore meals-only가 backup snapshot basis로 local snapshot을 교체함
replace mode가 교체 의도 없이 local/backup basis를 섞음
Recent aggregation이나 score version이 같이 바뀜
```

## 10. 아직 사용자 판단이 필요한 항목

아직 결정하지 않는다.

```text
1. known macroRangeContract를 backup payload에 저장할지.
2. macroRangeBasisSignature를 저장할지.
3. backup conflict preview에 basis conflict를 표시할지.
4. smart restore meals-only에서 basis conflict를 user-facing으로 보여줄지.
5. replace mode에서 range basis 적용을 어떻게 설명할지.
6. records-only import에서 unsupported range contract를 어떻게 보고할지.
```

## 12. 최종 판단

backup/restore old-new fixture design은 진행 가능하다.

하지만 아직:

```text
backup schema 변경 아님
restore policy 변경 아님
macroRangeContract roundtrip 구현 아님
macroRangeBasisSignature 저장 아님
conflict preview 변경 아님
production macro range 구현 아님
```

최종 결론:

```text
backup/restore fixture는 old fixed, current unknown range, future known range 후보를 분리해야 한다.
현재 단계에서는 unknown range field pruning과 no-leak이 정상이다.
future known range roundtrip은 explicit normalizer / signature / schema decision 후에만 검토한다.
```
