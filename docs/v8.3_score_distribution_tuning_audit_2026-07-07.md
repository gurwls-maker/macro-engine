## 3. 샘플링 방식

방법:
- repo 파일을 추가하지 않는 임시 Playwright evaluate로 `index.html?codexInternalTests=1`을 로드했다.
- 브라우저 안에서 `window.getMacroRangeProductionScoreSummary(...)`를 직접 호출했다.
- base target은 `2400kcal / protein 150g / carbs 300g / fat 70g / bodyWeight 75kg`로 두었다.
- 기본 training context는 `normal_resistance`다.
- 필요한 경우 `totalBurn`, `trainingContext`, `goal`을 sample별로 바꿨다.

코드 변경 없이 샘플링한 이유:
- 이번 작업의 전체 구현 대상은 score distribution audit이지 test surface 확장이 아니다.
- production helper가 이미 브라우저 window에서 호출 가능하다.
- 샘플 그룹 전체를 뽑는 데 새 permanent helper가 필요하지 않았다.
- 새 helper를 남기면 오히려 다음 작업자가 audit helper를 production helper처럼 오해할 수 있다.

분류 기준:
- PASS: current truth와 방향이 맞고, 다음 튜닝 없이 유지 가능.
- WATCH: 구조는 맞지만 점수 강도 또는 user-facing 해석을 다음 tuning decision에서 다시 볼 후보.
- FAIL: current truth와 충돌하거나, monotonicity / no plateau / null gate / axis separation이 깨짐.

## 4. 전체 요약

최종 판단:
- FAIL 없음.
- WATCH 있음.
- 따라서 이번 branch에서 anchor 값은 바꾸지 않는다.
- 다음은 `score distribution tuning decision docs-only`가 자연스럽다.

핵심 수용:
- target +1kcal는 near-zero다.
- alcohol 56g, 80g, 112g, 140g은 같은 bucket에 묶이지 않는다.
- alcohol kcal axis와 physiology risk axis가 분리되어 보인다.
- fat upper +1g은 near-zero다.
- fat 1000g은 fixed 감점에서 멈추지 않고 rawScore가 크게 내려간다.
- 운동은 finalScore 보너스가 아니라 carb context modifier로 작동한다.
- high-volume carb도 TDEE overload를 면제하지 않는다.
- missing macro는 null/blocked이며 sentinel 0이 아니다.

WATCH 후보:
- TDEE 2.0에서 finalScore가 0으로 clamp되지만 rawScore는 계속 악화된다.
- fat upper 3.0x에서 finalScore가 0 근처로 닫히는 강도는 다음 decision에서 다시 볼 만하다.
- UNKNOWN training context에서 450g carb가 98.54로 꽤 관대하다.
- missing TDEE full macro day가 100점으로 보일 수 있으므로 UI/storage를 열기 전 source visibility 정책을 다시 확인해야 한다.
- alcohol 112g은 25.43, 140g은 final 0으로 떨어진다. 방향은 맞지만 강도는 product tone 관점에서 다음 decision 후보로 둔다.
