# v8.3 implementation regression audit

작성일: 2026-07-07
기준 커밋: 47f9e00 v8.3 연속 점수 산식 구현
변경 성격: implementation QA / regression cleanup
구현 범위: minimal surface, complete model 유지

## 1. 내 판단

47f9e00은 유지한다.

v8.3 production score body는 `getV83ContinuousMacroScoreSummary(...)`이며,
`getMacroRangeProductionScoreSummary(...)`와 `getDailyAdherenceScore(...)`가 이 경로를 사용한다.

구조는 current truth와 맞다.

- fixed high/severe table은 production body가 아니다.
- final score는 continuous penalty 합산 후 마지막 0~100 clamp만 사용한다.
- alcohol은 kcal axis와 physiology risk axis로 분리된다.
- exercise는 score bonus가 아니라 carb context modifier다.
- TDEE overload는 target energy deviation과 별도 축이다.

그러나 구현 직후 QA에서 두 관성 리스크를 확인했다.

1. `runMacroRangeProductionScoreTransitionDecisionTests` 안에 v8.2 fixed-point helper가 남아 있었다.
2. v8.3 scoring context가 body weight와 totalBurn source를 명시하지 않았다.

이 둘은 production formula 자체를 되돌릴 문제는 아니지만, 다음 작업자가 v8.2 fixed table이나 hidden fallback을 현행 기준처럼 오해할 수 있는 리스크다.

## 2. 수정한 것

### 2-1. v8.2 fixed-point helper 격하

`runMacroRangeProductionScoreTransitionDecisionTests` 안의 fixed component table과 exact fixture는 삭제하지 않았다.
대신 historical helper로 격하했다.

이유:

- helper는 no-leak, blocked data, old fixed no-mutation, scoreDelta optional guard에 아직 쓸 수 있다.
- 하지만 helper가 `v8.3_anchor_continuous_macro_score_v1`을 반환하면 current production output처럼 보인다.

변경:

- `historicalFixedPointHelperVersion = "v8.2_fixed_component_transition_helper_historical"`
- `historicalFixedPointHelperRole = "historical_no_leak_and_old_record_guard_only"`
- `getMacroRangeTransitionScoreFromPoints(...)`는 current v8.3 version이 아니라 historical helper identity를 반환한다.
- new Records forward-only test는 fixed helper가 아니라 `getMacroRangeProductionScoreSummary(...)` 결과로 v8.3 record contract를 확인한다.

결론:

- v8.3 scoring correctness는 `runMacroRangeContinuousScoringTests`와 `runMacroRangeProductionScoreImplementationDecisionTests`가 담당한다.
- transition suite의 fixed table은 production target이 아니라 historical guard evidence다.

### 2-2. weight / TDEE source metadata 추가

v8.3 score는 g/kg 기반 axis와 TDEE overload axis를 사용한다.
따라서 body weight와 totalBurn이 어디에서 왔는지 숨기면 QA가 흐려진다.

변경:

- `getV83WeightContext(...)` 추가.
- `getV83TotalBurnContext(...)` 추가.
- `scoringContext.weightSource` 추가.
- `scoringContext.totalBurnSource` 추가.
- `scoringContext.tdeeAvailable` 추가.

weight source 우선순위:

1. `options.bodyWeightKg`
2. `balance.goalSnapshot.weight`
3. `currentResult.s.weight`
4. `targetProteinFallback`
5. `missing`

중요한 판단:

- saved goalSnapshot weight는 live currentResult weight보다 우선한다.
- targetProtein fallback은 숨겨진 authoritative source가 아니라 QA-visible fallback이다.
- totalBurn은 현재 goalSnapshot에 저장하지 않으므로, source를 명시하되 storage/schema는 열지 않는다.
- totalBurn missing은 `tdeeAvailable: false`로 보이게 하고, overload penalty를 발명하지 않는다.

## 3. 보강한 테스트

`runMacroRangeContinuousScoringTests`에 source metadata regression case를 추가했다.

검증:

- explicit option weight는 `weightSource: "options"`.
- saved goalSnapshot weight는 live currentResult weight보다 우선.
- live totalBurn은 `totalBurnSource: "currentResult.totalBurn"`.
- targetProtein fallback은 `weightSource: "targetProteinFallback"`로 노출.
- missing totalBurn은 `totalBurnSource: "missing"` / `tdeeAvailable: false`.
- missing totalBurn은 overload penalty를 조용히 만들어내지 않음.

`runMacroRangeProductionScoreTransitionDecisionTests`는 12 cases를 유지하되 의미를 정리했다.

검증:

- historical fixed helper는 v8.3 production version을 반환하지 않음.
- new Records forward-only contract는 v8.3 production summary를 사용.
- old fixed Records는 helper 평가와 무관하게 silent mutation 없음.
- scoreDeltaPreview는 optional audit-only이며 production transition blocker가 아님.
