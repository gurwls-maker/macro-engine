# V8 1단계 복구 감사

Date: 2026-06-03
Branch: `codex/v8-formula-audit`
Latest checked commit: `1641f49 v8.0-AS2 align today snapshot source values`

이 문서는 V8을 더 진행하기 전에, 지금까지의 작업이 실제 앱 production 동작인지, report-only 후보/감사인지, 문서/테스트/렌더 증거인지 분리하기 위한 1단계 복구 감사다.

## 3. 우회성 또는 오해 위험이 큰 항목

### F1. 다이어트 단백질 production 미해결

상태: 실제 미해결.

근거:
- `index.html`의 `getDualBasisProteinRange(goal)`는 diet에서 `{ min: 2.2, mid: 2.4, max: 2.6 }`을 반환한다.

위험:
- 문서상 "protein guard conflict resolved" 또는 "candidate-v2 productionReady"를 단백질 과다 해결로 읽으면 틀린다.
- 실제 사용자는 여전히 일반 diet 기본값에서 고단백을 본다.

다음 행동:

- 단순히 2.4를 2.0으로 낮추는 작업이 아니라, goal/profile/body-fat/LBM reliability/training load/fat floor/carb redistribution을 함께 설계해야 한다.

## 4. 1단계 결론

V8 자산은 남길 가치가 있다.

- runner / render analyzer / ledger / profile routine structure / snapshot path / candidate-v2 wiring은 실제로 유용한 기반이다.
- 이미 여러 단계가 report-only와 production을 분리하려고 노력한 흔적도 있다.

하지만 사용자가 우려한 "기준문서 통과용 우회" 위험도 실제로 존재했다.

- profileSession 중심 증거가 user-owned routine/session path를 대체하려 했던 전력이 있다.
- 단백질 과다 문제는 후보에서는 다뤄졌지만 production 일반 diet 산식에는 남아 있다.
- render/test 통과가 실제 사용성 완료로 과대해석될 수 있다.
- full Cartesian은 실행 완료가 아닌 도구/소량 evidence 상태다.

따라서 다음 작업은 새 feature stage가 아니라 다음 순서여야 한다.

1. production macro policy 2단계: diet protein/fat/carb 재설계와 실제 production 적용 여부 결정.
2. runner/status wording hardening: scoped production 적용과 full V8 completion이 섞이지 않게 명칭 정리.
3. profile/routine/session broad human UX review matrix.
4. full Cartesian closure strategy 또는 owner-approved 대체 gate 결정.

## 5. 다음 Codex 작업 금지선

- `proteinGuardConflictCount=0`을 diet protein over-resolved로 말하지 않는다.
- `productionTargetCalApplied=true`를 general diet macro production 완료로 말하지 않는다.
- render audit failed 0을 profile/routine/session UX 전체 완료로 말하지 않는다.
- full Cartesian tooling을 full Cartesian execution 완료로 말하지 않는다.
- `wholeStageEvidenceBoundaryAudit` pass를 "더 이상 shortcut 없음"으로 말하지 않는다.
- 다음 단계에서 기준문서가 앱 사용자 흐름과 충돌하면 기준문서를 고친다. 기준문서 통과를 위해 사용자 경로를 우회하지 않는다.

## 7. Follow-up: 2026-06-03 v8.0-AV

문서정리는 다음 작업 진입을 막지 않는 수준으로 닫는다. 이 문서의 1단계 감사 결론은 유지하되, AT에서 general diet production macro policy 복구가 이미 끝난 점을 반영해 다음 순서를 다시 고정한다.

다음 실제 작업은 기존 3단계다. 여기서 기존 3단계는 profile/routine/session broad human UX review matrix와 수준별 산식 범위 설계를 뜻한다. candidate-v2/scoped production status wording hardening은 이 3단계의 preflight로 처리하며, full V8 completion처럼 보이는 표현을 좁히는 목적을 넘겨 새 장시간 stage로 만들지 않는다.

금지선은 그대로 유지한다. AT의 `2.0g/kg` 복구를 수준별 산식 완료로 말하지 않고, candidate-v2 scoped production wiring을 모든 사용자/모든 운동 프로필의 전면 산식 승인으로 말하지 않는다. `/GOAL` 장시간 자동 재개 방식도 다시 살리지 않는다.

## 8. Follow-up: 2026-06-03 v8.0-AW

기존 3단계 보완 실행으로 `profileRoutineUserPathMatrix`를 추가했다. 이 matrix는 profile/routine/session 경로를 Settings/default Today path와 Today explicit override path로 나눠 `calculate()`와 Records `goalSnapshot`까지 확인한다.

확인한 핵심:

- 보디빌딩 advanced OFF 기본 경로는 단순 훈련/휴식이며 훈련 기본 강도는 주간 보정 전 `0.70`이다.
- AX에서 이 AW 해석은 바로잡혔다. 보디빌딩 advanced OFF는 사용자 화면과 계산 기본값 모두 기존 단순 훈련/휴식 흐름이며, 운동일 기본 강도는 주간 보정 전 `0.70`이다. `PPL-UL / PUSH 0.80` 같은 상세 루틴/세션 xw는 advanced ON 경로다.
- 비보디빌딩 advanced OFF도 화면에는 단순 `REST / PUSH`만 노출한다. 다만 단순 운동일의 내부 계산 기본값은 해당 프로필의 기본 세션을 적용해 xw, 웨이트 시간, 유산소 기본값을 잡는다. 비보디빌딩 상세 루틴/세션 선택은 advanced ON 경로다.
- AW는 수준별 산식을 구현하지 않았고, full human visual review나 full V8 completion도 닫지 않는다.

검증:

- `runV8ScenarioRunnerTests` = 1 suite / 30 cases / failed 0
- `runTodayQuickEditTests`, `runTodayCalculationOwnershipTests`, `runTodayRecordConfirmationTests` = 3 suites / 70 cases / failed 0
- core profile = 26 suites / 370 cases / failed 0
- AX re-verification: `runTodayCalculationOwnershipTests`, `runTodayQuickEditTests`, `runV8ScenarioRunnerTests` = 3 suites / 88 cases / failed 0; calibration profile = 14 suites / 138 cases / failed 0

## 9. Follow-up: 2026-06-04 v8.0-AY

AY is the fourth step of the manual recovery plan in this document.

Implementation:
