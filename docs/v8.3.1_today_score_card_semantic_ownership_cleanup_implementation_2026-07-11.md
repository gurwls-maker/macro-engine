# v8.3.1 Today score-card semantic ownership cleanup implementation

## 비개발자용 결과 설명

### 무엇을 바꿨는지

이전 Today 점수 화면은 탄수화물, 단백질, 지방, 총칼로리 네 항목을 보여 주면서도 실제 최종 점수에 들어가는 술의 생리적 부담과 비정상적으로 큰 섭취량 확인 감점을 완결되게 설명하지 못했다. 탄수화물과 지방의 조합이 함께 권장 범위를 벗어난 경우도 독립된 점수 근거로 분명하게 보이지 않았다.

반대로 점수 아래의 보조 문구와 `추가 확인` 영역은 점수 산식의 직접 근거뿐 아니라 DailyCoach의 행동 조언, 이전 기록, InBody 맥락, 고정 목표 기준의 단순 높고 낮음까지 섞어 가져왔다. 같은 열량 안에서 탄수화물과 지방을 정상적으로 교환한 경우에도 점수와 무관한 경고가 나올 수 있었고, 같은 내용이 Score와 Coach 양쪽에 중복될 수 있었다.

이번 구현은 Today 점수 화면을 다음처럼 정리했다.

- 탄수화물, 단백질, 지방, 총칼로리 네 항목은 계속 핵심 점수 근거로 보인다.
- `섭취량 확인`, `술 영향`, `탄수·지방 조합`은 실제 해당 감점이 0보다 클 때만 추가 점수 근거로 보인다.
- 세 추가 근거는 숨겨진 펼침 영역이 아니라 화면에 바로 보이는 목록이며, 순서는 `섭취량 확인 → 술 영향 → 탄수·지방 조합`으로 일정하다.
- 감점이 0이면 추가 항목을 만들지 않는다. 0보다 크지만 0.1점 미만이면 `감점 없음`으로 반올림하지 않고 `0.1점 미만 감점`이라고 표시한다.
- 술은 총칼로리에 포함되는 영향과 별개로, 기존 산식에 이미 있던 생리적 부담 감점만 `술 영향`에 표시한다.
- `섭취량 확인`은 기록이 틀렸다고 단정하지 않고, 일반적인 계산 범위를 크게 벗어났으니 기록값을 확인해 달라고 설명한다.
- 탄수화물과 지방의 개별 상태가 아니라 실제 joint-failure 감점이 있을 때만 `탄수·지방 조합`을 표시한다.

### 왜 바꿨는지

점수 화면의 역할은 “오늘 무엇을 하면 좋은가”를 다시 조언하는 것이 아니라, 화면에 보이는 최종 점수가 어떤 계산 근거에서 나왔는지 빠짐없이 설명하는 것이다. 행동 조언과 최근 맥락은 DailyCoach가 맡고, Score는 점수 계산 근거만 맡도록 책임을 분리해야 누락과 중복, 정상 교환 오경고를 함께 막을 수 있다.

### 실제 사용자 화면과 계산 결과가 어떻게 달라지는지

예를 들어 술의 생리적 부담 감점이 3.5점이고 탄수·지방 조합 감점은 0점이라면, 화면에는 `술 영향 · 3.5점 감점`만 추가로 보이고 `탄수·지방 조합`은 보이지 않는다. 데이터 확인 감점과 joint 감점도 같은 원칙을 따른다.

핵심 네 항목은 각각 별도 100점 점수로 바뀐 것이 아니다. 해당 항목에 속한 실제 감점 합계를 `감점 없음`, `0.1점 미만 감점`, `n.n점 감점`으로 보여 준다. 따라서 사용자가 보는 최종 점수 숫자는 이전과 같고, 그 숫자의 근거만 더 정직하게 보인다.

감점 원합계가 100점을 넘어서 최종 표시 점수가 0점이 되는 경우에는 원래의 0점 표시를 유지하면서 `감점 합계가 100점을 넘어 최종 점수는 0점으로 표시됐다`는 문맥을 함께 남긴다. 원감점을 100으로 잘라 근거를 숨기지 않는다.

### 정책·산식·데이터 해석이 바뀌었는지

바뀌지 않았다.

- scoring version은 계속 `v8.3_anchor_continuous_macro_score_v1`이다.
- `percent`, `rawScore`, 9개 `penaltyBreakdown` 계산, anchor, curve, adaptive target, 3일/7일 limiter를 바꾸지 않았다.
- 운동 보너스를 추가하지 않았고, 운동은 기존처럼 탄수화물 curve context에만 남는다.
- 술 감점을 새로 만들거나 v6.1 후처리를 되살리지 않았다.
- evidence model과 화면 렌더링은 점수 결과를 읽을 뿐, 계산 결과를 수정하지 않는다.

### 저장된 점수는 어떻게 처리하는지

현재 실제 저장 기록은 점수 정수, 출처, scoring version 등은 보존하지만 `penaltyBreakdown`, live `scoringContext`, compatibility contribution을 저장하지 않는다. `stored_auto_snapshot`은 상세 계산 snapshot이 아니라 저장된 정수 점수 상태다. 따라서 과거 저장 점수에 현재 식사나 현재 설정을 끼워 넣어 항목별 감점을 다시 추정하는 hidden recompute를 해서는 안 된다.

저장 점수에 breakdown이 없는 실제 경로에서는 다음처럼 처리한다.

- 저장된 점수 숫자는 그대로 보여 준다.
- 네 핵심 항목과 조건부 감점 항목을 새로 만들어 내지 않는다.
- live source warning도 붙이지 않는다.
- `저장 당시 점수입니다. 항목별 감점 내역은 함께 남아 있지 않습니다.`라고 설명한다.

이 제한을 풀기 위해 storage/schema/backup/old Records migration을 열지 않았다. 명시적인 breakdown이 전달된 경우에만 evidence adapter가 그 값을 읽을 수 있지만, 현재 저장 경로는 breakdown을 만들거나 복원하지 않는다.

### Score와 DailyCoach가 어떻게 분리됐는지

Score renderer는 이제 점수 결과 하나만 입력으로 받으며 DailyCoach 결과를 입력으로 받지 않는다. Coach 문구, 최근 기록, InBody, 다음 식사 행동, 기타 칼로리 슬롯, 식사 집중도 힌트는 점수 근거에 섞이지 않는다. Coach 결과만 바꿔도 Score DOM과 문구가 달라지지 않도록 테스트했다.

DailyCoach 자체의 행동 조언을 없앤 것은 아니다. 행동 조언은 Coach 화면에 그대로 남는다. Hero와 Coach 사이의 더 넓은 중복 정리, 탄수·지방 자동 조정의 원인·방향·누적 맥락 설명은 이후 DailyCoach semantic v2에서 다룬다.

### 최소로 제한한 표면과 이 기능 안에서 완결한 범위

최소 변경 표면은 Today 점수의 evidence adapter, 점수 renderer, 필요한 스타일, 관련 내부 테스트와 테스트 프로필 등록이다. 점수 계산 함수, Records 표시 정책, Settings, backup 형식, 저장 schema, old record migration, adaptive calculation, DailyCoach 생성 로직은 열지 않았다.

그러나 “화면만 조금 바꾼다”는 이유로 기능 내부 완결성을 줄이지 않았다. 이 기능 안에서는 9개 raw 감점 축이 정확히 한 번씩 매핑되는지, 그 합이 원래 감점 합과 일치하는지, 0점 clamp 전 원감점까지 보존되는지, stored breakdown 부재를 추정 없이 처리하는지, desktop/mobile과 접근성 구조가 유지되는지까지 닫았다.

### 사용자가 큰 틀에서 확인할 판단점

- 네 핵심 항목이 독립 점수처럼 오해되지 않고 “최종 점수에 반영된 감점”으로 읽히는가.
- 술, 섭취량 확인, 탄수·지방 조합이 실제 감점이 있을 때만 충분히 눈에 띄는가.
- 추가 근거가 너무 많아 화면을 압도하지 않으면서도 감점을 숨기지 않는가.
- 저장 점수에서 세부 근거가 없다는 정직한 안내가 억지 재계산보다 적절한가.

## PROMPT_SCOPE_AUDIT

### 평문으로 다시 말한 근본 문제

근본 문제는 점수 계산 자체가 아니라 “화면의 설명이 실제 계산과 일치하지 않고, Score와 Coach의 역할이 섞여 있다”는 것이다. 계산에 들어간 감점이 빠지는 한편 계산에 직접 들어가지 않은 조언이 점수 근거처럼 보이는 비대칭을 없애야 했다.

### next-gate 문구 밖에서 확인한 증거

- production score path가 최종 점수를 9개 raw penalty axis의 합으로 만드는지 확인했다.
- 기존 네 compatibility aggregate가 어떤 raw axis를 합치며 alcohol physiology와 data outlier를 완결하지 못하는지 확인했다.
- Today renderer가 Coach output, recent/InBody 맥락, meal concentration hint, non-formula `otherKcal` 슬롯을 가져오던 경로를 확인했다.
- 실제 record normalize/load 계약이 breakdown과 live scoring context를 저장하지 않는지 확인했다.
- Records가 저장 점수를 직접 표시하며 Today live evidence를 소비하지 않는 경계를 확인했다.
- 같은 kcal의 정상 carb-up/fat-down fixture에서 joint-failure가 0이어야 하는 production 동작을 확인했다.

### 검토한 대안

1. 기존 네 tile과 보조 문구를 그대로 두고 술/data 문구만 추가
   - 누락 일부는 줄지만 Coach 중복과 fixed-target 오경고, raw 합계 불일치 가능성이 남으므로 폐기했다.
2. 9개 감점 축을 항상 9개 행으로 표시
   - 내부 완결성과 화면 밀도를 혼동한다. 사용자는 0점짜리 조건부 항목까지 항상 볼 필요가 없으므로 폐기했다.
3. 조건부 근거를 접힌 `<details>` 안에 숨김
   - 실제 발생한 감점을 다시 숨기고 모바일 터치·접근성 부담을 늘리므로 폐기했다.
4. 저장 점수의 식사 기록을 현재 산식으로 재계산해 breakdown 복원
   - 당시 설정과 scoring context를 보존하지 않은 상태의 추론이며 stored snapshot 계약을 깨므로 금지했다.
5. 이번 작업에서 DailyCoach semantic v2와 adaptive help까지 함께 구현
   - 실제 문제보다 변경 범위를 넓히고 역할 경계를 다시 섞으므로 보류했다.

### 선택한 방식이 근본 문제를 푸는 이유

단일 evidence model이 raw 9축을 정확히 한 번 소유하고 원합계와 reconcile하므로 계산 근거 누락과 중복을 구조적으로 잡는다. UI는 그 모델을 네 핵심 aggregate와 nonzero 조건부 근거로만 표현하므로 내부 완결성을 유지하면서 화면 밀도를 제한한다. Score renderer에서 Coach 입력을 제거해 역할 혼합도 호출 계약 수준에서 끊었다.

### 이 선택이 틀렸음을 입증하는 조건

- 어떤 nonzero raw penalty가 화면 근거 어디에도 매핑되지 않거나 두 번 이상 매핑된다.
- evidence 합과 production raw penalty 합이 일치하지 않는다.
- evidence render 전후 `percent`, `rawScore`, `penaltyBreakdown`이 바뀐다.
- Coach 결과만 바꿨는데 Score DOM 또는 문구가 바뀐다.
- breakdown이 없는 stored score에서 세부 근거나 live source note가 만들어진다.
- 정상 same-kcal carb/fat exchange에 joint 조건부 경고가 생긴다.
- 작은 화면에서 가로 overflow가 생기거나 점수 근거의 list semantics/accessible label이 사라진다.

위 조건 중 하나라도 실제로 나오면 이 cleanup을 완료로 판정하지 않고 stable help gate로 넘어가지 않는다.

## 구현 계약

### 단일 score-evidence mapping

순수 adapter `buildTodayScoreEvidenceModel(adherenceResult)`가 Score 결과만 받아 아래 매핑과 attribution/clamp 문맥을 만든다. Coach output이나 current Records context는 입력으로 받지 않는다.

| 사용자 표시 | raw penalty axis | 표시 규칙 |
| --- | --- | --- |
| 탄수화물 | `carbExerciseContextPenalty` | 핵심 네 항목에 항상 표시 |
| 단백질 | `proteinShortagePenalty` + `proteinExcessEfficiencyPenalty` | 핵심 네 항목에 항상 표시 |
| 지방 | `fatRangePenalty` | 핵심 네 항목에 항상 표시 |
| 총칼로리 | `targetEnergyDeviationPenalty` + `tdeeOverloadPenalty` | 핵심 네 항목에 항상 표시 |
| 섭취량 확인 | `dataOutlierPenalty` | 0보다 클 때만 표시 |
| 술 영향 | `alcoholPhysiologyPenalty` | 0보다 클 때만 표시 |
| 탄수·지방 조합 | `carbFatExchangeFailurePenalty` | 0보다 클 때만 표시 |

이 매핑의 axis 슬롯 수는 정확히 9개이고 중복이 없다. `componentContributions`는 호환용 계산 결과이므로 evidence 원장으로 사용하지 않는다. `otherKcal`도 독립 감점 축이 아니므로 점수 행을 만들지 않는다.

### 완전성과 UI 밀도 분리

- evidence model은 9개 축의 존재, 유한한 비음수 값, 미등록 축 부재, 원합계와 매핑합의 일치를 확인한다.
- 화면은 9개 축을 9행으로 늘어놓지 않고 네 핵심 aggregate와 최대 세 개의 nonzero 조건부 항목으로 표현한다.
- breakdown이 불완전하거나 알 수 없는 축이 있으면 점수 숫자는 유지하되 억지 세부 귀속을 하지 않고 attribution limit을 표시한다.
- no-meal/pending 상태에는 full-day evidence를 만들지 않는다.
- live score의 기존 source validity note 경계는 유지하고 stored score에는 이를 붙이지 않는다.

### 접근성과 모바일

- 네 핵심 근거는 이름이 있는 list와 list item semantics를 갖는다.
- 각 핵심 item은 항목명과 감점 방향을 함께 읽을 수 있는 accessible label을 갖는다.
- 조건부 근거는 `<ul>/<li>` 직접 목록이며 별도 펼침 조작을 요구하지 않는다.
- desktop과 mobile test profile 모두에서 점수 영역의 가로 overflow를 확인한다.

## 테스트 정책 교정

기존 테스트가 어색하거나 잘못된 화면 책임을 고정하고 있으면 production 문구를 테스트에 맞춰 되돌리지 않았다. 다음과 같이 의미 중심으로 교정했다.

- generic secondary 문구의 정확한 문자열을 요구하던 기대 대신, 점수 계산 근거가 정확히 노출되는지를 검증한다.
- Coach/recent/InBody/action을 Score `추가 확인`에 복제하던 기대를 제거하고 Score DOM이 Coach 입력과 독립인지 검증한다.
- fixed-target high flag만으로 탄수·지방 경고를 기대하던 테스트를 실제 joint-failure axis 기준으로 바꿨다.
- 0.04점 같은 작은 감점을 `감점 없음`으로 보이게 만들던 반올림 위험을 별도 의미 테스트로 잠갔다.
- full-day available score를 검사하면서 실제로는 한 끼라 pending에 머물던 fixture는 두 끼 full-day fixture로 고쳐 테스트 이름과 제품 상태를 일치시켰다.
