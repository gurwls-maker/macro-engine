## 비개발자용 설명

### 무엇을 바꿨는지

기존 `오늘의 코치`는 계산 근거와 상태 이름을 나열하는 성격이 강했고, 실제 코치처럼 현재 상황을 설명하고 다음 행동을 한 번 제안하는 역할이 약했다. 이번에는 오늘의 식사, 방금 기록한 식사, 최근 완료 기록, 오늘 운동 맥락을 함께 보되 앱이 실제로 아는 사실만 사용해 아래 순서로 말하도록 바꿨다.

```text
현재 상태
왜 그런지
지금 할 행동 한 가지
```

첫 화면에는 가장 중요한 내용 하나만 보여 주고, 같이 볼 내용은 최대 두 개까지 `추가 확인` 안에 둔다. 같은 문장을 Hero와 Coach에서 반복하지 않으며, `행동:`이나 `이유:` 같은 기계적인 꼬리표도 화면에서 제거했다.

### 왜 바꿨는지

사용자가 원한 것은 점수표를 말로 다시 읽는 분석판이 아니라, 앱이 가진 정보 범위 안에서 오늘 무엇을 하면 되는지 자연스럽게 알려 주는 코치였다. 그러나 과거 구현은 다음 문제가 있었다.

- `회복 재료`, `성장 재료`, `흐름`처럼 사람이 일상적으로 잘 쓰지 않는 표현이 남아 있었다.
- 최근 며칠의 경향, 오늘의 문제, 운동 맥락이 한 화면에서 같은 종류의 경고처럼 섞일 수 있었다.
- 식사 시간이 저장되지 않는데도 아침이나 저녁처럼 추정하면 거짓 설명이 될 수 있었다.
- 100점인 날의 작은 최적화 제안을 현재 문제처럼 말하면 점수와 코치가 충돌했다.
- 문구 테스트를 맞추려고 어색한 기존 표현을 되살릴 위험이 있었다.

이번 구현은 현재 상태와 최근 경향을 분리하고, 근거가 부족하면 모른다고 처리하며, 테스트도 문장 형식보다 의미와 실제 화면 동작을 검증하도록 바꿨다.

### 실제 사용자 화면이나 계산 결과가 어떻게 달라지는지

예를 들어 오늘 단백질을 60g 먹었고 목표에 아직 부족하다면 아래처럼 보인다.

```text
지금까지 단백질은 60g이에요. 아직 부족해요.
현재 체중과 목표에 필요한 단백질보다 부족해요.
다음 식사에서 지방이 많지 않은 단백질을 먼저 보강하세요.
```

사용자가 앱에서 `아침`으로 고른 식사를 방금 저장했고 그 식사에 지방이 28g 들어 있다면 `아침 식사에서 지방을 28g 먹었어요.`라고 말할 수 있다. 사용자 태그가 없거나 페이지를 새로 연 뒤라면 식사 시간을 지어내지 않고 `지금까지 기록한 식사`라는 누적 표현을 쓴다. `createdAt`은 식사 정렬에만 쓰며 아침·점심·저녁 추정에는 쓰지 않는다.

최근 완료 기록에서 고지방 경향이 실제로 이어졌다면 오늘의 현재 문제와 분리해 `최근 기록에서는 지방 섭취가 높은 날이 이어졌어요.`처럼 말한다. 이 최근 경향은 오늘 점수를 다시 깎거나 카드의 무감점 범위를 좁히지 않는다. 오늘 점수는 오늘 입력으로 계산하고, 최근 기록은 코칭과 자동 목표 조정의 근거로만 사용한다.

오늘 운동이 PUSH라면 앱이 아는 사실인 `오늘은 상체 운동이 예정돼 있어요.` 정도만 말할 수 있다. 세트 구성, 부상 상태, 가동범위는 앱이 모르므로 특정 스트레칭이나 운동 처방을 꾸며 내지 않는다. 운동은 점수 보너스가 아니며 기존 탄수화물 곡선 맥락만 유지한다.

술은 기존처럼 두 갈래로 처리한다. 술 칼로리는 총칼로리 축에 포함되고, 술 자체의 생리학적 위험은 별도 alcohol 축에 남는다. 이번 코치가 이 감점을 새로 만들거나 이중 감점하지는 않는다.

### 정책, 산식, 데이터 해석이 바뀌었는지

점수 정책과 산식은 변경 없음이다.

- 현행 `v8.4_joint_axis_retired_continuous_macro_score_v1` 8축 점수와 각 축의 연속 곡선을 그대로 사용한다.
- 최근 3일/7일 경향을 점수 감점축으로 추가하지 않는다.
- 오늘 목표 자동 조정, 탄수·지방 joint allocation, 연속형 recent pressure와 conditional card ranges는 그대로다.
- 100점인 날에는 현재 문제를 만들지 않는다. 다만 현재 점수를 깎지 않는 작은 최적화가 실제 근거와 함께 있을 때만 선택적 조언을 할 수 있다.
- 현재 점수가 낮다면 실제 nonzero penalty 축을 오늘 문제의 근거로 사용한다. 과거 fixed severity나 계단식 상태표를 다시 쓰지 않는다.

최근 기록은 목표 대비 비율로 비교하고 날짜가 실제로 이어질 때만 `연속`이라고 판단한다. 단순히 7일 창에 세 건이 있다는 이유로 3일 연속이라고 말하지 않는다. 반복 압력은 횟수 계단이 아니라 크기, 최근성, 연속성을 함께 반영한 연속값을 읽는다.

### 바꾸지 않은 범위와 보류한 내용

- 점수 percent, rawScore, scoring version, formula, anchor, curve: 변경 없음.
- 목표 칼로리와 탄단지 숫자, 카드 범위, adaptive ON/OFF 동작: 변경 없음.
- storage/schema/backup/Records 저장 계약: 변경 없음.
- 마지막으로 저장·수정한 식사 맥락은 현재 페이지 세션 안에서만 사용하며 localStorage나 backup에 추가하지 않는다.
- 과거 기록 재계산, scoreDeltaPreview, component-score aggregation: 열지 않았다.
- 식사 시간 입력 필드: 추가하지 않았다. 사용자 식사 태그가 가장 정확한 출처이므로 우선 사용한다.
- selectable Coach voice와 broad tooltip/glossary: 자동 후속 구현으로 열지 않았다. 실제 사용자 사용에서 필요가 확인될 때 별도 판단한다.
- 운동 처방, 의료적 진단, 근거 없는 루틴별 조언: 추가하지 않았다.

### 사용자가 큰 틀에서 확인해야 할 판단점

이번 변경의 핵심 판단은 `최근 식습관도 오늘 점수에 직접 넣을 것인가`가 아니라, 오늘 점수와 최근 코칭의 역할을 분리할 것인가였다. 같은 95g 지방이라도 오늘 목표 조정에는 최근 경향이 이미 반영될 수 있지만, 최근 기록이라는 이유만으로 오늘 점수를 다시 감점하면 같은 정보를 두 번 벌할 수 있다. 따라서 현재 점수는 오늘 상태를 평가하고, 최근 패턴은 코치가 이유와 행동을 설명하도록 분리한 것이 더 일관적이라고 판단했다.

실제 화면에서 확인할 것은 세 가지다.

1. 첫 문장이 지금 가장 중요한 문제를 제대로 고르는가.
2. 이유와 행동이 한 번씩만 나오고 사람이 말하는 것처럼 자연스러운가.
3. 앱이 모르는 식사 시간이나 운동 처방을 지어내지 않는가.

추가 정책 판단은 필요 없다. 실제 사용 중 우선순위나 문구가 어색한 구체 사례가 생기면 그 사례를 새 이슈로 다룬다.

### 테스트에서 무엇을 검증했는지

- 아침/점심/저녁/간식/야식 태그와 중복·사용자 정의 태그를 모두 확인했다.
- 방금 저장하거나 수정한 식사는 현재 세션에서만 정확한 태그를 사용하고, 새로고침 뒤에는 누적 표현으로 안전하게 돌아가는지 확인했다.
- 세션 식사 맥락이 localStorage, 전체 백업, Records에 저장되지 않는지 확인했다.
- 식사가 없거나 한 끼뿐일 때 과도한 영양 판단 대신 정보 부족을 우선 말하는지 확인했다.
- 최근 3일 연속과 날짜가 띄엄띄엄인 3건을 구분하고, 최근 압력이 섭취 편차가 커질수록 연속적으로 커지는지 확인했다.
- 탄수·지방의 3x3 조합과 총칼로리·단백질의 3x3 조합을 모두 확인해 한 예시에만 맞춘 구현이 아닌지 검증했다.
- 100점 + 선택적 최적화와 낮은 점수 + 현재 문제를 구분했다.
- 자동 목표 조정의 요청량, 실제 적용량, recent limiter, 상·하한 도달, 조정 없음 이유를 각각 확인했다.
- 일반 운동, 휴식, PUSH, LEGS에서 앱이 아는 사실만 말하고 운동 보너스나 임의 처방을 만들지 않는지 확인했다.
- 술과 데이터 부족이 함께 있을 때 중요한 안전 정보를 놓치지 않는지 확인했다.
- 데스크톱과 모바일에서 문장이 겹치거나 넘치지 않고, 핵심 행동이 실제로 보이며, 같은 말이 Hero와 Coach에서 중복되지 않는지 확인했다.
- 구현 전후 점수, 자동 목표값, 카드 범위가 같아 코칭 변경이 산식 변경으로 새지 않았음을 확인했다.

### 구현 내용

- `getDailyCoachSignals`가 `dailycoach_semantic_v2_phase1` evidence를 함께 만든다.
- 현행 `penaltyBreakdown`의 nonzero 8축을 현재 문제의 source of truth로 사용한다.
- 마지막 식사 저장/수정 맥락은 메모리에서만 유지한다. reload 후에는 누적 today context로 돌아간다.
- recent evidence는 완료된 detailed auto records를 목표 대비 비율로 읽고, 날짜 연속성과 연속 pressure를 따로 계산한다.
- adaptive evidence는 requested/applied shift, recent limiter 효과, boundary hit, no-adjust reason을 포함한다. 최종 target 숫자는 바꾸지 않는다.
- canonical semantic output은 최대 3개이며 primary 1개와 supporting 최대 2개다.
- Hero와 Coach는 같은 output을 소비하되 화면에 같은 행동 문장이 두 번 보이지 않게 한다.
- legacy test caller 경로는 유지하되 live semantic renderer에서는 옛 context notice와 `행동:`/`이유:` label을 사용하지 않는다.

- 문구 테스트가 실패해도 기존 문장이 어색하면 테스트를 의미 계약에 맞춰 고친다.
- 이 장치가 문서를 계속 만드는 메타 작업이나 새 관성이 되지 않도록 대부분의 audit는 결과로그 안에서만 끝낸다.
