# 5.4.1 저장 타이밍 수정 계획

## 1. 5.4.1 목적

- `5.4_save_flow_map.md`와 현재 5.4.0 작업본을 대조해 `saveState()` 중복 호출 후보를 확정한다.
- 이번 단계에서는 HTML을 수정하지 않고, 5.4.1에서 실제로 건드릴 최소 저장 타이밍 후보만 좁힌다.
- 분석 기준 작업본은 워크스페이스 내 `macro_engine.html`이며, 파일 헤더에 `Version: 5.4.0`이 명시되어 있어 실질적인 5.4.0 본체로 판단했다. 현재 워크스페이스에는 `macro_engine_5.4.0.html` 파일명 자체는 존재하지 않았다.

## 2. 현재 중복 저장 후보 표

| 경로 | 현재 흐름 | 중복 성격 | 판단 메모 |
|---|---|---|---|
| `saveMealEntry()` | `upsertRecord()` -> `persistAutoAdherenceForDate()` -> `syncRecordInputsForDate()` -> `saveState()` -> `render()` -> `saveState()` | 높음 | 선행 저장과 렌더 마지막 저장이 모두 같은 `recordsState.items`를 저장한다. 후행 저장은 렌더 후 UI 입력 상태까지 포함한다. |
| `saveRecord()` | `upsertRecord()` -> `syncRecordInputsForDate()` -> `saveState()` -> `render()` -> `saveState()` | 높음 | 저장 직후 입력값을 다시 동기화한 뒤 또 렌더 저장까지 간다. |
| `deleteRecord()` | `recordsState.items` 재구성 -> `syncRecordInputsForDate()`(조건부) -> `saveState()` -> `render()` -> `saveState()` | 높음 | 삭제 반영은 이미 메모리에 끝난 상태라 렌더 마지막 저장으로 수렴 가능하다. |
| `deleteMealEntry()` | `upsertRecord()` -> `persistAutoAdherenceForDate()` -> `resetMealInputsForDate()`(조건부) -> `saveState()` -> `render()` -> `saveState()` | 높음 | 식사 삭제와 auto adherence 재계산이 모두 끝난 뒤 중복 저장이 발생한다. |
| `importRecordsJson()` | `recordsState.items` 일괄 대입 -> `syncRecordInputsForDate()` -> `saveState()` -> `render()` -> `saveState()` | 높음 | import 완료 상태를 두 번 저장한다. |
| `render()` 자체 | 대부분 입력 이벤트 종착점에서 마지막에 `saveState()` | 넓음 | 현재 구조상 전역 저장 종착점 역할이다. 5.4.1에서는 유지 쪽이 안전하다. |

## 3. saveState 유지/제거 판단표

| 위치 | 분류 | 판단 | 가장 안전한 조치 |
|---|---|---|---|
| `render()` 마지막 `saveState()` | 유지 | 현재 전역 저장 종착점이다. 일반 입력, 탭 복원, 리렌더 후 UI 동기화까지 한 번에 저장한다. | 5.4.1에서는 유지 |
| `saveMealEntry()` 내부 `saveState()` | 제거 후보 | 바로 다음 줄에서 `render()`가 다시 저장한다. 선행 저장은 중간 상태를 저장하고, 후행 저장은 동기화된 입력 UI까지 포함한다. | 내부 `saveState()`만 제거하고 `render()` 저장 유지 |
| `saveRecord()` 내부 `saveState()` | 제거 후보 | `syncRecordInputsForDate()` 후 즉시 `render()`가 이어져 같은 record를 다시 저장한다. | 내부 `saveState()`만 제거하고 `render()` 저장 유지 |
| `deleteRecord()` 내부 `saveState()` | 제거 후보 | 삭제 결과가 이미 `recordsState.items`에 반영된 상태라 렌더 마지막 저장만으로 충분하다. | 내부 `saveState()`만 제거하고 `render()` 저장 유지 |
| `deleteMealEntry()` 내부 `saveState()` | 제거 후보 | meal 삭제와 auto adherence 재계산이 끝난 뒤 곧바로 `render()`가 저장한다. | 내부 `saveState()`만 제거하고 `render()` 저장 유지 |
| `importRecordsJson()` 내부 `saveState()` | 제거 후보 | import 후 `syncRecordInputsForDate()`까지 끝난 뒤 `render()` 저장이 다시 실행된다. | 내부 `saveState()`만 제거하고 `render()` 저장 유지 |
| `recordWeight`/`recordAdherence`/`recordNote` draft 리스너의 `saveState()` | 유지 | 이 경로는 `render()`를 부르지 않는다. draft 보존 목적의 직접 저장이 필요하다. | 유지 |
| 탭 클릭 후 `saveState()` | 유지 | `setActiveTab()`은 렌더를 호출하지 않고 탭 상태만 바꾼다. | 유지 |

## 4. recordsState 변경 함수별 저장 필요 여부

| 함수 | `recordsState` 변경 여부 | 저장 시점 판단 | 비고 |
|---|---|---|---|
| `upsertRecord(rawRecord)` | 예 | 직접 저장 금지, 호출자 책임 | 공통 병합기다. 내부에서 `saveState()`를 넣으면 저장 경로가 더 넓어져 리스크가 커진다. |
| `persistAutoAdherenceForDate(date, currentResult, options)` | 예 | 직접 저장 금지, 호출자 책임 | auto adherence와 snapshot을 갱신하지만 helper 성격이라 저장은 상위 액션에서 마무리하는 편이 안전하다. |
| `saveMealEntry()` | 예 | `render()` 마지막 저장으로 충분 | action 안에서 record 변경과 UI 동기화가 끝난 뒤 바로 렌더 저장이 이어진다. |
| `saveRecord()` | 예 | `render()` 마지막 저장으로 충분 | 선행 `saveState()`는 중복이다. |
| `deleteRecord()` | 예 | `render()` 마지막 저장으로 충분 | 삭제 후 조건부 입력 동기화까지 끝나므로 렌더 저장으로 수렴 가능하다. |
| `deleteMealEntry()` | 예 | `render()` 마지막 저장으로 충분 | meal 삭제와 adherence 재계산 이후 렌더 저장으로 충분하다. |
| `importRecordsJson()` | 예 | `render()` 마지막 저장으로 충분 | import 직후 상태를 렌더 저장이 다시 반영한다. |
| `resetState()` | 예 | `render()` 마지막 저장 필요 | reset은 로컬 스토리지를 먼저 비운 뒤 렌더가 초기 상태를 다시 저장하는 구조다. 5.4.1 대상 아님. |

정리하면, 5.4.1에서 실제로 줄일 중복 저장은 "recordsState 변경 직후의 선행 `saveState()`"이며, helper 레벨 저장 추가가 아니라 action 레벨 선행 저장 제거가 맞다.

## 5. 5.4.1에서 실제 수정할 최소 후보

| 우선순위 | 후보 | 이유 |
|---|---|---|
| 1 | `saveMealEntry()` 내부 `saveState()` 제거 | 중복 저장 빈도가 높고, 식사 추가는 가장 자주 발생하는 쓰기 경로다. |
| 2 | `saveRecord()` 내부 `saveState()` 제거 | 체중/메모 저장도 동일한 중복 패턴이다. |
| 3 | `deleteMealEntry()` 내부 `saveState()` 제거 | meal 삭제 후 auto adherence 재계산까지 끝난 뒤 다시 저장하므로 제거 이득이 분명하다. |
| 4 | `deleteRecord()` 내부 `saveState()` 제거 | 삭제 액션도 구조가 단순해 안전하다. |
| 5 | `importRecordsJson()` 내부 `saveState()` 제거 | import 성공 시점 저장 중복 제거. 다만 import는 회귀 영향 확인이 필요하므로 마지막 순서가 안전하다. |

핵심 원칙:

- 5.4.1에서는 `render()` 마지막 `saveState()`는 유지한다.
- 제거 대상은 "액션 함수 내부 선행 `saveState()`"로 한정한다.
- `upsertRecord()`나 `persistAutoAdherenceForDate()` 내부에 새로운 저장 호출을 넣지 않는다.

## 6. 5.4.1에서 아직 수정하지 말아야 할 후보

| 후보 | 이유 |
|---|---|
| `render()` 마지막 `saveState()` 제거 또는 조건부화 | 저장 범위가 너무 넓지만 현재는 전역 일관성의 마지막 안전망이다. 먼저 액션 함수 내부 중복만 줄이는 편이 안전하다. |
| `recordWeight`/`recordAdherence`/`recordNote` draft 저장 리스너 변경 | 이 경로는 사용자가 저장 버튼을 누르기 전 draft 복원을 담당한다. 중복 저장 이슈와 성격이 다르다. |
| 탭 클릭 저장 로직 변경 | `activeTab` 복원 목적의 독립 저장 경로라 5.4.1 범위 밖이다. |
| `resetState()` 저장 순서 변경 | 초기화-재저장 시맨틱이 섞여 있어 별도 검토가 필요하다. |
| `exportRecordsJson()` 이후 `render()` 흐름 조정 | `lastBackupAt` 직접 저장과 UI 갱신을 같이 다루므로 현재 단계에서는 건드리지 않는다. |

## 7. 5.4.2로 넘길 항목

| 항목 | 이유 |
|---|---|
| `adherencePercent`와 manual `adherence`의 공존 구조 정리 | `getStoredAdherenceEntry()`가 auto 우선, manual fallback으로 읽는 구조다. 저장 타이밍 조정과는 별개의 데이터 모델 문제다. |
| `goalSnapshot` 생성 시점 정책 재검토 | 과거 날짜 식사 추가 시 `forceLateSnapshot`으로 1회 생성하는 정책은 5.4.1에서 바꾸면 회귀 위험이 크다. |
| `persistAutoAdherenceForDate()`의 책임 분리 | 현재는 snapshot 생성, adherence 계산, source/lock 저장을 한 번에 맡는다. 저장 타이밍 정리 후 분리 검토가 더 안전하다. |
| `upsertRecord()`의 필드 단위 덮어쓰기 안전장치 | 날짜만 맞으면 해당 날짜 record를 통째로 재구성하는 구조라, 향후 보호 장치나 좁은 patch 전략 검토가 필요하다. |
| import 후 snapshot/adherence 정합성 재평가 정책 | 현재 import는 normalize 후 그대로 수용한다. imported data의 source/lock 정책 점검은 후속 단계가 적절하다. |

## 8. 최종 권장 수정 순서

1. `saveMealEntry()`, `saveRecord()`, `deleteMealEntry()`, `deleteRecord()`, `importRecordsJson()`에서 액션 내부 선행 `saveState()`만 제거한다.
2. `render()` 마지막 `saveState()`는 그대로 둔다.
3. helper 함수인 `upsertRecord()`와 `persistAutoAdherenceForDate()`는 건드리지 않는다.
4. 회귀 확인 포인트는 `recordsState.items` 유지, draft 입력 복원, import 이후 목록 반영, 식사 추가/삭제 후 UI 동기화다.
5. `adherencePercent`, `goalSnapshot`, Weekly 해석 우선순위 변경은 5.4.2로 넘긴다.

부가 판단 메모:

- 과거 record 덮어쓰기 위험은 저장 타이밍보다 `date` 인자와 `upsertRecord()`의 날짜 단위 교체 방식에서 발생한다.
- 특히 `saveMealEntry()`는 과거 날짜에 대해 `persistAutoAdherenceForDate(... forceLateSnapshot: true)`를 통해 최초 snapshot을 생성할 수 있으므로, 5.4.1에서는 저장 호출 중복만 줄이고 snapshot 정책은 그대로 유지해야 한다.
- `deleteMealEntry()`는 기존 snapshot을 재사용하며 `forceLateSnapshot: false`라 snapshot 덮어쓰기 위험은 상대적으로 낮다.
- `persistAutoAdherenceForDate()`는 기존 snapshot이 있으면 재사용하므로, 이번 단계에서 이 함수 내부 저장 정책을 건드리는 것은 피하는 편이 안전하다.
