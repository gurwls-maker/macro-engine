## 1. 구현한 suite

추가 suite:

```text
runMacroRangeSignatureDecisionDesignTests
```

등록 위치:

```text
index.html window export
tools/render_audit/run_internal_tests.cjs core profile
package.json test:macro-policy
```

중요:

```text
window export는 suite 함수만 한다.
signature candidate helper는 suite 내부 local helper로만 존재한다.
GOAL_SNAPSHOT_SIGNATURE_KEYS는 바꾸지 않는다.
getGoalSnapshotSignature production output은 바꾸지 않는다.
```

## 2. test-local helper

suite 내부 helper:

```text
normalizeSignatureDecisionBasisContract(rawContract, fixedSnapshot, mode)
```

지원 mode:

```text
exclude
include
separate
```

역할:

```text
exclude mode가 current fixed signature를 유지하는지 확인
include mode가 basis mismatch를 감지할 수 있는지 확인
separate mode가 별도 basis signature 후보를 안정적으로 만들 수 있는지 확인
forbidden/debug/copy/evidence field가 signature 후보에 들어가지 않는지 확인
```

하지 않는 일:

```text
production signature 변경
Records 저장 비교 변경
goalSnapshot 저장 field 추가
backup payload field 추가
Recent basis field 추가
score 계산
UI copy 생성
```

## 3. 추가 테스트

### 3-1. current fixed signature ignores unknown range fields

확인:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS에 macroRangeContract 없음
GOAL_SNAPSHOT_SIGNATURE_KEYS에 rangeScoreCandidate 없음
GOAL_SNAPSHOT_SIGNATURE_KEYS에 candidateScorePreview 없음
unknown range field가 current production fixed signature를 바꾸지 않음
```

의미:

```text
현재 production signature는 fixed-only다.
```

### 3-2. candidateScorePreview exclusion

확인:

```text
candidateScorePreview는 signature candidate를 invalid로 만듦
basisSignature 생성 안 함
normalizedBasis에 candidateScorePreview 없음
candidateScorePreview 값이 signature candidate output에 유입되지 않음
```

### 3-3. exclude mode unchanged save stability

확인:

```text
exclude mode는 current fixed signature만 사용
candidate basis change가 effectiveSignature를 바꾸지 않음
production signature도 fixed-only 유지
```

의미:

```text
무변경 저장 안정성을 보존하는 가장 보수적인 후보를 확인했다.
```

### 3-4. include mode targetCal mismatch detection

확인:

```text
include mode는 range targetCal mismatch를 basis_change_candidate로 감지
basisSignature 후보 생성
effectiveSignature는 fixedSignature와 달라짐
production fixed signature는 그대로 유지
```

주의:

```text
이 테스트는 include mode를 production에 적용한 것이 아니다.
```

### 3-5. debug / copy / evidence field rejection

확인:

```text
debugReason
explanation
rawExternalEvidence
uiCopy
```

위 field는 include signature candidate에서 invalid 처리된다.

의미:

```text
문구, 근거 텍스트, debug 설명은 signature 대상이 아니다.
```

### 3-6. separate basis signature changes on allowed basis change

확인:

```text
allowed protein/fat range field가 바뀌면 separate basis signature가 바뀜
current fixed effectiveSignature는 유지
```

### 3-7. field order stability

확인:

```text
동일한 contract field를 다른 순서로 구성해도 separate basis signature 동일
```

### 3-8. numeric serialization stability

확인:

```text
150 vs 150.004
90 vs 90.004
420 vs 420.004
```

위 harmless serialization 차이는 0.01 기준 정규화 후 같은 separate basis signature를 만든다.

주의:

```text
이 precision은 정책 차이를 숨기는 tolerance가 아니다.
signature serialization 안정성을 위한 test-local 정규화다.
```

### 3-9. invalid range contract

확인:

```text
fat range order violation은 invalid
basisSignature 없음
fixedSignature는 계속 사용 가능
```

### 3-10. unsupported range contract

확인:

```text
unsupported version은 unsupported
basisSignature 없음
fixedSignature는 계속 사용 가능
```

### 3-11. backup roundtrip candidate

확인:

```text
test-local basisSignature candidate는 생성됨
normalized record fixed signature는 backup roundtrip 후 유지
backup payload에는 basisSignature 없음
backup payload에는 macroRangePolicy.v0.reportOnly 없음
normalized backup snapshot에는 macroRangeContract 없음
```

의미:

```text
backup schema를 열지 않은 현재 단계에서 candidate basis signature가 backup에 새지 않는다.
```

### 3-12. Recent mixed basis candidate

확인:

```text
test-local detector는 fixed/range split 감지
production Recent component counts는 변경 없음
production Recent summary에는 macroRangeContract 없음
```

의미:

```text
mixed basis detection 후보를 확인하되 Recent production aggregation은 바꾸지 않는다.
```
