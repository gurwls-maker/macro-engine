# v8.2 macro range snapshot compatibility test design

상태:

```text
문서 성격: snapshot compatibility test design
작업 범위: docs-only
구현 변경: 없음
index.html 변경: 없음
테스트 코드 추가: 없음
goalSnapshot schema 변경: 없음
backup / restore schema 변경: 없음
Recent range-aware logic 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
production macro range 구현: 금지
```

이 문서는 `v8.2_macro_range_snapshot_compatibility_design_2026-07-03.md`의 다음 단계다.

구현 기록:

```text
이 test design은 `v8.2_macro_range_snapshot_compatibility_design_tests_implementation_2026-07-03.md`에서
runMacroRangeSnapshotCompatibilityDesignTests suite로 구현됐다.

구현은 test suite / fixture / runner registration만 추가하며,
goalSnapshot schema, normalizer, backup/restore policy, Recent range-aware logic,
ADHERENCE_SCORING_VERSION, UI는 변경하지 않는다.
```

목적:

```text
macro range snapshot compatibility를 실제 테스트로 열기 전에
fixture, assertion, failure mode, 금지선, 실행 gate를 정밀하게 설계한다.
```

중요:

```text
이번 문서는 테스트 구현 지시서가 아니다.
이번 문서는 schema 구현 지시서가 아니다.
schema를 먼저 추가하고 나중에 테스트하는 순서는 금지한다.
```

## 1. 테스트 설계 원칙

테스트가 증명해야 하는 것:

```text
old fixed snapshot이 계속 source of truth다.
unknown range field는 현재 normalize에서 보존되지 않는다는 사실이 테스트로 명확해진다.
future known macroRangeContract는 explicit normalizer 없이는 도입할 수 없다.
invalid range contract가 fixed snapshot을 null로 만들면 안 된다.
signature include/exclude 정책은 구현 전 simulation으로 결정한다.
backup/restore/smart restore가 snapshot basis를 의도 없이 바꾸지 않는다.
Recent는 fixed/range mixed basis를 조용히 같은 fatLowDays로 합치지 않는다.
candidateScorePreview는 저장/복원/Recent/UI에 절대 새지 않는다.
ADHERENCE_SCORING_VERSION은 production score 결정 전까지 그대로다.
```

테스트가 하면 안 되는 것:

```text
macroRangeContract를 실제 schema로 추가
goalSnapshot에 range field 저장
backup payload에 range field 추가
Recent range-aware aggregation 구현
candidateScorePreview 숫자화
ADHERENCE_SCORING_VERSION 변경
```

## 2. suite 구성 후보

1차 test design 기준 suite:

```text
runMacroRangeSnapshotCompatibilityDesignTests
```

후속 구현 후보 suite:

```text
runMacroRangeSnapshotCompatibilityTests
runMacroRangeBackupCompatibilityTests
runMacroRangeSmartRestoreCompatibilityTests
runMacroRangeRecentCompatibilityTests
```

권장:

```text
처음에는 runMacroRangeSnapshotCompatibilityDesignTests 하나로 시작한다.
이 suite는 현재 production 동작을 바꾸지 않고, 현재 normalize/backup/Recent 계약을 확인한다.
```

## 3. fixture 공통 구조

### 3-1. old fixed snapshot fixture

목적:

```text
현재 v8.0/v8.1 fixed target snapshot이 baseline이다.
```

후보:

```js
const oldFixedSnapshot = {
  targetCal: 2400,
  protein: 150,
  carbs: 270,
  fat: 70,
  goal: "lean_bulk",
  exerciseManagementMode: "exercise",
  weight: 75,
  activityLevel: "moderate",
  workType: "office",
  sleepHours: 8,
  workHours: 8,
  lifestyleHours: 4,
  routine: "PUSH",
  weightDuration: 60
};
```

필수 assertion:

```text
normalizeGoalSnapshot(oldFixedSnapshot) !== null
normalized.targetCal/protein/carbs/fat preserved
getGoalSnapshotSignature(oldFixedSnapshot) stable
macroRangeContract field absent
```

### 3-2. unknown future field fixture

목적:

```text
현재 normalizeGoalSnapshot이 unknown optional field를 보존하지 않는다는 사실을 명시적으로 확인한다.
```

후보:

```js
const unknownFutureSnapshot = {
  ...oldFixedSnapshot,
  macroRangeContract: { version: "unknown_future" },
  rangeScoreCandidate: { candidateScorePreview: 88 },
  unknownFutureField: { shouldNotPassThrough: true }
};
```

현재 기대값:

```text
normalizeGoalSnapshot(unknownFutureSnapshot).macroRangeContract === undefined
normalizeGoalSnapshot(unknownFutureSnapshot).rangeScoreCandidate === undefined
normalizeGoalSnapshot(unknownFutureSnapshot).unknownFutureField === undefined
fixed fields remain valid
```

주의:

```text
이 테스트는 "unknown field를 보존하라"가 아니다.
현재 코드의 allowlist 계약을 명시해 future schema 구현이 필요함을 드러내는 테스트다.
```

### 3-3. invalid range contract fixture

목적:

```text
나중에 known macroRangeContract normalizer를 추가하더라도 invalid range contract가 fixed snapshot 전체를 깨면 안 된다.
```

후보:

```js
const invalidRangeSnapshot = {
  ...oldFixedSnapshot,
  macroRangeContract: {
    version: "macroRangePolicy.v0.reportOnly",
    targetCal: "not-a-number",
    fat: { actionFloorG: -10 }
  }
};
```

현재 기대값:

```text
현재 normalizeGoalSnapshot에서는 macroRangeContract가 pruned된다.
fixed snapshot은 정상이다.
```

future 기대값:

```text
known macroRangeContract normalizer가 추가되더라도 invalid contract만 null이 된다.
fixed target snapshot은 null이 되면 안 된다.
```

### 3-4. candidateScorePreview leakage fixture

목적:

```text
report-only score candidate output이 저장 구조에 새지 않는지 확인한다.
```

후보:

```js
const candidateLeakSnapshot = {
  ...oldFixedSnapshot,
  macroRangeContract: {
    scoring: {
      candidateScorePreview: 92,
      scoreDeltaPreview: 11
    }
  }
};
```

현재 기대값:

```text
candidateScorePreview is pruned
scoreDeltaPreview is pruned
backup payload does not contain candidateScorePreview
```

future 기대값:

```text
candidateScorePreview는 production scoring decision 전까지 저장 금지다.
```

## 4. test case 설계

### 4-1. old snapshot remains valid

테스트 이름 후보:

```text
v8.2 snapshot compatibility keeps old fixed snapshot valid
```

검증:

```text
old fixed snapshot normalize 성공
required fixed fields preserved
goal/type/activity fields preserved where currently supported
signature 생성 가능
macroRangeContract absent
```

실패 의미:

```text
old Records를 깨는 회귀
```

### 4-2. unknown range field is currently pruned

테스트 이름 후보:

```text
v8.2 snapshot compatibility documents current unknown range field pruning
```

검증:

```text
macroRangeContract unknown field pruned
rangeScoreCandidate unknown field pruned
unknownFutureField pruned
fixed target fields preserved
signature equals old fixed signature if only unknown fields differ
```

실패 의미:

```text
누군가 unknown field pass-through를 추가했거나 schema가 암묵적으로 열린 상태
```

주의:

```text
이 실패가 무조건 나쁜 것은 아니다.
하지만 schema가 열린 것이므로 반드시 별도 migration/backup/Recent 설계와 함께 검토해야 한다.
```

### 4-3. known future macroRangeContract requires explicit normalizer

테스트 이름 후보:

```text
v8.2 snapshot compatibility blocks known range contract without explicit normalizer
```

검증:

```text
현재 코드에는 normalizeMacroRangeContractForSnapshot 함수가 없다.
macroRangeContract는 normalizeGoalSnapshot output에 없다.
문서/테스트가 이 상태를 명시한다.
```

future 전환 조건:

```text
normalizeMacroRangeContractForSnapshot이 생기면 이 테스트는 새 contract validation test로 전환한다.
그때는 explicit field allowlist와 invalid contract fallback을 검증해야 한다.
```

### 4-4. invalid macroRangeContract does not null fixed snapshot

테스트 이름 후보:

```text
v8.2 snapshot compatibility keeps fixed snapshot alive when range contract is invalid
```

현재 검증:

```text
invalid macroRangeContract is pruned
normalizeGoalSnapshot result remains non-null
fixed target signature remains available
```

future 검증:

```text
invalid range contract -> normalized.macroRangeContract === null
normalized.targetCal/protein/carbs/fat remain valid
```

### 4-5. signature include/exclude behavior is explicit

테스트 이름 후보:

```text
v8.2 snapshot compatibility keeps range signature behavior explicit
```

현재 검증:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS does not include macroRangeContract
old fixed signature ignores pruned unknown range field
```

future simulation 후보:

```text
signature exclude mode:
  range field change does not trigger unchanged-save diff

signature include mode:
  range field change triggers explicit diff key
```

주의:

```text
include/exclude를 동시에 production에 둘 수 없다.
테스트 설계 단계에서는 두 mode를 simulation 후보로만 둔다.
```

### 4-6. full backup old/new basis roundtrip

테스트 이름 후보:

```text
v8.2 snapshot compatibility full backup keeps old fixed basis and rejects accidental range persistence
```

현재 검증:

```text
buildFullBackupPayload() old record payload contains fixed goalSnapshot
payload does not contain macroRangeContract
payload does not contain candidateScorePreview
normalizeFullBackupPayload() keeps old fixed record valid
```

future 검증:

```text
known macroRangeContract old/new fixture roundtrip
unknown future field behavior explicit
invalid range contract fallback explicit
```

### 4-7. records-only restore old/new basis roundtrip

테스트 이름 후보:

```text
v8.2 snapshot compatibility records-only restore keeps snapshot basis explicit
```

현재 검증:

```text
applyRecordsOnlyImportData() uses normalizeRecordsCollection()
old fixed snapshot survives
unknown range field pruned
candidateScorePreview not persisted
```

future 검증:

```text
known macroRangeContract survives records-only import only after explicit normalizer
```

### 4-8. smart restore keep/replace/meals modes

테스트 이름 후보:

```text
v8.2 snapshot compatibility smart restore modes keep snapshot basis intentional
```

mode별 기대:

```text
keep:
  local record remains local basis
  backup conflict does not replace local snapshot

replace:
  backup record replaces local basis intentionally

meals:
  meals are merged
  local record snapshot basis remains local
  backup snapshot basis does not silently replace local snapshot
```

현재 검증:

```text
applySmartRestoreImportData(data, "meals") calls mergeRecordMealsOnly()
merged record keeps localRecord basis
candidateScorePreview not introduced
```

중요:

```text
meals-only mode는 가장 위험한 케이스다.
식사만 합치는 흐름에서 snapshot basis가 바뀌면 안 된다.
```

### 4-9. Recent fixed/range mixed basis detection before aggregation

테스트 이름 후보:

```text
v8.2 snapshot compatibility blocks silent Recent mixed-basis aggregation
```

현재 검증:

```text
getRecentContextRecordSummary() reads fixed targetCal/protein/carbs/fat
getRecentContextComponentCounts() uses fixed target thresholds
macroRangeContract absent/pruned means range basis cannot be aggregated
```

future 검증:

```text
records with fixed basis and range basis produce mixedMacroBasisDetected before range-aware aggregation
fatLowDays does not silently combine fixed targetFat and range actionFloor
confidence/contextFit or basis summary changes only by explicit policy
```

### 4-10. candidateScorePreview not persisted

테스트 이름 후보:

```text
v8.2 snapshot compatibility keeps candidate score preview transient
```

검증:

```text
buildFullBackupPayload does not include candidateScorePreview
normalizeGoalSnapshot prunes candidateScorePreview
normalizeRecord does not expose candidate score fields
Recent summary does not expose candidateScorePreview
DOM text does not include range score candidate labels
```

### 4-11. scoring version remains unchanged

테스트 이름 후보:

```text
v8.2 snapshot compatibility keeps scoring version unchanged until production decision
```

검증:

```text
ADHERENCE_SCORING_VERSION === "v6.1_alcohol_penalty_v2"
old auto score basis remains current/legacy only
range score candidate does not create a stored scoring version
```

future 전환 조건:

```text
range-aware score production decision
new scoring version name approved
old/new mixed score basis UI/Recent policy approved
```

## 5. assertion matrix

| 축 | 현재 기대 | future 전환 조건 |
| --- | --- | --- |
| old fixed snapshot | valid | 항상 유지 |
| unknown range field | pruned | explicit schema decision 없이는 유지 |
| known macroRangeContract | not supported | explicit normalizer 필요 |
| invalid range contract | fixed snapshot survives | invalid contract만 null |
| signature | fixed key only | include/exclude 결정 필요 |
| full backup | fixed only | known range roundtrip test 필요 |
| records-only restore | fixed only | known range roundtrip test 필요 |
| smart restore meals | local snapshot basis 유지 | range basis merge policy 필요 |
| Recent | fixed target aggregation | mixed basis detection 필요 |
| candidateScorePreview | transient only | production score decision 전까지 저장 금지 |
| scoring version | v6.1 유지 | range-aware production score 승인 필요 |

## 6. false positive 방지

테스트가 오해하면 안 되는 것:

```text
unknown field pruning은 현재 결함이 아니라 현재 계약이다.
macroRangeContract가 없다는 것은 아직 production 미적용이라는 정상 상태다.
candidateScorePreview가 null인 것은 미완성이 아니라 의도된 report-only 안전장치다.
Recent가 range-aware가 아닌 것은 현재 정상이다.
backup이 range field를 보존하지 않는 것은 현재 정상이다.
```

하지만 다음 중 하나가 발견되면 blocker:

```text
candidateScorePreview가 backup에 들어감
macroRangeContract가 normalize 없이 살아남음
old snapshot이 range field 때문에 null이 됨
smart restore meals-only가 backup snapshot으로 local snapshot을 교체함
Recent가 fixed/range basis를 구분 없이 같은 component count로 합침
ADHERENCE_SCORING_VERSION이 바뀜
```

## 9. 구현 순서 후보

다음 구현 단계가 열릴 경우:

```text
1. runMacroRangeSnapshotCompatibilityDesignTests 추가
2. old fixed snapshot validity test
3. unknown range field pruning test
4. invalid range contract fixed snapshot survival test
5. signature behavior explicit test
6. backup payload no-leak test
7. smart restore meals-only local snapshot basis test
8. Recent fixed-only aggregation guard test
9. candidateScorePreview no-persist test
10. scoring version unchanged test
```

이 순서의 이유:

```text
먼저 현재 계약을 테스트로 고정한다.
그 다음에야 explicit normalizer simulation을 열 수 있다.
```

## 10. 아직 사용자 판단이 필요한 항목

아직 결정하지 않는다.

1. unknown future field를 장기적으로 계속 prune할지.
2. known macroRangeContract의 exact shape.
3. signature include/exclude.
4. smart restore meals-only에서 range basis 충돌을 표시할지.
5. Recent mixed basis를 internal confidence로만 낮출지 UI에 말할지.
6. candidateScorePreview를 언제 숫자로 만들지.
7. range-aware production scoring version 이름.
8. old Records에 range candidate를 report-only로 계산할지.
9. backup conflict preview에서 macro basis 차이를 보여줄지.
10. invalid range contract가 있을 때 user-facing warning을 낼지.

## 11. 최종 판단

snapshot compatibility test design은 진행 가능하다.

하지만 아직 금지:

```text
schema 구현
normalizer 구현
backup/restore 연결
Recent range-aware 연결
score version 변경
candidateScorePreview 숫자화
UI 노출
```

최종 결론:

```text
다음에 열 작업은 test implementation이다.
그 test implementation도 production schema 구현이 아니다.
테스트가 old/new basis 혼합 방지와 no-leak guard를 먼저 증명해야 한다.
```
