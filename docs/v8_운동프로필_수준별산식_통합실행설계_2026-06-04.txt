# V8 운동 프로필·수준별 산식 통합 실행설계

작성일: 2026-06-04
상태: 후속 구현 기준 / production 구현 전 실행설계
브랜치 기준: `codex/v8-formula-audit`

이 문서는 기존 `v8_수준별산식_설계1~8`, `v8_수준별산식_초정밀설계_2026-06-04.txt`, `v8_운동프로필_산식_정밀설계.txt`의 유효한 판단을 통합한 현재 기준 문서다.

기존 1~8 문서는 report-only 단계 기록이다. 다음 작업자는 그 문서들을 최종 기준처럼 따르지 말고, 이 통합문서를 먼저 읽은 뒤 필요한 경우 archive의 원문을 근거 확인용으로만 읽는다.

## 0. 가장 중요한 운영 원칙

이 문서는 정답지가 아니다.

이 문서는 다음 코덱스가 생각을 멈추지 않도록 만드는 작업헌장이다.

기준문서가 있어도 다음 상황에서는 문서를 그대로 따르지 않는다.

- 실제 앱 렌더와 문서가 다를 때
- 사용자 행동 흐름이 문서보다 더 자연스러운 방향을 보여줄 때
- 산식 결과가 생리학적으로 이상하거나 설명이 불가능할 때
- 테스트는 통과하지만 화면·기록·백업·코치 문구 중 하나가 어색할 때
- 구현 중 새 충돌, 새 위험, 새 좋은 방향이 발견될 때
- 이전 코덱스가 "완료"라고 썼지만 실제로는 report-only, scoped production, QA 부족 상태일 때

이 경우의 처리 규칙:

1. 현재 문서를 절대 기준으로 삼지 않는다.
2. 실제 코드, 실제 렌더, 실제 사용자 입력 흐름을 다시 확인한다.
3. 문제가 더 좋은 방향을 요구하면 문서와 다르게 구현할 수 있다.
4. 단, 왜 문서와 달라졌는지 코드 주석이나 문서의 결정 로그에 남긴다.
5. "문서에 맞췄다"는 이유만으로 성공이라고 말하지 않는다.

금지 표현:

- "기준문서대로 했으므로 완료"
- "테스트가 통과했으므로 완료"
- "기능이 생겼으므로 완성"
- "대표 케이스가 통과했으므로 모든 유형에서 안전"
- "report-only candidate가 좋으므로 production 적용 가능"

항상 구분해야 하는 말:

- 생겼다
- 연결됐다
- 계산된다
- 렌더된다
- 사용자가 이해할 수 있다
- 생리학적으로 방어 가능하다
- 기록·백업·코치·점수·Today 기준과 함께 믿을 만하다

마지막 단계까지 도달해야 "믿을 만하게 만들었다"고 말할 수 있다.

### 0-1. 단계 통과 판정 방식

각 단계의 통과 기준은 "구현됐는가"가 아니라 "의도대로 충분히 구현됐는가"다.

다음은 통과가 아니다.

- 코드에 함수나 필드가 생겼다.
- 화면에 UI가 렌더된다.
- 테스트가 특정 숫자만 통과한다.
- report-only 결과가 runner에 붙었다.
- 문서의 항목명이 코드에 등장한다.

통과하려면 아래를 함께 확인한다.

1. 깊이: 해당 기능이 사용자의 원래 의도를 충분히 반영하는가. 예를 들어 운동 수준 판단이면 주간 웨이트일수만 보지 않고 운동 프로필, 세션, 빈도, 시간, 강도, 유산소, 체성분 신뢰도, FFMI/LBM/골격근량 단서를 함께 본다.
2. 넓이: Today, Records, 최근 흐름, InBody, 설정, 백업/복원, Coach, Score 중 영향을 받는 영역이 어긋나지 않는가.
3. 설명 가능성: 사용자가 "왜 이 값/판단이 나왔는지" 물었을 때 내부 label이 아니라 사람 말로 설명 가능한가.
4. 실제 사용자 관점: 기능이 단순히 생긴 게 아니라 해당 운동을 오래 한 사람도 앱을 끄지 않을 만큼 믿을 만한가.
5. 회귀 위험: 기존 scoped production, 기록 snapshot, 기준갱신 signature, 테스트 기대값을 의도치 않게 깨지 않았는가.

따라서 "이미 구현되어 있음"은 중복 구현을 막기 위한 상태 판정일 뿐, 품질 통과 판정이 아니다. 이미 존재하는 기능도 의도 적합성 감사에서 깊이/넓이/설명 가능성이 부족하면 보강하거나 재설계한다.

## 1. 문서 우선순위

충돌이 생기면 아래 순서로 판단한다.

1. 사용자가 이번 대화에서 명시한 최신 의도
2. 앱의 실제 사용자 경험과 실제 렌더
3. 현재 `index.html` production 경로의 실제 동작
4. 이 통합 실행설계 문서
5. `docs/개발정책.txt`, `docs/내가-바라는-앱의-성격.txt`
6. archive로 이동된 설계 1~8 원문
7. 테스트 결과
8. 코덱스의 추정

주의:

- 테스트는 중요하지만 판단의 최상위가 아니다.
- 문서도 중요하지만 실제 앱의 불편함을 덮을 수 없다.
- 기존 코드도 중요하지만 생리학적·사용자적 타당성이 없으면 바꿔야 한다.

## 2. 현재 production 사실

2026-06-04 기준, production으로 봐도 되는 것:

- 일반 diet 단백질 기본값은 `2.0g/kg`이다.
- `GOALS.diet.proteinKg = 2.0`이다.
- `getDualBasisProteinRange("diet")`는 `{ min: 1.4, mid: 2.0, max: 2.0 }`이다.
- 목표는 diet / cut / recomp / maintain / lean_bulk / bulk로 확장되어 있다.
- 운동 프로필은 bodybuilding / powerbuilding / strength / running / mixed로 들어와 있다.
- 운동 프로필·루틴·세션은 Today 계산과 Records goalSnapshot에 일부 연결되어 있다.
- 전문가 LBM 반영률은 사용자가 보는 슬라이더값이 기본 source of truth이고, 가정용 InBody만 0.8 reliability factor가 적용된다.
- 기준갱신 snapshot에는 운동 프로필, 루틴/세션, 운동강도, 유산소, 전문가 LBM 관련 값이 포함되어 있다.

production으로 과대해석하면 안 되는 것:

- 운동 프로필이 생겼다고 profile-specific macro formula가 완성된 것은 아니다.
- 프로필별 세션명이 생겼다고 실제 운동인을 설득할 만큼 정확한 세션 모델이 완성된 것은 아니다.
- candidate-v2 scoped production은 모든 사용자 수준별 산식 승인이 아니다.
- report-only runner 결과는 production formula가 아니다.
- 18개 또는 146개 케이스는 full Cartesian 검증이 아니다.

## 3. 현재 열린 gate

다음은 아직 닫히지 않았다.

1. user-level profile-specific macro formula production implementation
2. broad profile/routine/session human UX review
3. full 8-2 Cartesian execution 또는 owner-approved 대체 gate
4. 모든 목표·모든 사용자 유형·모든 진행상황에 대한 산식 신뢰성 감사
5. 프로필별 실제 수행능력 입력 설계
6. 프로필별 루틴/세션 정확도 검증
7. 사용자 설명문·Coach·Records·백업 스냅샷까지 이어지는 신뢰성 검증

## 4. 현재 구현의 핵심 문제

### 4-1. 운동일수만으로 수준을 판단하는 위험

현재 production 쪽에는 `getTargetRateTrainingBand()`처럼 주간 웨이트일수에 크게 의존하는 판단이 남아 있다.

이 방식의 위험:

- 체성분이 좋은 사람이 주 2회만 운동하면 초급자로 오판될 수 있다.
- 운동 경력이 길지만 현재 감량기라 빈도를 낮춘 사용자가 낮은 수준으로 오판될 수 있다.
- 초보자가 주 5회 운동한다고 고급자로 오판될 수 있다.
- 러닝, 스트렝스, 파워빌딩, mixed는 "웨이트일수" 하나로 설명되지 않는다.

따라서 후속 구현에서는 주간 운동일수를 단독 수준 판단 근거로 쓰지 않는다.

주간 운동일수는 하나의 signal일 뿐이다.

### 4-2. 프로필·세션이 생겼지만 아직 믿을 만하지 않다

현재 프로필별 세션은 이름, 기본 운동시간, xw, 유산소 기본값 중심이다.

이것은 "프로필별 입력 경로가 생겼다"에 가깝다.

아직 부족한 것:

- 각 운동 종목의 실제 수행능력 입력
- 운동 경력
- 세션별 부하·회복·연료 요구량의 타당한 기준
- 사용자가 봤을 때 "이 앱이 내 운동을 이해한다"는 신뢰
- 프로필별 루틴/세션 용어의 정확도
- Today 단순 모드와 세부 모드의 설명

사용자가 지적한 것처럼 각 프로필별 루틴/세션도 현재는 모자라 보인다.

후속 구현에서는 "일단 생긴 프로필"을 믿지 말고, 실제 사용자가 프로필 설정 단계에서 앱을 끄지 않을 정도로 정확한지 검토해야 한다.

## 5. 목표별 산식 방향

목표는 단순 이름이 아니라 산식 정책을 바꿔야 한다.

목표별 기본 성격:

- diet: 일반 감량. 지속 가능성과 과단백 방지를 우선한다.
- cut: 더 명확한 감량. 저체지방·훈련 수준·기록 품질 gate가 필요하다.
- recomp: 체중 변화보다 체성분 개선과 기록 안정성을 우선한다.
- maintain: 체중 유지와 수행 안정성을 우선한다.
- lean_bulk: 지방 증가를 제한하며 점진적 증가를 목표로 한다.
- bulk: 더 적극적 증가지만 체지방 증가와 식단 실행률을 같이 본다.

원칙:

- 목표별 rate는 하나의 고정 기본값만으로 끝내지 않는다.
- 단, 기록이 부족한 사용자의 targetCal을 자동으로 크게 바꾸지 않는다.
- 14일은 관찰과 안내 중심이다.
- 28일은 제안 후보를 만들 수 있는 최소 단위로 본다.
- production 자동 적용은 별도 gate가 닫히기 전까지 금지한다.

## 6. 다차원 사용자 컨텍스트

후속 산식은 다음 컨텍스트를 함께 묶어야 한다.

### 6-1. 체성분 컨텍스트

필수 또는 강력 권장 입력:

- 키
- 체중
- 체지방률
- 골격근량 또는 제지방량
- 체지방량
- InBody 출처: 전문 InBody / 가정용 InBody / 직접 입력 / 없음

파생값:

- BMI
- FFMI
- LBM
- 체지방률 band
- 체성분 신뢰도

체성분 사용 원칙:

- 체지방률은 명확한 팩트에 가깝지만, 측정 출처 신뢰도는 다르다.
- 골격근량과 키를 같이 봐야 체격 대비 근육 수준을 더 잘 볼 수 있다.
- InBody 단일 값만으로 targetCal이나 단백질을 공격적으로 올리지 않는다.
- 가정용 InBody는 신뢰도 보정이 필요하다.

### 6-2. 운동 프로필 컨텍스트

기본 프로필:

- 기본값은 bodybuilding이 자연스럽다.
- 앱이 기본적으로 보디빌딩/체성분 기반 영양 코치 성격을 갖고 있기 때문이다.
- 단, 사용자가 다른 프로필을 고르면 그 프로필이 실제 산식과 설명에 반영되어야 한다.

프로필 후보:

- bodybuilding
- powerbuilding
- strength
- running
- mixed

후속 검토 후보:

- crossfit/metcon이 mixed에 충분히 포함되는지
- endurance running과 general running을 나눌 필요가 있는지
- rehabilitation/recovery 성격을 별도 프로필로 둘 필요가 있는지

### 6-3. 실제 수행능력 컨텍스트

주간 운동일수보다 중요한 입력 후보:

공통:

- 운동 경력
- 최근 4주 평균 운동 빈도
- 최근 4주 훈련 지속성
- 주된 운동 목적
- 부상/회복 이슈

bodybuilding:

- 주요 운동 8~12RM 또는 working set 기준 중량
- 주당 부위별 빈도
- 세트 수 또는 대략적 볼륨
- 훈련 RPE/RIR 성격

powerbuilding:

- 주요 리프트 1RM 또는 e1RM
- 보디빌딩식 보조운동 볼륨
- heavy/volume/accessory 비율

strength:

- squat / bench / deadlift / optional OHP 1RM 또는 e1RM
- 훈련 목적: maximal strength / volume / technique / deload
- 최근 고중량 빈도

running:

- 주간 거리
- 5km 또는 10km 기록
- easy pace
- long run 거리 또는 시간
- tempo/interval 빈도

mixed:

- 저항운동 요소
- 유산소/메트콘 요소
- 세션 내 강도
- 주당 hybrid session 빈도

주의:

- 이 입력을 모두 처음부터 강제로 받으면 사용자가 피곤해진다.
- 그러나 앱이 전문적인 산식을 주장하려면 최소한 어떤 입력이 없어서 추정 중인지 정직하게 보여줘야 한다.

## 7. 운동 수준 판단 방식

`performanceLevel`은 사용자가 직접 고르는 버튼이 아니다.

파생 컨텍스트다.

입력 신호:

- 주간 운동 빈도
- 운동 경력
- 체성분 수준
- FFMI 또는 키 대비 골격근량
- 프로필별 수행능력
- 오늘 세션 종류
- 최근 기록 지속성
- 체중/체성분 변화 흐름

출력 예시:

- insufficient_evidence
- beginner_context
- recreational_trained
- intermediate_context
- advanced_context
- specialist_context

주의:

- 이 label을 사용자에게 그대로 노출하지 않는다.
- 사용자에게는 "현재는 보디빌딩 기본 기준으로 계산 중입니다", "러닝 기록이 없어 유산소 부하를 보수적으로 반영합니다"처럼 설명한다.
- 내부 label은 산식과 감사용이다.

## 8. 프로필별 세션 모델

세션은 이름이 아니라 아래 dimension을 가져야 한다.

- resistanceLoad
- cardioLoad
- recoveryNeed
- fuelNeed
- targetReliefPressure
- proteinNeedContext
- carbFloorContext
- fatProtectionContext
- restOrTraining
- confidence

예시:

bodybuilding PUSH:

- resistanceLoad: moderate_high
- cardioLoad: none/supplemental
- recoveryNeed: moderate
- fuelNeed: moderate
- carbFloorContext: moderate

strength heavy lower:

- resistanceLoad: high
- cardioLoad: none
- recoveryNeed: high
- fuelNeed: moderate_high
- carbFloorContext: moderate

running long:

- resistanceLoad: none
- cardioLoad: high
- recoveryNeed: high
- fuelNeed: high
- carbFloorContext: high

mixed circuit:

- resistanceLoad: moderate
- cardioLoad: moderate_high
- recoveryNeed: moderate_high
- fuelNeed: high
- carbFloorContext: moderate_high

REST:

- 공통 휴식 세션이다.
- 보디빌딩 fallback 토큰으로 오해하지 않는다.
- 운동시간과 훈련강도는 0으로 처리한다.

## 9. 산식 레이어 순서

후속 산식은 다음 순서로 계산한다.

1. 입력 수집
2. 체성분 신뢰도 계산
3. 운동 프로필/세션 컨텍스트 계산
4. 수행능력/운동 수준 컨텍스트 계산
5. 최근 기록 품질 계산
6. 목표별 기본 target range 계산
7. 안전선 계산
8. protein anchor 계산
9. fat protection 계산
10. carb remainder 계산
11. profile/session fuel need와 carb floor 충돌 검사
12. target relief 후보 생성
13. 기록 기반 14/28일 검토
14. 사용자 설명문 생성
15. Records goalSnapshot 저장
16. Backup/restore roundtrip 확인

금지:

- protein을 남는 kcal를 채우는 filler로 쓰지 않는다.
- running/mixed라고 무조건 탄수를 크게 올리지 않는다.
- 최근 흐름이 부족한데 targetCal을 자동 변경하지 않는다.
- 체성분 값 하나만 보고 전문 운동인처럼 계산하지 않는다.
- "기본값"이라는 이유만으로 생리학적 타당성을 생략하지 않는다.

## 10. 단백질 정책

현재 일반 diet 기본값은 2.0g/kg이다.

이전 2.4g/kg 정책의 의도:

- 감량 중 근손실을 줄이고 싶었다.
- 운동인이 단백질을 충분히 먹게 하려는 보수적 의도였다.
- 체성분 기반 앱의 전문성을 강조하려는 의도가 있었다.

왜 문제가 됐는가:

- 모든 diet 사용자에게 2.4g/kg는 과할 수 있다.
- 사용자의 체지방률, 훈련 수준, 제지방 신뢰도, 감량 강도, 수행 맥락을 보지 않고 높게 잡았다.
- 단백질이 높아지면 탄수화물이 과하게 밀려, 운동 수행과 식단 지속성이 떨어질 수 있다.
- 고단백이 필요한 맥락과 일반 감량 맥락을 구분하지 못했다.

개정 원칙:

- 일반 diet 기본은 2.0g/kg을 유지한다.
- 더 높은 단백질은 저체지방, 높은 훈련 수준, 강한 감량, 신뢰 가능한 체성분, 충분한 탄수/지방 여유가 있을 때만 후보로 둔다.
- 단백질은 목표/체성분/훈련 맥락의 결과여야 한다.
- "칼로리가 남아서 단백질을 올림"은 금지한다.

## 11. 목표 rate 정책

rate는 목표별 범위로 다룬다.

단일 기본값보다 범위가 타당한 이유:

- 사람마다 체지방률, 훈련 수준, 목표, 지속성, 기록 품질이 다르다.
- 같은 diet라도 초고도비만 사용자와 저체지방 운동인은 적절한 감량 속도가 다르다.
- 같은 bulk라도 초보자와 고급자는 지방 증가 허용 폭이 다르다.

하지만 범위 적용은 조심해야 한다.

자동 조정 조건:

- 최근 기록 커버리지 충분
- 체중 흐름 충분
- 식단 실행률 충분
- 체성분 신뢰도 충분
- 설명문 준비
- 사용자가 바꿀 수 있는 회수/확인 UX 존재

권장 구조:

- 14일: 관찰, 경고, 설명
- 28일: 제안 후보
- 사용자가 승인한 뒤 반영

## 12. 사용자 설명 원칙

사용자에게 내부 용어를 보여주지 않는다.

보여주면 안 되는 말:

- candidate-v2
- profileSession
- targetReliefRisk
- selectedMacroBasis
- formulaAuthority
- recordConfidence
- windowFit
- contextFit

대신 보여줄 말:

- "최근 기록이 부족해서 목표 변경은 제안하지 않습니다."
- "러닝 기록이 없어 유산소 부하는 보수적으로 반영했습니다."
- "가정용 InBody 기준이라 체성분 반영을 낮춰 계산했습니다."
- "이번 28일 흐름에서는 체중 변화가 목표보다 빠릅니다. 다음 주 목표 칼로리 조정을 검토할 수 있습니다."
- "오늘은 하체 고강도 세션이라 탄수화물 여유를 더 보존했습니다."

Score와 Coach는 역할을 분리한다.

- Score: 왜 점수가 나왔는지 근거를 설명한다.
- Coach: 다음 행동을 말한다.

## 13. Records / Today / InBody / Settings 책임

Today:

- 오늘 계산 기준
- 오늘 세션
- 오늘 운동/유산소
- 오늘 목표 섭취량
- 기준 갱신 여부

Records:

- 날짜별 실제 기록
- 당시 목표와 당시 계산기준
- 식사, 체중, 메모, InBody 기록 연결
- 수정 시 Today 기준과 다른 값이 생기면 정직하게 표시

InBody:

- 체성분 원천
- 측정 기록 목록
- 변화 그래프
- 체성분 신뢰도 판단 재료

Settings:

- 기본 운동 프로필
- 기본 루틴/세션
- 전문가 조정 기본값
- 백업/복구

주의:

- 책임이 섞이면 기준갱신이 어색하게 뜨거나, Records 수정만 했는데 Today가 바뀐 것처럼 보일 수 있다.
- 기준갱신 감사는 별도 작업으로 유지한다.

### 14-1. 필수 수동 대표 사용자

- 근육 많고 체지방도 높은 사용자

- 식단 실행률 높음

### 14-2. 자동 검증

자동 검증은 다음을 본다.

- 목표 kcal가 안전선을 깨지 않는가
- 단백질이 과도하게 올라가지 않는가
- 지방이 지나치게 낮아지지 않는가
- 러닝/고강도 세션에서 탄수가 지나치게 낮아지지 않는가
- 기록이 부족한데 자동으로 targetCal을 바꾸지 않는가
- Records snapshot과 Today 기준이 의도 없이 섞이지 않는가
- Backup/restore 후 같은 결과가 나오는가

### 14-3. 렌더 검증

코드와 테스트만으로 끝내지 않는다.

확인해야 하는 화면:

- Today 기본/펼침/기준갱신
- Records 기록 있음/없음/수정/날짜선택/상세
- Recent Flow 3/7/14/28
- InBody 요약/상세/변화/기록선택
- Settings 기본/전문가/백업/복구
- 모바일 주요 화면

## 15. 작업 중 새 문제가 생겼을 때

새 문제가 생기면 아래처럼 분류한다.

- P0: 현재 산식 결과가 명백히 위험하거나 완전히 틀림
- P1: 사용자가 잘못 이해할 가능성이 높음
- P2: UI/설명/흐름이 어색하지만 계산 자체는 유지 가능
- P3: 후속 개선 후보

처리 규칙:

- P0/P1은 문서 계획보다 우선할 수 있다.
- P2는 현재 단계와 충돌하면 기록하고 단계 종료 후 처리한다.
- P3는 후속 후보로 둔다.

중요:

- 새 문제를 봤는데 문서에 없다는 이유로 무시하지 않는다.
- 반대로 새 생각이 떠올랐다고 현재 작업을 무작정 갈아엎지 않는다.
- 현재 변경이 다른 탭, 기록, 백업, 코치 문구에 어떤 영향을 주는지 먼저 본다.

## 16. 실제 작업 운영 루프

이 섹션의 목적은 코덱스가 기준문서를 고정된 정답처럼 따라가는 것을 막고, 사람이 작업할 때처럼 관찰, 판단, 수정, 재검증을 반복하게 만드는 것이다.

따라서 이 루프는 유연성을 제한하는 규칙이 아니라, 코덱스가 스스로 멈춰서 다시 판단하도록 만드는 안전장치다. 사람은 작업 중 새 문제를 보면 자연스럽게 계획을 고치지만, 코덱스는 그 중단점과 재판단 조건을 명시해야 같은 방식으로 움직일 가능성이 높아진다.

각 단계는 아래 루프를 따른다.

### 16-1. 단계 시작 전

작업을 시작하기 전에 반드시 다음을 짧게 고정한다.

- 이번 단계의 목적
- 이번 단계가 production 변경인지, report-only 비교 레이어인지, 문서/감사 작업인지
- 이번 단계에서 건드릴 수 있는 파일과 건드리면 안 되는 파일
- 예상되는 숫자 변화
- 예상되는 UI/문구 변화
- Records, Today, Coach, Score, Backup 중 영향을 받을 수 있는 영역
- 이번 단계에서 발견될 수 있는 위험

이 단계에서 "문서에 있으니까 구현한다"는 식으로 시작하지 않는다.

먼저 현재 앱이 실제로 어떻게 동작하는지 확인한다.

### 16-2. 관찰 먼저, 구현은 그 다음

구현 전에 확인한다.

- 현재 `index.html`의 실제 production 경로
- 현재 렌더 화면
- 현재 테스트가 무엇을 보장하고 무엇을 보장하지 않는지
- 현재 사용자가 실제로 보는 문구
- 현재 Records snapshot과 기준갱신 signature

문서와 코드가 다르면 코드가 무조건 옳다는 뜻도 아니고, 문서가 무조건 옳다는 뜻도 아니다.

둘 중 어느 쪽이 사용자에게 더 자연스럽고 생리학적으로 방어 가능한지 판단한다.

### 16-3. 작게 붙이고 비교한다

V8 수준별 산식은 한 번에 production으로 갈아엎지 않는다.

권장 진행:

1. report-only context를 만든다.
2. 기존 production 결과와 새 후보 결과를 나란히 비교한다.
3. 대표 사용자와 경계 사용자에서 차이를 본다.
4. 차이가 생긴 이유를 설명할 수 있는지 확인한다.
5. 설명할 수 없는 차이는 production에 반영하지 않는다.

특히 `performanceContext`, `profileEvidence`, `sessionModel`, `targetReliefCandidate`는 먼저 report-only로 붙인다. 단, 이미 report-only로 존재하는 브리지는 재구현하지 않고 상태 확인/보강 대상으로 읽는다.

### 16-4. 예상 변화와 실제 변화 비교

구현 후에는 반드시 다음 질문에 답한다.

- 예상한 kcal 변화와 실제 kcal 변화가 같은가?
- 단백질·지방·탄수 변화가 예상 범위 안인가?
- targetCal은 변하지 않아야 하는 상황에서 변하지 않았는가?
- Records만 수정했는데 Today 기준갱신이 뜨는 등 책임 혼선이 없는가?
- 0으로 보정하면 안 되는 값을 0으로 보정하지 않았는가?
- 가정용 InBody 보정, 전문가 LBM 반영률, 운동세션 기본값이 서로 중복 보정되지 않았는가?
- 운동일수 하나만으로 사용자 수준이 결정되지 않았는가?

이 답을 못 하면 다음 작업으로 넘어가지 않는다.

### 16-5. 새 발견을 처리하는 방식

작업 중 새 문제가 나오면 아래 중 하나로 분류한다.

- 즉시 수정: 현재 작업의 정확성을 깨는 문제
- 같은 커밋에 포함: 현재 변경의 자연스러운 일부
- 별도 커밋으로 분리: 같은 영역이지만 다른 책임의 문제
- 후속 후보: 지금 처리하면 범위가 커지는 문제
- 문서 수정: 기존 기준이 틀렸거나 부족한 문제

분류 기준:

- 사용자가 지금 직접 겪는 오류인가?
- 산식 결과를 틀리게 만드는가?
- 사용자가 잘못 이해하게 만드는가?
- 다른 탭이나 백업에 파급되는가?
- 지금 고치면 현재 단계 검증이 흐려지는가?

새 문제를 발견했는데 그냥 넘어가면 안 된다.

지금 고치지 않는다면 왜 미루는지 남긴다.

### 16-6. 문서와 다르게 가는 조건

아래 경우에는 문서와 다르게 가도 된다.

- 실제 렌더가 문서의 흐름보다 더 나은 방향을 보여줄 때
- 사용자 입력 체감이 문서 설계보다 무겁거나 불편할 때
- 운동 프로필 사용자에게 문서의 세션명이 부정확해 보일 때
- 산식이 문서대로면 특정 사용자군에서 단백질, 지방, 탄수, kcal이 어색해질 때
- 기록/백업/기준갱신 책임이 문서보다 더 명확한 구조를 요구할 때

단, 이탈할 때는 다음을 남긴다.

- 기존 문서 기준
- 실제 발견
- 왜 문서보다 새 판단이 더 나은지
- 영향을 받는 코드/화면/테스트
- 후속 위험

### 16-7. 커밋 단위

커밋은 의미 단위로 나눈다.

섞으면 안 되는 것:

- 문서 정리와 production 산식 변경
- report-only candidate와 production 적용
- UI 정리와 산식 변경
- 테스트 기대값 변경과 실제 계산 로직 변경
- 백업/스냅샷 변경과 단순 문구 변경

권장 커밋 단위:

- 문서/기준 정리
- 현재 코드 경로 감사
- report-only context 추가
- candidate 비교 추가
- production 반영
- 렌더/문구 반영
- 테스트/회귀 보강

### 16-8. 단계 종료 조건

한 단계를 끝냈다고 말하려면 아래가 확인되어야 한다.

- git status가 확인됐다.
- 변경된 파일 범위가 설명 가능하다.
- 예상 변화와 실제 변화가 비교됐다.
- 테스트를 돌렸거나, 왜 돌리지 않았는지 설명할 수 있다.
- 필요한 경우 렌더를 봤다.
- 새로 생긴 위험과 미룬 작업이 기록됐다.
- production 변경인지 report-only 변경인지 분명히 말했다.

이 조건이 없으면 "완료"라고 말하지 않는다.

### 18-1. 7단계 실행 블록

기본 작업 방식은 `/GOAL` 자동 진행이 아니라 아래 7단계를 수동으로 끊어 진행하는 것이다. `/GOAL`은 사용자가 명시적으로 원할 때만 이 7단계 중 한 단계 단위로 사용한다. V8 전체 완료를 한 번에 맡기지 않는다.

사용자 실행 명령:

`다음 한 단계만, 충돌 발견 시 멈추고 재판단`

이 문장이 나오면 다음처럼 해석한다.

1. 아래 7단계 중 다음 미완료 단계 하나만 선택한다.
2. 단계 시작 전에 목적, production/report-only/문서 작업 여부, 건드릴 파일, 예상 변화, 확인 방법을 짧게 고정한다.
3. 단계 진행 중 문서·코드·렌더·테스트·사용자 최신 의도가 충돌하면 다음 단계로 넘어가지 않는다.
4. 충돌은 먼저 `문서 수정`, `코드 수정`, `설계 재판단`, `사용자 승인 필요` 중 하나로 분류한다.
5. 충돌이 없고 단계가 닫히면 그 단계의 결과만 보고한다. 사용자가 다시 지시하기 전까지 다음 단계 구현을 시작하지 않는다.
6. git stage/commit/push/tag는 작업 경계, 테스트 결과, 다른 컴퓨터 인수인계 필요성을 보고 Codex가 판단한다. 사용자가 git 판단을 맡겼다면 매 단계마다 승인 질문을 반복하지 않는다.

이 실행 명령은 기능 구현과 별개인 운영 규칙이다. 따라서 이 규칙을 보강하는 작업은 산식/프로필 production 변경과 섞지 않는 것이 원칙이다. 다만 이미 같은 단계의 report-only 보강과 함께 문서 경계를 닫아야 할 때는 커밋 메시지에서 두 범위를 명확히 적는다.

각 단계가 끝날 때마다 실제 앱/코드/렌더/테스트/문서를 다시 대조한다.

2026-06-04 수동 진행 상태:

- 1단계 현재 산식/프로필 입력 경로 재감사는 완료됐다.
- 2단계 `performanceContext` report-only 브리지는 현재 `index.html`의 `v8.0-BI_user_level_production_context_bridge_v0`로 이미 존재한다.
- 따라서 2단계는 새 구현이 아니라 의도 적합성 감사/필요 보강으로 진행한다. `performanceContext`, `profileEvidence`, `bodyCompositionContext`, `riskFlags`, 테스트 기대값, 사용자 설명 가능성이 원래 의도에 맞는지 확인하기 전에는 3단계로 넘어가지 않는다.
- 2026-06-04 보강: 2단계 브리지에 `levelConflictSignals`를 추가했다. 현재 production `trainingBand`가 주간 웨이트일수에 크게 의존하는 상태에서 `좋은 체성분 + 낮은 주간 빈도`, `낮은 체성분 + 높은 주간 빈도`가 생기면 `productionBandEvidenceConflict`, `goodBodyCompositionLowFrequencyReview`, `lowBodyCompositionHighFrequencyReview`를 report-only로 남긴다. 이것은 production 산식 변경이 아니라 다음 단계 판단을 안전하게 하기 위한 설명/감사 증거다.
- "다음 단계가 남아 있음" 자체는 충돌이 아니다. 충돌은 문서가 미구현처럼 말하는데 코드에 이미 구현되어 있거나, 문서가 완료처럼 말하는데 실제 앱/테스트/렌더가 그렇지 않을 때 선언한다.

1. 현재 산식/프로필 입력 경로 재감사
   - production `calculateMacroTargets()`, `buildGoalTargetRateContext()`, `getTargetRateTrainingBand()`가 현재 사용자 입력을 어떻게 읽는지 확인한다.
   - Settings/Today/Records/Backup의 `exerciseProfile`, `routinePlan`, `routine`, `goalSnapshot`, 기준갱신 signature가 같은 책임 체계를 유지하는지 본다.
   - 아무 값도 바꾸지 않았는데 기준갱신이 뜨는 경우, 값을 바꿨다가 원상복구했는데 갱신이 남는 경우를 실제 렌더로 확인한다.

2. `performanceContext` report-only 상태 확인/보강
   - 주간 웨이트일수만으로 beginner/intermediate/advanced를 추정하는 구조를 그대로 믿지 않는다.
   - 운동 프로필, 세션, 주간 빈도, 운동 시간, 강도 보정, 유산소, 체성분 신뢰도, FFMI/LBM/골격근량 단서를 묶어 report-only 수준 판단 근거를 만든다.
   - 현재 브리지는 이미 존재하므로 재구현하지 않는다. 대신 `performanceContext`, `profileEvidence`, `bodyCompositionContext`, `riskFlags`가 실제 사용자 설명에 충분한지 감사하고, 부족한 근거만 보강한다.
   - 감사는 "필드가 있다"에서 멈추지 않는다. 사용자가 요구한 다차원 운동 수준 판단 의도, 즉 운동 프로필/세션/빈도/시간/강도/유산소/체성분/FFMI/LBM/골격근량 단서가 주간 웨이트일수 단독 판단을 실제로 대체할 만큼 잘 묶였는지 본다.
   - 이 단계는 production 산식을 바꾸지 않는다. 숫자 산출보다 "왜 이런 수준 판단을 했는지"를 추적 가능하게 만드는 것이 목적이다.

3. 프로필별 세션/루틴 정밀화
   - 보디빌딩 기본값을 유지하되, 파워빌딩/스트렝스/러닝/혼합 프로필이 보디빌딩 세션을 억지로 재사용하지 않게 한다.
   - 각 프로필의 세션 이름, 기본 강도, 휴식/운동일 처리, 유산소 재사용 가능 범위를 실제 사용자 관점에서 검토한다.
   - 세션이 "생겼다"와 해당 운동을 오래 한 사람이 "믿고 시작할 수 있다"를 구분한다.

4. candidate 산식 비교와 충돌 감사
   - 기존 production, 기존 candidate, 새 candidate를 같은 입력으로 나란히 비교한다.
   - 단백질 과다, 지방 하한, 탄수 floor, target relief, 감량/린매스업/벌크업/유지 rate가 서로 충돌하는지 본다.
   - 기존 산식이 왜 그렇게 만들어졌는지, 왜 유지/개정/폐기해야 하는지 근거를 남긴다.

5. Records / Coach / Score / Backup 연결
   - 산식 후보가 Today 숫자만 바꾸고 Records snapshot, Coach 문구, 식단 점수, 기준갱신, 백업/복원과 어긋나면 실패로 본다.
   - user-facing 문구에는 내부 label(`candidate-v8`, `profileSession`, `formulaAuthority` 등)을 노출하지 않는다.
   - 기록 불러오기/수정/복원 후에도 같은 입력은 같은 판단으로 돌아와야 한다.

6. 렌더/시나리오/테스트 검증
   - 대표 사용자 수동 케이스, pairwise/targeted stress, 필요한 자동 테스트를 분리해서 본다.
   - full Cartesian을 실행하지 않았다면 full Cartesian을 실행했다고 말하지 않는다.
   - 렌더 감사는 화면 안정화 후 실제 뷰포트로 확인하며, 모바일/웹에서 정보가 숨겨져서 기능이 사라진 것처럼 보이지 않는지 확인한다.

7. 승인 범위 production 적용
   - report-only 결과가 좋아도 곧바로 전면 production으로 넣지 않는다.
   - 사용자가 승인한 범위, 검증된 목표/프로필/세션 조합, 영향 범위가 닫힌 부분만 production에 적용한다.
   - 적용 후에는 같은 시나리오를 다시 돌려 production 변경 전후의 예상 변화/예상하지 못한 변화를 문서와 보고에 남긴다.

각 단계 공통 중단 조건:

- 문서와 실제 앱 동작이 충돌한다.
- 더 좋은 사용자 흐름이 보이는데 문서가 막고 있다.
- report-only 결과를 production 완료처럼 말하고 싶어지는 지점이 생긴다.
- 테스트 수치만 통과했는데 실제 사용자 설명이 납득되지 않는다.
- 프로필/세션/수준 판단이 실제 운동 사용자에게 부끄러운 수준으로 단순화된다.

이 중 하나라도 발견하면 다음 단계로 넘어가지 말고, "문서 수정", "코드 수정", "설계 재판단", "사용자 승인 필요" 중 어디에 해당하는지 먼저 분류한다.

`/GOAL` 사용 기준:

- 일반적인 현재 작업은 `/GOAL` 없이 이 7단계를 순서대로 진행한다.
- `/GOAL`은 장시간 자동 진행이 필요하거나, 다른 컴퓨터/다른 코덱스에게 특정 한 단계를 명확히 넘길 때만 쓴다.
- `/GOAL`을 쓰더라도 목표는 "7단계 전체"가 아니라 "1단계 현재 산식/프로필 입력 경로 재감사"처럼 한 단계로 제한한다.
- `/GOAL` 중에도 공통 중단 조건이 발견되면 완료를 향해 계속 달리지 말고 중단/재판단한다.

## 20. 현재 결론

V8 수준별 산식은 아직 production 완료가 아니다.

현재 필요한 일은 기준문서를 더 많이 만드는 것이 아니라, 기준문서를 줄이고 더 좋은 판단이 가능한 구조로 만드는 것이다.

이 통합문서는 그 출발점이다.

후속 코덱스는 이 문서를 따라야 하지만, 동시에 이 문서를 의심해야 한다.

앱이 더 좋은 방향을 보여주면 앱과 사용자 경험을 우선한다.
