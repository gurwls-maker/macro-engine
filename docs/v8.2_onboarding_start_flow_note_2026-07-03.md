# v8.2 온보딩 시작 흐름 후보

> 상태: 후속 후보 / 현재 구현 지시서 아님
>
> 분류: 첫 실행 UX / Settings 기본값 초기화 / Today 첫 계산 진입 후보
>
> 현재 지시: v8.1 copy/help 구현 범위에 섞지 않는다. 구현 금지. 별도 UX 설계 감사 필요.
>
> 예상 트랙: v8.2 또는 v8.3 후보. 실제 구현 전에는 Settings 기본값, Today 직접 수정값 우선순위, Records `goalSnapshot`, InBody 적용 흐름, backup/restore 호환성을 먼저 감사한다.

## 1. 현재 기준

온보딩은 삭제된 아이디어가 아니라 후속 후보로 살아 있다.

현재 기준문서 위치:

- `00_현재작업기준_2026-06-16.txt`
  - 온보딩이 생기기 전까지는 Settings에서 운동 여부를 바꾼다.
  - 온보딩은 운동 여부만 간단히 묻는 시작 흐름을 후속 후보로 둔다.
  - 현재 앱은 Settings에서 운동 여부를 바꾸는 흐름이 살아 있으므로, 설명문/도움말 정리보다 앞서 열 필요는 낮다.
- `02_대화의도_근거표_2026-06-16.txt`
  - 온보딩에서는 운동 여부만 간단히 묻는다.
  - 온보딩 전에는 Settings에서 운동 여부를 바꾼다.
  - 온보딩, 설명문 전수조사, 도움말 체계, 데이터 관리, InBody -> Today 반영은 삭제된 후보가 아니다.
- `99_v8.1_takeover_audit_2026-06-30.md`
  - 온보딩에서 운동 여부만 간단히 묻는 흐름 설계는 v8.1 후반 또는 별도 UX 설계 후보로 둔다.
  - 설명문/도움말 정리 이후 검토한다.

## 2. 왜 지금 구현하지 않는가

온보딩은 앱의 첫 실행 흐름을 결정한다. 따라서 단순 copy/help가 아니라 Settings 기본값, Today 첫 계산, 운동 여부 모드, 체성분 입력, InBody 적용, backup/restore 해석까지 영향을 줄 수 있다.

지금 바로 구현하지 않는 이유:

- v8.1 copy/help pass는 기존 의미를 보존하면서 사용자-facing 문구를 줄이는 흐름이다.
- 온보딩은 새 사용자 시작 흐름과 초기값 저장 방식을 만드는 별도 UX 작업이다.
- macro range rebalance 같은 후속 산식/점수 정책 후보와 섞이면 첫 질문과 기본값 설계가 다시 바뀔 수 있다.
- Records `goalSnapshot`과 Today 직접 수정값 우선순위가 흔들리면 과거 기록 해석이 불안정해질 수 있다.

따라서 현재 단계에서는 문서 후보로만 보관하고, v8.1 copy/help 라인을 닫은 뒤 별도 audit으로 연다.

## 3. 온보딩 v1 목표

온보딩 v1의 목표는 고급 설정 마법사가 아니다.

핵심 목표:

```text
처음 앱을 켠 사용자가 Today 계산을 바로 시작할 수 있게
현재 기준에서는 운동 여부만 간단히 묻는 흐름을 우선 검토한다.
```

현재 잠긴 최소 v1:

- 운동 여부를 먼저 묻는다.
- 온보딩 전에는 Settings에서 운동 여부를 바꾸는 현재 흐름을 유지한다.
- 운동 안 함 사용자는 운동 입력을 보지 않아도 된다.
- 운동함 사용자는 현재 Settings/Today 흐름으로 이어진다.

audit에서만 열 수 있는 확장 목표:

- Settings 기본값과 첫 Today 계산에 필요한 최소값을 함께 받을지 판단한다.
- 모르는 값은 건너뛸 수 있게 둘지 판단한다.
- 운동함 사용자의 기본 루틴을 온보딩에서 받을지, Settings로 넘길지 판단한다.

## 4. 입력 후보 분류

현재 기준으로 잠긴 최소 입력:

- 운동 여부

audit 후에만 열 수 있는 확장 후보:

- 목표
- 키
- 성별
- 나이
- 체중
- 기본 생활활동
- 기본 업무 유형

audit 후에만 열 수 있는 선택/건너뛰기 후보:

- 골격근량
- 체지방률
- 체지방량
- InBody 있음 / 없음 / 모르겠음

audit 후에만 운동함 사용자에게 열 수 있는 후보:

- 기본 루틴 종류
- 주당 웨이트일수
- 기본 웨이트 시간
- 유산소 기본값 또는 나중에 설정

현재 v1에서 묻지 않을 후보:

- 단백질 하한/표준/상한 선택
- 탄수/지방 범위 재배분 설정
- 고급 계수
- 식사 태그
- 점수 정책 선호도
- Records 과거 기록 보정

## 5. 구현 전 audit 항목

실제 구현 전에 다음을 먼저 확인한다.

1. Settings 기본값과 Today 직접 수정값의 우선순위
2. 첫 Today 기록 생성 전후에 어떤 값이 저장되는지
3. 온보딩 값을 Settings에만 저장할지, 첫 Today draft에도 채울지
4. 사용자가 온보딩을 건너뛰었을 때 현재 Settings 흐름이 그대로 유지되는지
5. 운동 여부를 바꿨을 때 운동 UI와 운동 계산이 어떻게 숨김/fallback 처리되는지
6. 체성분을 모르는 사용자가 앱을 계속 쓸 수 있는지
7. InBody가 있는 사용자는 온보딩에서 직접 입력할지, InBody 탭으로 유도할지
8. backup/restore 이후 온보딩을 다시 띄우면 안 되는 조건
9. 기존 사용자에게 온보딩을 강제로 띄우지 않는 조건
10. mobile 390px에서 단계 진행, 뒤로가기, 건너뛰기가 안전한지

## 6. 다른 후속 후보와의 경계

macro range rebalance와 섞지 않는다.

- 온보딩 v1은 현재 고정 target macro 정책을 전제로 최소 시작 흐름만 만든다.
- 단백질 기준 선택, 탄수/지방 범위 재배분, 점수 산식 변경은 `v8.2_macro_range_rebalance_note_2026-07-02.md`의 별도 후보로 둔다.

copy/help pass와 섞지 않는다.

- copy/help는 이미 있는 화면의 설명을 줄이는 작업이다.
- 온보딩은 새 첫 실행 흐름이므로 별도 UX 설계, 테스트, mobile QA가 필요하다.

Records/Backup 위험 action copy와 섞지 않는다.

- 온보딩은 삭제/복원 warning 문구를 바꾸지 않는다.
- 온보딩 구현 중 Records 저장 구조, backup schema, smart restore를 바꾸지 않는다.

## 7. 구현 시 테스트 후보

실제 구현 단계에서 필요한 테스트 후보:

- 첫 실행 온보딩 노출 조건
- 기존 사용자에게 온보딩이 다시 뜨지 않는 조건
- 온보딩 건너뛰기 후 기존 Settings 흐름 유지
- 운동함 선택 시 운동 관련 기본값 저장
- 운동 안 함 선택 시 운동 UI/계산 숨김 유지
- 필수값 누락 시 Today 계산 진입 보호
- 체성분 건너뛰기 가능
- InBody가 없어도 Today 계산 가능
- 온보딩 완료 후 Settings 기본값 반영
- 온보딩 완료 후 첫 Today 계산 입력 반영 여부
- backup/restore 이후 온보딩 재노출 방지
- mobile 390px 단계 진행/뒤로가기/건너뛰기

예상 보호선:

- `npm run test:smoke`
- `npm run test:ui`
- `npm run test:mobile`
- `npm run test:core`
- 온보딩 전용 targeted tests
- Settings/Today 관련 regression tests
- 필요 시 backup/restore targeted tests
