# v8.2 macro range score candidate report-only design

상태:

```text
문서 성격: report-only score candidate 설계
구현 변경: 없음
index.html 변경: 없음
production score 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
goalSnapshot / backup / Recent / Today UI 변경: 없음
현재 구현 지시서 아님
```

이 문서는 `v8.2_macro_range_report_only_helper_implementation_2026-07-03.md` 이후 단계다.

목적은 고정 매크로 목표 기반 기존 점수와, range candidate 기반 점수 후보를 같은 입력으로 나란히 비교하기 위한 debug/report-only 설계를 만드는 것이다.

중요:

```text
range score candidate는 production score가 아니다.
getDailyAdherenceScore()를 바꾸지 않는다.
ADHERENCE_SCORING_VERSION을 바꾸지 않는다.
Records 과거 점수를 새 기준으로 재해석하지 않는다.
Today UI에 range score를 노출하지 않는다.
DailyCoach action을 range score로 바꾸지 않는다.
```

## 1. 배경

현재 report-only helper는 fixed target과 range candidate를 비교한다.

이미 닫힌 것:

```text
fixed macro target summary 생성
range candidate summary 생성
fat status 분리
expanded sample matrix 10행
production / score / snapshot / backup / Recent / DOM 무영향 guard
```

아직 열지 않은 것:

```text
range-aware score 계산
range score version
range score evidence copy
Records snapshot range contract
Recent mixed-basis 해석
backup/restore compatibility
Today UI 표시
```

따라서 다음 단계는 production score 구현이 아니라, range-aware score 후보를 별도 debug object로 계산했을 때 어떤 차이가 나는지 관찰하는 설계다.

## 2. 금지선

이번 설계에서 금지:

- `getDailyAdherenceScore()` 변경.
- `ADHERENCE_SCORING_VERSION` 변경.
- fixed score weighting 변경.
- current score evidence copy 변경.
- DailyCoach decision matrix 변경.
- Today score UI 변경.
- Records score display 변경.
- `goalSnapshot`에 score candidate 저장.
- backup / restore payload에 score candidate 저장.
- Recent range-aware logic 연결.
- range score candidate를 final score처럼 이름 붙이기.
- 숫자 정책을 임의 확정하기.

허용:

- 새 문서로 input/output 설계.
- 나중에 구현할 경우의 helper 함수 후보명 정의.
- test 후보명 정의.
- sample matrix와 guard 기준 정의.
- 미결정 값을 `unresolved` / `needsDecision`으로 남기는 계약 정의.

## 3. 기존 fixed score와의 관계

기존 fixed score는 그대로 source of truth다.

```text
current score = production result
range score candidate = report-only comparison result
```

range candidate가 더 좋아 보이는 케이스가 있어도 production score를 대체하지 않는다.

예:

```text
fixed target:
fat target 78g
consumed fat 42g
remaining fat 36g

range candidate:
fat action floor 40g
recommendedMin 50g
status below_recommended

해석:
fixed score는 기존 기준대로 유지한다.
range score candidate는 "fixed target보다 지방 미달 압박이 작을 수 있음"을 report-only로만 표시한다.
```

## 4. helper 후보명

구현이 열릴 경우 후보명:

```text
buildMacroRangeScoreCandidateReportOnly()
```

보조 후보:

```text

buildMacroRangeScoreCandidateComponents()
buildMacroRangeScoreCandidateComparison()
buildMacroRangeScoreCandidateGuards()
```

이름에 반드시 `ReportOnly` 또는 `Candidate`가 들어가야 한다.

금지 이름:

```text
getDailyAdherenceScoreV2
getRangeAdherenceScore
calculateAdherenceScore
buildProductionMacroRangeScore
```

score candidate helper input은 기존 production score input을 직접 바꾸지 않고, 복사된 debug input으로만 받는다.

## 7. fixedScoreSummary 후보

fixed score summary는 기존 결과를 관찰만 한다.

후보 필드:

```text
version: ADHERENCE_SCORING_VERSION
score
label
componentReasons
macroEvidence
kcalDelta
proteinDelta
carbsDelta
fatDelta
dataQualityState
```

주의:

```text
fixedScoreSummary는 getDailyAdherenceScore() 결과를 다시 계산하거나 수정하지 않는다.
이미 production에서 나온 결과를 clone해서 비교 기준으로 둔다.
```

## 8. rangeScoreCandidateSummary 후보

range score candidate는 component별 preview다.

후보:

```text
candidateVersion: "range_score_candidate_report_only_v0"
candidateScorePreview: number | null
componentPreviews: {
  kcal: { status, pointsPreview, reasonIds },
  protein: { status, pointsPreview, reasonIds },
  fat: { status, pointsPreview, reasonIds },
  carbExchange: { status, pointsPreview, reasonIds },
  dataQuality: { status, pointsPreview, reasonIds }
}
impossibleState
unresolved
needsDecision
```

`candidateVersion`은 `ADHERENCE_SCORING_VERSION`이 아니다.

## 9. component 원칙

### 9-1. kcal

`targetCal`은 authoritative다.

range score candidate도 총 kcal mismatch를 무시하면 안 된다.

```text
total kcal matched -> candidate score에서 주요 안정 신호
total kcal high/low -> range 안이어도 감점 후보
```

금지:

```text
fat/carb range가 targetCal을 다시 올리거나 내리는 해석
```

### 9-2. protein

단백질은 anchor다.

```text
protein target candidate satisfied -> 안정 신호
protein low + kcal matched -> 중요한 감점 후보
protein low + one-meal pending -> 단정 약화
```

단백질 숫자 정책은 아직 확정하지 않는다.

미결정이면:

```text
status: "unresolved"
needsDecision: ["protein_policy"]
```

### 9-3. fat

fat은 fixed target을 exact target으로 맞추는 구조에서 range-aware 구조로 바뀔 수 있는 핵심 후보다.

status 후보:

```text
below_action_floor
below_recommended
inside_recommended
above_recommended
above_flexible
unresolved
```

score 후보 의미:

```text
below_action_floor -> 더 강한 감점 후보
below_recommended -> 약한 감점 또는 관찰 후보
inside_recommended -> 안정 신호
above_recommended -> 약한 감점 또는 관찰 후보
above_flexible -> 더 강한 감점 후보
```

아직 결정하지 않는 것:

```text
below_recommended와 above_recommended의 감점 대칭 여부
actionFloor와 scoreLowThreshold 동일 여부
flexibleMax 초과가 fixed fat overage와 같은 severity인지
```

### 9-4. carb exchange

carb는 fat과 교환 가능한 잔여 에너지 후보로 본다.

하지만 다음 조건에서는 carb high를 무조건 허용하지 않는다.

```text
kcal high
protein low
fat below_action_floor
endurance/training support minimum 미충족
data quality insufficient
```

carb high가 감점인지 아닌지는 range candidate와 goal/activity context에 따라 달라질 수 있다.

미결정이면:

```text
needsDecision: ["carb_high_penalty_policy"]
```

### 9-5. data quality

기록이 부족하면 candidate도 단정하지 않는다.

```text
one-meal pending -> strong shortage/overage verdict 금지
snapshotless saved record -> saved record 재해석 금지
insufficientRecord -> score preview null 가능
```

## 10. comparison 후보

fixed score와 range candidate는 나란히 비교한다.

후보:

```text
fixedScore
candidateScorePreview
scoreDeltaPreview
fixedMainReasonIds
candidateMainReasonIds
candidateReducesFatRemainingPressure
candidateExplainsCarbFatExchange
candidateStillFlagsProteinLow
candidateStillFlagsKcalMismatch
candidateBlockedByImpossibleState
```

중요:

```text
scoreDeltaPreview가 양수여도 production score를 바꾸지 않는다.
scoreDeltaPreview가 음수여도 current score를 바꾸지 않는다.
```

## 11. impossible state 후보

range score candidate는 불가능한 조합을 숫자로 억지 해결하지 않는다.

후보 code:

```text
insufficient_policy_numbers
insufficient_data_quality
protein_anchor_consumes_targetCal
fat_floor_breaks_carb_training_minimum
fat_recommended_max_makes_carb_min_negative
targetCal_macro_range_infeasible
fixed_target_and_range_candidate_conflict
```

주의:

```text
targetCal_macro_range_infeasible는 아직 독립 구현 code가 아니다.
현재 helper 구현은 낮은 targetCal 조건에서 fat floor / carb minimum 충돌과 negative carbMin 후보를 관찰한다.
독립 code가 필요할지는 sample matrix와 정책값 검토 후 판단한다.
```

## 12. guard

구현이 열릴 경우 반드시 확인할 guard:

```text
targetCal unchanged
fixed macro target unchanged
getDailyAdherenceScore output unchanged
ADHERENCE_SCORING_VERSION unchanged
goalSnapshot signature unchanged
backup payload unchanged
Recent component counts unchanged
Today DOM text unchanged
DailyCoach production decision unchanged
Coach copy unchanged
Records saved score unchanged
```

helper output의 `guards`는 선언값이다.

실제 무영향 보증은 테스트의 before/after assertion이 담당한다.

## 13. sample matrix 후보

기본 축:

```text
maintain / diet / cut / recomp / lean_bulk / bulk
운동 안 함 / 저항운동 / mixed / endurance
highBF / average / lean
low targetCal / normal targetCal / high targetCal
```

문제 케이스:

```text
fat below_action_floor + carbs high + kcal matched
fat below_recommended + carbs high + kcal/protein matched
fat inside_recommended + carbs high + kcal matched
fat above_recommended + carbs low
fat above_flexible + carbs low
protein low + kcal matched
kcal high + macros inside range
kcal low + protein satisfied
one-meal pending
snapshotless saved record
targetCal low + protein/fat floors infeasible
```

각 case는 다음을 확인해야 한다.

```text
candidate is report-only
candidate does not change fixed score
candidate does not persist
candidate does not update UI
candidate preserves unresolved decisions
```

## 15. 문구 / UI 후보 금지

이번 단계에서 사용자-facing copy를 만들지 않는다.

금지 문구:

```text
범위 점수
새 점수
지방은 더 먹지 않아도 됩니다
탄수/지방 교환 점수
```

이유:

```text
range score candidate는 아직 debug/report-only다.
UI copy를 먼저 만들면 사용자는 production 정책으로 이해한다.
```

## 16. 아직 사용자 판단이 필요한 항목

아직 결정하지 않는다.

1. range score candidate를 숫자로 낼지, component preview까지만 낼지.
2. fixed score와 candidate score의 delta를 표시할지.
3. `below_recommended`를 감점으로 볼지 관찰로 볼지.
4. `above_recommended`를 감점으로 볼지 관찰로 볼지.
5. carb high 감점 조건.
6. protein low severity.
7. data quality pending 상태에서 score preview를 null로 둘지.
8. old Records에 candidate를 계산할지, Today/debug fixture에만 둘지.
9. candidate version 이름.
10. candidate evidence reason id 체계.

## 18. 최종 판단

range score candidate report-only 설계는 진행 가능하다.

하지만 아직 금지:

```text
production range score 구현
score version 변경
getDailyAdherenceScore 변경
Records/Recent 재해석
goalSnapshot schema 변경
backup/restore 변경
Today UI 노출
DailyCoach action 변경
```

이번 문서의 결론:

```text
fixed score는 계속 source of truth다.
range score candidate는 비교용 debug output 후보일 뿐이다.
점수 정책 숫자와 severity는 아직 확정하지 않는다.
```
