# v8.1 개발환경 재현성 결정

작성일: 2026-07-02

이 문서는 v8.1 인수인계 감사와 docs correction 이후, 다음 작업을 시작하기 전에 개발환경 재현성 문제를 어떻게 다룰지 결정하기 위한 운영 문서다.

이번 단계는 개발환경을 바로 바꾸는 작업이 아니다. `index.html`, 앱 산식, UI, 저장 schema, 테스트 기대값, `package.json`은 변경하지 않는다.

## 1. 현재 확인한 상태

| 항목 | 확인 결과 |
| --- | --- |
| 기준 브랜치 | `docs/v8.1-dev-env-reproducibility` |
| 시작 기준 | `1c7a3c2 v8.1 문서 기준 정합성 보정` |
| Node.js | 현재 환경에서 `v24.15.0` |
| npm | 현재 환경에서 `11.12.1` |
| `package.json` | 없음 |
| lockfile | 없음 |
| repo `node_modules` | 없음 |
| 직접 `require("playwright")` | repo 루트에서는 실패 |
| runner 내부 Playwright resolve | 성공. Codex runtime cache 후보를 `NODE_PATH`에 추가함 |
| generated render audit 산출물 | `tools/render_audit/manifest.json` 등 ignored 경로에 존재 가능 |

현재 `run_internal_tests.cjs`, `capture_render_audit.cjs`, `run_v8_full_cartesian_shard.cjs`는 공통적으로 다음 위치를 `NODE_PATH` 후보로 넣는다.

1. repo `node_modules`
2. 실행 중인 Node.js 주변 `node_modules`
3. Codex runtime cache: `%USERPROFILE%\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\node_modules`

따라서 현재 Codex 환경에서는 테스트가 실행된다. 하지만 새 컴퓨터에서 같은 runtime cache가 없거나 Playwright/브라우저 channel 상태가 다르면 실패할 수 있다.

## 2. 테스트 도구 구조

주요 도구:

| 파일 | 역할 |
| --- | --- |
| `tools/render_audit/run_internal_tests.cjs` | iframe으로 `index.html`을 열어 exported browser tests 실행 |
| `tools/render_audit/capture_render_audit.cjs` | desktop/mobile 화면 캡처와 manifest 생성 |
| `tools/render_audit/analyze_render_audit.cjs` | ignored manifest/screenshots 분석 |
| `tools/render_audit/run_v8_full_cartesian_shard.cjs` | bounded full Cartesian shard evidence 실행 |
| `tools/render_audit/*cartesian*` | shard/batch/plan/campaign/ledger 보조 도구 |

`run_internal_tests.cjs`의 profile:

| profile | 의미 |
| --- | --- |
| `smoke` | 빠른 핵심 회귀 |
| `core` | 주요 기능 회귀 |
| `ui` | 화면/문구/모바일 UI 관련 회귀 |
| `mobile --mobile` | 390px 모바일 회귀 |
| `calibration` | 산식/매크로/calibration 계열 |
| `full` | exported `run*Tests` 전체와 `runDailyCoachTestCases` |

브라우저 channel 선택은 `PLAYWRIGHT_BROWSER_CHANNEL`, `msedge`, `chrome`, bundled Playwright 순으로 시도한다.

## 3. 재현성 리스크

현재 구조의 장점:

- repo에 package 파일이 없어도 Codex 런타임 cache가 있으면 바로 테스트가 돈다.
- 기존 runner가 Playwright channel fallback을 갖고 있어 Edge/Chrome 설치 차이에 어느 정도 버틴다.
- generated output은 `.gitignore`에 포함되어 기준 source와 분리된다.

현재 구조의 한계:

- 새 컴퓨터에서 Codex runtime cache가 없으면 `require("playwright")`가 실패할 수 있다.
- Node.js 버전과 Playwright 버전이 repo에 고정되어 있지 않다.
- `npm test`, `npm run smoke` 같은 표준 진입점이 없다.
- 브라우저 설치 상태가 문서만으로는 충분히 재현되지 않는다.
- render audit analyze는 기존 ignored manifest를 읽을 수 있으므로, source 변경 후 capture 없이 analyze만 실행하면 잘못된 UI 증거가 될 수 있다.

## 4. 이번 단계 결정

이번 커밋에서는 `package.json`을 추가하지 않는다.

이유:

- 지금 단계는 문서/운영 기준 결정이며 앱 동작이나 개발환경 기준을 동시에 바꾸는 단계가 아니다.
- `package.json`과 lockfile 도입은 작은 문서 보정이 아니라 개발환경 기준 변경이다.
- package 도입은 설치 방식, Playwright browser install 범위, Windows PowerShell 명령, CI 여부까지 함께 결정해야 한다.
- 설명문/도움말 inventory를 시작하기 전에는 현 구조로 smoke/core/ui/mobile을 실행할 수 있지만, v8.1 구현이 커질수록 repo-local dependency 기준이 필요해진다.

결정:

```text
현재 커밋: docs-only 개발환경 재현성 결정
package.json: 이번 커밋에서는 미도입
다음 운영 후보: 별도 브랜치에서 package.json / lockfile / npm scripts 도입 여부 결정
v8.1 구현 전 권장: package 도입 또는 최소한 설치 절차 문서화를 먼저 닫기
```

## 5. package 도입 판단 기준

다음 조건 중 하나라도 해당하면 `package.json` 도입을 별도 작업으로 여는 것이 맞다.

- 다른 컴퓨터에서 `run_internal_tests.cjs`가 Playwright resolve 실패로 막힌다.
- 설명문/도움말 작업이 여러 커밋으로 이어져 smoke/ui/mobile을 반복 실행해야 한다.
- release 후보에서 full exported, DailyCoach, render audit capture/analyze를 다시 gate로 써야 한다.
- Codex runtime cache에 의존하지 않는 재현 가능한 온보딩 절차가 필요하다.
- GitHub Actions 또는 다른 CI에서 테스트를 실행하려 한다.

도입한다면 한 커밋에 섞지 않는다.

권장 분리:

1. package 도입 결정 문서 또는 짧은 계획
2. `package.json` + lockfile + npm scripts 추가
3. Playwright browser 설치 절차 문서화
4. smoke/core/ui/mobile/calibration 대표 실행 검증

## 6. 현재 권장 QA 명령

문서만 바꾼 경우:

```powershell
git diff --check
git status -sb
git diff --name-only
node tools/render_audit/run_internal_tests.cjs --profile=smoke
```

문구/도움말 inventory 또는 UI copy 작업:

```powershell
node tools/render_audit/run_internal_tests.cjs --profile=smoke
node tools/render_audit/run_internal_tests.cjs --profile=ui
node tools/render_audit/run_internal_tests.cjs --profile=mobile --mobile
```

산식/매크로 작업:

```powershell
node tools/render_audit/run_internal_tests.cjs --profile=core
node tools/render_audit/run_internal_tests.cjs --profile=calibration
node tools/render_audit/run_internal_tests.cjs --suite=runTargetMacroProductionPolicyTests,runAuthoritativeTargetCalTests,runTargetCalSemanticGuardTests,runDualBasisProductionTests,runModeFreeCalculationContextTests
```

release 후보:

```powershell
node tools/render_audit/run_internal_tests.cjs --profile=smoke
node tools/render_audit/run_internal_tests.cjs --profile=core
node tools/render_audit/run_internal_tests.cjs --profile=ui
node tools/render_audit/run_internal_tests.cjs --profile=mobile --mobile
node tools/render_audit/run_internal_tests.cjs --profile=calibration
node tools/render_audit/run_internal_tests.cjs --profile=full
node tools/render_audit/capture_render_audit.cjs
node tools/render_audit/analyze_render_audit.cjs
```

주의:

- render audit analyze는 같은 source 시점에서 생성된 capture/manifest만 UI 증거로 인정한다.
- source 변경 후에는 `capture_render_audit.cjs`부터 다시 실행한다.
- generated output은 커밋하지 않는다.
- `tools/render_audit/manifest.json`, `screenshots/`, `pages/`, `_debug/`는 ignored 산출물이다.
