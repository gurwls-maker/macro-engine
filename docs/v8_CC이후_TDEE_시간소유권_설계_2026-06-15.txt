v8 CC 이후 TDEE 시간 소유권 설계
작성일: 2026-06-15

2026-06-16 정정
----------------
이 문서는 CC 이후 TDEE 시간 소유권을 정리한 중요한 설계 문서지만, 작성 시점의 중간 판단도 함께 들어 있다.
특히 두 기준을 사용자 선택 토글로 병렬 유지한다는 문장은 이후 대화에서 최신 결론으로 폐기되었다.

최신 기준:
- 사용자에게 Today에서 "가이드 기준 / 활동량 기준"을 직접 고르게 하지 않는다.
- 최상위 선택은 "운동을 관리 기준에 포함할지"다.
- 운동함 모드는 기존 가이드 기준의 운동생리학 성격을 흡수한다.
- 운동 안 함 모드는 기존 활동량 기준의 일반 식단 / 체중 관리 성격을 흡수한다.
- TDEE 시간 소유권 설계는 계속 유효하지만, 그 결과를 예전 mini toggle 구조에 억지로 끼워 넣으면 안 된다.

따라서 이 문서의 TDEE 철학은 유지하되, 가이드/활동량 사용자 토글 유지 부분은 `02_대화의도_근거표_2026-06-16.txt`가 덮어쓴다.

2026-06-21 CP/CQ 적용 확인
----------------
시간 소유권 TDEE는 더 이상 "앞으로 만들 report-only 후보" 상태가 아니다.

현재 상태:
- `v8.0-CP_time_ownership_runtime_formula_application_v0`에서 production `calculate()`가 시간 소유권 목표 칼로리를 사용한다.
- `v8.0-CQ_time_ownership_runtime_formula_validation_after_production_application_v0`에서 적용 후 검증이 통과했다.
- 22단계 이후 계산은 `BMR + 훈련기본값 + 업무 순소모 + 입력한 생활활동 순소모 + 운동 순소모`를 목표 칼로리 기준으로 쓴다.
- 웨이트, 유산소, EPOC는 운동 순소모 하나로 접어 target-rate / energy-availability 경로에 한 번만 들어간다.
- 생활활동 시간은 직접 입력한다. 남은 시간은 자동 휴식시간이며 별도 활동 kcal를 더하지 않는다.
- 수면+업무+운동+생활활동 요청 시간이 24시간을 넘으면 생활활동 반영 시간은 남은 시간까지만 제한하고, 화면에서는 24시간 초과를 안내한다.

검증값:
- 앉아서 일함 + 낮은 생활활동 + 휴식: 목표 1,907kcal, 소비 기준 2,253kcal
- 앉아서 일함 + 보통 생활활동 + 휴식: 목표 2,141kcal, 소비 기준 2,553kcal
- 고강도 현장 + 매우 활동적 + 휴식: 목표 4,088kcal, 소비 기준 4,892kcal
- 14단계 당시에는 생활활동을 남은 시간 평균으로 검증했다. 22단계 이후에는 생활활동 시간을 직접 입력하고, 수면/업무/운동 변화는 자동 휴식시간을 움직인다.
- 고강도 업무 4h: 목표 2,651kcal, 업무 순소모 900kcal
- 고강도 업무 12h: 목표 3,897kcal, 업무 순소모 2,699kcal
- 웨이트 60분 + 유산소 30분: 목표 2,685kcal, 운동 순소모 353kcal

따라서 다음 작업은 production 산식 연결이 아니라 사용자 입력/화면 정합성이다.
22~30단계에서 생활활동 시간, 자동 휴식시간, 요일별 기본 시간, Today 시간 배분 요약까지 반영했다.
이후 작업은 입력된 시간 기준이 Records/Recent/문구/검증 흐름에서 계속 자연스럽게 이어지는지 보는 것이다.

2026-06-21 22~25단계 입력 정합성 반영
----------------
생활활동 입력 방식은 아래로 확정해 구현했다.

- Today와 Settings에는 생활활동 시간이 있다. 기본값은 2시간이다.
- 생활활동 강도는 입력한 생활활동 시간 동안의 평균 강도다.
- 하루 시간은 `수면 + 업무 + 운동 + 생활활동 + 자동 휴식`으로 본다.
- 자동 휴식시간은 남은 시간이며, BMR이 이미 하루 기본 에너지를 소유하므로 별도 활동 kcal를 더하지 않는다.
- Records 상세와 기준 수정에도 생활활동 시간과 자동 휴식시간을 표시한다.
- 생활활동 안내는 항상 보이는 본문이 아니라 `?` 도움말로 둔다.
- 업무 유형 가이드도 Today 빠른 설정에서는 선택칸 아래 고정 문구가 아니라 `?` 도움말로 둔다.

따라서 이 문서 아래쪽의 "생활활동은 업무·운동·수면을 뺀 나머지 시간의 평균"이라는 과거 표현은 설계 히스토리로만 읽고, 현재 앱 기준으로 쓰지 않는다.

2026-06-21 26~30단계 기본 시간/요약 반영
----------------
생활 패턴 입력과 Today 결과 설명은 아래로 정리했다.

- Settings에서 요일별 기본 수면/업무/생활활동 시간을 저장한다.
- Today는 해당 요일 기본값을 가져오고, 사용자는 오늘 다른 값만 바꾼다.
- 수면 시간, 업무 시간, 생활활동 시간 설명은 긴 본문이 아니라 `?` 도움말로 둔다.
- Today 계산 요약에는 `시간 배분`과 `자동 휴식`을 표시한다.
- `시간 배분`은 수면/업무/생활/운동 시간이 어떻게 나뉘었는지 보여주는 결과 설명이다. 사용자에게 새 기준을 고르게 하는 토글이 아니다.
- 29단계 화면 확인에서 데스크톱 1280px와 모바일 390px 모두 가로 넘침이나 콘솔 오류 없이 표시됐다.

따라서 다음 작업자가 다시 확인해야 할 것은 "시간 소유권 산식이 붙었는가"가 아니다.
사용자가 입력한 시간과 저장된 기록 기준이 화면마다 같은 의미로 이어지는지, 그리고 필요한 설명이 `?` 도움말이나 결과 요약처럼 적절한 위치에 있는지다.

2026-06-21 16~19단계 후속 확인
----------------
Records 쪽도 시간 소유권 기준이 이어지는지 확인했다.

확인 결과:
- Records 상세와 수정 기준은 production 시간 소유권 산식으로 재계산한다. legacy runtime으로 돌아가면 실패다.
- Records `goalSnapshot`은 당시 목표, 탄단지, 수면, 업무, 생활활동, 운동 기준과 자동 휴식시간을 확인할 수 있는 시간 기준을 보존한다.
- Today 기준 갱신은 기록의 식사, 공복체중, 메모를 보존하고 `goalSnapshot`과 자동 점수만 명시적으로 갱신한다.
- 저장된 `goalSnapshot`은 이후 식단 점수와 Recent Flow의 기준이다. 이후 Today live 값이 과거 기록 해석을 흔들면 실패다.
- 19단계 데스크톱/모바일 화면 확인에서 Today 기준 갱신 안내와 Records 당시 계산 기준은 가로 넘침이나 카드 깨짐 없이 표시됐다.

따라서 다음 복귀 작업은 "시간 소유권 산식이 붙었는가"가 아니다.
업무, 생활활동, 수면, 운동 계수와 직접입력 필요성을 생리학 근거와 실제 화면 기준으로 다시 보는 것이다.

2026-06-17 CJ 결과 반영(과거 경계)
----------------
`v8.0-CJ_time_ownership_production_formula_scope_decision_v0`는 리포트 전용으로 완료됐다.

이 단계에서 실제 목표 칼로리, 탄단지 배분, Today 화면 동작은 바꾸지 않았다. CJ가 닫은 것은 "production 산식에 바로 연결해도 되는가"가 아니라, "다음 실제 연결 단계가 어떤 조건을 지켜야 하는가"다.

CJ만 보고 실제 반영 완료로 해석하면 안 된다는 경계였다. 2026-06-21 기준으로는 그 뒤 CP/CQ 단계가 production 적용과 적용 후 검증을 닫았다.

CJ 당시에는 아래 해석이 실패였다.

- Today에 예전 `가이드 기준 / 활동량 기준` mini toggle을 다시 노출한다.
- 업무나 생활활동을 목표 칼로리에 반영하지 않는 것이 최종 안전장치라고 본다.
- CJ 리포트 전용 후보만 보고 production 산식이 이미 바뀐 것으로 말한다.

이 문서는 v8.0-CC 이후 대화에서 다시 확정된 의도를 기준으로 작성한다.
기존 README의 BY/BZ/CA/CC 메모에는 "목표 칼로리는 똑같습니다", "업무유형은 오늘 소비 기준만 바꾸고 목표칼로리는 바꾸지 않는다"는 결론이 남아 있다.
그러나 CC 이후 대화에서 이 결론은 최종 기준이 아니라 재판단 대상으로 바뀌었다.

핵심 결론:
업무, 운동, 생활활동이 실제 하루 소비량을 올렸다면 목표칼로리도 그 소비량을 기준으로 움직일 수 있어야 한다.
문제는 "많이 움직였으니 무조건 막자"가 아니라, 업무 / 운동 / 생활활동이 같은 시간을 중복 계산하지 않도록 하루 시간의 소유권을 나누는 것이다.

────────────────────────
1. 이번 단계의 목적
────────────────────────

작성 당시 이 단계는 production 산식을 바로 바꾸는 단계가 아니었다.
2026-06-21 기준으로 아래 질문은 CP/CQ까지 거쳐 production 적용과 검증이 완료됐다. 다음 작업자는 이 절을 "앞으로 해야 할 구현 순서"가 아니라 설계 근거로 읽는다.

1. 하루 소비량을 BMR * 활동계수로 뭉뚱그릴 것인가, 아니면 BMR + 시간별 활동 부하로 나눌 것인가?
2. 업무유형과 생활활동을 목표칼로리에서 제외할 타당한 이유가 있는가?
3. 제외 이유가 중복 계산 때문이라면, 중복을 막는 구조를 만들면 되는 것 아닌가?
4. Settings 기본값과 Today 예외값은 어떻게 나뉘어야 사용자가 피곤하지 않은가?
5. 기존 두 기준의 성격을 새 TDEE 구조에서 어떻게 흡수할 것인가?

답:
이 앱의 방향은 일반 TDEE 앱처럼 BMR * 활동계수 하나로 편하게 끝내는 것이 아니다.
조금 더 입력이 필요하더라도 BMR, 업무, 운동, 생활활동, 수면/휴식을 더 정확하게 나누는 쪽이다.
다만 매일 24시간을 전부 입력하게 만들면 사용자가 지친다.
따라서 Settings에 기본 하루 구조를 저장하고, Today에서는 오늘 달라진 예외만 조정하는 구조가 맞다.

────────────────────────
2. 기존 CC 결론의 문제
────────────────────────

현재 index.html의 핵심 구조는 대략 아래와 같다.

- BMR을 계산한다.
- 활동량 계수와 업무유형을 섞어 totalBurn을 만든다.
- targetTotalBurn에서는 업무유형과 업무 보정을 제외한다.
- 그래서 업무유형을 바꾸면 "오늘 소비 기준"은 바뀌지만 목표칼로리는 그대로 남을 수 있다.

이 구조가 생긴 이유는 이해할 수 있다.
업무유형, 활동량, 운동이 모두 같은 움직임을 가리키면 중복 계산이 생긴다.
특히 에너지 가용성 하한이 걸린 다이어트 케이스에서 업무유형만 바꿨다고 목표칼로리가 계속 완화되면 앱이 흔들려 보일 수 있다.

하지만 이 해결책은 제품 의도와 맞지 않는다.
실제로 업무나 생활활동이 하루 소비량을 올렸다면, 목표는 그날의 실제 소비량에서 목표별 감량/증량 폭을 적용해야 한다.
예를 들어 하루 -150kcal를 목표로 하는 사용자가 갑자기 3~4시간 육체노동을 했다면, 그 노동은 "오늘 소비 기준 설명"에만 남을 것이 아니라 실제 목표에도 반영될 수 있어야 한다.

따라서 CC의 "업무유형은 targetTotalBurn에서 제외"는 최종 방향이 아니다.
CC는 중복 계산을 피하려고 만든 임시 보호막으로 보고, 새 시간 소유권 설계로 대체해야 한다.

────────────────────────
3. 새 기준: 하루 시간 소유권
────────────────────────

새 TDEE 구조는 하루 24시간을 아래 축으로 나눈다.

1. 수면
2. 업무
3. 운동
4. 사용자가 입력한 생활활동
5. 자동 휴식 시간

원칙:
- 같은 시간은 두 번 계산하지 않는다.
- 운동 시간은 운동이 소유한다.
- 업무 시간은 업무가 소유한다.
- 수면 시간은 수면이 소유한다.
- 생활활동은 사용자가 입력한 생활활동 시간의 평균 움직임이다.
- 총합이 24시간을 넘으면 24시간 초과 안내가 필요하다. 계산에서는 생활활동 반영 시간이 남은 시간까지만 제한된다.
- 총합이 24시간보다 작으면 남은 시간은 자동 휴식시간으로 처리한다.

현재 production 계산 방향:
- BMR은 하루 전체의 기본 생존 에너지 기준이다.
- 업무, 생활활동, 운동은 BMR 위에 얹히는 시간별 추가 부하로 계산한다.
- 수면은 추가 kcal를 더하는 항목이 아니라, 자동 휴식시간을 줄이는 시간 소유권으로 작동한다.
- 활동 강도 계수는 현재 코드에 들어와 있다. 값을 바꾸려면 외부근거와 전체 시나리오 검증을 다시 거친다.

중요:
숫자가 테스트를 통과했다는 이유만으로 완성 선언하지 않는다.
이후 작업은 실제 화면에서 사용자가 입력 의미를 자연스럽게 이해하는지, 운동함/운동 안 함 상위 모드와 충돌하지 않는지 확인해야 한다.

────────────────────────
4. Settings와 Today의 역할
────────────────────────

Settings는 장기 기본값을 가진다.

Settings 기본값 후보:
- 기본 수면 시간
- 기본 업무 시간
- 기본 업무 유형
- 기본 생활활동 시간
- 기본 생활활동 강도
- 기본 운동 프로필 / 루틴 / 세션 / 주간 빈도

Today는 오늘 예외값을 가진다.

- 오늘 운동 세션, 운동 시간, 유산소가 달랐는지

5. 생활활동과 추가활동 판단
────────────────────────

처음에는 "추가활동" 카드를 따로 둘 수 있다고 봤다.
하지만 CC 이후 대화에서 추가활동과 생활활동은 같은 시간을 나눠 먹는 축이라는 점이 정리됐다.

따라서 상시 카드로는 "추가활동"을 따로 두지 않는다.
기본 구조는 하나의 "생활활동"이다.

생활활동의 뜻:
입력한 생활활동 시간 동안의 평균 활동량.

예:
- 퇴근 후 대부분 쉬었고 30분만 집안일을 했다면 생활활동 시간은 0.5h 정도로 넣고 강도는 낮음 또는 보통에 가깝다.
- 이삿짐을 3~4시간 옮기고 나머지 시간을 쉬었다면 생활활동 시간은 3~4h로 넣고 강도는 활동 많음 또는 매우 활동적으로 판단한다.
- 매우 활동적은 입력한 생활활동 시간 대부분이 빠른 걸음, 반복 운반, 숨찬 활동일 때만 고른다.
- 운동으로 따로 입력한 풋살, 러닝, 웨이트를 생활활동에서 다시 높게 잡으면 중복 계산이다.

UI 원칙:
- 설명은 기본 노출하지 않는다. 카드 높이가 늘어나면 안 된다.
- Today 목표 카드의 ? 도움말 구조를 재사용한다.
- 생활활동 카드의 ?에는 "평균 활동량"이라는 점과 중복 입력 주의를 설명한다.
- 필요하면 "직접 입력" 모달 또는 bottom sheet를 열어 예외 활동 시간을 나눌 수 있다.
- 하지만 카드 안에는 `생활활동 3시간 · 활동 많음`처럼 짧은 상태만 표시한다.
- 인라인 패널을 카드 안에 펼쳐 카드 높이를 늘리는 방식은 피한다.

7. 운동수준 / 수행능력 / 운동강도와의 관계
────────────────────────

운동수준과 수행능력은 중요하다.
하지만 "고급자니까 같은 운동시간에 kcal를 자동으로 더 준다"는 단순 곱셈은 위험하다.

현재 유지할 원칙:
- 실제 운동 칼로리는 운동 종류, 시간, 강도, 유산소 속도/경사처럼 workload에 가까운 값이 소유한다.
- 운동수준, 수행능력, 프로필은 회복, 세션 추천, 코칭, 매크로 배분, 기록 해석에 우선 사용한다.
- 운동수준이 kcal에 직접 들어가려면 외부근거와 시나리오 검증이 필요하다.

예:
초급자가 10km/h 5분에 매우 힘들어하고 고급자가 편하게 느끼더라도, 같은 속도/시간의 물리적 이동량은 비슷하다.
차이는 주로 상대 피로도, 회복 부담, 지속 가능성, 훈련 처방에 나타난다.
따라서 운동수준은 kcal multiplier가 아니라 "상대 강도 해석"으로 먼저 쓰는 것이 맞다.

  단계를 늘리기 위한 추가가 아니라, 실제 앱 완성도를 맞추기 위한 화면 정합성 단계다.

보류:
- 수면 kcal 보정값
- 생활활동 강도별 계수 변경 여부
- 업무유형 시간별 계수 변경 여부
- 고급 예외 직접입력 UI
- 수행능력의 kcal 직접 반영 여부

────────────────────────
10. 다음 작업자가 지켜야 할 것
────────────────────────

문서나 테스트가 "목표칼로리는 똑같습니다"를 정상으로 잠가 두고 있더라도 그대로 따르지 않는다.
그 문구는 CC 당시의 임시 결론이지, CC 이후 대화의 최종 의도가 아니다.

새 기준은 아래다.

1. 실제 하루 소비량이 바뀌면 목표칼로리도 바뀔 수 있다.
2. 막아야 하는 것은 반영 자체가 아니라 중복 계산이다.
3. 중복은 시간 소유권으로 해결한다.
4. 사용자는 매일 24시간을 전부 입력하지 않는다.
5. Settings 기본값 + Today 예외값 구조로 입력 피로도를 낮춘다.
6. UI는 카드 높이가 늘어나지 않게 설계한다.
7. production 반영은 이미 CP/CQ와 14단계 재검증에서 닫혔다. 다음에는 Records snapshot, 기준갱신 문구, 수면/업무/생활활동 입력 흐름의 화면 자연스러움을 본다.

────────────────────────
11. 운동 여부 상위 모드 계약
────────────────────────

2026-06-15 추가 대화에서 "운동 여부"는 가이드 기준 / 활동량 기준보다 상위에 있는 사용자 모드로 정리한다.
온보딩이 생기면 사용자는 복잡한 기준명을 고르는 것이 아니라 운동을 관리 기준에 포함할지 먼저 고른다.
온보딩 전에는 Settings에서만 바꿀 수 있게 둔다.

2026-06-16 정정:
아래 항목 중 두 기준을 계속 사용자 선택 토글처럼 유지한다는 표현은 최신 대화 기준으로 폐기한다.
최신 결정은 "기존 가이드/활동량 개념을 운동 여부 상위 모드에 흡수한다"이다.
운동함 모드와 운동 안 함 모드가 같은 화면과 같은 해석을 억지로 공유하면 안 된다는 원칙은 그대로 유효하다.

운동함 모드:
- 현재 앱의 운동 프로필, 루틴, 세션, 웨이트, 유산소, 체성분 기반 해석을 사용할 수 있다.
- 기존 가이드 기준의 운동생리학 / 운동인 기준 성격을 흡수한다.
- Today에서 별도의 가이드/활동량 mini toggle을 보여주지 않는다.
- 운동 관련 값은 Records 스냅샷, Coach, 최근 흐름, 점수와 연결될 수 있다.

운동 안 함 모드:
- 운동 프로필, 루틴, 세션, 웨이트 시간, 훈련 강도, 유산소 같은 운동 관련 입력은 기본 화면에서 숨긴다.
- 숨긴 값을 저장소에서 삭제하지 않는다.
- 계산과 렌더에서만 운동 관련 값은 0, REST, 없음, fallback으로 처리한다.
- 기존 활동량 기준의 일반 식단 / 체중 관리 기준 성격을 흡수한다.
- 운동 안 함 상태에서 저장한 기록이 나중에 운동함 모드로 전환됐을 때 과거 운동 데이터처럼 갑자기 살아나면 안 된다.
- 반대로 운동함 상태에서 저장한 과거 운동값을 운동 안 함 모드로 잠시 바꿨다는 이유로 0으로 덮어쓰면 안 된다.
- 체성분 데이터는 체중 / 건강 기록으로 보존할 수 있지만, 운동 성과 해석처럼 말하지 않는다.

데이터 / 백업 원칙:
- 운동함과 운동 안 함은 같은 내보내기 / 불러오기 형식을 쓴다.
- 한쪽 모드에서 읽지 않는 값은 저장소에서 지우지 않고 계산 / 렌더 / 해석 단계에서만 fallback 처리한다.
- schema를 둘로 나누지 않는다.
- import/export는 양쪽 모드에서 round-trip 되어야 한다.

문구 원칙:
- Coach / 최근 흐름 본문에 앱 정책 설명을 쓰지 않는다.
- 피해야 할 문구: "운동 기록은 현재 관리 기준에 포함하지 않습니다", "일반 권장 범위 안에서 관리합니다", "체성분 변화와 식단 기록을 함께 봅니다".
- 정책 설명은 Settings 도움말, ? 툴팁, 온보딩, 기준문서에 둔다.
- Coach / 최근 흐름은 사용자가 지금 볼 수 있는 상태와 다음 행동을 말한다.

다음 구현 전 감사 대상:
1. Settings에 운동 여부를 둘 위치.
2. Today에서 운동 안 함 모드일 때 숨겨야 할 카드와 남겨야 할 카드.
3. Records 스냅샷에서 운동 관련 값 fallback과 원본 보존을 분리하는 방식.
4. Coach / 최근 흐름 / 점수에서 운동 신호를 제외할 때 문구가 정책 설명으로 새지 않는지.
5. backup/import에서 숨긴 운동값이 삭제되거나 가짜 운동값으로 살아나지 않는지.
6. 기존 가이드/활동량 개념이 운동 여부 상위 모드 아래에서 계산 내부로만 남고, 사용자-facing Today mini toggle이 제거되는지.

중단 조건:
- 구현 중 "운동 여부 상위 모드만으로 Today 화면과 계산 기준을 자연스럽게 설명할 수 없는 경우"에는 사용자가 이해할 수 있는 말로 다시 질문한다.
- 운동값을 실제 저장소에서 0으로 덮어써야만 구현 가능한 구조라면 그대로 진행하지 말고 재설계한다.
- Coach / 최근 흐름이 정책 설명 문구를 다시 노출해야만 구현되는 구조라면 그대로 진행하지 말고 문구 구조를 다시 잡는다.
