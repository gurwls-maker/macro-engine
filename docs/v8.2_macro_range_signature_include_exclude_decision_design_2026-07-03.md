## 1. 현재 기준

현재 snapshot signature:

```text
getGoalSnapshotSignature()
GOAL_SNAPSHOT_SIGNATURE_KEYS allowlist
```

현재 range 관련 상태:

```text
macroRangeContract 저장 안 함
rangeScoreCandidate 저장 안 함
candidateScorePreview 저장 안 함
unknown range field는 normalizeGoalSnapshot에서 prune
GOAL_SNAPSHOT_SIGNATURE_KEYS에 range field 없음
```

현재 테스트로 고정된 것:

```text
old fixed snapshot normalize 가능
unknown range field는 current contract에서 prune
unknown range field는 current fixed signature를 바꾸지 않음
simulation output은 backup / Recent / score / UI에 연결되지 않음
```

## 2. signature decision이 위험한 이유

signature는 단순 hash가 아니다.

영향 영역:

```text
Records 상세 수정 무변경 저장
Today / Records 갱신 필요 판단
goalSnapshot drift 감지
backup / restore roundtrip 비교
old/new basis 해석
Recent mixed basis 판단
```

잘못 넣으면 생길 수 있는 문제:

```text
1. 사용자가 아무것도 바꾸지 않았는데 Records 갱신 필요 경고가 뜸
2. optional field order / floating precision 차이로 무변경 저장이 변경 저장처럼 보임
3. policy version만 바뀌어도 과거 record가 새 basis처럼 해석됨
4. range contract 변화가 실제 의미 변경인데 signature가 감지하지 못함
5. backup/restore 후 signature가 달라져 같은 기록이 충돌처럼 보임
```

특히 RC1에서 고친 체성분 정밀도 / 무변경 저장 문제와 같은 유형의 회귀가 생길 수 있다.

## 3. 선택지 A: range field를 current goalSnapshot signature에서 제외

정의:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS에는 fixed fields만 유지한다.
macroRangeContract / rangeScoreCandidate / candidateScorePreview는 current signature에 넣지 않는다.
```

장점:

```text
Records 상세 수정 무변경 저장 안정성이 높다.
range candidate meta / debug field / policy minor update가 fixed snapshot diff를 만들지 않는다.
old fixed record와 current fixed signature의 호환성이 좋다.
RC1 무변경 저장 회귀 가능성이 낮다.
```

위험:

```text
range contract가 실제 저장 의미를 바꾸는 경우 current signature만으로 감지하지 못한다.
new range record의 basis 변경을 fixed signature가 표현하지 못한다.
Recent / backup / smart restore에서 mixed basis 감지를 별도 field에 의존해야 한다.
```

적합한 경우:

```text
range contract가 아직 report-only이거나 candidate basis인 경우
range score가 production score가 아닌 경우
old fixed target이 계속 source of truth인 경우
```

부적합한 경우:

```text
range contract가 저장 당시 macro 해석의 source of truth가 되는 경우
range-aware score가 production score가 되는 경우
backup/restore conflict 판단에 range basis 차이를 보여야 하는 경우
```

## 4. 선택지 B: range field를 current goalSnapshot signature에 직접 포함

정의:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS에 macroRangeContract 또는 그 하위 field를 추가한다.
```

장점:

```text
저장 당시 range basis 변경을 current signature가 직접 감지한다.
backup/restore roundtrip에서 range field 손실을 signature mismatch로 잡을 수 있다.
Records 상세에서 range basis 변경이 fixed target 변경과 함께 드러난다.
```

위험:

```text
optional field order / floating precision / policy version 변화가 무변경 저장을 깨뜨릴 수 있다.
candidate/debug field가 끼면 signature가 불필요하게 흔들린다.
old fixed record와 new range record 비교가 복잡해진다.
unknown future field 정책과 충돌할 수 있다.
Records 갱신 필요 표시가 실제 사용자 변경 없이 뜰 수 있다.
```

현재 판단:

```text
직접 포함은 아직 이르다.
production range score / schema / backup / Recent basis가 확정되기 전에는 선택하지 않는다.
```

## 5. 선택지 C: current fixed signature와 별도 basis signature를 분리

정의:

```text
current getGoalSnapshotSignature는 fixed fields 기준으로 유지한다.
future range basis에는 별도 후보 signature를 둔다.
```

후보 이름:

```text
getMacroRangeBasisSignatureCandidate()
getGoalSnapshotBasisSignatureCandidate()
```

또는 저장 구조 후보:

```text
goalSnapshot.fixedSignature
goalSnapshot.macroRangeBasisSignature
```

단, 현재는 저장하지 않는다.

장점:

```text
Records 무변경 저장 안정성을 보존한다.
range basis 변경 감지를 별도 경로로 설계할 수 있다.
old fixed basis와 new range basis를 명시적으로 분리할 수 있다.
Recent mixed basis detection에 필요한 signal을 만들 수 있다.
```

위험:

```text
signature가 두 개가 되면 어떤 비교가 사용자-facing인지 정해야 한다.
backup/restore conflict preview가 복잡해진다.
Records 상세 수정에서 fixed unchanged / basis changed 상태를 따로 설명해야 한다.
```

현재 판단:

```text
가장 안전한 후보는 C다.
하지만 아직 구현하지 않는다.
다음 단계에서 simulation test로 A/B/C를 비교할 수 있다.
```

## 6. signature 대상 field 분류

절대 포함 금지:

```text
candidateScorePreview
scoreDeltaPreview
reportOnly
uiVisible
persisted
debugReason
explanation
rawExternalEvidence
warnings
```

포함 후보가 될 수 있는 field:

```text
macroRangeContract.version
macroRangeContract.targetCal
protein target / min / upper
protein mode / basis / appPolicyKey
fat floor / range / threshold
carbs trainingSupportMin / range
scoring.candidateVersion
scoring.appliesToSavedRecord
```

단, `scoring.appliesToSavedRecord`는 production range score 결정 전에는 반드시 false여야 한다.

주의:

```text
policy display label, evidence text, debug reason은 signature 대상이 아니다.
사용자-facing copy 변경이 snapshot signature를 바꾸면 안 된다.
```

## 7. stable serialization 후보

range basis signature가 필요해지면 다음이 필요하다.

```text
field allowlist
stable field order
finite number normalization
precision policy
null / absent 처리
version handling
unknown field pruning
forbidden field rejection
```

숫자 precision 후보:

```text
kcal: 0.01
grams: 0.01
percent: 0.01
```

주의:

```text
precision은 저장값을 숨기기 위한 큰 tolerance가 아니다.
floating serialization 차이로 같은 basis가 다른 signature가 되는 것을 막기 위한 정규화다.
```

## 8. Records 무변경 저장 기준

현재 지켜야 할 원칙:

```text
사용자가 값을 실제로 수정하지 않았다면 저장해도 goalSnapshot이 변하면 안 된다.
표시 반올림 때문에 signature가 바뀌면 안 된다.
unknown/report-only field 때문에 갱신 필요 경고가 뜨면 안 된다.
```

future range field가 생길 때 필요한 상태 분리:

```text
fixedUnchanged: true / false
rangeBasisUnchanged: true / false / absent / unsupported
signatureMode: fixed_only / range_basis_candidate / combined
```

사용자-facing으로 바로 노출하지 않는다.

## 9. backup / restore 기준

signature 결정은 backup/restore와 분리할 수 없다.

필수 비교 후보:

```text
old fixed record backup roundtrip signature unchanged
new range record backup roundtrip basis signature unchanged
unknown future field pruned or preserved behavior explicit
invalid range contract fallback does not alter fixed signature
records-only restore keeps expected signature mode
smart restore meals-only does not replace local basis signature
replace mode intentionally applies backup basis signature
```

아직 금지:

```text
backup payload에 range basis signature 추가
restore conflict preview에 basis signature 노출
smart restore policy 변경
```

## 10. Recent 기준

Recent는 여러 날짜를 섞기 때문에 signature 결정의 영향을 크게 받는다.

future signal 후보:

```text
fixedSignatureDays
rangeBasisSignatureDays
mixedBasisDetected
unsupportedRangeBasisDays
invalidRangeBasisDays
```

현재 금지:

```text
getRecentContextRecordSummary 변경
getRecentContextComponentCounts 변경
Recent component count를 range basis로 재해석
Recent UI 문구 변경
```

원칙:

```text
old fixed record와 new range record를 같은 fatLowDays 의미로 조용히 합치지 않는다.
```

## 11. decision matrix

| 기준 | A. exclude | B. include current signature | C. separate basis signature |
| --- | --- | --- | --- |
| 무변경 저장 안정성 | 높음 | 낮음~중간 | 높음 |
| range basis 변경 감지 | 낮음 | 높음 | 높음 |
| old fixed compatibility | 높음 | 중간 | 높음 |
| backup/restore roundtrip 설명성 | 낮음 | 중간 | 높음 |
| Recent mixed basis 감지 | 낮음 | 중간 | 높음 |
| 구현 복잡도 | 낮음 | 중간 | 높음 |
| 현재 단계 적합성 | 높음 | 낮음 | 설계 후보 |

현재 결론:

```text
production current signature는 그대로 둔다.
직접 include는 아직 선택하지 않는다.
future 후보로는 C를 우선 검토한다.
```

테스트 후보:

```text

8. separate basis signature normalizes numeric serialization
9. invalid range contract has no range basis signature
10. unsupported range contract reports unsupported basis signature status

12. Recent mixed basis fixture candidates detect fixed/range split without changing production counts
```

필수 병행 테스트:

## 13. false positive 방지

정상:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS에 range field가 없음
unknown range field가 current signature를 바꾸지 않음
candidate basis signature가 아직 저장되지 않음
Records 무변경 저장 판정이 그대로임
```

blocker:

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS를 변경함
getGoalSnapshotSignature production output을 변경함
Records 무변경 저장 판정을 range field로 바꿈
candidateScorePreview를 signature 후보에 넣음
copy/evidence/debug field가 signature에 들어감
signature design이라는 이름으로 backup/Recent/score/UI를 바꿈
```

## 14. 아직 사용자 판단이 필요한 항목

아직 결정하지 않는다.

```text
1. range basis를 current goalSnapshot signature에 포함할지.
2. 별도 macroRangeBasisSignature를 저장할지.
3. basis signature를 user-facing warning에 쓸지.
4. backup conflict preview에 basis 차이를 보여줄지.
5. smart restore meals-only에서 range basis conflict를 어떻게 표시할지.
6. production range score version과 basis signature를 어떻게 연결할지.
```

## 16. 최종 판단

signature include/exclude decision design은 진행 가능하다.

하지만 아직:

```text
signature 정책 확정 아님
GOAL_SNAPSHOT_SIGNATURE_KEYS 변경 아님
range basis signature 저장 아님
backup/restore roundtrip 구현 아님
Recent mixed basis 구현 아님
production macro range 구현 아님
```

최종 결론:

```text

range field 직접 include는 아직 위험하다.

```
