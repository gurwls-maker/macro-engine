# V8 수준별 산식 초정밀 설계

작성일: 2026-06-04
상태: report-only precision design / production 산식 미변경
관련 stage: `v8.0-AZ_user_level_formula_precision_design_v0`
브랜치: `codex/v8-formula-audit`

이 문서는 사용자 수준별 profile-specific macro formula를 구현하기 전, 어떤 입력과 근거와 gate가 필요한지 고정하는 설계 문서다.
이번 단계는 production targetCal/protein/carbs/fat를 바꾸지 않는다. `runV8ScenarioRunner()`의 `userLevelFormulaPrecisionDesign`도 같은 경계를 가진다.

## 1. 결론

다음 production 산식은 단순히 "단백질 몇 g/kg", "러닝이면 탄수 많이"처럼 바꾸면 안 된다.

수준별 산식은 아래 다섯 가지를 하나의 context로 묶은 뒤에만 targetCal과 탄단지를 조정해야 한다.

1. 사용자 목표: diet / cut / recomp / maintain / lean_bulk / bulk
2. 현재 운동 프로필과 오늘 세션: Settings 기본 프로필, Today 루틴/세션, 파생 `profileSession`
3. 실제 훈련 수준: 주간 빈도, 오늘 시간, 세션 강도, 유산소 부하, 최근 지속성
4. 체성분 신뢰도: 직접 입력 / 가정용 InBody / 최신 InBody / 누락 또는 불안정
5. 최근 흐름: 최근 14일 기록 커버리지와 최근 28일 체중/목표 흐름

금지선:

- 일반인 / 운동인 / 전문가 같은 모드 버튼을 다시 핵심 입력으로 부활시키지 않는다.
- `profileSession`을 사용자가 고르는 별도 shortcut으로 만들지 않는다.
- 18개 대표 케이스나 report-only candidate 결과를 full V8 산식 완료로 말하지 않는다.
- 내부 코드의 기존 숫자를 외부 운동생리학 근거처럼 말하지 않는다.
- 기록이 부족한 사용자의 최근 추세를 근거로 targetCal을 자동 변경하지 않는다.

## 2. 현재 production 기준

현재 실제 production으로 적용된 기준:

- 일반 diet 단백질 기본값은 `GOALS.diet.proteinKg = 2.0`이다.
- `getDualBasisProteinRange("diet")`도 `{ min: 1.4, mid: 2.0, max: 2.0 }` 기준이다.
- `calculateMacroTargets()`는 `getProteinBasisWeight()`를 사용해 protein basis weight를 잡고, `GOALS[goal].proteinKg`를 곱한 뒤 kcal 35% cap과 plausibility layer를 통과시킨다.
- `buildGoalTargetRateContext()`는 goal별 weekly rate, delta guard, BMR floor, energy availability floor를 이미 적용한다.
- 비보디빌딩 profile/routine/session 경로는 production 입력 경로에 들어왔지만, 그 profile별 macro formula가 전면 적용된 것은 아니다.
- candidate-v2는 active runtime target-delta case에 scoped production으로 적용될 수 있으나, 모든 사용자 수준별 산식이 아니다.

열린 gate:

- user-level profile-specific macro formula implementation
- broad profile/routine/session human UX review
- full 8-2 Cartesian execution 또는 owner-approved 대체 gate
- full V8 completion

## 3. 외부 근거와 앱 내부 근거

### 3-1. 외부 스포츠영양 근거

ISSN protein and exercise 2017:

- 건강한 운동 사용자 대부분의 단백질 anchor는 대략 1.4-2.0 g/kg/day 범위다.
- 더 높은 섭취는 특정 저열량/저체지방/저항훈련 맥락에서만 후보로 다룬다.
- 앱 적용: 일반 diet 기본값을 2.4 g/kg로 되돌리지 않는다. 높은 단백질은 "운동 수준 + 저열량 + 신뢰 가능한 제지방/체성분" gate가 필요하다.
- 출처: https://link.springer.com/article/10.1186/s12970-017-0177-8

ISSN diets and body composition 2017:

- 감량은 지속적인 열량 적자가 핵심이며, 높은 체지방에서는 더 큰 적자가 가능하지만, 마른 사용자일수록 느린 감량이 lean mass 보존에 유리하다.
- 2.3-3.1 g/kg FFM 수준의 높은 단백질은 lean resistance-trained hypocaloric context에서만 후보가 된다.
- 앱 적용: "커팅/고급 운동인"이라고 바로 고단백을 주는 것이 아니라, 체성분 신뢰도와 저체지방/훈련 맥락을 함께 확인한다.
- 출처: https://jissn.biomedcentral.com/articles/10.1186/s12970-017-0174-y

Academy/DC/ACSM Nutrition and Athletic Performance 2016:

- 탄수화물은 훈련 부하별로 대략 light 3-5, moderate 5-7, high 6-10, very high 8-12 g/kg/day 범위를 쓴다.
- 에너지 가용성은 건강과 스포츠영양 전략의 기반이며, 낮은 에너지 가용성은 피해야 한다.
- 앱 적용: running/mixed 세션은 cardio 부하와 세션 종류를 보고 carb floor 또는 target relief를 요청한다. 단, 모든 running day에 고정 고탄수를 주는 방식은 금지한다.
- 출처: https://pubmed.ncbi.nlm.nih.gov/26891166/

IOM/NASEM DRI AMDR:

- 성인 AMDR은 carbohydrate 45-65%, fat 20-35%, protein 10-35% energy range다.
- 앱 적용: AMDR은 population plausibility boundary로 사용한다. 스포츠 세션 때문에 벗어날 수는 있지만, 그때는 profile/session reason과 Coach/Records 설명이 필요하다.
- 출처: https://www.ncbi.nlm.nih.gov/books/NBK208874/

BIA/InBody 신뢰도 근거:

- BIA는 널리 쓰이지만 측정 정확도와 표준화 문제가 있다. 단일 측정값을 절대 진실로 쓰지 않는다.
- 앱 적용: InBody/가정용 InBody는 LBM alpha, trend confidence, review copy를 조절하는 근거다. targetCal을 공격적으로 자동 변경하는 단독 근거가 아니다.
- 출처: https://www.nature.com/articles/s41430-018-0335-3

### 3-2. 내부 코드 근거

내부 코드 근거는 "현재 앱이 그렇게 움직인다"는 증거다.
운동생리학적으로 맞다는 증거가 아니다.

현재 내부 코드 근거:

- `GOALS`, `GOAL_RATE_POLICIES`
- `buildGoalTargetRateContext()`
- `calculateMacroTargets()`
- `resolveTrainingEvidence()`
- `resolveBodyCompositionReliability()`
- `buildProfileCandidateV2TargetDeltaGate()`
- `profileRoutineUserPathMatrix`
- Records `goalSnapshot` / signature / backup roundtrip contracts

### 3-3. 제품 정책 근거

제품 정책 근거:

- Today는 오늘 계산과 오늘 행동의 작업공간이다.
- Settings는 기본값과 장기 기준이다.
- Records는 확정 기록과 당시 계산 기준 snapshot이다.
- 최근 흐름은 3/7/14/28일을 중심으로 해석한다.
- 이 앱은 "오늘 작업을 끝내고 최근 흐름을 해석하는 기록 기반 코치 앱"이다.

## 4. userLevelFormulaContext

새 production 산식을 구현하기 전에 아래 context를 먼저 만든다.

```text
userLevelFormulaContext = {
  goal,
  exerciseProfile,
  routinePlan,
  routineSession,
  profileSession,
  trainingEvidence,
  performanceLevel,
  bodyCompReliability,
  recent14RecordCoverage,
  recent28Trend,
  activityWorkload,
  cardioLoad,
  advancedTuning,
  formulaInputConfidence,
  autoApplyPermission
}
```

### 4-1. performanceLevel 산출

`performanceLevel`은 독립 UI selector가 아니라 derived context로 시작한다.

초기 후보:

```text
none:
  no training session and no cardio load, or profileSession=rest

low:
  light trainingEvidence, short session, weeklyTrainingDays <= 2, or recent coverage insufficient

normal:
  regular trainingEvidence, coherent profile/session, ordinary session duration/intensity

high:
  high trainingEvidence, long/high-intensity session, and stable recent record context
```

구현 후보 score는 내부 앱 정책이다. 외부 근거로 말하지 않는다.

```text
trainingLoadScore candidate =
  weeklyFrequencyScore * 0.25
  + sessionDurationScore * 0.25
  + intensityScore * 0.20
  + cardioLoadScore * 0.20
  + profileSessionSpecificityScore * 0.10
```

이 score는 바로 production에 쓰지 않는다.
다음 단계에서 report-only candidate로 각 coefficient를 stress test하고, 사용자가 보기 어색한 사례를 먼저 찾는다.

### 4-2. bodyCompReliability

현재 runtime context의 신뢰도를 유지한다.

```text
none:
  체성분 입력 없음 또는 계산에 쓸 수 없음

partial:
  사용자가 직접 입력했지만 source/control이 약함

home:
  가정용 InBody 또는 homeInbody path

inbody:
  Today에 적용한 최신 InBody / controlled source
```

사용처:

- LBM alpha와 protein basis weight
- recent body-composition trend copy
- targetRate 자동 적용 permission

금지:

- 체성분 신뢰도 하나만으로 targetCal을 올리거나 내리지 않는다.
- skeletal muscle이 높다는 이유만으로 단백질/칼로리를 과하게 올리지 않는다.

## 5. 산식 레이어

### 5-1. Target Energy Layer

기본값은 기존 `buildGoalTargetRateContext()`가 만든다.

수준별 layer는 그 위에 아래 후보를 얹는다.

```text
baseTarget = current buildGoalTargetRateContext().targetCal

candidateTargetDelta =
  profileSessionFuelDelta
  + recentTrendDelta
  + levelRecoveryDelta

finalTarget =
  guard(
    baseTarget + candidateTargetDelta,
    goal delta min/max,
    BMR floor,
    energy availability floor,
    recentContext gate
  )
```

초기 delta grid:

```text
recentTrendDeltaGrid: -150, -100, -75, 0, +75, +100, +150 kcal
profileFuelReliefGrid: 0, +75, +125, +175, +250 kcal
levelRecoveryDeltaGrid: 0, +50, +100 kcal
```

이 grid는 external evidence가 아니라 앱 내부 calibration 후보다.
다음 단계에서 report-only로만 돌리고, 실제 production 적용은 approval gate 이후다.

자동 적용 금지 조건:

- 최근 목표 변경 감지
- 최근 28일 체중일수 부족
- 상세 diet-day coverage 부족
- recordConfidence low
- energy availability caution/risk
- 오늘 세션이 rest인데 profile fuel delta가 양수
- running/mixed가 cardio load 없이 high-carb/target relief를 요구

### 5-2. Protein Layer

기본은 현재 production:

```text
proteinBasisWeight = getProteinBasisWeight(s, lbmBase, lbmEffective, modeCfg)
proteinRaw = GOALS[goal].proteinKg * proteinBasisWeight
proteinCap = targetCal * 0.35 / 4
```

수준별 후보:

```text
general exercising anchor:
  1.4-2.0 g/kg/day

general diet production:
  2.0 g/kg basis 유지

lean trained hypocaloric exception:
  2.3-3.1 g/kg FFM candidate
  only if bodyCompReliability is home/inbody or controlled enough,
  bodyFatBand lean/very_lean,
  resistance profile/session,
  diet/cut deficit,
  and carb/fat floors remain explainable
```

프로필별 방향:

- bodybuilding/powerbuilding/strength: protein adequacy를 우선하되, high-carb 세션에서 탄수를 무의미하게 밀어내지 않는다.
- running: 일반적으로 1.4-2.0 anchor 안에서 처리한다. deficit 또는 mixed strength가 있으면 상단으로 이동할 수 있다.
- mixed: strength 비중과 cardio 비중을 모두 보고 protein은 보존하되 carb floor와 충돌하면 target relief 후보를 먼저 검토한다.

금지:

- "기존 코드에 있던 값"이라는 이유로 2.4 g/kg를 일반 diet에 적용하지 않는다.
- protein cap pass를 생리학적 타당성으로 말하지 않는다.

### 5-3. Fat Layer

기본은 `FAT_TARGET_PROFILES`와 action/physio floor를 유지한다.

수준별 후보:

```text
fatTarget = current getFatTargetContext()

guard:
  adult AMDR 20-35% kcal boundary as broad check
  app action floor by gender/weight
  do not let protein/carb floors push fat below action floor silently
```

러닝/혼합 운동에서 탄수 floor가 높아질 때:

1. fat을 바로 낮추지 않는다.
2. protein이 예외적으로 높은지 먼저 확인한다.
3. 그래도 부족하면 target relief 후보를 만든다.
4. target relief도 recentContext/energy gate를 통과해야 한다.

### 5-4. Carbohydrate Layer

탄수는 단순 remainder가 아니라 session fuel layer를 가진다.

외부 근거 anchor:

```text
light training: 3-5 g/kg/day
moderate about 1h/day: 5-7 g/kg/day
high endurance 1-3h/day: 6-10 g/kg/day
very high 4-5h+/day: 8-12 g/kg/day
```

앱 적용:

- bodybuilding: resistance session fuel. 일반 운동일은 무조건 endurance high-carb로 올리지 않는다.
- powerbuilding: heavy lower/compound는 carb priority를 높이고 accessory는 낮춘다.
- strength: heavy/volume은 recovery carb 후보, technique/deload는 dampen.
- running: easy/recovery/tempo/interval/long run을 다르게 본다. weightDuration=0인 non-rest running session을 rest로 취급하지 않는다.
- mixed: strength+cardio combined load를 본다. cardio_focus/circuit은 carb priority가 높고 recovery는 낮다.

## 6. 프로필별 설계

| profile | load owner | high priority | advanced OFF | advanced ON |
| --- | --- | --- | --- | --- |
| bodybuilding | routineSession + weightDuration + xw | protein, fat floor, resistance carbs | REST/PUSH simple, PUSH xw 0.70 | PPL/split sessions |
| powerbuilding | upper/lower/main/accessory | heavy session carbs, protein | REST/PUSH simple + internal default | upper/lower/accessory plans |
| strength | heavy/volume/technique/deload | recovery, protein, carb support | REST/PUSH simple + internal default | heavy/deload difference preserved |
| running | cardio duration/speed/incline/session | carbs, energy availability | REST/PUSH visible, cardio-owned default internally | easy/tempo/interval/long/recovery |
| mixed | resistance + cardio combined | target relief, carbs, protein | REST/PUSH visible, combined default internally | strength_focus/cardio_focus/circuit/recovery |

## 7. Recent Gate

recent14:

- Coach/action copy와 adherence interpretation에 쓴다.
- targetCal 자동 변경 근거로는 부족하다.

recent28:

- target-rate adjustment 후보가 될 수 있다.
- 단, 아래 조건을 통과해야 한다.

```text
required:
  weightDays >= current recentContext gate threshold
  usable diet days enough
  recordConfidence not low
  no goalChangeDetected
  contextFit/windowFit acceptable
  Today target basis preserved in goalSnapshot/signature
```

해석:

- diet/cut에서 너무 빠르게 빠지면 target을 올리는 후보를 우선한다.
- diet/cut에서 너무 느리게 빠져도 기록 신뢰도가 낮으면 target을 내리지 않고 기록 품질 안내를 우선한다.
- lean_bulk/bulk에서 너무 빠르게 늘면 target을 낮추는 후보를 만든다.
- maintain에서 흔들림이 크면 trend copy와 보수 조정 후보를 만든다.

## 8. 구현 순서

다음 구현은 아래 순서로만 진행한다.

1. `userLevelFormulaContext` builder 추가
   - production 계산값 미변경
   - Today state + recentContext + profile/routine/session ownership 확인

2. report-only `levelAwareMacroCandidateV0` 추가
   - targetEnergy/protein/fat/carbohydrate layer를 계산
   - current production과 delta 비교
   - 외부 근거 / 내부 정책 / 제품 정책 근거를 분리 표기

3. Records/Score/Coach/Backup contract 추가
   - model/stage
   - selected basis
   - target delta
   - profile/session reason
   - trend gate reason

4. 대표 human-eye case 보강
   - 단순 18개가 아니라 프로필별 실사용 흐름 케이스 추가
   - running tempo가 rest/push처럼 보이는지
   - bodybuilding advanced OFF/ON이 기존 의도와 맞는지
   - mixed circuit/recovery가 같은 식으로 뭉개지지 않는지

5. Cartesian closure 전략 확정
   - 80,621,568,000개 full execution을 실제로 돌릴지
   - 수학적 contract + bounded shard + owner-approved representative gate로 대체할지
   - Codex가 임의로 대체하면 안 된다.

6. production approval
   - report-only candidate가 충분히 검증된 뒤 별도 승인 단계에서만 production 적용
