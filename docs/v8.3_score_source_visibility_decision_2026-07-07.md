# v8.3 score source visibility decision

## 1. 문서 목적

이 문서는 `docs/v8.3_score_distribution_tuning_decision_2026-07-07.md`에서 W06으로 분리한 source visibility 문제를 닫는다.

이번 decision에서 닫는 것:
- missing TDEE일 때 점수를 낼지, null/blocked할지, warning/copy로 둘지 판단한다.
- `totalBurnSource`, `tdeeAvailable`, `weightSource`, `trainingContext`, alcohol derive source, stored score source를 source confidence 관점으로 분류한다.
- source warning이 필요한 항목과 null/blocked여야 하는 항목을 분리한다.
- 다음 implementation이 필요한지, 추가 docs-only gate가 필요한지 판단한다.
- permanent regression test 후보를 정한다.

이번 decision에서 열지 않는 것:
- `index.html` 산식 변경.
- anchor/penalty 값 변경.
- `ADHERENCE_SCORING_VERSION` 변경.
- `MACRO_RANGE_PRODUCTION_SCORING_VERSION` 변경.
- UI/storage/schema/backup/Recent 변경.
- scoreDeltaPreview 재개 금지.
- old records migration/recompute/reset.

## 2. 실제 코드 상태 확인

확인한 production identity:
- `MACRO_RANGE_PRODUCTION_SCORING_VERSION = "v8.3_anchor_continuous_macro_score_v1"`
- `ADHERENCE_SCORING_VERSION = MACRO_RANGE_PRODUCTION_SCORING_VERSION`
- `MACRO_RANGE_SCORE_FORMULA_NAME = "anchor_continuous_macro_score_v1"`

확인한 source helper:
- `getV83WeightContext(...)`
- `getV83TotalBurnContext(...)`
- `getV83TrainingContext(...)`
- `buildV83ContinuousScoringContext(...)`
- `getV83ContinuousMacroScoreSummary(...)`
- `getMacroRangeProductionScoreSummary(...)`
- `getDailyAdherenceScore(...)`

현재 weight source 우선순위:
1. `options.bodyWeightKg` -> `weightSource: "options"`
2. `balance.goalSnapshot.weight` -> `weightSource: "goalSnapshot"`
3. `currentResult.s.weight` -> `weightSource: "currentResult"`
4. `targetProtein / 1.8` -> `weightSource: "targetProteinFallback"`
5. missing -> `weightSource: "missing"`

현재 totalBurn source 우선순위:
1. `options.totalBurn` -> `totalBurnSource: "options"`
2. `currentResult.totalBurn` -> `totalBurnSource: "currentResult.totalBurn"`
3. `currentResult.targetTotalBurn` -> `totalBurnSource: "currentResult.targetTotalBurn"`
4. `currentResult.targetBurnBase` -> `totalBurnSource: "currentResult.targetBurnBase"`
5. missing -> `totalBurnSource: "missing"`

확인한 동작:
- missing TDEE이면 `context.tdeeRatio`가 null이므로 `tdeeOverloadPenalty`는 0이다.
- missing TDEE이어도 target/macro/weight가 충분하면 `percent`는 numeric으로 나올 수 있다.
- 이때 `scoringContext.tdeeAvailable === false`와 `totalBurnSource === "missing"`은 detail에 남는다.
- `targetProteinFallback`은 체중 source가 없을 때 g/kg 계산을 가능하게 하지만, authoritative body weight source가 아니다.
- stored current-version auto score는 조회 시 저장된 `adherencePercent`를 우선한다.
- old fixed records는 no silent mutation 정책을 유지한다.

판단:
- 현재 구현은 source metadata를 숨기지 않는 방향으로 이미 일부 준비되어 있다.
- 그러나 user-facing layer가 이 metadata를 해석하지 않으면 missing TDEE 100점이 완전한 100점처럼 보일 수 있다.
- 따라서 문제는 score curve가 아니라 source visibility / confidence policy다.

## 4. Source case decisions

### S01. full macro day + missing TDEE

Current behavior:
- full macro가 있고 target도 있으면 numeric score가 가능하다.
- TDEE가 없으면 `tdeeAvailable:false`, `totalBurnSource:"missing"`으로 남는다.
- `tdeeOverloadPenalty`는 0이다.

Risk:
- `percent:100`이 "완전한 100점"처럼 보일 수 있다.
- 실제로는 TDEE overload axis가 빠진 partial-source score다.

Decision:
- classification: source-availability
- decision: allow-with-user-warning
- next implementation needed: yes
- test needed: yes
- UI/storage needed: UI/copy yes, storage schema no

Reason:
- TDEE가 없다고 target adherence까지 막을 필요는 없다.
- 하지만 TDEE 축이 빠진 점수는 완전한 score confidence로 표시하면 안 된다.
- `detail.scoringContext.tdeeAvailable:false`가 이미 있으므로 storage schema를 열지 않고도 warning을 만들 수 있다.

Next action:
- next implementation에서 Today score card 또는 score reason detail에 "총소비량/TDEE 축이 없어 목표 대비 점수로만 해석"하는 warning을 붙인다.
- Records 저장 schema는 열지 않는다.
- backup/Recent aggregation 구조도 열지 않는다.

### S02. totalBurnSource missing but target match

Current behavior:
- targetEnergyDeviationPenalty만으로 높은 점수가 가능하다.
- TDEE overload 축은 빠진다.

Risk:
- target을 잘 맞춘 날과 TDEE까지 검증된 날이 같은 confidence처럼 보인다.

Decision:
- classification: source-availability
- decision: allow-with-user-warning
- next implementation needed: yes, together with S01
- test needed: yes
- UI/storage needed: UI/copy yes, storage schema no

Reason:
- target match는 유효한 정보다.
- 그러나 TDEE overload absence는 source confidence 차이다.
- 점수 numeric 자체를 null로 바꾸기보다 "TDEE 축 제외"를 드러내는 쪽이 제품적으로 더 정확하다.

Next action:
- source warning copy는 score를 낮추는 penalty가 아니다.
- warning은 "계산 축이 빠진 점수"를 알리는 역할이다.

### S03. weightSource targetProteinFallback

Current behavior:
- 실제 체중 source가 없으면 `targetProtein / 1.8`로 weight를 추정한다.
- 이 추정값은 protein/carb/fat/alcohol g/kg 계산에 쓰인다.
- `weightSource:"targetProteinFallback"`으로 detail에 남는다.

Risk:
- 실제 체중이 아니라 목표 단백질에서 역산한 체중인데, g/kg 기반 penalty가 authoritative처럼 보일 수 있다.
- 특히 protein shortage/excess, carb g/kg, alcohol g/kg pressure가 모두 영향을 받을 수 있다.

Decision:
- classification: source-confidence
- decision: allow-with-source-warning
- next implementation needed: yes
- test needed: yes
- UI/storage needed: UI/copy yes, storage schema no

Reason:
- target protein은 사용자의 목표 설계에 연결된 의미 있는 fallback이다.
- 그래서 score를 null로 막을 필요는 없다.
- 하지만 actual body weight source와 같은 confidence로 표시하면 안 된다.
- saved `goalSnapshot.weight`와 explicit `options.bodyWeightKg`가 fallback보다 항상 우선해야 한다.

Next action:
- next implementation에서 `weightSource:"targetProteinFallback"`이면 score detail/source note에 "체중은 단백질 목표 기반 추정"임을 표시한다.
- g/kg penalty 자체를 지금 바꾸지는 않는다.

### S04. trainingContext UNKNOWN

Current behavior:
- 명시적 `trainingContext`가 있으면 options 값이 우선한다.
- 운동 관리가 꺼져 있거나 rest로 판정되면 rest context가 된다.
- 운동 정보가 애매하면 unknown context가 가능하다.

Risk:
- UNKNOWN high carb가 training-friendly하게 보이면 고탄수 면제가 과해질 수 있다.
- 다만 이 문제는 source visibility와 carb context tuning이 섞여 있다.

Decision:
- classification: context-uncertainty
- decision: allow-with-internal-source
- next implementation needed: not for source warning alone
- test needed: later if carb context tuning changes
- UI/storage needed: no now

Reason:
- UNKNOWN은 계산 불가가 아니다.
- UNKNOWN은 carb curve context policy 문제이며, 이미 tuning decision에서 `tune-later`로 분류됐다.
- source visibility branch에서 UNKNOWN을 user warning으로 과하게 띄우면 운동기록이 없는 정상 rest day까지 시끄러워질 수 있다.

Next action:
- source visibility implementation은 `trainingContext`를 internal detail/source note로 유지한다.
- high-carb UNKNOWN 점수 강도는 future score tuning에서 본다.

### S05. alcohol source

Current behavior:
- `alcoholG = alcoholKcal / 7`로 derive한다.
- `standardDrinks = alcoholG / 14`로 physiology risk를 계산한다.
- drinking duration은 현재 source에 없다.

Risk:
- NIAAA binge/high-intensity anchor는 duration 맥락이 있는데, 앱은 duration 없이 pure alcohol amount만 본다.
- 숫자 점수는 맞아도 copy가 과도하게 확정적으로 읽힐 수 있다.

Decision:
- classification: source-confidence + copy-policy
- decision: allow-with-copy-caveat-later
- next implementation needed: no for source visibility branch
- test needed: later if alcohol copy changes
- UI/storage needed: no now

Reason:
- alcohol kcal에서 alcoholG를 derive하는 것은 현재 입력 구조에서 합리적이다.
- duration 없음은 score blocked 사유가 아니다.
- 다만 user-facing copy에서는 "술량 기준 위험"으로 표현하고, duration을 아는 것처럼 말하면 안 된다.

Next action:
- DailyCoach/copy tone pass에서 alcohol duration caveat를 본다.
- v6.1 alcoholImpactPenalty 후처리 감산은 되살리지 않는다.

### S06. no meals / one meal / insufficient / snapshotless / range missing

Current behavior:
- no meals, one meal pending, missing macro, missing body weight context 등은 null/blocked로 반환된다.
- sentinel 0을 쓰지 않는다.

Risk:
- source warning으로 대체하면 계산 불가 상태가 계산 가능한 낮은 점수처럼 보일 수 있다.

Decision:
- classification: null-gate
- decision: null-or-blocked
- next implementation needed: no, existing behavior keep
- test needed: existing tests keep, add source visibility regression if implementation touches this area
- UI/storage needed: no

Reason:
- source warning은 "계산은 가능하지만 source confidence가 낮다"는 경우에만 쓴다.
- no meals/one meal/insufficient/snapshotless/range missing은 source confidence 문제가 아니라 calculation availability 문제다.
- 0점도 warning도 아닌 null/blocked가 맞다.

Next action:
- source visibility implementation에서 이 케이스를 warning score로 승격하지 않는다.

### S07. stored current-version auto score

Current behavior:
- current `ADHERENCE_SCORING_VERSION`으로 저장된 auto score는 조회 시 saved `adherencePercent`를 우선한다.
- Today live source 변화가 stored record score를 흔들지 않도록 guard가 있다.

Risk:
- source visibility를 구현하면서 stored score를 live `tdeeAvailable`이나 live weight source로 다시 판정하면 past record가 흔들릴 수 있다.

Decision:
- classification: record-stability
- decision: preserve-stored-score
- next implementation needed: yes, as a guard in source visibility implementation
- test needed: yes
- UI/storage needed: no schema change

Reason:
- 저장된 current-version auto score는 저장 당시의 record contract다.
- source visibility implementation은 Today/live score에는 적용하되, stored record를 조용히 재계산하는 근거가 되면 안 된다.

Next action:
- implementation test에서 stored current-version auto score가 Today live source 변화로 흔들리지 않는지 유지한다.
- stored score detail에 source context가 없더라도 hidden recompute하지 않는다.

### S08. old fixed records

Current behavior:
- old fixed records는 no silent mutation을 유지한다.
- old fixed records는 v8.3 scoring policy를 막는 제품 요구가 아니다.

Risk:
- source visibility policy를 이유로 old fixed records를 자동 v8.3으로 재계산하거나, 반대로 old fixed records 보존을 이유로 source warning implementation을 막을 수 있다.

Decision:
- classification: record-stability
- decision: no-change
- next implementation needed: no
- test needed: existing guards keep
- UI/storage needed: no

Reason:
- old fixed records는 migration concern이다.
- source visibility는 new/live v8.3 scoring confidence를 드러내는 정책이지, old fixed records migration policy가 아니다.

Next action:
- old fixed records migration/recompute/reset은 계속 별도 decision으로 둔다.
