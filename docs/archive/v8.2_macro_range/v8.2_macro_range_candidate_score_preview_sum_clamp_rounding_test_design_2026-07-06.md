## 6. exact sum assertion

formula:

```text
rawCandidateScorePreview = 100 + kcal + protein + fat + carbExchange
candidateScorePreview = clamp(rawCandidateScorePreview, 0, 100)
```

expected exact values:

```text
[0, 0, 0, 0] -> raw 100 -> candidate 100
[-10, 0, -5, 0] -> raw 85 -> candidate 85
[-16, -24, -12, -8] -> raw 40 -> candidate 40
[-10, -16, 0, -6] -> raw 68 -> candidate 68
[-16, 0, -5, -6] -> raw 73 -> candidate 73
```

assertion:

```text
exact expected value를 검증한다.
range label이나 severity label만으로 값을 추론하지 않는다.
candidateScorePreview는 integer 0..100이어야 한다.
```

## 7. lower clamp assertion

lower clamp fixture:

```text
[-80, -40, -30, -10] -> raw -60 -> candidate 0
[-120, 0, 0, 0] -> raw -20 -> candidate 0
```

assertion:

```text
rawCandidateScorePreview < 0이면 candidateScorePreview 후보는 0이다.
단, blocked / invalid / insufficient 상태를 0으로 표현하면 안 된다.
lower clamp 0은 eligible numeric path에서만 의미가 있다.
```

## 8. upper clamp defensive-only assertion

정상 component formula v0에서는 component points가 모두 <= 0이므로 raw candidate가 100을 초과할 수 없다.

따라서 test design은 다음을 구분한다.

```text
defensive clamp boundary: clamp(104, 0, 100) 자체는 100일 수 있음
approved formula path: positive component point가 있으면 candidateScorePreview null / violation
```

future suite가 검증할 것:

```text
positive component pointsPreview는 invalid payload로 reject
positive payload를 100으로 clamp해서 accepted candidate로 만들지 않음
upper clamp는 positive bonus 허용 장치가 아님
```

negative fixture:

```text
[4, 0, 0, 0] -> invalid positive component point
[0, 0, 2, 0] -> invalid positive component point
```

expected:

```text
candidateScorePreview null
violation includes positive_component_point
rangeScoreCandidateSummary.candidateScorePreview remains null
```
