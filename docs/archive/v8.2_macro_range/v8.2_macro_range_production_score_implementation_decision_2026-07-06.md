상태:

```text
문서 성격: docs-only production implementation decision
작업 범위: docs-only
index.html 변경: 없음
package.json 변경: 없음
tools/render_audit 변경: 없음
production score 구현: 없음
getDailyAdherenceScore 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
storage/schema 변경: 없음
backup / Recent / UI / DailyCoach 코드 변경: 없음
scoreDeltaPreview numeric line 진행: 없음
```

이 문서는 `v8.2_macro_range_production_score_transition_tests_implementation_2026-07-06.md` 이후 range-aware production score implementation decision이다.

검토 기준:

```text
docs/v8.2_macro_range*.md 전체 62개 문서 흐름
초기 제품 문제 정의
현재 앱의 실사용 데이터 보존 부담
현재 getDailyAdherenceScore / Records 저장 경로
scoreDeltaPreview realignment 이후의 본류
```

최종 판단:

```text
다음 production implementation은 getDailyAdherenceScore replacement와 ADHERENCE_SCORING_VERSION bump를 같은 implementation gate에서 열어야 한다.
v8.2 production score formula는 component contribution 기반 range-aware score다.
저장/표시 계약은 기존 adherencePercent / adherenceScoringVersion을 우선 사용한다.
pointsPreview / candidateScorePreview / scoreDeltaPreview는 저장 field로 열지 않는다.
scoreDeltaPreview는 계속 optional audit-only / non-blocking이다.
old fixed Records는 no silent mutation만 유지하고 production blocker로 부풀리지 않는다.
componentScores / alcohol penalty / Today-DailyCoach consumer alignment는 implementation 범위에서 명시적으로 처리해야 한다.
```

## 1. 이번 decision이 필요한 이유

직전 test-only suite는 다음을 고정했다.

```text
future scoring version target: v8.2_macro_range_score_v1
productionScore = clamp(100 + sum(kcal/protein/fat/carbExchange contribution), 0, 100)
blocked/null policy
invalid numeric input rejection
new Records forward-only contract
old fixed no silent mutation / non-blocking
scoreDeltaPreview optional audit-only / non-blocking
```

이제 남은 문제는 "formula가 맞는가"가 아니라 "현재 앱의 production score path를 어떻게 안전하게 교체할 것인가"다.

현재 코드에서 `getDailyAdherenceScore()`는 아래를 동시에 담당한다.

```text
Today score percent
Today score label / tone / reason
Records auto adherencePercent 저장
Records adherenceScoringVersion 저장
DailyCoach / Today card consumer detail
fixed componentScores
alcohol penalty detail
```

따라서 실제 구현에서 version bump만 하거나 helper만 추가하면 안 된다. score formula, scoring version, reason copy, consumer detail을 같이 닫아야 한다.

## 2. production implementation 기본 결정

다음 production implementation에서 열 것:

```text
ADHERENCE_SCORING_VERSION = "v8.2_macro_range_score_v1"
getDailyAdherenceScore의 primary score path를 range-aware formula로 교체
new detailed auto Records에 v8.2 score 저장
Today score / Records auto score / DailyCoach score context가 같은 v8.2 result를 읽도록 정렬
range-aware reason copy와 status를 runtime detail로 제공
```

같은 implementation에서 반드시 같이 일어나야 하는 것:

```text
formula switch
ADHERENCE_SCORING_VERSION bump
Records new auto score write path 확인
Today / Records / DailyCoach consumer alignment
기존 fixed componentScores 의존 테스트 갱신
alcohol penalty score semantics 갱신
```

같이 일어나면 안 되는 것:

```text
pointsPreview saved field 추가
candidateScorePreview saved field 추가
scoreDeltaPreview saved/display field 추가
goalSnapshot range schema 확장
backup schema 확장
Recent range-aware aggregation 확장
old fixed Records 일괄 재계산
old fixed Records hidden migration
```

## 3. version bump timing

결정:

```text
ADHERENCE_SCORING_VERSION bump와 getDailyAdherenceScore replacement는 같은 production implementation gate에서 처리한다.
```

금지:

```text
formula를 바꾸고 version을 v6.1_alcohol_penalty_v2로 유지
version만 v8.2_macro_range_score_v1로 바꾸고 fixed score formula 유지
candidate preview version을 production scoring version으로 재사용
testLocal preview version을 production scoring version으로 재사용
scoreDeltaPreview identity를 production scoring version으로 사용
```

이유:

```text
저장된 adherencePercent의 의미는 adherenceScoringVersion과 같이 읽힌다.
score 의미가 바뀌면 version도 같이 바뀌어야 한다.
version이 바뀌면 score path도 실제로 바뀌어야 한다.
```

## 4. production score formula

production formula:

```text
productionScore = clamp(
  100 + kcalContribution + proteinContribution + fatContribution + carbExchangeContribution,
  0,
  100
)
```

합산 대상:

```text
kcal
protein
fat
carbExchange
```

합산 제외:

```text
dataQuality
scoreDeltaPreview
fixedScoreSummary
saved componentScores
testLocalNumericPreview
```

component contribution source of truth:

```text
kcal matched: 0
kcal high / low: -10
kcal severe_high / severe_low: -16

protein satisfied: 0
protein low: -16
protein severe_low: -24

fat inside_recommended: 0
fat below_recommended / above_recommended: -5
fat below_action_floor / above_flexible: -12

carbExchange exchange_explained / inside_candidate_range: 0
carbExchange below_training_support: -8
carbExchange above_candidate_range: -6

dataQuality: null-only eligibility gate
```

## 5. blocked/null policy

numeric score를 만들 수 있는 상태:

```text
full-day detailed macro evidence
goal / targetCal resolved
protein anchor basis resolved
kcal anchor basis resolved
fat range basis resolved
carb exchange basis resolved
invalid/impossible state 없음
```

numeric score를 만들면 안 되는 상태:

```text
no meals
one_meal_pending
insufficient_record
snapshotless old/debug-only record
impossible_state
unsupported_range_basis
invalid_range_basis
range_policy_missing
```

blocked output:

```text
percent: null
isAvailable: false
status: blocked reason
detail.scoringVersion: null 또는 omitted
adherencePercent 저장: null
adherenceScoringVersion 저장: null
sentinel 0: 금지
```

중요:

```text
0점은 가능한 최저 numeric score다.
계산 불가 상태를 0점으로 저장하거나 표시하지 않는다.
```

## 6. current componentScores 의존성 처리

현재 `detail.componentScores`는 fixed target score의 0..1 component ratio다.

v8.2 production score에서 금지:

```text
fixed componentScores를 v8.2 score source로 사용
saved componentScores를 fallback으로 사용
component contribution integer를 기존 componentScores 의미로 끼워 넣기
componentScores를 유지한 채 copy만 range-aware처럼 바꾸기
```

v8.2 implementation 방향:

```text
detail.componentContributions:
  kcal / protein / fat / carbExchange integer contribution

detail.componentStatuses:
  kcal / protein / fat / carbExchange status enum

detail.rangeScoreFormula:
  macro_range_score_v1

detail.componentScores:
  v8.2 score source가 아님
```

판단:

```text
가능하면 v8.2 result에서는 Today / DailyCoach consumer가 componentScores 대신 reasons,
componentStatuses, componentContributions를 읽도록 교체한다.
호환을 위해 componentScores를 잠시 남긴다면 display-only compatibility field로 제한하고,
score source / fallback / storage source로 쓰지 않는 guard가 필요하다.
```

## 7. alcohol penalty 처리 결정

현재 production version 이름은 `v6.1_alcohol_penalty_v2`이고, 현재 `getDailyAdherenceScore()`는 fixed score 산출 후 `alcoholImpactPenalty`를 별도 감산한다.

v8.2 macro range score v1 결정:

```text
별도 alcoholImpactPenalty를 range-aware productionScore에서 후처리 감산하지 않는다.
```

허용:

```text
alcohol kcal가 total/scoring kcal에 포함되어 kcal anchor status에 영향을 주는 것
DailyCoach alcohol signal이 별도로 유지되는 것
alcohol 관련 warning / action copy가 DailyCoach에서 유지되는 것
```

금지:

```text
range-aware formula 이후 alcoholImpactPenalty를 추가로 빼기
scoreBeforeAlcoholPenalty / scoreAfterAlcoholPenalty를 v8.2 production score source로 유지
v6.1 alcohol penalty detail을 v8.2 score 의미처럼 저장
```

이유:

```text
v8.2 score formula는 이미 component contribution table로 닫혔다.
별도 alcohol penalty를 얹으면 exact fixtures와 formula identity가 흔들린다.
알코올 행동 안내는 DailyCoach signal로 유지할 수 있고, 점수 버전 변경으로 score semantics 변화는 명시된다.
```

후속 test에서 필요한 것:

```text
기존 alcohol penalty scoring tests 갱신
alcohol이 kcal overshoot를 만들면 kcal contribution으로 반영되는지 확인
alcohol-only warning은 DailyCoach signal로 유지되는지 확인
별도 alcoholImpactPenalty가 productionScore에서 감산되지 않는지 확인
```

## 8. getDailyAdherenceScore replacement boundary

replacement 범위:

```text
getDailyAdherenceScore(date, currentResult, options)
  -> getMacroBalanceForDate(...)
  -> range-aware eligibility / contribution classification
  -> productionScore number 또는 null
  -> label / tone / status / reasons / detail 반환
```

권장 internal helper:

```text
getMacroRangeProductionScoreInput(balance, currentResult, options)
getMacroRangeComponentContributions(input)
getMacroRangeProductionScoreFromContributions(contributions)
buildMacroRangeAdherenceReasons(contributions, balance)
```

helper는 production path 내부 helper이지 preview field writer가 아니다.

반환 방향:

```text
numeric:
  percent: 0..100 integer
  source: auto
  isAvailable: true
  status: available 또는 full_day_candidate
  label / tone: percent 기반
  reasons: range-aware reasons
  detail.scoringVersion: v8.2_macro_range_score_v1
  detail.componentContributions: internal runtime detail
  detail.componentStatuses: internal runtime detail
  detail.rangeScoreFormula: macro_range_score_v1

blocked:
  percent: null
  source: auto
  isAvailable: false
  status: blocked reason
  reasons: blocked guidance
  detail.scoringVersion: null 또는 omitted
```

## 9. Records write path

현재 detailed auto Records는 `persistAutoAdherenceForDate`, `syncTodayRecordGoalSnapshot`, `saveRecord`, Today input save flows를 통해 `getDailyAdherenceScore()` 결과를 저장한다.

v8.2 implementation에서 유지할 저장 계약:

```text
adherencePercent
adherenceScoringVersion
adherenceSource
adherenceLockedAt
goalSnapshot
snapshotSource
```

new detailed auto Record:

```text
adherencePercent = range-aware score number
adherenceScoringVersion = v8.2_macro_range_score_v1
```

blocked detailed auto Record:

```text
adherencePercent = null
adherenceScoringVersion = null
```

simple / weight_only / manual:

```text
기존 manual/simple/weight_only 정책 유지
range score로 강제 승격하지 않음
```

old fixed Records:

```text
saved adherencePercent 유지
saved adherenceScoringVersion 유지
goalSnapshot 유지
no silent mutation
production 전환 blocker 아님
```

## 10. Today / Records / DailyCoach consumer alignment

v8.2 implementation에서 같이 확인할 consumer:

```text
Today score card
Today macro component cards
Records list score display
Records detail score copy
Recent/weekly summary basis labels
DailyCoach signals / decisions / output
```

필수 정렬:

```text
Today copy가 fixed g 목표 exact miss를 전제로 말하지 않을 것
fat low + carb exchange explained case에서 이중 처벌 copy가 나오지 않을 것
protein low case에서 anchor penalty copy가 유지될 것
blocked/null case에서 0점처럼 보이지 않을 것
old fixed Records가 range-aware score로 조용히 재해석되지 않을 것
DailyCoach가 candidateScorePreview / pointsPreview / scoreDeltaPreview를 읽지 않을 것
DailyCoach alcohol signal은 유지하되 scoreDelta나 old fixed comparison을 전제로 하지 않을 것
```

이번 implementation decision은 UI layout redesign을 열지 않는다. copy/consumer alignment는 production score 의미와 충돌하지 않게 하는 최소 범위다.
