# v8.2 macro range snapshot compatibility design

상태:

```text
문서 성격: snapshot compatibility 설계 / 중간점검
작업 범위: docs-only
구현 변경: 없음
index.html 변경: 없음
goalSnapshot schema 변경: 없음
backup / restore schema 변경: 없음
Recent range-aware logic 변경: 없음
ADHERENCE_SCORING_VERSION 변경: 없음
production macro range 구현: 금지
```

이 문서는 `v8.2_macro_range_score_candidate_report_only_implementation_2026-07-03.md` 이후 단계다.

목적:

```text
macro range 후보가 나중에 저장 가능한 contract로 승격될 때,
기존 Records / goalSnapshot / backup / restore / Recent / score basis를 깨지 않도록
사전 compatibility 계약을 정리한다.
```

중요:

```text
이 문서는 구현 지시서가 아니다.
macroRangeContract를 저장하라는 뜻이 아니다.
현재 fixed score와 fixed goalSnapshot이 source of truth다.
```

## 1. 중간점검

현재까지 닫힌 단계:

```text
feasibility audit
policy/data contract design
report-only helper design
report-only helper implementation
sample matrix 보정
range score candidate report-only design
range score candidate report-only implementation
```

현재까지 유지한 금지선:

```text
calculateMacroTargets 변경 없음
buildExternalMacroActiveProductionApplication 변경 없음
getDailyAdherenceScore 변경 없음
ADHERENCE_SCORING_VERSION 변경 없음
goalSnapshot schema 변경 없음
backup / restore schema 변경 없음
Recent range-aware logic 변경 없음
Today UI range 표시 없음
Records 과거 점수 재해석 없음
```

이번 문서의 역할:

```text
다음 사람이 "이제 schema만 붙이면 되나?"라고 오해하지 않게 막는다.
현재 코드가 어떤 필드를 보존하고 어떤 필드를 버리는지 명확히 적는다.
future optional field가 필요한 경우 어떤 함수와 테스트를 동시에 열어야 하는지 정리한다.
```

## 2. 현재 코드 관찰

### 2-1. normalizeGoalSnapshot()

현재 `normalizeGoalSnapshot(raw)`는 allowlist 방식이다.

필수:

```text
targetCal
protein
carbs
fat
```

이 네 값이 finite가 아니면 snapshot 전체가 `null`이다.

반환 필드는 명시된 필드만이다.

따라서 현재 구조에서:

```text
raw.goalSnapshot.macroRangeContract
raw.goalSnapshot.rangeScoreCandidate
raw.goalSnapshot.unknownFutureField
```

같은 unknown field는 normalize 과정에서 보존되지 않는다.

판단:

```text
현재 구조는 unknown optional field pass-through가 아니다.
future range snapshot field가 필요하면 normalizeGoalSnapshot에 명시적으로 추가해야 한다.
```

### 2-2. normalizeRecord()

현재 `normalizeRecord(raw)`는 record field도 allowlist 방식이다.

보존:

```text
date
recordMode
weight
adherence
adherencePercent
adherenceSource
adherenceScoringVersion
goalSnapshot
adherenceLockedAt
snapshotSource
note
meals
```

`goalSnapshot`은 다시 `normalizeGoalSnapshot(raw.goalSnapshot)`을 통과한다.

판단:

```text
record-level unknown field도 보존하지 않는다.
snapshot-level unknown field도 보존하지 않는다.
```

### 2-3. buildGoalSnapshotFromCurrentState()

현재 `buildGoalSnapshotFromCurrentState(currentResult, snapshotSource)`는 current production result에서 fixed target snapshot을 만든다.

현재 포함:

```text
targetCal
protein
carbs
fat
body/profile/activity/time ownership fields
profileMacroCandidateModel
profileMacroCandidateStage
profileTargetDeltaKcal
profileTargetRateDeltaEquivalentKgPerWeek
```

현재 미포함:

```text
macroRangeContract
rangeScoreCandidate
candidateScorePreview
macroTargetBasisVersion
rangeScoreVersion
```

판단:

```text
현재 저장 snapshot은 fixed target basis다.
report-only helper output은 buildGoalSnapshotFromCurrentState에 연결되어 있지 않다.
```

### 2-4. getGoalSnapshotSignature()

현재 signature는 `GOAL_SNAPSHOT_SIGNATURE_KEYS` allowlist 기준이다.

현재 range 관련 field는 없다.

의미:

```text
range field를 normalize에 추가해도 signature key에 넣지 않으면 무변경 저장 비교에는 반영되지 않는다.
signature key에 넣으면 Records 상세 수정 무변경 저장, Today/Records 갱신 필요 판단, backup roundtrip 테스트에 영향이 생긴다.
```

판단:

```text
future range field를 signature에 넣을지 여부는 별도 결정이다.
대충 넣으면 RC1에서 고친 goalSnapshot 정밀도/무변경 저장 문제가 재발할 수 있다.
```

### 2-5. backup / restore

현재 full backup:

```text
buildFullBackupPayload()
-> records: getSortedRecords()
```

restore:

```text
normalizeFullBackupPayload()
-> normalizeRecordsCollection()
-> normalizeRecord()
-> normalizeGoalSnapshot()

restoreFullBackupData()
-> normalizeRecordsCollection(data.records)

applySmartRestoreImportData()
-> normalizeRecordsCollection(data.records)
-> normalizeRecord(record)

applyRecordsOnlyImportData()
-> normalizeRecordsCollection(data.records)
```

판단:

```text
현재 backup/restore는 unknown goalSnapshot optional field를 보존하는 구조가 아니다.
future macroRangeContract를 backup roundtrip으로 살리고 싶으면 normalizeGoalSnapshot과 backup/restore 테스트를 같이 열어야 한다.
```

### 2-6. Recent

현재 `getRecentContextRecordSummary(record)`는 normalized record에서 fixed target snapshot을 읽는다.

사용:

```text
targetCal: normalized.goalSnapshot.targetCal
targetProtein: normalized.goalSnapshot.protein
targetCarbs: normalized.goalSnapshot.carbs
targetFat: normalized.goalSnapshot.fat
goalType
activity/time ownership fields
adherenceScoringVersion
```

현재 `getRecentContextComponentCounts()`는 fixed target 기준이다.

```text
proteinLowDays: protein < targetProtein * 0.90
carbLowTrainingDays/restDays: carbs < targetCarbs * 0.85
fatLowDays: fat < targetFat * 0.75
fatHighDays: fat > targetFat * 1.25
```

판단:

```text
Recent는 현재 range-aware가 아니다.
range field를 snapshot에 저장해도 Recent가 자동으로 이해하지 않는다.
Recent에 range basis를 연결하려면 mixed basis 해석과 UI 문구까지 별도 설계가 필요하다.
```

### 2-7. Weekly adherence / scoring version

현재 `getStoredAdherenceEntry(record)`는 `adherenceScoringVersion`을 읽는다.

`getWeeklyAdherenceSnapshot()`은 auto score basis를:

```text
ADHERENCE_SCORING_VERSION과 일치하면 current basis
아니면 legacy
```

로 묶는다.

판단:

```text
range score production이 열리면 scoring version 변경 없이는 old/new score basis를 구분하기 어렵다.
하지만 지금은 ADHERENCE_SCORING_VERSION 변경 금지다.
```

## 3. compatibility 목표

목표:

```text
old fixed snapshot은 계속 정상 동작한다.
new optional range snapshot이 생겨도 old record를 재해석하지 않는다.
backup/restore는 old/new record를 모두 안전하게 roundtrip한다.
Recent는 basis가 섞인 데이터를 조용히 같은 기준으로 취급하지 않는다.
Records 상세 수정 무변경 저장은 range optional field 때문에 깨지지 않는다.
```

비목표:

```text
이번 문서에서 macroRangeContract 저장 구조 확정
이번 문서에서 range score version 확정
이번 문서에서 Recent range-aware 계산 확정
이번 문서에서 backup schema migration 구현
```

## 4. old/new snapshot 원칙

### 4-1. old snapshot

정의:

```text
goalSnapshot에 targetCal/protein/carbs/fat만 fixed target basis로 있는 현재 v8.0 구조
```

원칙:

```text
old snapshot은 fixed target basis로 읽는다.
old snapshot에는 macroRangeContract가 없어도 정상이다.
old snapshot을 range 기준으로 조용히 재해석하지 않는다.
old record의 adherencePercent/adherenceScoringVersion은 기존 의미를 유지한다.
```

### 4-2. new optional snapshot 후보

정의 후보:

```text
goalSnapshot 안에 range basis optional contract가 있는 미래 구조
```

주의:

```text
이 후보는 아직 구현하지 않는다.
이 후보 이름도 아직 확정하지 않는다.
```

후보 형태:

```js
goalSnapshot: {
  targetCal,
  protein,
  carbs,
  fat,
  macroTargetBasisVersion: "fixed_v8" | "macro_range_v8_2_candidate",
  macroRangeContract: { ...optionalKnownShape },
  rangeScoreCandidateVersion: "range_score_candidate_report_only_v0" | null
}
```

단, 현재 단계에서는 위 형태를 저장하지 않는다.

## 5. macroRangeContract 저장 후보의 최소 조건

`macroRangeContract`가 저장 field로 승격되려면 최소 조건이 필요하다.

필수 조건:

```text
1. normalizeMacroRangeContractForSnapshot() 설계
2. normalizeGoalSnapshot()에 optional field 명시 추가
3. GOAL_SNAPSHOT_SIGNATURE_KEYS 포함 여부 결정
4. buildGoalSnapshotFromCurrentState() write 조건 결정
5. getDailyAdherenceScore() old/new score basis 분리 설계
6. getStoredAdherenceEntry() / weekly basis summary 설계
7. Recent mixed basis 해석 설계
8. full backup roundtrip 테스트
9. records-only restore 테스트
10. smart restore conflict 테스트
```

금지:

```text
unknown field pass-through로 대충 보존
helper output 전체를 snapshot에 복사
candidateScorePreview를 snapshot에 저장
macroRangeContract를 backup field처럼 독립 저장
old Records에 range field를 사후 주입
```

## 6. normalize 설계 후보

후보 함수:

```text
normalizeMacroRangeContractForSnapshot(raw)
```

원칙:

```text
allowlist 방식
version 필수
targetCal은 snapshot targetCal과 일치해야 함
protein/fat/carbs 하위 구조는 known field만 허용
candidateScorePreview 저장 금지
reportOnly meta 저장 금지
needsDecision 저장 여부 별도 결정
invalid range contract는 null
invalid range contract가 core fixed snapshot 전체를 null로 만들면 안 됨
```

핵심:

```text
targetCal/protein/carbs/fat fixed fields는 여전히 snapshot의 최소 생존 조건이다.
macroRangeContract가 깨져도 old fixed snapshot은 읽을 수 있어야 한다.
```

## 7. signature 설계 후보

선택지가 있다.

### A. range field를 signature에서 제외

장점:

```text
Records 상세 수정 무변경 저장 안정성이 높다.
range candidate meta 변화가 기존 fixed snapshot diff를 만들지 않는다.
```

위험:

```text
range contract가 실제 저장 의미를 바꾸는 경우 변경 감지가 안 된다.
```

### B. range field를 signature에 포함

장점:

```text
저장 당시 range basis 변경을 감지할 수 있다.
```

위험:

```text
floating precision / optional field order / policy version 변화가 Records 갱신 필요 경고를 만들 수 있다.
RC1에서 고친 무변경 저장 문제가 재발할 수 있다.
```

판단:

```text
production 적용 전에는 A/B를 결정하지 않는다.
snapshot compatibility test에서 둘 다 simulation해야 한다.
```

## 8. backup / restore 설계 후보

현재 사실:

```text
unknown optional field는 restore normalize 과정에서 보존되지 않는다.
```

future 목표:

```text
known optional macroRangeContract는 full backup / records-only / smart restore에서 roundtrip한다.
unknown future field는 보존하지 않을 수 있지만, known range field는 명시적으로 보존한다.
```

필수 테스트:

```text
full backup old fixed record roundtrip
full backup new range record roundtrip
records-only old fixed record roundtrip
records-only new range record roundtrip
smart restore keep mode preserves local basis
smart restore replace mode applies backup basis
smart restore meals-only mode does not accidentally replace local snapshot basis
conflict preview indicates mixed basis only if user-facing policy is approved
```

주의:

```text
smart restore meals-only mode는 특히 위험하다.
식사만 합치기인데 snapshot basis까지 바뀌면 사용자 기대와 다르다.
```

## 9. Recent compatibility 설계 후보

현재 Recent는 fixed target basis다.

range-aware Recent를 열 경우 후보:

```text
fixedBasisDays
rangeBasisDays
legacyScoreDays
currentFixedScoreDays
rangeCandidateDays
mixedMacroBasisDetected
```

원칙:

```text
old fixed record와 new range record를 같은 component count로 조용히 합치지 않는다.
range basis가 섞이면 confidence/contextFit을 낮추거나, 별도 internal basis summary를 둔다.
user-facing 문구는 나중에 결정한다.
```

아직 금지:

```text
Recent 3/7/14/28 trend를 range 기준으로 재계산
fatLowDays를 range floor로 대체
carb high/low interpretation 변경
Recent UI 문구 변경
```

## 10. score compatibility 설계 후보

현재:

```text
ADHERENCE_SCORING_VERSION = v6.1_alcohol_penalty_v2
```

range score production이 열리면 새 scoring version 후보가 필요하다.

하지만 지금:

```text
ADHERENCE_SCORING_VERSION 변경 금지
candidateScorePreview 숫자화 금지
getDailyAdherenceScore 변경 금지
```

future 원칙:

```text
old auto score는 old scoring version으로 유지
new range-aware score는 새 version 후보가 필요
weekly/recent는 mixed scoring basis를 감지해야 함
Records 상세는 저장 당시 score basis를 표시/해석해야 함
```

## 11. failure mode

막아야 할 실패:

```text
1. old Records가 range 기준으로 조용히 재해석됨
2. backup/restore 후 macroRangeContract가 사라짐
3. smart restore meals-only가 snapshot basis까지 바꿈
4. Records 상세 무변경 저장에서 optional field 때문에 갱신 경고 발생
5. Recent가 fixed/range basis를 섞어 같은 fatLowDays로 집계함
6. candidateScorePreview가 저장되거나 UI에 노출됨
7. ADHERENCE_SCORING_VERSION 없이 새 score 의미가 섞임
8. normalizeGoalSnapshot invalid range contract 때문에 fixed snapshot까지 null이 됨
9. full backup diff에 report-only helper output이 들어감
10. old/new mixed records에서 Coach/Recent가 확정적 행동 조언을 함
```

## 14. 아직 사용자 판단이 필요한 항목

아직 결정하지 않는다.

1. `macroRangeContract`를 실제 goalSnapshot 하위 필드로 둘지.
2. range field를 `GOAL_SNAPSHOT_SIGNATURE_KEYS`에 포함할지.
3. invalid range contract가 있을 때 warning을 남길지 조용히 null 처리할지.
4. full backup conflict preview에서 range basis 차이를 보여줄지.
5. smart restore meals-only에서 backup snapshot basis를 절대 무시할지.
6. Recent mixed basis를 사용자에게 보여줄지 internal confidence로만 낮출지.
7. old Records에 report-only range candidate를 계산해 보여줄지.
8. new range-aware score의 scoring version 이름.
9. candidateScorePreview 숫자화를 언제 허용할지.
10. range score와 fixed score가 다른 경우 Records 상세에서 어떤 기준을 표시할지.

## 16. 최종 판단

현재 macro range 라인은 아직 production schema로 갈 준비가 아니다.

하지만 compatibility 설계 방향은 명확하다.

```text
fixed snapshot은 계속 source of truth다.
macroRangeContract는 아직 저장하지 않는다.
unknown optional field는 현재 보존되지 않는다.
future range field는 explicit normalizer + signature decision + backup/restore/Recent tests 없이는 열 수 없다.
```

최종 결론:

```text
snapshot compatibility 설계는 진행 가능.
production snapshot schema 구현은 아직 금지.
backup/restore/Recent 연결은 아직 금지.
```
