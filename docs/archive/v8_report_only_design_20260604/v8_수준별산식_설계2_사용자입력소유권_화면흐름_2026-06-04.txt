V8 수준별 산출식 설계 2단계 - 사용자 입력 소유권 / 화면 흐름 확정
작성일: 2026-06-04
기준 브랜치: codex/v8-formula-audit
기준 커밋 직전 상태: v8.0-BB report-only contract 추가 작업 중

이 문서는 1단계 `현 상태 진실지도` 다음 단계다.
목적은 수준별 산출식을 구현하기 전에 사용자가 어떤 값을 어디서 고르고, 어떤 값은 보여주기만 하고, 어떤 값은 자동 파생하는지 확정하는 것이다.

이 단계도 production 산식 구현이 아니다.
다만 문서만으로 끝내면 다음 작업자가 다시 문서를 자기 편한 쪽으로 읽을 수 있으므로, `index.html`의 `runV8ScenarioRunner()`에 `userInputOwnershipFlowDesign` report-only contract를 추가했다.

## 1. 직접 확인한 코드 사실

### 1-1. Today 계산은 Settings 원본값을 그대로 쓰지 않는다

현재 `calculate()`는 `collectState()`를 호출한다.
`collectState()`는 다시 `getEffectiveTodayCalculationValues()`를 통해 오늘 날짜의 effective calculation values를 만든다.

즉 production 계산 입력은 대략 아래 순서다.

1. Settings / state 기본값
2. Today 날짜별 draft
3. Today draft에 없는 경우 InBody / cardio preset / routine default materialization
4. 정규화된 effective Today values
5. `calculate()`
6. `buildGoalSnapshotFromCurrentState()`
7. Records `goalSnapshot`

따라서 Settings 기본값과 Today 오늘 값은 같은 것이 아니다.
Settings는 기본값이고, Today는 오늘 계산 기준이다.

### 1-2. Today 프로필은 현재 수정 권한이 없다

`syncTodayQuickProfileControls()`는 Today quick exercise profile select를 렌더하지만 disabled 처리한다.

현재 정책:

- 운동 프로필은 Settings-owned default다.
- Today에서는 프로필을 보여줄 수 있다.
- Today에서 프로필을 바꾸는 기능은 현재 승인되지 않았다.
- Today에서 프로필까지 date-owned override로 열려면 별도 UX / 저장 / snapshot 계약이 필요하다.

이 결정은 사용자가 제안한 “Today에는 프로필을 보여주되 수정은 못 하게 한다”와 맞는다.

### 1-3. Today 루틴/세션은 날짜 소유다

Today draft에는 `todayRoutinePlan`, `todayRoutineSession`, `todayProfileSession`, `todayRoutineExplicit`이 들어갈 수 있다.
`routineSession`은 Today에서 날짜별로 바뀔 수 있고, Records snapshot에는 당시 `routine`으로 남는다.

현재 정책:

- Settings는 기본 운동 프로필, 기본 루틴 플랜, 요일별 세션 계획을 가진다.
- Today는 그 날짜에 실제 적용할 루틴/세션을 가진다.
- Records는 저장 시점의 루틴/세션 snapshot을 보존한다.

### 1-4. `profileSession`은 사용자가 고르는 값이 아니다

`profileSession`은 `deriveProfileSessionFromRoutine(exerciseProfile, routinePlan, routineSession)`에서 파생된다.
이 값은 runner / snapshot / future formula context에서 사용할 수 있지만, 사용자에게 별도 shortcut selector로 다시 살리면 안 된다.

금지:

- `PROFILESESSION`을 사용자가 고르는 핵심 입력처럼 복구
- 루틴/세션이 실제로 존재하지 않는데 `profileSession`만 넣어서 기준 통과
- 보디빌딩 세션으로 러닝/혼합운동을 fallback

### 1-5. 고급설정 OFF / ON 의미

현재 코드 원리는 다음과 같다.

고급설정 OFF:

- visible routine/session control은 단순 REST / PUSH 흐름이다.
- 보디빌딩 OFF는 기존 간단 훈련/휴식 흐름을 유지한다.
- 보디빌딩 OFF의 기본 훈련 강도는 주간 보정 전 `0.70`이다.
- 비보디빌딩 OFF도 화면상 단순 흐름을 유지하지만, 내부 기본값은 해당 프로필의 기본 훈련 session default를 참조할 수 있다.

고급설정 ON:

- 프로필별 상세 routine plan / session이 노출된다.
- session xw, defaultWeightDuration, cardioDefaults가 세션별로 반영된다.
- running / mixed처럼 weightDuration이 0이어도 cardioDefaults가 있는 세션은 휴식으로 해석하면 안 된다.

남은 UX 검토:

- 비보디빌딩 OFF에서 화면은 단순 REST/PUSH인데 내부는 프로필 default를 반영하는 구조가 사용자가 이해하기 쉬운지 broad human UX review가 아직 닫히지 않았다.
- 이 단계는 그 UX 검토를 닫지 않는다.

### 1-6. Records는 dated snapshot이지 Today 입력창이 아니다

Records는 식사, 공복 체중, 메모, 당시 계산 기준 snapshot을 저장한다.
Records 식사 추가/수정은 Today 계산 값을 자동으로 바꾸면 안 된다.

예외:

- 사용자가 Records/Today 기록 시작 flow에서 “오늘 계산에 반영”을 명시적으로 선택하면 Records 공복 체중을 Today calculation draft에 반영할 수 있다.
- 오늘 상세 식사 추가 hotfix 이후, 공복 체중을 계산에 반영하기로 한 경우에는 식사 snapshot 저장 전에 Today calculation weight가 먼저 맞춰진다.

### 1-7. InBody는 측정 source이고 Today를 seed할 수 있다

InBody 기록은 `state.inbodyRecords`에 보관된다.
Today가 아직 사용자 body draft를 갖고 있지 않을 때 최신 InBody 값이 Today default로 materialize될 수 있다.

하지만:

- InBody가 Records 공복 체중을 직접 바꾸면 안 된다.
- InBody가 Settings 기본 체중/체성분을 직접 바꾸면 안 된다.
- InBody 값은 bodyCompReliability의 근거가 될 수 있지만, 단독으로 target calorie를 자동 변경하는 권한은 없다.

### 1-8. 최근 기록은 입력창이 아니라 evidence window다

`buildRecentContext()`는 Records에서 1/3/7/14/28일 window를 만든다.
recent14는 주로 coverage / interpretation evidence다.
recent28은 target-rate review / gate context로 쓸 수 있지만, 목표 변경, weight-day 부족, snapshot coverage 부족, confidence 부족이 있으면 자동 변경 근거로 쓰면 안 된다.

BA candidate의 recent14/recent28은 전부 `productionAutoApplyAllowedCount = 0`으로 막혀 있다.

## 3. 화면 흐름 확정

### 3-1. Settings

역할:

- 장기 기본값
- Today가 열릴 때 쓰는 default candidate
- 사용자 기본 운동 프로필
- 기본 루틴 플랜
- 요일별 세션 계획
- weeklyTrainingDays 자동/직접 기준
- 고급설정 ON/OFF
- cardio preset/default
- backup/data management

Settings에서 하지 않는 것:

- 특정 날짜의 실제 세션 확정
- `profileSession` 직접 선택
- performanceLevel shortcut 선택
- Records snapshot 직접 갱신

### 3-2. Today

역할:

- 오늘 계산 기준 작업공간
- 계산용 체중/체성분의 date-owned override
- 오늘 activityLevel/workType override
- 오늘 routinePlan/routineSession override
- 오늘 cardioLoad override
- guide/activity macro basis 선택
- 오늘 기록 시작 flow

Today에서 보여주되 수정하지 않는 것:

- 운동 프로필

Today에서 파생만 하는 것:

- profileSession
- trainingEvidence
- performanceLevel
- bodyCompReliability
- recent window context

Today의 확정 UX:

- 상단 또는 운동 카드에서 `보디빌딩 - 루틴 / 세션`, `러닝 - 루틴 / 세션`처럼 프로필을 문맥으로 보여준다.
- 프로필 자체는 Settings-owned이므로 Today quick edit에서 disabled context다.
- 루틴/세션은 Today date-owned로 선택 가능하다.
- 고급설정 OFF에서는 단순 REST/PUSH 흐름이다.
- 고급설정 ON에서는 프로필별 상세 루틴/세션을 노출한다.

### 3-3. Records

역할:

- 저장된 날짜별 evidence
- 식사 / 공복 체중 / 메모
- 당시 계산 기준 snapshot
- 최근 흐름 / Coach / Score의 근거

Records에서 하지 않는 것:

- Today 현재 계산 기준 자동 변경
- Settings 기본값 자동 변경
- 과거 snapshot 자동 재계산

예외:

- 사용자가 “오늘 계산에 반영”을 명시적으로 선택한 one-shot flow
- 사용자가 record detail basis edit를 명시적으로 연 경우

### 3-4. InBody

역할:

- 체성분 source record
- Today 계산값으로 가져올 수 있는 source
- bodyCompReliability의 근거

InBody에서 하지 않는 것:

- Records 공복 체중 자동 변경
- Settings default 자동 변경
- targetCal 자동 변경

### 3-5. Recent / Coach

역할:

- 기록 기반 해석
- coverage / confidence / trend gate
- 사용자 설명

하지 않는 것:

- 사용자 입력값처럼 직접 수정
- 낮은 coverage에서 target 자동 변경
- Records snapshot 없이 근거 주장

### 3-6. Backup

역할:

- Settings, Records, InBody, Today calculation drafts, weekly training state, presets를 보존
- future level-aware formula context가 production에 들어가기 전 model/stage/reason이 roundtrip되는지 확인

## 4. 수준별 산출식 입력 설계 결론

수준별 산출식은 아래처럼 입력을 받아야 한다.

1. 직접 사용자 입력
   - goal
   - Settings exerciseProfile
   - Today routinePlan/routineSession
   - Today calculationWeight/bodyComposition
   - Today activityLevel/workType/cardioLoad
   - Settings weeklyTrainingDays manual override 또는 Records auto weekly count

2. source에서 가져온 입력
   - InBody source record
   - cardio preset
   - Records goalSnapshot

3. 자동 파생 context
   - profileSession
   - bodyCompReliability
   - trainingEvidence
   - performanceLevel
   - recent14RecordCoverage
   - recent28Trend

4. 저장만 되는 당시 기준
   - Records goalSnapshot
   - goalSnapshot signature
   - future candidate model/stage/delta/reason

5. production 전 반드시 붙어야 하는 사용자 설명
   - 왜 targetCal이 바뀌었는지
   - 왜 protein/fat/carbs가 바뀌었는지
   - 어떤 입력이 그 변화를 만들었는지
   - recent trend가 실제 자동 변경인지, review signal인지
   - 프로필/세션 fuel request가 target relief인지, 단순 설명인지

## 6. 다음 단계로 넘기는 기준

다음 단계는 구현이 아니라 3단계다.

3단계 이름:

V8 수준별 산출식 설계 3단계 - 운동 프로필별 훈련 모델 설계

3단계에서 해야 할 일:

- 보디빌딩 / 파워빌딩 / 스트렝스 / 러닝 / 혼합운동의 실제 훈련 모델을 각각 설계한다.
- 각 프로필별 routine/session이 어떤 load, recovery, cardio/fuel need, target relief risk를 갖는지 설계한다.
- 고급설정 OFF/ON에서 같은 프로필이 어떻게 다르게 해석되는지 명확히 정한다.
- running / mixed를 bodybuilding fallback으로 처리하지 않는다.
- profileSession shortcut이 아니라 `exerciseProfile + routinePlan + routineSession + cardioLoad + weightDuration + intensity + weekly consistency` 조합으로 판단한다.

3단계 진입 전 금지:

- `performanceLevel` selector부터 만들기
- profile별 기본세션 1개로 통과시키기
- recent trend를 target 자동 변경 근거로 바로 쓰기
- InBody reliability만 보고 protein/target을 공격적으로 바꾸기
- Records snapshot/Coach/Backup 계약 없이 production 산식 적용하기
