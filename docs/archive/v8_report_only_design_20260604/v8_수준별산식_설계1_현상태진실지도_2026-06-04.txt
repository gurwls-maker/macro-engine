# V8 수준별 산출식 설계 1단계 - 현 상태 진실지도

작성일: 2026-06-04
브랜치: `codex/v8-formula-audit`
기준 커밋: `1a9b5f1 v8.0-BA add level-aware macro candidate`
목적: 수준별 산출식 설계를 완성하기 전에 production, scoped production, report-only, 문서 설계, open gate를 서로 헷갈리지 않게 고정한다.

## 2. 현재 production / scoped production

### 2-1. 현재 production으로 봐도 되는 것

직접 추출 기준:

```text
GOALS.diet.proteinKg = 2.0
getDualBasisProteinRange("diet") = { min: 1.4, mid: 2.0, max: 2.0 }
GOALS.maintain exists = true
GOALS.maintain.proteinKg = 2.0
CARDIO_TARGET_REFLECTION_RATES.diet = 0.40
CARDIO_TARGET_REFLECTION_RATES.maintain = 0.50
```

현재 production으로 인정되는 범위:

- 일반 diet 단백질 기본값은 `2.0g/kg`로 복구됐다.
- maintain goal key는 production `GOALS`에 존재한다.
- active runtime candidate-v2 target-delta wiring은 이미 production 적용/QA 완료로 기록되어 있다.
- 운동 프로필/루틴/세션 input path는 `calculate()`와 Records snapshot까지 연결되는 code-path evidence가 있다.

### 2-2. scoped production으로만 봐야 하는 것

`profileFormulaApprovalDecision.summary` 직접 추출 기준:


이 값은 전체 수준별 산출식 완성이 아니다.

정확한 범위:

- candidate-v2는 active runtime target-delta case에 scoped production으로 적용된 것이다.
- candidate-v2 approval은 모든 사용자, 모든 프로필, 모든 수준별 macro formula approval이 아니다.
- `productionFormulaChanged=true`가 보인다고 해서 BA level-aware candidate나 user-level formula가 production 적용됐다는 뜻이 아니다.

## 4. 아직 문서 설계 또는 미구현으로 봐야 하는 것

아직 production 구현으로 보면 안 되는 항목:

- user-level profile-specific macro formula production implementation
- BA candidate의 Records snapshot/signature contract
- BA candidate의 Score/Coach explanation contract
- BA candidate의 backup/import roundtrip contract
- 사용자가 Today/Records에서 수준별 산출 이유를 자연스럽게 이해하는 UX flow
- expanded human-eye profile/routine/session case set
- full Cartesian 실행 또는 owner-approved closure 대체 gate
- 최종 production approval

아직 문서 설계 성격이 강한 항목:

- 수준별 산출식 layer별 최종 수식 순서
- profile별 targetEnergy/protein/fat/carbohydrate 조정 정책의 production 승격 조건
- body-composition reliability가 낮을 때의 damping 강도
- recent14/recent28이 target change에 개입하는 최종 조건
- Score/Coach/Records에 보여줄 사용자 언어

## 6. 사용자가 바로 느낄 수 있는 UX 상태

현재 사용자 흐름에서 좋아진 것:

- Settings는 기본 운동 프로필을 갖는다.
- Today는 date-owned routine/session override를 갖는다.
- advanced OFF/ON 의미가 복구되어 있다.
- 러닝/혼합운동이 weightDuration=0이라는 이유로 generic rest처럼 보이는 위험은 줄었다.
- Today top training summary는 profile-aware로 바뀌었다.
- Records snapshot 쪽에도 운동 프로필/루틴/세션 code path evidence가 있다.

하지만 아직 완벽하지 않은 것:

- 사용자는 아직 “수준별 산출식이 왜 이 칼로리/탄단지를 냈는지” 완성된 흐름으로 설명받지 못한다.
- BA candidate의 model/stage/context가 Records/Score/Coach/Backup에 production-grade로 보존되지 않는다.
- 사용자 입장에서 profile/routine/session 전체 조합을 눈으로 검수한 broad UX signoff가 없다.
- 수준별 산출식의 최종 production 숫자 정책이 아직 승인되지 않았다.

## 8. 다음 설계 단계

다음 단계는 구현이 아니라 설계 2단계다.

설계 2단계 이름:

```text
V8 수준별 산출식 설계 2단계 - 사용자 입력 소유권/화면 흐름 확정
```

2단계에서 반드시 결정할 것:

- advanced OFF/ON에 따라 바뀌는 것

- 이 문서의 truth map을 먼저 기준으로 삼는다.
- production과 report-only를 섞지 않는다.
- UI 편의나 테스트 통과를 위해 hidden shortcut을 만들지 않는다.
