## 1. 구현 범위

추가한 suite:

```text
runMacroRangeBackupRestoreFixtureDesignTests
```

추가 위치:

```text
index.html
```

runner 연결:

```text

  test:macro-policy

tools/render_audit/run_internal_tests.cjs
  core profile
```

이 suite는 backup/restore production 로직을 바꾸지 않고, raw fixture에 future range 후보 필드를 섞은 뒤 현재 normalize / backup / restore / smart restore 경로가 기존 계약대로 prune 또는 no-leak 하는지 확인한다.

## 2. fixture 계층

테스트 fixture는 세 층으로 나눴다.

```text
old fixed fixture
  현재 저장 가능한 fixed goalSnapshot만 가진 기록

current unknown range fixture
  raw 입력에는 macroRangeContract / macroRangeBasisSignature / rangeScoreCandidate 등이 있지만
  현재 normalizeGoalSnapshot allowlist 계약에 따라 prune되어야 하는 기록

future known range fixture candidate
  아직 저장 schema가 아니며, candidate basis signature나 contract roundtrip을 통과시키지 않는다.
```

중요:

```text
현재 단계의 성공 조건은 macroRangeContract roundtrip이 아니다.
현재 단계의 성공 조건은 old fixed basis 보존과 unknown range field no-leak이다.
```

2026-07-05 latest-policy correction:

```text
old fixed basis 보존은 backup / restore 저장값 보호를 뜻한다.
충분한 detailed old fixed Record의 non-persisted latest-policy preview 계산 금지가 아니다.
```

## 3. 추가 테스트

`runMacroRangeBackupRestoreFixtureDesignTests`는 12개 케이스로 구성했다.

```text
1. full backup old fixed roundtrip preserves fixed snapshot
2. full backup current unknown range fixture prunes range fields
3. candidate basis signature does not persist before schema approval
4. records-only old fixed import preserves fixed snapshot
5. records-only current unknown range import prunes range fields
6. smart restore keep mode keeps local fixed basis
7. smart restore replace mode applies backup fixed basis and prunes unknown range field
8. smart restore meals-only merges meals but keeps local snapshot basis
9. smart restore meals-only does not introduce macroRangeContract / macroRangeBasisSignature
10. conflict preview candidate fixture does not alter current preview output
11. candidateScorePreview never persists through backup/restore paths
12. ADHERENCE_SCORING_VERSION remains unchanged
```

## 4. no-leak guard

현재 backup/restore output에 아래 값이 들어가면 실패다.

```text
macroRangeContract
macroRangeBasisSignature
rangeScoreCandidate
candidateScorePreview
scoreDeltaPreview
normalizedContract
macroRangePolicy.v0.reportOnly
range_score_candidate_report_only_v0
candidate-only-basis-signature
```

이 guard는 full backup payload, normalized backup, records-only imported record, smart restore merged/replaced record, conflict preview output, post-restore backup payload에 적용했다.

## 5. smart restore 기준

smart restore는 세 흐름을 분리해 확인했다.

```text
keep:
  same-date local conflict record는 유지한다.
  backup-only record는 추가되지만 unknown range fields는 prune된다.

replace:
  same-date conflict record는 backup fixed basis로 교체된다.
  backup raw record에 range 후보 필드가 있어도 normalized restored record에는 남지 않는다.

meals-only:
  backup meal만 local record에 병합한다.
  local goalSnapshot basis는 유지한다.
  backup snapshot basis, macroRangeContract, macroRangeBasisSignature는 들어오지 않는다.
```

특히 `meals-only`는 식사 병합 동작이지 snapshot basis 교체 동작이 아니므로, local targetCal / protein / carbs / fat이 그대로 유지되어야 한다.

## 6. conflict preview 기준

current conflict preview는 range basis를 아직 모른다.

따라서 current unknown range fixture를 넣어도:

```text
buildBackupImportComparison()
buildSmartRestoreImportPreview()
```

의 결과는 같은 fixed fixture preview와 동일해야 한다.

이번 단계에서는 `basisConflict`, range basis warning, conflict preview 문구 변경을 추가하지 않았다.

## 7. signature / score / UI 무영향

이번 suite는 다음을 바꾸지 않는다.

```text
GOAL_SNAPSHOT_SIGNATURE_KEYS
getGoalSnapshotSignature
Records 무변경 저장 판정
getDailyAdherenceScore
ADHERENCE_SCORING_VERSION
Recent aggregation
Today / Records / Recent UI
```

`candidateScorePreview`와 `scoreDeltaPreview`는 raw fixture에는 들어갈 수 있지만, normalized snapshot, backup payload, restored record, smart restore output에는 남으면 안 된다.
