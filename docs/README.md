# 탄단지 다이어리 문서 입구

이 README는 긴 기준문서가 아니라 문서 색인이다.
작업자는 여기서 읽을 순서를 확인한 뒤, 반드시 최신 기준과 실제 앱 상태를 함께 본다.

## 1. 가장 먼저 읽을 문서

1. `docs/00_현재작업기준_2026-06-16.txt`
   - 현재 최상위 기준이다.
   - 최신 대화 의도, 실제 앱 상태, 문서 우선순위, 운동 여부 상위 모드, TDEE 시간 소유권 방향을 먼저 고정한다.

2. `docs/02_대화의도_근거표_2026-06-16.txt`
   - 최신 대화에서 실제로 확정된 방향을 시간순으로 고정한 근거표다.
   - 특히 운동 여부 상위 모드와 가이드/활동량 기준 흡수 관계를 다시 흔들지 않기 위해 먼저 읽는다.

3. `docs/개발정책.txt`
   - 작업 태도와 우선순위 기준이다.
   - 문서보다 현재 앱과 사용자의 의도가 우선이라는 원칙을 확인한다.

4. `docs/내가-바라는-앱의-성격.txt`
   - 제품 철학 문서다.
   - 이 앱이 단순 계산기가 아니라 사용자가 믿고 쓰는 영양 코치가 되어야 한다는 방향을 담고 있다.

## 2. 현재 구현축 문서

아래 문서는 최신 작업과 직접 연결된다.
단, 어떤 문서도 `00_현재작업기준_2026-06-16.txt`보다 앞서지 않는다.

- `docs/v8_운동여부_코드영향감사_2026-06-15.txt`
  - 운동 여부 상위 모드를 코드에 넣을 때 봐야 하는 감사 문서다.
  - 운동함/운동안함 모드, 숨김 처리, 저장값 보존, fallback 계산 위험을 확인한다.

- `docs/v8_CC이후_TDEE_시간소유권_설계_2026-06-15.txt`
  - 수면, 업무, 운동, 생활활동이 같은 하루 24시간을 나눠 쓰는 문제를 다룬다.
  - 업무/활동량/운동을 무제한으로 더하거나 반대로 과하게 막는 둘 다 피하기 위한 참고 문서다.

- `docs/v8_운동프로필_수준별산식_통합실행설계_2026-06-04.txt`
  - 운동하는 사람 모드 안에서 운동 프로필, 세션, 수준별 산식, 수행능력 맥락을 다룰 때 참고한다.
  - 최신 기준에서는 운동 여부 상위 모드보다 아래에 있는 문서로 본다.

- `docs/앱-문구-기준.txt`
  - 문구 톤과 표현 기준 문서다.
  - 다만 오래된 정책 설명형 문구가 남아 있을 수 있으므로 그대로 베끼지 않는다.
  - Coach와 최근 흐름은 앱 정책 설명이 아니라 현재 상태와 다음 행동을 말해야 한다.

## 5. 과거 감사와 긴 기록

과거 단계 로그와 report-only 설계 원문은 현재 기준처럼 읽히는 위험이 커서 삭제했다.
필요하면 Git 이력에서 복구한다.

삭제된 원문에서 살린 교훈:

- 테스트 통과, 렌더 캡처, 후보 산식, 실제 production 적용은 서로 다르다.
- `recordMode`는 과거 기록/백업/최근 흐름 해석용 내부값이지, 신규 사용자 선택지가 아니다.
- report-only 산식은 실제 앱에 적용된 산식이 아니다.
- 오래된 단계 로그를 다음 실행 순서처럼 되살리지 않는다.

## 6. 문서 충돌 처리

문서끼리 충돌하면 아래 순서로 판단한다.

1. 최신 대화에서 확정된 의도
2. `02_대화의도_근거표_2026-06-16.txt`
3. 현재 앱의 실제 동작과 렌더
4. `00_현재작업기준_2026-06-16.txt`
5. 제품 철학과 개발정책 문서
6. 산식 근거 문서
7. Git 이력에 남아 있는 과거 감사 원문

문서가 오래된 방향을 가리키면 코드를 억지로 문서에 맞추지 않는다.
먼저 문서가 왜 틀어졌는지 기록하고, 필요한 경우 기준문서를 갱신한다.

## 6-1. 진행 번호 규칙

사용자가 말한 1~5 번호는 별도 작업 묶음이 아니다.
기존 개발 순서를 유지하면서 보고 번호만 1부터 5까지 순환하고, 번호가 5에 해당하는 시점에는 기준문서를 갱신한다.
문서 갱신 때문에 원래 하던 개발 단계를 쪼개거나 늘리지 않는다.

## 7. 현재 문서 정리 상태

문서 정리 현황은 `docs/01_문서정리_분류표_2026-06-16.txt`를 본다.

현재 원칙:

- 삭제보다 먼저 분류한다.
- 병합할 내용은 최신 기준으로 흡수한 뒤 삭제한다.
- archive는 사용자가 명시적으로 원하거나 Git 이력만으로 부족한 예외 상황에만 쓴다.
- 오래된 stage note는 현재 작업 기준처럼 읽지 않는다.
- `코덱스-인텔리전스.txt`처럼 앱 문서가 아닌 파일은 사용자 확인 후 삭제하거나 별도 보관한다.
