# 02. 대화 의도 근거표

작성일: 2026-06-16
용도: 최근 대화에서 확인된 의도를 짧게 남긴다. 문서끼리 충돌하면 이 표와 최신 대화 의도를 우선한다.

## 1. 가이드 기준 / 활동량 기준

대화에서 확인된 의도:

- 처음에는 탄단지 배분 기준을 둘로 나누려 했다.
- `가이드 기준`은 운동생리학/운동인 기준에 가까웠다.
- `활동량 기준`은 일반적인 식단/체중 관리 기준에 가까웠다.
- 현재는 이 둘을 Today의 독립 토글로 두는 것이 아니라, 더 상위 개념인 `운동 여부`에 흡수한다.

현재 판단:

- 운동함 = 기존 가이드 기준의 역할을 흡수한다.
- 운동 안 함 = 기존 활동량 기준의 역할을 흡수한다.
- Today에서 사용자가 `가이드 기준 / 활동량 기준`을 직접 고르는 구조는 제거한다.

## 2. 운동 여부 모드

대화에서 확인된 의도:

- 온보딩에서는 운동 여부만 간단히 묻는다.
- 온보딩 전에는 Settings에서 운동 여부를 바꾼다.
- 사용자를 “운동 기반 관리” 같은 이름으로 어렵게 분류하지 않는다.

현재 판단:

- Settings에 운동 여부가 있어야 한다.
- 운동함일 때는 운동 관련 입력, 운동 프로필, 운동 수준, 체성분 기준을 사용한다.
- 운동 안 함일 때는 운동 관련 UI를 숨기고 일반 기준을 사용한다.

## 3. 데이터 호환

대화에서 확인된 의도:

- 운동함과 운동 안 함은 같은 앱 안의 두 모드다.
- 기록과 백업 형식은 공유한다.
- 한 모드에서 쓰지 않는 값은 화면/계산에서 숨기거나 fallback 처리한다.
- 의미 있는 과거 데이터는 지우면 안 된다.

현재 판단:

- 운동 안 함 모드가 운동 데이터를 0으로 덮어쓰면 실패다.
- 운동함으로 바꿨을 때 없는 운동 데이터가 갑자기 생기면 실패다.
- 백업/복원은 두 모드 모두 읽을 수 있어야 한다.

## 4. 코치와 흐름 문구

대화에서 확인된 의도:

- 사용자는 내부 정책 설명이 아니라 현재 상태와 다음 행동을 보고 싶어 한다.
- “현재 관리 기준에 포함하지 않습니다” 같은 문구는 앱 내부 구조를 설명할 뿐이다.

현재 판단:

- 코치와 최근 흐름 문구는 정책 설명보다 상태/근거/행동을 우선한다.
- 운동 안 함 모드에서도 “운동 기록은 제외됩니다” 같은 내부 문구를 반복하지 않는다.
- 문구를 바꿀 때는 `앱-문구-기준.txt`를 참고하되, 현재 화면에서 자연스러운지를 우선한다.

## 5. TDEE 시간 소유권

대화에서 확인된 의도:

- 업무, 생활활동, 운동이 실제 소비량을 올렸다면 목표 칼로리에 반영되어야 한다.
- 문제는 반영 자체가 아니라 같은 시간을 중복 계산하는 것이다.
- `v8.0-CC`의 보수적 처리 또는 target 제외는 임시 보호막이었다.

현재 판단:

- 2026-06-21 14단계 재검증 기준으로 TDEE 시간 소유권 산식은 production `calculate()`에 적용되어 있다.
- 수면, 업무, 운동, 생활활동, 휴식 시간을 나누어 계산한다.
- 업무/생활활동을 작은 보정으로 격하하지 않는다.
- 업무/생활활동/운동 소모가 실제로 커지면 목표 칼로리도 움직인다.
- 웨이트, 유산소, EPOC는 운동 순소모 하나로 접어 중복 계산하지 않는다.
- 22단계 이후 생활활동은 사용자가 직접 시간을 입력한다. 남은 시간은 자동 휴식시간으로 처리하고 별도 활동 칼로리를 올리지 않는다.
- 수면+업무+운동+생활활동 요청 시간이 24시간을 넘으면 생활활동 반영 시간은 남은 시간까지만 제한하고, 화면에서는 24시간 초과를 안내한다.
- 14단계 값/화면 재검증에서 관련 시나리오 테스트 87/87이 통과했고, Today 데스크톱/모바일 화면에서도 시간 요약과 목표 칼로리 요약이 깨지지 않았다.
- 16단계에서 Records 상세와 저장 당시 계산 기준도 production 시간 소유권을 사용하도록 정리했다.
- 17단계에서 Today 기준 갱신은 식사, 공복체중, 메모를 보존하고 저장 계산 기준과 자동 점수만 갱신하는 흐름으로 확인했다.
- 18단계에서 저장된 `goalSnapshot`이 식단 점수와 Recent Flow의 기준으로 유지되는지 검증했다. Today live 값이 바뀌어도 과거 기록 해석을 바꾸면 실패다.
- 19단계에서 Today 기준 갱신 안내와 Records 당시 계산 기준을 데스크톱/모바일 실제 화면으로 확인했고, 배치 문제 없이 닫았다.
- 22~34단계에서 생활활동 시간 입력, 자동 휴식시간, 요일별 기본 수면/업무/생활활동 시간, 업무 유형/생활활동/수면·업무·생활활동 시간 `?` 도움말, Today `시간 배분`/`자동 휴식` 요약을 Today/Settings/Records 흐름에 반영하고 검증했다.
- 31단계 전체 검증에서는 core/ui/calibration/mobile 기준 총 828개 케이스가 통과했다.
- 32단계 외부근거 감사에서는 현재 MET 기반 시간 소유권 구조를 즉시 바꿀 필요는 없다고 봤다. 수면 kcal 보정, 업무/생활활동 계수 재조정은 다음 산식 백로그다.
- 33단계에서 업무 유형 설명은 Today와 Settings 모두 `?` 도움말로 정리했고, 숨겨진 도움말이 화면 가로폭을 밀지 않도록 고쳤다.
- 34단계 감사 결과, 지금 사용자에게 다시 물어야 하는 결정은 없다. 고급 예외 직접입력 UI와 수행능력 kcal 직접 반영은 현재 흐름을 막지 않는 백로그다.
- 36~39단계에서 최근 흐름과 Records 상세가 저장된 `goalSnapshot` 시간 기준을 계속 쓰는지 확인했다. Today 현재값으로 과거 기록을 다시 해석하면 실패다.
- 38단계 전체 검증에서는 core/ui/calibration/mobile 기준 총 828개 케이스가 다시 통과했고, Today/Settings 화면의 시간 도움말과 요일별 기본 시간 표시도 깨지지 않았다.
- 39단계 감사 결과도 같다. 지금 사용자에게 다시 물어야 하는 결정은 없고, 수면 kcal 보정/업무·생활활동 계수/고급 예외 UI/수행능력 kcal 직접 반영은 다음 산식·화면 백로그다.
- 다음 작업은 산식 연결 여부 확인이 아니라, 남은 계수와 직접입력 필요성을 생리학 근거와 실제 화면 기준으로 다시 보는 것이다.

## 6. 생활활동

대화에서 확인된 의도:

- 생활활동과 추가활동은 같은 남은 시간을 나눠 쓰므로 기본적으로 분리하지 않는다.
- 생활활동은 업무, 운동, 수면과 겹치지 않는 별도 생활활동 시간의 평균 강도다.
- 높은 단계는 “잠깐 힘든 일”이 아니라 입력한 생활활동 시간 대부분이 실제로 높은 강도일 때만 선택해야 한다.

현재 판단:

- 별도 추가활동 카드는 기본 UI로 만들지 않는다.
- 생활활동 시간은 직접 입력한다. 남은 시간은 자동 휴식시간이다.
- 생활활동 카드에 `?` 도움말을 둔다.
- 도움말은 입력 시간 기준, 중복 입력 방지, 과대 선택 위험을 설명한다.
- 업무 유형 가이드도 Today 빠른 설정에서는 항상 보이는 본문이 아니라 `?` 도움말로 둔다.
- 요일별 기본 시간은 Settings에서 관리하고, Today는 해당 요일 기본값을 가져온다.
- Today 계산 요약에는 `시간 배분`과 `자동 휴식`으로 현재 적용된 시간을 보여준다. 이 표시는 새 선택지가 아니라 계산 결과 설명이다.
- 숨겨진 `?` 도움말은 레이아웃 가로폭을 밀면 안 된다. 숨김 상태에서는 화면 공간을 차지하지 않게 처리한다.
- 고급 예외 직접입력 UI는 지금 기본 흐름에 추가하지 않는다. 사용자가 다시 필요성을 열 때만 별도 결정 질문으로 다룬다.

## 7. 문서 정리

대화에서 확인된 의도:

- 오래된 문서를 아카이브로 쌓아두면 다시 읽지 않고 혼선을 만든다.
- 현재 판단에 쓰이지 않는 문서는 삭제해도 된다.
- 실행 조건이 없는 선언문보다 실제 체크 규칙이 필요하다.

현재 판단:

- 현재 기준 문서는 짧고 실행 가능한 규칙 중심으로 유지한다.
- 오래된 의사결정은 Git 이력에 맡긴다.
- 문서가 앱 의도와 충돌하면 문서를 고친다.

## 8. 매크로 정책 분리

대화에서 확인된 의도:

- 운동 여부는 단순 UI 숨김이 아니라 탄단지 정책의 상위 기준이다.
- 운동함은 기존 가이드 기준의 스포츠/운동생리학 기준을 쓴다.
- 운동 안 함은 기존 활동량 기준의 일반 식단/체중 관리 기준을 쓴다.
- 운동 안 함에서 운동 소모를 0으로 빼더라도, 단백질 정책이 운동함과 같으면 의도와 맞지 않는다.

현재 판단:

- 운동함/운동 안 함은 같은 기록/백업 형식을 유지하되, production 매크로 정책표는 분리한다.
- 운동 안 함에서 `diet 1.8g/kg`, `cut 2.2g/kg`, `maintain 1.8g/kg` 같은 운동인 단백질값이 나오면 실패다.
- 운동 안 함은 훈련 탄수 g/kg 하한을 적용하지 않는다. 탄수는 남은 칼로리로 배분하고 일반 성인 AMDR 비율을 sanity check로 본다.
- 다음 구현은 `exerciseManagementMode`가 매크로 정책표 선택을 소유하게 만드는 것이다.
