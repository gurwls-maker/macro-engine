v8 운동 여부 상위 모드 코드 영향 감사
작성일: 2026-06-15

2026-06-16 정정
----------------
이 문서는 운동 여부 상위 모드가 코드에 들어가기 전 작성된 감사 문서다.
따라서 아래 본문에는 당시 코드 상태와 당시 미결정 사항이 섞여 있다.

최신 기준은 `02_대화의도_근거표_2026-06-16.txt`와 `00_현재작업기준_2026-06-16.txt`가 우선한다.

최신 확정:
- 사용자가 Today에서 "가이드 기준 / 활동량 기준"을 직접 고르는 구조는 유지하지 않는다.
- 사용자가 고르는 최상위 기준은 "운동을 관리 기준에 포함할지"다.
- 운동함 모드는 기존 가이드 기준의 운동생리학 성격을 흡수한다.
- 운동 안 함 모드는 기존 활동량 기준의 일반 식단 / 체중 관리 성격을 흡수한다.
- 온보딩 전에는 Settings에서 운동 여부를 바꾼다.
- 숨긴 운동값은 저장소에서 지우지 않고 계산 / 렌더 / 해석 단계에서만 fallback 처리한다.

이 문서에서 두 기준을 사용자 선택 토글로 병렬 유지하거나 아직 방향 결정이 필요하다고 적힌 부분은 최신 기준으로 폐기된 중간 판단이다.

목적
----
이 문서는 "운동을 관리 기준에 포함할지"를 가이드 기준 / 활동량 기준보다 상위에 두는 설계를 실제 코드에 넣기 전에, 어떤 코드 경로가 영향을 받는지 감사한 문서다.
이번 단계는 감사 단계이며 앱 동작을 바꾸지 않았다.

가장 중요한 결론
----------------
운동 여부 모드는 단순히 운동 카드 몇 개를 숨기는 기능이 아니다.
현재 앱에서는 운동 프로필, 루틴, 세션, 주간 웨이트일수, 웨이트 시간, 훈련 강도, 유산소 입력이 목표 칼로리, 탄단지 배분, 오늘 소비 기준, Records 스냅샷, 기준갱신 비교, 최근 흐름, Coach, 테스트까지 이어진다.

따라서 운동 안 함 모드는 아래 두 가지를 동시에 만족해야 한다.

1. 계산과 화면 해석에서는 운동값을 0 / REST / 없음 / fallback으로 본다.
2. 사용자가 저장해 둔 운동 설정값은 지우지 않는다.

둘 중 하나만 하면 버그가 된다.
카드만 숨기면 보이지 않는 운동값이 계산에 계속 반영된다.
반대로 저장값을 0으로 덮어쓰면 사용자가 나중에 운동함 모드로 돌아왔을 때 기존 운동 설정이 사라진다.

현재 코드 상태
--------------
1. 운동 여부 값은 최신 코드에는 들어와 있다.

이 문서 작성 당시에는 `exerciseManagementMode`가 없었다.
하지만 2026-06-16 기준 최신 코드에는 `exerciseManagementMode`가 들어와 있으며, Settings의 "운동 기준 사용"과 연결되어 있다.
따라서 이 항목은 역사적 감사 기록으로만 본다.

여전히 유효한 원칙:
- 기존 `state.mode = "general"`은 사용자-facing 운동 여부 모드로 재사용하지 않는다.
- `generalAdvancedSettings`도 운동 여부가 아니라 운동 설정 확장 / 단순 세션 노출과 관련된 값으로 본다.

2. 운동 입력은 Settings와 Today 계산값에 모두 연결되어 있다.

대표 입력:
- `weeklyTrainingDays`
- `exerciseProfile`
- `profileSession`
- `routinePlan`
- `routine`
- `intensityOverride`
- `weightDuration`
- `cardioType`
- `cardioDuration`
- `cardioSpeed`
- `cardioIncline`

이 값들은 저장소, Today draft, Settings 기본값, Today 계산값에 걸쳐 사용된다.

3. 계산식은 운동값을 직접 사용한다.

`collectState()`는 Today 유효값을 모아 `calculate()`로 넘긴다.
`calculate()`는 다음 항목을 목표 칼로리와 오늘 소비 기준에 반영한다.

- 주간 웨이트일수 기반 baseline
- 오늘 루틴 / 세션
- 웨이트 시간
- 훈련 강도
- 유산소 종류 / 시간 / 속도 / 경사
- 유산소 EPOC

즉 운동 안 함 모드를 UI만 숨기면 숨은 운동값이 그대로 계산에 들어간다.

4. Records 스냅샷과 기준갱신 비교에도 운동값이 들어간다.

`buildGoalSnapshotFromCurrentState()`는 현재 계산 기준을 Records에 저장할 때 운동 관련 값을 함께 저장한다.
`GOAL_SNAPSHOT_SIGNATURE_KEYS`도 운동 관련 값을 기준갱신 비교에 포함한다.

따라서 운동 안 함 모드에서는 "실제 저장값"과 "계산에 사용한 유효값"을 분리해야 한다.
예를 들어 저장소에는 예전 PUSH 60분 값이 남아 있어도, 운동 안 함 모드에서 저장한 오늘 기록의 스냅샷은 REST / 0분 기준이어야 한다.

5. Coach / 최근 흐름은 운동 신호를 해석 재료로 쓴다.

코드에는 운동 부족, 운동 여부, 운동 전후 탄수화물, 주간 웨이트일수, 실제 웨이트 기록일수 같은 신호가 남아 있다.
운동 안 함 모드에서는 이런 신호가 사용자에게 "운동을 하라"는 식으로 새면 안 된다.
다만 식사, 체중, 체성분 자체는 계속 볼 수 있다.

구현 원칙
---------
1. 새 상위 모드 값을 추가한다.

권장 이름:
- 저장값: `exerciseManagementMode` 또는 `includesExerciseInPlan`
- 값 예시: `exercise` / `general`

기존 `state.mode`를 재사용하지 않는다.
기존 `generalAdvancedSettings`를 재사용하지 않는다.

2. 저장값과 계산 유효값을 분리한다.

운동함 모드:
- 저장된 운동 프로필 / 루틴 / 세션 / 웨이트 / 유산소 값을 그대로 계산과 화면에 사용한다.

운동 안 함 모드:
- 저장값은 그대로 둔다.
- 계산과 화면에는 아래 유효값을 넘긴다.
  - `weeklyTrainingDays = 0`
  - `routine = "REST"`
  - `profileSession = "rest"`
  - `weightDuration = 0`
  - `intensityOverride = 0`
  - `cardioDuration = 0`
  - `cardioSpeed = 0`
  - `cardioIncline = 0`
  - `cardioType`은 저장 선호값으로만 보존하고 계산에는 영향 없게 처리한다.

이 처리는 계산 직전 또는 Today 유효값을 만드는 단일 함수에서 해야 한다.
여러 화면에서 각각 0으로 바꾸는 방식은 나중에 값이 새거나 저장값을 덮어쓸 위험이 크다.

3. Records 스냅샷에 운동 여부를 남긴다.

운동 안 함 모드에서 저장한 기록은 나중에 운동함 모드로 돌아와도 당시에는 운동을 계산 기준에 넣지 않았다는 사실을 알아야 한다.
따라서 goalSnapshot에는 운동 여부 모드가 함께 저장되어야 한다.

예상 필드:
- `exerciseManagementMode`

스냅샷 비교도 이 모드-aware 유효값 기준으로 비교해야 한다.
숨겨진 원본 Settings 운동값이 바뀌었다는 이유로 운동 안 함 기록에 갱신 필요가 뜨면 안 된다.

4. Backup / Import schema는 둘로 나누지 않는다.

운동함과 운동 안 함은 같은 내보내기 / 불러오기 형식을 쓴다.
옛 백업에는 운동 여부 필드가 없으므로 기본값은 운동함으로 본다.
운동 안 함 모드에서 읽지 않는 운동 필드는 삭제하지 않는다.

5. Coach / 최근 흐름 문구는 정책 설명을 하지 않는다.

금지 성격:
- 운동 기록은 현재 관리 기준에 포함하지 않습니다.
- 일반 권장 범위 안에서 관리합니다.
- 체성분 변화와 식단 기록을 함께 봅니다.
- 운동 관련 값은 계산에서 제외됩니다.

허용 방향:
- 오늘은 식사 기록부터 채우세요.
- 최근 섭취 변동이 큽니다.
- 단백질이 며칠째 낮습니다.
- 식사 기록이 부족해 원인 판단은 아직 이릅니다.

사용자는 앱 내부 정책을 읽고 싶은 것이 아니라, 지금 무엇을 보면 되는지 알고 싶다.

탭별 영향
---------
Today:
- Settings에서 운동 안 함으로 바꾸면 Today의 운동 프로필 / 루틴 / 세션 / 웨이트 / 훈련 강도 / 유산소 입력은 기본 화면에서 숨긴다.
- 오늘 목표 섭취량, 식사 기록, 식단 점수는 계속 보여준다.
- 계산에는 운동 유효값 0 / REST가 들어가야 한다.
- 기존 숨은 운동 설정값은 삭제하지 않는다.

Records:
- 스냅샷에는 당시 운동 여부 모드와 모드-aware 계산값을 저장한다.
- 운동 안 함 기록은 나중에 운동함 모드에서 봐도 갑자기 운동 기록처럼 해석되면 안 된다.
- 기준갱신 비교는 당시 모드 기준으로 해야 한다.

Recent Flow:
- 운동 안 함 모드에서는 운동 부족, 웨이트 기록 부족, 운동일수 같은 판단을 숨기거나 제외한다.
- 식사 실행률, 체중 흐름, 식사 기록 충분성, 체성분 변화는 계속 사용할 수 있다.
- "운동을 제외했다"는 정책 설명 대신, 현재 볼 수 있는 데이터 중심으로 말한다.

InBody:
- 체성분 기록은 운동 안 함 모드에서도 보존할 수 있다.
- 다만 골격근 변화나 운동성과처럼 읽히는 해석은 조심해야 한다.
- 체중 / 체지방 / 건강 기록 중심으로 보여줄 수 있다.

Settings:
- 온보딩 전에는 Settings에서만 운동 여부를 바꾼다.
- 운동 안 함 선택 시 운동 관련 설정 영역은 접거나 숨기되, "값을 삭제하지 않는다"는 점은 도움말 / ? / 안내로 설명한다.

테스트 / 검증 필요 항목
----------------------
구현 단계에서는 아래를 테스트해야 한다.

1. 기본값
- 옛 사용자 / 옛 백업은 운동함 모드로 열린다.

2. 운동 안 함 전환
- Today 운동 카드가 숨겨진다.
- 목표 칼로리와 탄단지 계산에서 웨이트 / 유산소 / 주간 웨이트일수가 빠진다.
- 저장된 운동 설정값은 localStorage / backup에서 사라지지 않는다.

3. 운동함 복귀
- 숨겨져 있던 운동 프로필 / 루틴 / 세션 / 유산소 값이 복원된다.
- 운동 안 함 기간에 저장한 과거 기록이 갑자기 운동 기록처럼 바뀌지 않는다.

4. Records 기준갱신
- 운동 안 함으로 저장한 기록은 숨은 운동 설정값이 달라져도 갱신 필요가 뜨지 않는다.
- 식사 / 체중 / 목표 / 실제 계산 기준이 달라진 경우에는 갱신 필요가 뜬다.

5. Coach / 최근 흐름
- 운동 안 함 모드에서 운동 부족, 운동 강도, 운동 전후 탄수화물 같은 문구가 새지 않는다.
- 정책 설명 문구가 Coach / 최근 흐름 본문에 나오지 않는다.

6. Backup / Import
- 운동함에서 내보내고 운동 안 함에서 불러와도 값이 보존된다.
- 운동 안 함에서 내보내고 운동함에서 불러와도 숨겨진 운동 설정값이 보존된다.

2026-06-16 최신 결정 반영
--------------------------
아래 내용은 이 문서 작성 당시에는 미결정이었지만, 이후 대화에서 방향이 정리되었다.

1. 운동 안 함 모드에서 기존 기준 토글을 어떻게 제거하고 어떤 일반 관리 화면을 보여줄 것인가?

최신 결정:
- Today에서 "가이드 기준 / 활동량 기준"을 직접 고르는 mini toggle은 사용자-facing으로 유지하지 않는다.
- 운동함 모드가 기존 가이드 기준의 운동생리학 성격을 흡수한다.
- 운동 안 함 모드가 기존 활동량 기준의 일반 식단 / 체중 관리 성격을 흡수한다.
- 따라서 "운동 여부 토글 + 가이드/활동량 토글"처럼 2층 선택지를 사용자에게 노출하지 않는다.

구현 의미:
- 계산 내부에서는 기존 dual basis 계층을 재사용할 수 있다.
- 하지만 사용자가 기준 버튼을 눌러 목표 탄단지를 바꾸는 구조는 제거한다.
- Records 스냅샷과 백업에는 당시 운동 여부와 계산에 사용된 유효값이 남아야 한다.

2. InBody 해석 범위

최신 결정:
- InBody 기록 자체는 운동 안 함 모드에서도 보존한다.
- 다만 운동 안 함 모드에서는 골격근 증가, 훈련 반응, 운동성과처럼 읽히는 문구를 조심한다.
- 구체적인 InBody 화면 재구성은 후속 UI/해석 단계에서 별도로 다룬다.
