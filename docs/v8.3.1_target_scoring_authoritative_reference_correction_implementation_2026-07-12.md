# v8.3.1 target/scoring authoritative-reference correction implementation

## 비개발자용 설명

### 무엇을 바꿨는지

앱이 사용자에게 계산해 준 정상 목표를 사용자가 그대로 지켰는데도 식단 점수가 깎이던 문제를 바로잡았다.

- 단백질 점수는 사용자가 선택한 `낮게 / 중간 / 높게` 단계와 운동 맥락을 반영해 앱이 실제로 만든 단백질 목표 범위를 따른다.
- 칼로리 초과 판단은 오늘 목표 칼로리와 총소비량 중 더 높은 값을 기준으로 본다. 증량 목표처럼 의도적으로 총소비량보다 높게 잡힌 정상 목표를 다시 초과로 벌하지 않는다.
- 목표가 정상적으로 만들어졌다는 근거가 없거나 숫자·계산 기준이 서로 맞지 않으면 100점으로 덮지 않고 점수 계산을 막는다.
- 점수 산식 버전을 새 버전으로 올려, 이전 산식과 이번 교정을 구별할 수 있게 했다.
- 과거에 저장된 자동 점수는 화면을 열었다는 이유만으로 새 산식으로 바꾸지 않는다. 사용자가 식사를 수정하거나 명시적으로 다시 계산해 저장할 때만 새 버전으로 갱신한다.

### 왜 바꿨는지

이전 점수기는 목표 생성기가 정한 값을 그대로 믿지 않고 두 기준을 다시 덮어썼다.

첫째, 75kg 사용자가 `유지 / 휴식 / 단백질 낮게`를 선택하면 앱은 단백질 75g을 정상 목표로 만들었다. 하지만 점수기는 별도로 105g 하한을 적용해, 75g을 정확히 먹어도 단백질 부족으로 17.694점을 깎았다. 결과는 82.306점이었다.

둘째, `벌크업 / 휴식`에서는 총소비량 1,928.75kcal보다 높은 2,300kcal를 의도된 목표로 만들었다. 사용자가 2,300kcal를 지켜도 점수기가 총소비량 초과로 7.942점을 다시 깎았다. 낮은 단백질 단계의 중복 감점까지 합치면 86.098점이었다.

이 문제는 항목별 점수 UI나 코치 문구가 없어도 존재하는 현재 계산 오류였다. 그래서 component-score 설계나 DailyCoach 작업에 묶어 기다리지 않고 독립적인 release blocker로 먼저 닫았다.

### 실제 사용자 화면이나 계산 결과가 어떻게 달라지는지

화면 배치나 문구는 바뀌지 않는다. 달라지는 것은 정상 목표를 지킨 날의 점수다.

실제 production 계산의 다음 54개 조합을 모두 확인했다.

```text
목표 6개 × 운동 맥락 3개 × 단백질 단계 3개 = 54개
```

변경 전에는 54개 중 33개만 100점이고 21개는 100점이 아니었다. 변경 후에는 앱이 만든 탄수화물·단백질·지방 g을 그대로 기록했을 때 54개 모두 정확히 100점이며, 9개 감점 항목도 모두 0이다.

반대로 목표보다 실제로 더 먹으면 기존처럼 거리에 따라 점수가 내려간다. 총섭취량이 물리적 총소비량의 3배를 넘는 극단값을 확인하는 데이터 이상 판정도 유지했다. 즉 정상 목표만 이중 처벌하지 않으며, 실제 초과나 이상값을 무조건 100점으로 만드는 변경은 아니다.

### 정책, 산식, 데이터 해석이 바뀌었는지

점수 정책과 저장 점수 해석이 좁게 바뀌었다.

- active scoring version: `v8.3.1_target_authority_continuous_macro_score_v2`
- formula identity: `anchor_continuous_macro_score_v2_target_authority`
- 단백질 부족·초과: 앱이 만든 target-aligned protein range 기준
- 에너지 overload: `오늘 목표 kcal`와 `총소비 kcal` 중 더 높은 기준을 넘은 정도
- 물리적 TDEE ratio: 데이터 이상 판정용으로 계속 별도 유지
- 과거 저장 자동점수: 버전이 v6.1, 교정 전 v1, 미기록(null)이더라도 저장된 숫자와 버전을 우선 표시
- 명시적 재평가 후 저장: 그때만 current v2 점수와 버전으로 갱신

9개 감점 축, 각 곡선의 anchor와 기울기, 마지막 0~100 표시 clamp는 바꾸지 않았다. 운동을 점수 보너스로 추가하지 않았고, 술의 kcal 축과 생리적 부담 축도 그대로 분리했다.

### 계산 경계

1. target authority
   - target kcal은 양수이고 protein은 양수, carbs/fat은 0 이상이어야 한다.
   - target macro kcal 합은 target kcal와 8kcal 이내에서 닫혀야 한다.
   - live target은 current calculation 네 macro와 일치해야 한다.
   - adaptive target은 current base target 소유권과 adaptive target payload가 모두 일치해야 한다.
   - frozen target은 normalized full goal snapshot과 일치해야 한다.
   - rate contract는 delta, min/max guard, targetCal, minimumTargetCal이 모두 유한하고 minimumTargetCal이 0 이상이어야 한다.
   - protein target은 self-derived selected 값이 아니라 independent requestedGkg 및 constraint metadata로 검증한다.
   - external macro application은 applied flag뿐 아니라 final targetCal/protein/carbs/fat 일치까지 검증한다.

2. energy reference
   - `energyOverloadReferenceKcal = max(valid targetKcal, totalBurn)`.
   - target deviation penalty는 계속 target 대비 거리를 본다.
   - dataOutlierPenalty는 계속 physical `scoringKcal / totalBurn`을 본다.

3. protein reference
   - shortage/excess penalty는 `scoringContext.proteinRange.min/max`를 사용한다.
   - generated selected target은 이 range 안에 포함된다.

4. Records and snapshot
   - passive stored auto lookup에는 current-version equality gate가 없다.
   - explicit regrade/persistence 경계에서만 current v2 계산과 저장 갱신을 허용한다.
   - stored formula label은 current v2, known pre-correction v1, legacy/unknown을 구분한다. legacy/unknown에는 current formula를 거짓으로 표시하지 않는다.
   - frozen snapshot이 current burn을 빌리는 조건은 일부 필드 비교가 아니라 full goal snapshot signature equivalence다.
   - targetAuthority, scoringContext, penaltyBreakdown은 storage에 추가하지 않았다.
