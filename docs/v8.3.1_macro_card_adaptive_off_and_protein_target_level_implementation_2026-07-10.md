# v8.3.1 macro card adaptive OFF and protein target level implementation

Date: 2026-07-10

## Plain explanation

자동 조정 ON이면 탄수화물과 지방은 오늘 범위와 기본 권장값을 같이 보여주고, OFF이면 예전처럼 고정 권장값까지 얼마나 남았는지만 보여준다. 단백질은 탄수/지방처럼 실시간 교환되는 영양소가 아니므로 범위 chip 대신 사용자가 고른 단백질 목표 수준을 보여준다. 단백질 선택은 낮게·중간·높게 세 단계이며, 처음값은 중간이다.

## Implemented

- Added persistent setting `proteinTargetLevel`.
- `proteinTargetLevel` is a persistent setting surface, not a temporary UI-only choice.
- Added Settings select options:
  - `low`
  - `medium`
  - `high`
- `medium` is the only default/automatic UI choice.
- Legacy stored `default` values normalize to `medium`, so existing settings and backups remain compatible without exposing a duplicate choice.
- `low` and `high` select the lower/upper bound of the existing mode/context policy range.
- Exercise/general mode and eligible lean/advanced context are resolved before the level is selected.
- Generic `high` does not expose the 2.3-3.1g/kg FFM contest-prep exception.
- Protein level selection now happens inside `buildExternalMacroProductionCandidate`; the former post-policy final macro overwrite was removed.
- Final protein/carbs/fat and `externalMacroActiveProductionApplication` metadata now describe the same values.
- If a high protein request would displace training carbohydrate below its guard, high is constrained back toward the automatic target and the target-relief pressure remains visible.
- Fixed a pre-existing context bug where the broad `/A/i` check classified the word `general` as an advanced profile.
- Protein target level is saved, loaded, included in full backup, restored from full backup, and preserved by records-only import.
- Today macro card display now separates adaptive ON and OFF:
  - adaptive ON: carbs/fat keep compact conditional range chips and show the base target in the bottom status line.
  - adaptive OFF: carbs/fat range chips are removed and the card returns to fixed-target remaining/over wording.
- Protein card no longer shows a range chip.
- Protein card header shows the selected protein target level instead.
- Settings uses a compact 58px selector, matching the ON/OFF control width and height while preserving the native select arrow.
- `proteinTargetLevelHelpTip` explains that medium is the automatic recommendation and that low/high remain inside the current evidence-based policy range without displacing carb/fat minimums.

## Product boundary

Accepted:
- Carb/fat are still handled by the v8.3.1 protein-reserved iso-calorie joint allocation model.
- Protein remains an anchor, not a carb/fat exchange participant.
- Adaptive OFF means fixed Today target display.
- Adaptive ON keeps current joint conditional range display.
- Showing the base target is useful because it lets the user see how far the live target moved.

Rejected:
- Explaining internal carb/fat calorie-exchange math on the card.
- Keeping protein range chips just because older v8.3 card range work showed them.
- Making protein level a new score penalty bucket.
- Keeping `default` and `medium` as two UI choices even though they mean the same automatic recommendation.

## External evidence boundary

- ISSN's protein-and-exercise position stand describes 1.4-2.0g/kg body weight/day as sufficient for most exercising people: https://pubmed.ncbi.nlm.nih.gov/28642676/
- The 2.3-3.1 range comes from energy-restricted, resistance-trained, lean athletes and is expressed per kg of FFM: https://pubmed.ncbi.nlm.nih.gov/24092765/
- Therefore the ordinary `high` setting stays inside the app's existing body-weight policy range. The FFM exception is not generalized into this setting.
