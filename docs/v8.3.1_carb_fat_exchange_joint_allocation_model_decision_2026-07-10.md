# v8.3.1 carb-fat exchange joint allocation model decision

작성일: 2026-07-10

문서 성격: docs-only model decision/design

관련 직전 문서:
- docs/v8.3.1_carb_fat_exchange_range_consistency_audit_design_2026-07-10.md

## 결론

다음 구현은 카드 도움말이나 문구로 사용자를 설득하는 작업이 아니다.

채택 모델은 다음이다.

```text
protein-reserved iso-calorie carb/fat joint allocation
with conditional feasible display ranges
```

뜻은 단순하다.

```text
1. protein은 먼저 예약한다.
2. 남은 target kcal 안에서 carbs와 fat을 같이 배분한다.
3. carbs가 올라가면 fat 여유는 kcal 기준으로 내려간다.
4. fat이 올라가면 carbs 여유는 kcal 기준으로 내려간다.
5. 화면은 이 계산 결과 숫자만 자연스럽게 보여준다.
6. 사용자에게 "이건 1:1 교환이 아니고..." 같은 내부 계산 설명으로 해결하지 않는다.
7. No explanation-only shortcut.
```

현재 앱은 `getV83AdaptiveTodayMacroTarget`에서 이미 calorie-equivalent exchange 방향을 일부 구현했다.
하지만 card range chip은 여전히 독립 marginal range처럼 보일 수 있다.
따라서 다음 구현은 target / scoring / card display가 같은 joint allocation helper를 보게 해야 한다.

## PROMPT_SCOPE_AUDIT

이번 요청을 그대로 좁게 따르면 "GPT가 다음 문서라고 했으니 문서 추가"가 된다.
그건 부족하다.

이번에 실제로 다시 판단한 문제는 이것이다.

```text
문제:
carb range와 fat range를 각각 독립 범위처럼 보여주면,
carb upper와 fat lower/upper가 모두 동시에 가능한 조합처럼 보일 수 있다.

실제:
protein과 target kcal가 고정되면 carbs와 fat은 같은 kcal 선 위에서 함께 움직인다.
carb 1g은 fat 약 0.44g,
fat 1g은 carb 약 2.25g에 해당한다.

따라서:
copy/help/tooltip이 아니라 joint allocation model이 먼저 닫혀야 한다.
```

이 문서는 구현을 미루기 위한 절차 문서가 아니다.
다음 작업은 별도 test-design 문서가 아니라 model implementation with tests가 맞다.

## 현재 코드 기준 확인

정적 확인:

```text
calculateMacroTargets:
- protein을 먼저 계산한다.
- fat target을 계산한다.
- carbs는 target kcal에서 protein/fat kcal를 뺀 residual로 계산한다.

getV83TargetAlignedMacroRanges:
- proteinRange / carbRange / fatRange를 만든다.
- training load와 fat range source를 반영한다.
- 하지만 carb/fat display range를 완전한 joint feasible set으로 만들지는 않는다.

getV83AdaptiveTodayMacroTarget:
- consumed carbs/fat progress 차이를 보고 target carbs/fat을 움직인다.
- protein은 clamp 후 사실상 reserve로 둔다.
- fat이 range 밖이면 fat을 range 안으로 돌리고 carbs를 다시 계산한다.
- macro kcal gap을 작게 유지하려고 한다.

getTodayMacroCardRange:
- card chip은 key별 range를 반환한다.
- 이 range는 현재 joint pair의 모든 조합 가능성을 보장하지 않는다.
```

즉 방향은 맞지만 완성은 아니다.

## 왜 설명문으로 풀면 안 되는가

사용자가 앱에서 봐야 하는 것은 계산 설명이 아니라 계산 결과다.

폐기:

```text
- 카드에 "탄수/지방은 1:1 교환이 아닙니다" 같은 문구를 붙여 해결
- 도움말에 internal calorie exchange arithmetic을 설명해서 책임 전가
- 사용자에게 내부 calorie exchange arithmetic을 설명하지 말고 앱이 계산을 맞춘다.
- 현재 independent range를 유지하면서 copy만 보강
- UI-only display cap으로 어색한 숫자만 숨김
```

허용:

```text
- card header 옆 compact range 숫자
- main target number가 adaptive pair 결과로 움직이는 것
- DailyCoach가 이미 있는 과탄수/과지방/저탄수/저지방 맥락을 자연스럽게 말하는 것
- 전문 tooltip은 별도 copy/spec gate에서만, 이 모델의 핵심 해결책으로 쓰지 않음
```

## 모델 선택지

### Option A. Independent marginal ranges 유지

판정: reject

이 방식은 carbs와 fat을 각각 독립 범위로 둔다.
계산은 쉬우나 사용자가 carb upper와 fat upper/lower를 동시에 가능한 조합으로 읽을 수 있다.
이번 문제를 만든 구조와 같다.

### Option B. Moving target pair만 유지하고 range는 기존대로 둠

판정: incomplete

adaptive target pair가 움직이는 것은 맞다.
하지만 range chip이 계속 독립 marginal range라면, target과 range의 의미가 분리된다.
사용자에게 계산 설명을 하지 않더라도 화면 숫자 자체가 덜 맞는다.

### Option C. Protein-reserved iso-calorie pair + conditional feasible display range

판정: accept

이 모델을 채택한다.

핵심:

```text
proteinReservedG = clamp(baseProteinTarget, proteinRange)
availableCarbFatKcal = targetKcal - proteinReservedG * 4

base/adaptive target pair:
4 * carbsG + 9 * fatG = availableCarbFatKcal

display range:
carb/fat 각각의 raw physiological range를 보되,
상대 macro와 같은 kcal 선에서 동시에 가능한 조건으로 다시 제한한다.
```

display range는 "독립 사각형"이 아니라 "현재 상태에서 가능한 조건부 구간"이어야 한다.

예:

```text
target kcal: 2549
protein: 127.4g
available carb/fat kcal: 2039.4

carb 448g과 fat 52g을 동시에 두면:
448*4 + 52*9 + 127.4*4 = 2769.6kcal
target보다 약 220.6kcal 높다.

따라서 "carb 292~448g"과 "fat 52~94g"을 독립적으로 보여주면
모든 조합이 가능한 것처럼 읽힐 위험이 있다.
```

### Option D. Full coupled visual redesign

판정: later / not now

2D carb/fat tradeoff chart나 slider 같은 시각화는 이론상 더 정확하다.
하지만 지금 앱의 카드 구조와 사용성에 비해 과하다.
이번 구현은 shared helper와 compact numbers 정합성을 먼저 맞춘다.

### Option E. Copy/help/cue solution

판정: reject

이건 산식/모델 문제를 사용자 설명으로 넘기는 방식이다.
이번 문제의 해법이 아니다.

## 채택 모델 상세

### 1. Protein reserve

protein은 carb/fat 교환의 실시간 조정 대상이 아니다.

```text
proteinReservedG =
  clamp(targetProteinG, proteinRange.min, proteinRange.max)
```

protein이 부족하거나 과한 것은 proteinShortagePenalty / proteinExcessEfficiencyPenalty가 따로 본다.
carb/fat exchange가 protein 부족을 자동으로 보정하지 않는다.

### 2. Available kcal

기준선:

```text
availableCarbFatKcal = targetKcal - proteinReservedG * 4
```

alcohol kcal는 carb/fat target allocation budget에서 "자리를 만들어주는" 항목이 아니다.

이유:

```text
술을 마셨다고 앱이 탄수/지방 목표를 자동으로 내려서
alcohol kcal를 합법적인 macro target처럼 흡수하면 안 된다.
```

alcohol은 기존 v8.3 원칙대로:

```text
- totalKcal / TDEE overload 축에 반영
- alcohol physiology risk 축에 반영
- v6.1 alcoholImpactPenalty 후처리 감산은 되살리지 않음
```

### 3. Joint iso-calorie line

carb/fat target pair는 아래 선 위에 있어야 한다.

```text
4 * carbTargetG + 9 * fatTargetG = availableCarbFatKcal
```

rounding 때문에 tolerance는 허용한다.

```text
allowedGapKcal <= 2
```

이 gap을 넘으면 implementation/test 실패다.

### 4. Current intake reaction

현재 섭취값은 target pair를 움직일 수 있다.
하지만 먹은 만큼 목표가 무한히 따라가는 면제 구조가 되면 실패다.

채택:

```text
carb/fat consumed progress 차이
-> continuous skew
-> 3일/7일 trend limiter
-> target pair movement
-> iso-calorie line 재투영
```

금지:

```text
- 많이 먹을수록 목표가 계속 따라 올라가 점수를 방어
- totalKcal/TDEE overload를 adaptive target이 숨김
- high-volume context가 700g carb flat permission이 됨
- step bucket으로 갑자기 target/range가 뛰는 구조
```

### 5. Conditional feasible display range

card range chip을 계속 표시한다면, 다음 의미여야 한다.

```text
displayed carb range:
현재 protein reserve와 target kcal 안에서,
fat이 허용 범위 안에 남을 수 있는 carb 총량 구간

displayed fat range:
현재 protein reserve와 target kcal 안에서,
carbs가 허용 범위 안에 남을 수 있는 fat 총량 구간
```

즉:

```text
display range = raw physiological range ∩ iso-calorie feasible interval
```

현재까지 먹은 counterpart macro도 고려해야 한다.

예:

```text
carbs를 이미 많이 먹었다면 fat upper는 내려가야 한다.
fat을 이미 많이 먹었다면 carb upper는 내려가야 한다.
둘 다 과하면 targetEnergyDeviationPenalty / tdeeOverloadPenalty가 숨겨지면 안 된다.
```

이 계산은 화면에서 길게 설명하지 않는다.
화면은 결과 숫자를 보여주고, DailyCoach는 이미 있는 과탄수/과지방 맥락을 자연스럽게 말한다.

### 6. 3일/7일 limiter

3일/7일 기록은 "금지"가 아니라 "조정폭 제한"이다.

채택:

```text
continuous recency-weighted excess pressure
```

금지:

```text
- 3일 연속이면 갑자기 막고 2일이면 허용하는 count bucket
- 반복 고탄수/고지방 흐름에도 adaptive target movement가 그대로 유지되는 구조
- trend limiter를 final score bonus/penalty로 별도 부과
```

### 7. 운동 맥락

운동은 final score bonus가 아니다.
운동은 carb context / target source / training load factor로만 반영한다.

고강도/고볼륨 운동일에도:

```text
- target kcal가 올라갈 수 있음
- carb context가 연속적으로 열릴 수 있음
- 하지만 high-volume label 하나로 700g carb가 자동 무감점이 되면 실패
```

### 8. Storage / Records

이 모델은 Today live target/display/scoring helper 정합성 문제다.

다음 구현에서도 기본 금지선은 유지한다.

```text
- Records goalSnapshot을 adaptive live target으로 덮어쓰지 않음
- old records migration/recompute/reset 열지 않음
- storage/schema 확장 열지 않음
- scoreDeltaPreview product path 열지 않음
```

`오늘 목표 자동 조정`은 이미 persistent setting surface다.
다음 결과로그에서도 이 사실을 숨기지 않는다.

다음 구현은 최소가 아니라 전체 정합성을 봐야 한다.

```text

J02 impossible corner rejection: carb upper + fat lower/upper가 target kcal를 넘으면 display feasible range에서 동시에 허용하지 않음
J03 moving target example: carb가 올라가면 fat target이 kcal 기준으로 내려감
J04 fat-leading intake: fat이 올라가면 carb target/range가 내려감
J05 mixed overage: carb와 fat이 둘 다 높으면 adaptive target이 과식 방어로 작동하지 않고 kcal/TDEE penalty가 유지됨
J06 repeated 3-day/7-day high-carb pressure: carb target relaxation이 연속적으로 줄어듦
J07 repeated high-fat pressure: fat target relaxation이 연속적으로 줄어듦
J08 high-volume carb: 700g carb가 label 하나로 flat no-penalty가 되지 않음
J09 adaptive setting OFF: fixed target 유지
J10 Records snapshot: selected calculation target 보존, adaptive live target 저장 안 함
J11 card display range = joint helper output
J12 no user-facing internal exchange arithmetic
```
