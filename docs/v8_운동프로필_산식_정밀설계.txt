2026-06-03 v8.0-AU 문서 계층 정리 기준:
- 이 문서는 원래 V8 순서와 설계 의도를 보존하는 living design ledger다. "구현 전 인계서"라는 원래 상태 문구는 작성 당시의 성격이며, 아래 handoff/stage note가 현재 진행 상태를 갱신한다.
- `/GOAL` 장시간 재개 방식은 폐기한다. 이후 작업은 `docs/README.md`의 문서 태그와 이 문서의 원래 순서를 함께 보고, 수동으로 작게 범위를 잡아 진행한다.
- AT는 일반 diet production protein 기본값을 `2.0g/kg`으로 복구한 단계다. 이것은 수준별 산식 완성이 아니라, 수준별 산식을 올리기 전 일반 기본값의 shortcut을 제거한 것이다.
- V8의 수준별 산식 목표는 여전히 open이다. 운동 프로필, 루틴/세션, 수행 수준, 체성분 신뢰도, 최근 기록/최근 추세를 실제 UI/Records/Coach/backup 경로와 연결해야 한다.
- v8.0-AV 이후 문서정리는 다음 작업 진입을 막지 않는다. 다음 실제 작업은 기존 3단계인 profile/routine/session broad human UX review matrix와 수준별 산식 범위 설계로 들어가며, candidate-v2/scoped production wording hardening은 그 3단계의 preflight로 처리한다.
- v8.0-AW는 그 기존 3단계의 첫 user-path matrix다. `profileRoutineUserPathMatrix`가 Settings/default Today path와 Today explicit routine/session override path를 분리해 모든 profile/routine/session/advanced ON-OFF row를 `calculate()`와 Records `goalSnapshot`까지 확인한다.
- AX에서 AW의 advanced-toggle 해석을 바로잡았다. advanced OFF는 모든 프로필에서 사용자 화면상 단순 `REST / PUSH`만 노출한다. 보디빌딩 OFF 운동일은 기존 단순 기본 강도 `0.70`을 유지하고, `PPL-UL / PUSH 0.80` 같은 상세 세션 xw는 advanced ON에서만 적용한다.
- 비보디빌딩 advanced OFF는 전용 루틴/세션을 화면에 열지 않는다. 대신 단순 운동일의 내부 계산 기본값은 해당 프로필의 기본 훈련 세션을 적용해 xw, 웨이트 시간, 유산소 기본값을 잡는다. 상세 루틴/세션 선택과 수동 세션 강도 조정은 advanced ON 경로다.
- AW는 수준별 산식 구현이 아니다. `performanceLevel`, 체성분 신뢰도, 최근 기록 커버리지, 최근 추세를 묶는 사용자별 production formula는 계속 open gate다.

이 문서는 “방향성 메모”가 아니라 다른 컴퓨터의 다른 Codex가 처음 투입되어도 같은 의도로 작업할 수 있게 만든 구현 인계서다.
문서에 없는 내용을 Codex가 임의로 채우면 안 된다.
구현 중 더 좋은 판단이 생기면 먼저 문서를 갱신하고, 왜 기존 기준을 바꿨는지 남긴 뒤 구현한다.

────────────────────────
0. 절대 전제
────────────────────────

0-1. v8의 핵심 목표

v8의 핵심은 UI polish가 아니라 산출값 신뢰도다.
사용자가 앱의 목표 칼로리와 탄단지를 그대로 믿고 행동할 수 있는지가 v8의 중심이다.

v8에서 반드시 풀어야 하는 문제:

1. 다양한 체형과 운동 수준에서 유지 칼로리가 납득 가능한가.
2. 목표별 칼로리 변화율이 다이어트 / 커팅 / 유지 / 린매스업 / 벌크업마다 다르게 작동하는가.
3. 단백질 / 지방 / 탄수 배분이 목표와 운동 상태에 맞고, 한 영양소가 과하게 다른 영양소를 밀어내지 않는가.
4. 운동 수준을 주간 웨이트일수 하나로 오판하지 않는가.
5. InBody / 직접 입력 / 가정용 InBody / 기록 부족 상태에서 계산 신뢰도를 구분하는가.
6. Today와 Records의 값 책임이 섞여 “기준갱신” 같은 상태 표시가 어색하게 뜨지 않는가.
7. 산식 변경 후 대표 케이스가 아니라 넓은 시나리오 격자에서 사람이 봐도 납득 가능한가.

0-2. v8에서 당장 하지 않을 것

아래는 v8 산식 안정화 전에는 하지 않는다.

- 전체 UI 리디자인
- Records / 최근 흐름 / InBody / Settings의 대형 화면 재배치
- 점수 산식 전면 재설계
- backup schema 대형 변경
- InBody 탭 안에 목표별 행동 조언을 직접 넣기
- 설명문을 많이 늘려서 산식 불안을 문구로 덮기

필요하면 해도 되는 것:

- Settings의 운동 프로필 / 운동 수행 능력 입력 구조 최소 추가
- Today의 세션 / 운동 기준 선택 구조 보완
- 산식 검증용 내부 리포트 / 테스트 추가
- 고급 설정 도움말 보강
- Today-Records 기준갱신 상태가 잘못 뜨는 경우의 핫픽스

1. `docs/README.md`
2. `docs/내가-바라는-앱의-성격.txt`
3. `docs/개발정책.txt`
4. 이 문서
5. `docs/v8_현재산식_기준갱신_감사_2026-06-02.txt`
   - v8 1단계 현재 산식 / Today-Records 기준갱신 감사 완료 이후의 다음 작업자는 반드시 확인한다.

archive 내용을 새 기준으로 쓰려면 먼저 현재 앱 구조와 충돌 여부를 확인한다.

0-4. 근거 등급 원칙

v8 산식 작업에서는 근거의 종류를 반드시 분리한다.

1. 외부 운동생리학 / 스포츠영양 근거
   - 논문, 공식 가이드, position stand 등 외부 출처가 명시되어 있고, 앱의 대상 사용자와 적용범위까지 검토한 근거다.
2. 내부 코드 근거
   - 현재 `index.html` 구현, snapshot, 테스트, report 출력처럼 앱이 실제로 어떻게 동작하는지 보여주는 근거다.
   - 구현 동작의 증거일 뿐, 운동생리학적으로 맞다는 증거가 아니다.
3. 제품 정책 근거
   - 사용자가 원하는 앱의 성격, Today / Records / Coach 책임, UX 상태 구분처럼 앱이 어떻게 행동해야 하는지에 대한 근거다.
   - 제품 정책으로는 중요하지만 산식의 생리학적 타당성 근거로 말하면 안 된다.
4. 미검증 생리학 후보
   - 그럴듯한 내부 의도 또는 기존 산식이지만 외부 근거와 앱 적합성 검토가 끝나지 않은 항목이다.

외부 근거라고 명시되지 않은 기존 작업 근거는 기본적으로 내부 코드 근거 또는 제품 정책 근거로 본다.
특히 `docs/# 2026 ACSM 근력운동 가이드 한국어 해설판.txt`는 보조 문서이지 산식 숫자를 직접 확정하는 문서가 아니다.
어떤 수치를 산식에 반영하려면 해당 수치의 외부 출처, 대상군, 앱 적용범위, Today / Records / Coach 영향까지 별도로 남긴다.

1. 현재 앱의 값 책임
────────────────────────

1-1. 탭별 책임

Today:
- 오늘 계산값의 작업공간이다.
- 오늘 몸 상태, 오늘 활동, 오늘 운동, 오늘 유산소, 오늘 목표 칼로리, 오늘 탄단지, 오늘 기록 저장/갱신을 다룬다.
- Today의 입력값은 “오늘 계산 기준값”이다.
- Today 값이 바뀌었다고 해서 저장된 Records가 자동으로 바뀌면 안 된다.

Records:
- 날짜별 확정 기록 보관함이다.
- 식사, 공복 체중, 메모, 해당 날짜의 저장된 계산 기준 snapshot을 확인하고 수정한다.
- Records에서 식사나 메모를 수정했다고 Today 계산 기준이 바뀐 것으로 처리하면 안 된다.

InBody:
- 측정값과 변화 추세를 보는 탭이다.
- 목표별 행동 조언은 직접 하지 않는다.
- Today 계산에 InBody 값을 적용할 수는 있지만, InBody 탭 자체가 목표 칼로리를 고쳐서는 안 된다.

Settings:
- 기본값, 장기 기준, 운동 프로필, 전문가 설정, 데이터 관리를 맡는다.
- Today의 오늘 값과 Settings의 기본값은 다를 수 있다.

1-2. 값 종류

이 앱에는 비슷해 보이지만 서로 다른 값이 있다.
구현자는 반드시 구분해야 한다.

1. 측정값
   - 예: 공복 체중, InBody 체중, 골격근량, 체지방량, 체지방률
   - 사용자가 측정하거나 기록한 사실값이다.

2. Today 계산 기준값
   - 예: 계산용 체중, 오늘 활동량, 오늘 업무 유형, 오늘 세션, 오늘 웨이트 시간, 오늘 유산소 조건
   - 오늘 목표 칼로리와 탄단지를 계산하는 입력값이다.

3. 저장 snapshot
   - Records에 저장된 당시 목표 / 당시 계산 기준이다.
   - 과거 기록을 다시 열었을 때 “그날 어떤 기준으로 계산했는지”를 보존한다.

4. Settings 기본값
   - 다음 날짜나 기본 시작값으로 쓰는 장기 기준이다.
   - Today를 열 때 기본 후보로 들어올 수 있지만, 현재 Today 값과 항상 같아야 하는 것은 아니다.

5. 계산용 보정값
   - 예: 계산용 보정 LBM
   - 내부 계산에서 목표 보정을 위해 쓰는 값이다.
   - 생리학적 안전선 자체를 낮추는 용도로 쓰면 안 된다.

────────────────────────
2. Today-Records 기준갱신 감사 설계
────────────────────────

2-1. 이 감사를 v8에 반드시 넣는 이유

사용자 관찰:
- Today 계산 기준을 아무것도 건드리지 않았는데 Records 탭에서 기록만 수정한 뒤 Today에 “기준갱신”이 뜨는 것처럼 보이는 사례가 있다.
- 계산 기준 입력값을 바꿨다가 원래 값으로 되돌렸을 때도 갱신 버튼이 남아 있으면 어색하다.

이 문제는 단순 UI 문제가 아니다.
값의 책임이 섞였거나 dirty 판단이 정규화되지 않았거나, Records 수정 경로가 Today 기준 snapshot까지 건드리는 문제일 수 있다.

2-2. 기준갱신의 정확한 의미

기준갱신 표시는 아래 조건에서만 떠야 한다.

“저장된 오늘 날짜 Records의 계산 기준 snapshot과 현재 Today 계산 기준값이 실제로 다르며, 사용자가 현재 Today 기준을 저장 기록에 반영할 수 있는 상태”

아래 조건에서는 기준갱신이 뜨면 안 된다.

- 식사만 수정했다.
- 메모만 수정했다.
- 식사 점수만 다시 계산되었다.
- Records의 표시 카드가 다시 렌더되었을 뿐이다.
- 입력값을 바꿨다가 원래 기준값과 같은 값으로 되돌렸다.
- 문자열 표현만 다르고 실제 수치가 같다.
- 숫자 소수점 표시 자리만 다르다.
- REST로 인해 웨이트 시간이 0으로 정규화된 상태와 사용자가 0을 입력한 상태가 같은 의미다.
- 빈 문자열, null, undefined, 0이 같은 의미로 취급되어야 하는 필드에서 표현만 다르다.

2-3. 별도 상태로 분리해야 하는 것

다음은 “기준갱신”이 아니라 별도 mismatch 또는 안내로 다룰 수 있다.

- Today 계산용 체중과 Records 공복 체중이 다름
- 최신 InBody 값과 Today 계산용 체성분이 다름
- Settings 기본값과 Today 오늘 값이 다름
- 저장 기록의 식사량이 바뀌어서 식단 점수만 달라짐

중요:
Records에서 공복 체중을 바꾸면 “기록 측정값”이 바뀐 것이다.
그 변화가 Today 계산용 체중에 자동 반영되어야 하는지는 별도 사용자 선택 문제다.
기준갱신 표시로 뭉뚱그리면 안 된다.

2-4. 감사할 경우의 수

아래 케이스는 반드시 자동 테스트 또는 수동 QA로 확인한다.

A. Today 입력값 변경/복원
1. 활동량 변경 → 기준갱신 표시 여부 확인.
2. 활동량을 원래 값으로 복원 → 기준갱신 사라짐.
3. 업무 유형 변경 → 기준갱신 표시 여부 확인.
4. 업무 유형을 원래 값으로 복원 → 기준갱신 사라짐.
5. 업무 강도 슬라이더 변경 → 기준갱신 표시.
6. 업무 강도 슬라이더를 원래 값으로 복원 → 기준갱신 사라짐.
7. 루틴 변경 → 기준갱신 표시.
8. 루틴 원복 → 기준갱신 사라짐.
9. 세션 변경 → 기준갱신 표시.
10. 세션 원복 → 기준갱신 사라짐.
11. REST 선택 → 웨이트 시간 0, 입력 비활성, 기준갱신 판단 정규화.
12. REST에서 운동 세션으로 변경 → Settings 기본 웨이트 시간이 복원되는지 확인.
13. 운동 세션에서 REST로 변경했다가 원래 운동 세션으로 복원 → 원래 기준과 같으면 기준갱신 사라짐.
14. 웨이트 시간 변경 → 기준갱신 표시.
15. 웨이트 시간을 원래 값으로 복원 → 기준갱신 사라짐.
16. 훈련 강도 슬라이더 변경 → 기준갱신 표시.
17. 훈련 강도 슬라이더 원복 → 기준갱신 사라짐.
18. 유산소 종류 변경 → 기준갱신 표시.
19. 유산소 종류 원복 → 기준갱신 사라짐.
20. 유산소 시간/속도/경사 변경 후 원복 → 기준갱신 사라짐.
21. guide/activity basis 변경 후 원복 → 기준갱신 사라짐.
22. InBody 최신값 적용 후 다시 저장 snapshot과 같은 값으로 복원 → 기준갱신 사라짐.

B. Records 수정
1. 식사 추가만 함 → Today 기준갱신 표시 금지.
2. 식사 삭제만 함 → Today 기준갱신 표시 금지.
3. 식사 수량/탄단지/칼로리 수정 → Today 기준갱신 표시 금지.
4. 메모 수정 → Today 기준갱신 표시 금지.
5. 공복 체중 수정 → 기준갱신이 아니라 체중 mismatch 안내 후보로만 처리.
6. 기록 방식 변경 → 기준갱신 표시 금지. 단, 저장 snapshot 자체를 사용자가 명시적으로 바꾼 경우는 예외.
7. 과거 날짜 기록 수정 → 오늘 날짜 Today 기준갱신 표시 금지.
8. 오늘 날짜 Records 상세에서 “당시 목표 / 당시 계산 기준”을 직접 수정하는 기능이 있다면, 그때만 기준 snapshot 변경으로 판단.

C. 값 정규화
1. `74.7`, `74.70`, `74.700`은 같은 값으로 본다.
2. `0`, `0.0`, 빈 숫자 입력 후 정규화된 0은 같은 값으로 본다.
3. select 값의 label 변경과 value 변경을 구분한다.
4. REST의 웨이트 시간은 항상 0으로 비교한다.
5. 세션별 기본 강도는 session value 기준으로 비교한다. 표시 문구만 바뀌어서는 dirty가 되면 안 된다.
6. 부동소수점 비교는 raw string이 아니라 정규화된 number로 한다.
7. 소수점 표시 자리 변경 때문에 dirty가 생기면 안 된다.

2-5. 구현 원칙

기준갱신 판단은 DOM 문자열 비교로 하지 않는다.
반드시 하나의 정규화 함수가 Today 계산 기준 객체와 저장 snapshot 객체를 canonical form으로 바꾼 뒤 비교한다.

권장 구조:

1. `buildTodayBasisCanonical(state)`
   - 현재 Today 계산 기준값을 정규화한 객체를 만든다.

2. `buildRecordBasisCanonical(record)`
   - Records에 저장된 당시 계산 기준 snapshot을 정규화한 객체를 만든다.

3. `diffCalculationBasis(todayCanonical, recordCanonical)`
   - 실제 의미가 다른 필드만 diff로 반환한다.

4. `shouldShowBasisRefresh(diff)`
   - diff가 비어 있으면 false.
   - diff가 식사/메모/점수/렌더 표시 차이뿐이면 false.
   - 계산 기준 필드 차이가 있으면 true.

5. `getRecordValueMismatch(...)`
   - 공복 체중 mismatch, InBody mismatch 같은 별도 안내는 이쪽으로 분리한다.

금지:
- 버튼 표시 여부를 렌더 직전 임시 문자열로 판단하지 않는다.
- 특정 입력 이벤트마다 개별 땜질로 기준갱신을 숨기지 않는다.
- “Records 수정 후 Today에 갱신이 뜨는 문제”를 CSS hide로 처리하지 않는다.

────────────────────────
3. 유지 칼로리와 목표 칼로리
────────────────────────

3-1. 개념 분리

유지 칼로리:
- 현재 몸 상태와 생활/운동 조건에서 체중이 대략 유지될 것으로 보는 기준값이다.
- 목표와 무관하게 먼저 계산한다.

목표 칼로리:
- 유지 칼로리에 목표별 변화율을 적용한 값이다.
- 다이어트 / 커팅 / 린매스업 / 벌크업 / 유지에 따라 달라진다.

목표 탄단지:
- 목표 칼로리를 단백질, 지방, 탄수로 배분한 값이다.
- 칼로리만 맞고 단백질/지방/탄수가 비정상이면 실패다.

3-2. 현재 구조를 먼저 감사할 것

v8 구현자는 새 산식을 바로 만들지 않는다.
먼저 현재 `index.html`에서 아래 흐름을 추적한다.

1. 입력값 수집
   - Settings 기본값
   - Today 계산 기준값
   - Records 저장 snapshot
   - InBody / 직접 입력 체성분
   - 전문가 설정

2. 유지 칼로리 계산
   - BMR
   - 활동량 보정
   - 업무 강도 보정
   - 주간 웨이트일수 보정
   - 오늘 세션 보정
   - 웨이트 시간 / 훈련 강도 보정
   - 유산소 보정

3. 목표 칼로리 계산
   - 목표별 deficit/surplus
   - 안전선
   - 계산용 보정 LBM
   - 실제 LBM 안전선

4. 탄단지 배분
   - 단백질 기준
   - 지방 기준
   - 탄수 잔여 배정
   - guide/activity basis 차이

감사 결과는 “코드 위치 / 입력 / 중간값 / 출력 / 왜 그렇게 되는지” 형태로 남긴다.

3-3. 안전선

Energy availability 같은 생리학적 안전선은 실제 제지방량 기준으로 둔다.
계산용 보정 LBM은 목표 칼로리 보정에 쓸 수 있지만, 안전선 자체를 낮추는 데 쓰면 안 된다.

근거 후보:
- IOC RED-S consensus는 Energy Availability를 `섭취 에너지 - 운동 에너지 소비`를 FFM으로 나눈 값으로 다룬다.
- IOC 2018 update는 45 kcal/kg FFM/day를 일반적인 건강 기능의 기준으로, 30 kcal/kg FFM/day 미만을 LEA의 역사적 기준으로 언급한다.
- URL: https://bjsm.bmj.com/content/52/11/687

주의:
- 이 앱은 의료 진단 앱이 아니다.
- 안전선은 “위험 가능성 경고 / 산식 하한 방어” 용도이지 진단 문구로 쓰지 않는다.

────────────────────────
4. 단백질 / 지방 / 탄수 배분 감사
────────────────────────

4-1. 사용자가 제기한 실제 케이스

현재 사용자 기준 입력값에서 아래 출력이 관찰되었다.

- 목표 칼로리: 1913 kcal
- 탄수화물: 196.3 g
- 단백질: 180.5 g
- 지방: 45.1 g

이 케이스는 v8 산식 감사의 필수 회귀 케이스다.
단백질 180.5g은 체중 약 75kg 기준 약 2.4g/kg 수준이다.
감량기 고단백으로 설명은 가능하지만, 기본 산출값으로 자동 제시하기에는 상한 쪽이다.
특히 지방이 45.1g까지 낮아진 상태라면 단백질이 지방/탄수를 과하게 밀어내는지 확인해야 한다.

4-1-1. 기존 단백질 산식의 의도 복원

단백질 180.5g 같은 값이 보이면 바로 “버그”라고 단정하지 않는다.
먼저 기존 산식이 왜 그런 결정을 했는지 복원해야 한다.

복원해야 할 내용:

1. 기존 산식의 의도
   - 근육 보존을 우선하려고 했는가.
   - 감량기 단백질을 높게 주려는 의도였는가.
   - 린매스업/커팅/운동인에게 고단백을 주려는 의도였는가.
   - 체중 기준이 아니라 LBM 기준 또는 보정 LBM 기준으로 잡으려 했는가.
   - guide/activity basis 차이 때문에 단백질이 높아진 것인가.
   - 지방 하한과 탄수 배정 이후 남는 칼로리 처리 과정에서 단백질이 커진 것인가.

2. 기존 의도 중 내부 정책상 보존할 만한 부분
   - 감량 중 근육 손실을 줄이려는 방향은 외부 근거 후보와 맞춰 검토할 가치가 있다.
   - 운동하는 사용자에게 일반인보다 높은 단백질을 주려는 방향은 목표 / 훈련수준 / 체형에 따라 후보가 될 수 있다.
   - 체지방률이 높은 사용자에게 실제 체중 기준 단백질을 그대로 적용하지 않으려는 보정 의도도 필요할 수 있다.
   - 커팅처럼 근육 보존이 중요한 목표에서는 단백질 상향이 필요할 수 있다.
   - 단, 이 항목들은 기존 산식을 바로 통과시키는 외부 타당성 근거가 아니라 candidate 설계에서 보존 여부를 검토할 내부 의도다.

3. 기존 의도가 틀렸거나 부족할 수 있는 부분
   - “근육 보존”이라는 이유로 모든 다이어트 사용자에게 커팅/대회준비급 단백질을 주면 안 된다.
   - 단백질이 올라가면서 지방이 너무 낮아지면 지속성과 호르몬/건강 측면에서 불리할 수 있다.
   - 단백질이 올라가면서 탄수가 지나치게 줄면 훈련 수행과 식단 지속성이 떨어질 수 있다.
   - 체중 기준, LBM 기준, 보정 LBM 기준이 섞여 있으면 사용자가 왜 이 값이 나왔는지 이해할 수 없다.
   - 단백질 상한이 없거나 soft cap만 있고 재분배가 약하면 “고단백이 항상 좋은 것처럼” 작동할 수 있다.

4. 개정 시 비교해야 할 것
   - 기존 산식 결과와 새 후보 산식 결과를 같은 입력으로 나란히 비교한다.
   - 단백질만 낮아졌다고 개선으로 보지 않는다.
   - 지방 하한, 탄수 최소량, 총칼로리, 목표별 성격, 운동 수행 유지가 함께 좋아졌는지 본다.
   - 기존 산식의 장점인 근육 보존 의도를 잃지 않아야 한다.
   - 새 산식은 “더 낮은 단백질”이 목표가 아니라 “목표/체형/운동수준에 맞는 단백질”이 목표다.

필수 결론 형식:

- 기존 산식이 단백질을 높게 잡은 직접 원인
- 기존 산식의 원래 의도
- 그 의도가 현재 사용자 케이스에서 타당한 부분
- 그 의도가 현재 사용자 케이스에서 과하거나 틀린 부분
- 새 후보 산식이 기존 의도를 어떻게 보존하는지
- 새 후보 산식이 기존 부작용을 어떻게 줄이는지

금지:
- “180.5g은 높아 보이니 낮춘다”처럼 감각만으로 고치지 않는다.
- “고단백은 근손실 방지에 좋다”는 이유만으로 모든 케이스를 통과시키지 않는다.
- 기존 산식의 의도를 복원하지 않고 새 산식으로 덮어쓰지 않는다.

4-2. 단백질 기준 설계 원칙

단백질은 단일 고정값으로 두지 않는다.
아래 기준을 조합한다.

1. 체중 기준 g/kg
2. 제지방량 기준 g/kg LBM 또는 FFM
3. 목표
   - 유지
   - 다이어트
   - 커팅
   - 린매스업
   - 벌크업
4. 체지방률
5. 운동 프로필
6. 운동 수행 수준
7. 칼로리 제한 정도
8. 기록 신뢰도

근거 후보:
- ISSN protein position stand는 운동하는 사람 대부분의 단백질 섭취를 대략 1.4~2.0 g/kg/day 범위로 제시한다.
- URL: https://jissn.biomedcentral.com/articles/10.1186/s12970-017-0177-8
- Helms et al. natural bodybuilding contest prep review는 칼로리 제한 중인 자연 보디빌더에게 2.3~3.1 g/kg LBM/day와 지방 15~30% kcal를 제안한다.
- URL: https://pubmed.ncbi.nlm.nih.gov/24864135/
- 2026-06-03 v8.0-E에서 위 외부 근거를 다시 확인했다. ISSN 범위는 일반 운동인 검토 기준이고, Helms 범위는 자연 보디빌딩 contest prep 맥락이다. 둘 다 "현재 앱의 모든 diet 사용자에게 bodyweight 2.4g/kg을 기본 적용해도 된다"는 직접 근거가 아니다.

이 근거는 “항상 이렇게 넣어라”가 아니다.
v8에서는 아래처럼 해석한다.

- 일반 운동인 기본값은 ISSN의 보편 범위 쪽에 가깝게 둔다.
- 매우 마른 커팅 / 대회 준비형 / 고훈련자 / 큰 칼로리 제한 상황은 Helms식 LBM 기준 상향 범위를 후보로 둔다.
- 일반 다이어트 사용자가 자동으로 contest prep 수준 단백질을 받으면 안 된다.

4-3. 단백질 상한 후보

상한은 하나의 절대값이 아니라 상황별 soft cap / hard cap으로 설계한다.

필수 판단:

1. 일반 다이어트
   - 단백질이 체중 대비 과도하게 올라가면 soft warning.
   - 체지방률이 높으면 실제 체중 기준보다 LBM 기준을 더 강하게 고려한다.

2. 커팅
   - 운동 수행이 높고 체지방률이 낮을수록 단백질 상한을 높게 허용한다.
   - 그래도 지방 하한과 탄수 최소량을 밀어내면 재분배한다.

3. 린매스업 / 벌크업
   - 단백질을 무작정 높이는 것보다 탄수와 총칼로리 여유가 중요하다.
   - 단백질이 이미 충분하면 초과분은 탄수나 지방으로 재분배한다.

4. 비만 / 고도비만
   - 체중 기준 g/kg만 쓰면 단백질이 비현실적으로 커질 수 있다.
   - LBM / 목표체중 / 조정체중 기준을 검토한다.

4-4. 지방 하한

단백질을 높이느라 지방이 지나치게 낮아지면 실패다.
Helms review는 contest prep 맥락에서 지방 15~30% kcal를 언급한다.
이 값은 모든 사용자에게 그대로 적용하는 절대 규칙은 아니지만, 지방이 너무 낮아지는지 확인하는 후보 기준으로 쓸 수 있다.

v8 구현 시 지방 기준은 아래를 출력해야 한다.

- 지방 g
- 지방 kcal
- 전체 칼로리 대비 %
- 하한 적용 여부
- 하한 적용 때문에 탄수/단백질이 어떻게 바뀌었는지

4-5. 탄수 배정

탄수는 “남는 칼로리”로만 취급하면 안 된다.
운동 프로필과 수행 유지를 고려해야 한다.

예:
- 보디빌딩 / 파워빌딩 / 스트렝스는 훈련일 탄수 최소량이 중요하다.
- 러닝 / 지구력은 탄수 요구량이 더 커질 수 있다.
- 휴식일은 탄수 요구량이 낮을 수 있지만, 너무 낮아져서 식단 지속성이 떨어지면 안 된다.

4-6. 필수 출력

모든 산식 검증 리포트는 아래를 출력한다.

- protein_g
- protein_g_per_kg_bodyweight
- protein_g_per_kg_lbm
- protein_rule_source
- protein_soft_cap_hit
- protein_hard_cap_hit
- fat_g
- fat_percent_kcal
- fat_floor_hit
- carb_g
- carb_min_hit
- redistribution_reason

────────────────────────
5. 사용자 유형 판단
────────────────────────

5-1. 금지 원칙

주간 웨이트일수만으로 초급/중급/상급을 판단하면 안 된다.

오판 예:
- 체성분이 매우 좋은 사람이 주 2회만 운동함 → 단순 빈도 기준이면 초급으로 오판될 수 있다.
- 체성분이 낮지만 주 6회 운동함 → 단순 빈도 기준이면 상급으로 과대평가될 수 있다.
- 중량 수행은 높지만 체성분이 낮음 → 수행형 강점은 인정하되 체성분 목표에서는 다른 판단이 필요하다.
- 체성분은 좋지만 수행 입력이 없음 → 신뢰도 부족으로 보수적으로 판단해야 한다.

5-2. 필수 판단 축

3. 키
4. 체중
5. 체지방률
6. 체지방량
7. 제지방량
8. 골격근량
9. FFMI 또는 유사 근육량 지표
10. 주간 웨이트일수
11. 세션 길이
12. 훈련 강도
13. 운동 프로필

5-3. 사용자 유형 라벨은 내부용

앱 내부에서는 유형 라벨을 만들 수 있다.
하지만 사용자에게 “당신은 초급자입니다”처럼 단정적으로 보여주는 것은 v8 초반 목표가 아니다.

- low_muscle_lean
- high_muscle_lean
- average_body
- overweight_active
- high_muscle_high_fat
- obese
- severe_obese
- endurance_focused
- strength_focused
- bodybuilding_focused
- low_reliability_data

각 라벨은 최종 산식의 작은 신호일 뿐이다.
하나의 라벨이 목표 칼로리 전체를 지배하면 안 된다.

────────────────────────
6. 운동 프로필 설계
────────────────────────

6-1. 기본값

기본 운동 프로필은 보디빌딩으로 둔다.
이 앱의 출발점이 탄단지, 린매스업, 벌크업, 커팅 같은 체성분 목표에 가깝기 때문이다.

단, 기본값이 보디빌딩이라는 사실을 Settings에서 사용자가 알 수 있어야 한다.
사용자가 다른 운동을 주 운동으로 한다면 프로필을 바꿀 수 있어야 한다.

6-2. 프로필 후보와 책임

1. 보디빌딩
   - 기본 프로필.
   - PUSH / PULL / LEGS / UPPER / LOWER / REST 구조와 잘 맞는다.
   - 근육량 유지/증가와 체지방 조절을 중심으로 본다.
   - 수행 입력 후보: 8~12회 반복 기준 주요 리프트 또는 대표 운동 수행 수준.

2. 파워빌딩
   - 근비대와 중량 향상을 함께 본다.
   - 세션 구조는 보디빌딩과 유사할 수 있으나 주요 리프트 수행 지표를 더 중요하게 본다.
   - 칼로리 부족이 수행 저하로 이어지는 위험을 더 강하게 본다.

3. 스트렝스 / 파워리프팅
   - 1RM 또는 추정 1RM, 훈련 강도, 피로 관리가 중요하다.
   - 체성분 목표가 있어도 성능 저하 위험을 함께 봐야 한다.
   - 세션 후보는 SQUAT / BENCH / DEADLIFT / UPPER / LOWER / FULL BODY / REST처럼 별도 설계가 필요할 수 있다.

4. 러닝 / 지구력
   - 웨이트보다 거리, 페이스, 주간 총량, 회복이 중요하다.
   - 기존 유산소 입력을 재사용할 수 있지만 “보조 유산소”가 아니라 “주 운동”으로 해석해야 한다.
   - 수행 입력 후보: 5km 기록, 10km 기록, 주간 러닝 거리, 장거리 빈도.

5. 크로스핏 / 혼합형
   - 고강도 혼합 운동이다.
   - 단순 웨이트 MET나 단순 유산소 MET로 보기 어렵다.
   - 초기에는 보수적 보정과 사용자 확인을 강하게 둔다.

6-3. 세션 구현 주의

운동 프로필을 추가한다면 세션 구현도 정확해야 한다.
대충 만든 세션은 해당 운동을 오래 한 사용자에게 앱 신뢰도를 떨어뜨린다.

보디빌딩 외 프로필을 구현할 때는 아래를 반드시 설계한다.

- 표시 세션 이름
- 내부 value
- 기본 강도
- 기본 시간
- REST 처리
- Today 목표 칼로리에 미치는 영향
- Settings 주간 운동 기준과의 연결
- 유산소 입력 재사용 여부
- 기록 저장 snapshot 구조

프로필 이름만 추가하고 세션/산식은 보디빌딩 그대로 쓰는 것은 금지한다.
단, v8 초반에 모든 프로필을 완성하기 어렵다면 아래처럼 처리한다.

- 보디빌딩만 active
- 나머지는 후속 후보 또는 disabled 상태
- 화면에 가짜 선택지를 노출하지 않음
- 문서에는 왜 보류했는지 남김

────────────────────────
7. 목표별 산식 방향
────────────────────────

7-1. 목표 후보

- 유지
- 다이어트
- 커팅
- 린매스업
- 벌크업

다이어트와 커팅은 둘 다 감량이지만 성격이 다르다.

다이어트:
- 일반 감량.
- 체지방률이 높거나 운동 수준이 낮아도 쓸 수 있다.
- 과도한 저칼로리 방지와 지속성이 중요하다.

커팅:
- 운동 수행과 근육 보존을 전제로 한 감량.
- 체지방률이 이미 낮거나 운동 수행이 높은 사용자에게 더 적합하다.
- 감량 속도는 다이어트보다 보수적이어야 한다.

유지:
- 체중 변화보다 기록 안정성과 운동 수행 유지가 중요하다.
- 변동이 있더라도 자동 조정은 조심한다.

린매스업:
- 체지방 증가를 억제하면서 느리게 증량한다.
- 체성분 기록과 주간 체중 변화가 중요하다.

벌크업:
- 증량을 허용하되 체지방 증가가 지나치면 조정한다.
- 운동 수행과 체중 변화가 함께 봐야 한다.

7-2. 변화율은 범위로 설계한다

목표별 변화율은 고정 기본값 하나보다 범위가 타당하다.

이유:
- 체지방률, 근육량, 운동 수행, 목표, 기록 신뢰도에 따라 적절한 속도가 달라진다.
- 한 사람에게 맞는 값이 다른 사람에게는 과하거나 약할 수 있다.
- 진행 상황이 쌓이면 28일 기준으로 범위 안 위치를 조정 제안할 수 있다.

설계 원칙:

1. 각 목표별 권장 변화율 범위를 둔다.
2. 범위 안 위치를 고르는 점수/룰을 둔다.
3. 14일에는 자동 조정하지 않는다.
4. 28일에는 제안만 만들고, 사용자 승인 후 반영한다.
5. 기록 신뢰도가 낮으면 변화율 조정을 보류한다.

7-3. 14일 / 28일 정책

14일:
- 흐름 관찰.
- 경고 또는 기록 보완 안내 가능.
- 칼로리 자동 조정 금지.
- “설정을 바꾸세요”보다 “기록을 더 쌓아 판단합니다”에 가깝다.

28일:
- 목표 속도와 실제 변화 속도 비교.
- 목표 범위를 벗어나면 조정 제안 생성.
- 제안에는 이유, 기대 변화, 적용 전후 칼로리/탄단지가 있어야 한다.
- 사용자가 승인해야 반영된다.

────────────────────────
8. 시나리오 검증 설계
────────────────────────

8-1. 대표값 테스트만으로 통과 금지

v8 산식은 대표 케이스 몇 개만 보고 통과시키지 않는다.
“극단값이 안전선을 안 깼다”도 충분하지 않다.
사용자가 실제로 자기 값을 넣었을 때 결과가 납득되어야 한다.

8-2. 시나리오 축

필수 축:

1. 성별: 남 / 여
2. 키: 낮음 / 평균 / 높음
3. 체중: 저체중 / 일반 / 과체중 / 비만 / 고도비만
4. 체지방률: 낮음 / 보통 / 높음 / 매우 높음
5. 골격근량: 낮음 / 보통 / 높음
6. FFMI 또는 근육량 지표: 낮음 / 보통 / 높음
7. 목표: 유지 / 다이어트 / 커팅 / 린매스업 / 벌크업
8. 활동량: 낮음 / 보통 / 높음
9. 업무 유형: 앉아서 일함 / 서서 일함 / 많이 움직임
10. 업무 강도 보정: 낮춤 / 기본 / 높임
11. 주간 웨이트일수: 0 / 2 / 4 / 6
12. 운동 프로필: 보디빌딩 / 파워빌딩 / 스트렝스 / 러닝 / 혼합형
13. 수행 능력: 없음 / 낮음 / 보통 / 높음
14. 오늘 세션: REST / 보디빌딩 세션 / 파워빌딩 세션 / 스트렝스 세션 / 러닝 세션 / 혼합형 세션
15. 웨이트 시간: 0 / 30 / 60 / 90
16. 훈련 강도: 낮음 / 기본 / 높음
17. 유산소: 없음 / 짧음 / 보통 / 많음
18. InBody 신뢰도: 없음 / 직접입력 / 가정용 / 전문가 장비
19. 최근 14일 기록: 부족 / 충분
20. 최근 28일 변화: 목표보다 느림 / 적정 / 목표보다 빠름

중요:
- 14번 오늘 세션 축은 운동 프로필을 반영해야 한다. 러닝 사용자에게 `PUSH`를 오늘 세션처럼 보여주거나, 스트렝스 사용자에게 보디빌딩 분할만 보여주면 안 된다.
- v8.0-D/H 당시 production routine의 `REST / PUSH / PULL / LEGS / UPPER / LOWER` fallback은 actual profile session과 분리해서 기록했다. v8.0-AG 이후 현재 기준은 더 엄격하다. 러닝/스트렝스/파워빌딩/혼합형은 보디빌딩 fallback에 매핑하지 않고 각 프로필의 routine plan/session이 실제 계산 입력이어야 한다.
- 위 필수축을 단순 곱하면 80,621,568,000개 조합이 된다. 이 숫자를 18개 human-review 케이스로 대체했다고 주장하면 안 된다.
- 8-2 필수축 검증과 8-3 사람이 반드시 볼 18개 케이스는 서로 다른 레이어다.
- v8.0-D 1차 러너는 8-2 필수축의 각 카테고리를 한 번씩 실제 계산하는 category sweep과 8-3의 18개 human-review 표를 분리한다. 전체 80,621,568,000 cross-product를 실행했다고 주장하지 않는다.
- 80,621,568,000개 full cross-product는 단일 브라우저 내부 테스트로 돌릴 대상이 아니다. 1ms/케이스여도 약 933일이 걸리며, 결과 저장과 사람 검토도 불가능하다. 후속 검증은 full 축 정의를 보존한 상태에서 category sweep, pairwise/orthogonal coverage, targeted stress, 필요 시 shard runner 순서로 설계한다.
- v8.0-L부터 pairwise/targeted stress는 별도 report layer로 계산한다. 이 레이어도 full cross-product가 아니며, 18개 human-review를 대체하지 않는다. 제약상 말이 안 되는 profile/session/rest/cardio pair는 제외 사유를 집계하고, 포함 가능한 pair는 covering array로 실제 계산한다.
- `유지` 목표는 8-2 필수축에 남긴다.
  - v8.0-D/I의 `axisGoal=maintain`, `productionGoalFallback=recomp` 기록은 당시 production `GOALS.maintain`이 없다는 히스토리다. 유지 목표를 리컴프로 확정했다는 뜻이 아니며, v8.0-J 이후 현재 상태로 읽으면 안 된다.
  - v8.0-J 현재는 production `GOALS.maintain`을 추가했고, rate / cardio reflection / target floor / macro soft range / fat target / score / Coach threshold·decision·copy / `goalSnapshot` / backup-export-import 계약을 함께 닫는 1차 구현 상태다.
  - 단, v8.0-J는 maintain 목표를 production 선택지로 여는 작업이지, 운동 프로필별 macro formula 또는 전체 v8 산식 승인이 아니다.
  - maintain 수치 근거는 섞어 말하지 않는다. 단백질 범위는 ISSN 2017 sports nutrition reference, macro distribution guard는 IOM DRI Table C-5 AMDR의 broad population reference를 참고한다. `deltaMin=-120`, `deltaMax=120`, cardio reflection 0.50, score/Coach tolerance는 외부 운동생리학 근거가 아니라 앱의 제품 정책 근거다.
- 업무 강도 보정축은 감각적으로 "차이를 보이게" 고르면 안 된다.
  - v8.0-D의 `active` fixture 근거는 내부 산식 coverage 근거다. 출처는 현재 `index.html`의 `WORK_TYPE_Y` 값과 `workloadY = clamp(WORK_TYPE_Y[workType] + workAdj, 0, 1)` 경로다.
  - `office(0)`에서 `low(-0.2)`를 검증하면 clamp로 0이 되어 base와 같은 케이스가 된다. `active(0.55)`를 기준으로 두면 low/base/high가 0.35/0.55/0.75로 모두 유효 범위 안에 남아, 업무 보정축의 카테고리 누락 여부를 확인할 수 있다.
  - 이 근거는 "현재 구현의 workAdj 축이 report-only category sweep에서 사라지지 않게 하는 fixture 근거"일 뿐이다. `WORK_TYPE_Y` 계수, `0.06` 업무 가산폭, 실제 직업군별 활동량 모델이 현실적으로 타당하다는 외부 근거가 아니다.
  - v8.0-D1에서 이 구분을 report/docs에 명시한다. 실제 업무활동 계수의 타당성은 v8.0-D가 아니라 9-5/9-6 candidate 설계에서 외부 근거, 앱 사용성, Records/Today/Coach 영향까지 함께 검토해야 한다.

8-3. 사람 눈으로 반드시 볼 케이스

자동 scan과 별개로 아래 케이스는 사람이 표로 확인한다.

1. 마른데 근육 적은 남성
2. 마른데 근육 많은 남성
3. 일반 남성
4. 과체중이지만 운동하는 남성
5. 근육 많고 체지방도 높은 남성
6. 비만 남성
7. 고도비만 남성
8. 여성 저체중
9. 여성 일반
10. 여성 운동인
11. 체성분은 좋은데 주 2회 운동하는 사람
12. 체성분은 낮은데 주 6회 운동하는 사람
13. 러닝 중심 사용자
14. 스트렝스 중심 사용자
15. 보디빌딩 커팅 사용자
16. 벌크업 사용자
17. 린매스업 사용자
18. 기록 신뢰도 낮은 사용자

8-4. 각 케이스 출력 형식

각 케이스는 반드시 아래 형식으로 출력한다.

1. 입력 요약
   - 성별 / 나이 / 키 / 체중
   - 체지방률 / 골격근량 / LBM / FFMI
   - 목표
   - 활동량 / 업무 / 운동 프로필 / 주간 운동 / 오늘 운동
   - InBody 신뢰도

2. 유지 칼로리 분해
   - BMR
   - 활동 보정
   - 업무 보정
   - 주간 운동 보정
   - 오늘 운동 보정
   - 유산소 보정
   - 최종 유지 칼로리

3. 목표 칼로리 분해
   - 목표별 변화율 범위
   - 선택된 변화율
   - 선택 이유
   - 안전선 적용 여부
   - 최종 목표 칼로리

4. 탄단지 분해
   - 탄수 g / kcal / 비율
   - 단백질 g / g/kg 체중 / g/kg LBM / kcal / 비율
   - 지방 g / kcal / 비율
   - 단백질 cap 적용 여부
   - 지방 floor 적용 여부
   - 탄수 최소량 적용 여부
   - 재분배 이유

5. 경고 / 판단
   - 너무 낮은 칼로리
   - Energy availability 위험
   - 단백질 과다/과소
   - 지방 과다/과소
   - 탄수 과소
   - 운동량 대비 회복 부족
   - 기록 신뢰도 부족
   - 사람이 봐도 이상한 점

8-5. 필수 회귀 케이스

아래 사용자 실제 케이스는 반드시 포함한다.

현재 입력에서 휴식일 / 다이어트 기준:
- 목표 칼로리가 1900kcal 전후로 나오는지 여부를 확인한다.
- 단백질이 180.5g처럼 체중 대비 약 2.4g/kg까지 올라가는지 확인한다.
- 지방이 45g 전후까지 낮아지는 경우 단백질/지방/탄수 재분배가 타당한지 확인한다.

주의:
- “1900에 억지로 맞추기”가 목적이 아니다.
- 목적은 생리학적으로, 스포츠영양학적으로, 사용자 체감상 납득 가능한 값인지 확인하는 것이다.
- 주간 웨이트일수 때문에 1844 / 1907 / 2025처럼 차이가 생길 수 있다면, 그 차이가 의도인지 과보정인지 분해 리포트로 설명되어야 한다.

────────────────────────
10. 작업 중 금지사항
────────────────────────

1. UI부터 고치지 않는다.
2. 현재 코드가 그렇다는 이유로 산식 구조를 합리화하지 않는다.
3. 테스트 통과만으로 산식이 맞다고 판단하지 않는다.
4. 대표 케이스 몇 개로 전체 통과 처리하지 않는다.
5. 주간 웨이트일수를 운동 수준의 단독 기준으로 쓰지 않는다.
6. 계산용 보정 LBM으로 생리학적 안전선을 낮추지 않는다.
7. Records 수정 문제를 CSS hide나 렌더 숨김으로 처리하지 않는다.
8. 단백질 과다 문제를 문구로만 설명하고 산식은 그대로 두지 않는다.
9. 문서와 구현이 충돌하는데 조용히 넘어가지 않는다.
10. 다른 탭의 점수 / 저장 / import-export / InBody gate를 산식 작업과 한 번에 섞지 않는다.
