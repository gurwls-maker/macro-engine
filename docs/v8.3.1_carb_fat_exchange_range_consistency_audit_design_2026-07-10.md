## 1. Purpose

The current v8.3.1 app shows compact Today macro ranges beside carbohydrate, protein, and fat.

This solved the earlier problem where the card showed only one fixed target while the score was range-aware. A new product question remains:

```text
If carb and fat each show an independent range, will a user read every corner combination as possible at the same time?
```

The user raised the concrete concern:

```text
If carbs move up, should fat move down?
If carbs are at the upper range, can fat also stay anywhere in its displayed range?
Can carb/fat exchange be 1:1?
```

This document answers that question before implementation.

Decision:

```text
Current independent range chips are acceptable only as marginal ranges.
They are not a joint feasible carb/fat rectangle.
The next gate must decide the joint carb/fat allocation model before any UI or copy implementation.
```

## 3. Current behavior audit

Static code facts:

```text
getTodayMacroCardRange(key, ...)
- returns one displayed range per macro card key
- carb uses context.carbRange
- fat uses context.fatRange
- protein uses context.proteinRange

formatTodayMacroRangeCompactText(range)
- displays "min~maxg"

renderTodayRemainingPanel(...)
- shows the compact range chip beside each macro label
```

The displayed chips are therefore independent per nutrient:

```text
carbs: carbRange.min ~ carbRange.max
fat: fatRange.min ~ fatRange.max
protein: proteinRange.min ~ proteinRange.max
```

Static target/adaptive facts:

```text
getV83TargetAlignedMacroRanges(...)
- creates carbRange and fatRange from target/scoring context
- does not build a full joint feasible carb/fat polygon

getV83AdaptiveTodayMacroTarget(...)
- adjusts carbs and fat inside the same kcal budget
- protein is clamped first
- fat is recomputed from target kcal after carb moves
- if fat breaches range, fat is clamped and carbs are recomputed
- macroKcalGap must stay <= 2 kcal
```

So the implementation already knows that carb/fat exchange is calorie-linked.
The open risk is display interpretation, not basic arithmetic.

## 4. Why 1:1 exchange is impossible

Carbohydrate and fat do not have the same kcal density:

```text
carbohydrate: 4 kcal/g
protein:      4 kcal/g
fat:          9 kcal/g
```

At a fixed calorie target and fixed protein:

```text
fat -1g frees about carb +2.25g
carb +1g requires about fat -0.44g
```

Therefore:

```text
carb +1g <-> fat -1g
```

is not a valid exchange rule.

The correct exchange is calorie-equivalent, not gram-equivalent.

## 5. Concrete example

Example based on the current screenshot/session case:

```text
target kcal:    2549 kcal
target protein: 127.4g
target carbs:   331.5g
target fat:      79.3g

displayed carb range: 292g~448g
displayed fat range:   52g~94g
```

Protein kcal:

```text
127.4g * 4 = 509.6 kcal
```

If a user reads the chips as a joint rectangle and picks:

```text
carbs 448g
fat    52g
protein 127.4g
```

Total macro kcal becomes:

```text
448*4 + 52*9 + 127.4*4
= 1792 + 468 + 509.6
= 2769.6 kcal
```

That is about:

```text
2769.6 - 2549 = +220.6 kcal
```

above the target.

If protein stays 127.4g and carbs are 448g, the fat that fits 2549 kcal is:

```text
(2549 - 127.4*4 - 448*4) / 9
= 27.5g
```

That is below the displayed fat lower bound of 52g.

If fat is fixed at the displayed lower bound 52g, the carb amount that fits 2549 kcal is:

```text
(2549 - 127.4*4 - 52*9) / 4
= 392.9g
```

That is below the displayed carb upper bound of 448g.

Conclusion:

```text
292g~448g carbs and 52g~94g fat can both be useful marginal ranges,
but they cannot mean every carb/fat corner is valid at the same time.
```

## 7. Options considered

### Option A. Keep current chips as-is

Meaning:
- Do nothing.
- Treat carb and fat chip values as already obvious marginal ranges.

Pros:
- No UI churn.
- No code risk.

Cons:
- The user can reasonably read the chips as jointly valid.
- It ignores the specific concern that triggered this review.
- It leaves a known semantic mismatch to be rediscovered later.

Decision:

```text
Reject as complete answer.
```

### Option B. Decide the joint carb/fat allocation model first

Meaning:
- Do not implement a UI/copy workaround yet.
- Decide how carb and fat targets/ranges should move on a shared kcal budget.
- Decide whether protein is fixed first, clamped first, or part of the exchange.
- Decide whether the card shows marginal ranges, remaining feasible ranges, or a movable target point.
- Decide how current intake, exercise context, and 3-day/7-day history limit the movement.

Candidate model to evaluate, not yet final:

```text
reserve protein target/range first
keep total kcal budget fixed
move carb/fat on an iso-calorie line:
  4 * carbs + 9 * fat = remaining kcal after protein
bound the movement with physiological/scoring ranges and 3-day/7-day pressure
```

Pros:
- Answers the actual math/product question.
- Prevents explanatory copy from becoming a bandage over a model ambiguity.
- Gives implementation a testable source of truth.
- Keeps the user's product direction intact: targets can move, but not as an unlimited excuse.

Cons:
- Requires one more model decision step before implementation.
- Needs concrete examples rather than vague "range-aware" language.

Decision:

```text
Accept as the next gate.
```

### Option C. Explain the mismatch to the user

Meaning:
- Keep current ranges and add user-facing text about carb/fat exchange.

Pros:
- Low implementation risk.
- May reduce immediate confusion.

Cons:
- It skips the deeper question: what should the moving target/range model actually be?
- It asks the user to understand internal arithmetic that the app should handle.
- It is the exact kind of too-conservative patch that created previous inertia.

Decision:

```text
Reject as a solution.
No explanation-only shortcut.
```

### Option D. Narrow the displayed ranges so all corners are feasible

Meaning:
- Reduce the displayed carb/fat ranges until the whole displayed rectangle fits kcal constraints.

Pros:
- Simple to explain mathematically.

Cons:
- It can become another UI-only display cap.
- It may hide the real scoring boundary.
- It repeats the earlier mistake of changing display to avoid explaining the model.

Decision:

```text
Reject unless scoring and display source are changed together by a separate explicit decision.
```

### Option E. Move the burden to DailyCoach

Meaning:
- Keep card as-is.
- Let DailyCoach talk around the carb/fat exchange behavior.

Pros:
- Card stays very clean.
- DailyCoach can reflect practical outcomes.

Cons:
- It still asks copy to carry a model problem.
- It does not answer how targets and ranges should move together.

Decision:

```text
Reject as a solution.
DailyCoach may reflect the final result later, but it must not carry internal exchange math.
```

## 9. Accepted direction

Accepted:

```text
Option B.
```

Meaning:
- The next step is to define the carb/fat joint allocation model.
- The model must decide how target carbs and target fat move together on the same calorie budget.
- The model must decide whether displayed ranges are marginal ranges, remaining feasible ranges, or a movable target point plus bounds.
- The model must decide how 3-day/7-day high-carb or high-fat pressure restricts movement.
- The app should produce coherent target/range outputs without requiring the user to understand calorie exchange arithmetic.
