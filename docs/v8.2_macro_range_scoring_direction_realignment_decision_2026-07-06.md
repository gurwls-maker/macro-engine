## 1. 왜 재정렬이 필요한가

초기 v8.2 문제 정의는 고정 g 목표 기반 점수의 의미 문제였다.

핵심:

```text
fat target exact miss
carb/fat exchange 설명 불가
targetCal / protein은 맞았는데 fat low + carb high로 과도하게 나쁜 식단처럼 보임
copy-only 완화로는 score / Coach / Recent 의미가 닫히지 않음
```

따라서 v8.2의 본류는 다음이다.

```text
fixed macro target score를 계속 유지하며 비교 필드를 늘리는 것
=> 아님

range-aware score policy를 확정하고 production score로 전환하는 것
=> 맞음
```

`scoreDeltaPreview`는 이 본류를 검증하기 위한 보조 비교 도구일 수는 있지만, 새 점수 체계의 필수 구성요소는 아니다.

## 2. 유지할 본류

아래 라인은 계속 유지한다.

```text
component pointsPreview formula 후보
candidateScorePreview sum / clamp / rounding 후보
dataQuality eligibility gate
positive bonus 금지
fractional / non-finite / string numeric input rejection
blocked / unresolved null
storage / UI / Recent / DailyCoach no-leak
```

이유:

```text
component pointsPreview는 새 score의 component-local contribution 후보를 정의한다.
candidateScorePreview는 component contribution을 하나의 range-aware candidate total로 합산하는 후보를 정의한다.
이 둘은 fixed score와 비교하기 위해서가 아니라, 새 production score 자체를 만들기 위한 본체 후보다.
```

현재 유지할 formula 후보:

```text
kcal / protein / fat / carbExchange pointsPreview:
integer <= 0 또는 null

candidateScorePreview:
clamp(100 + sum(kcal/protein/fat/carbExchange pointsPreview), 0, 100)

dataQuality:
score component 아님
합산 제외
pointsPreview null-only
```

## 3. 재분류할 라인

`scoreDeltaPreview`는 아래처럼 재분류한다.

```text
기존 해석:
candidateScorePreview 이후 자연스러운 다음 numeric formula gate

재정렬 후 해석:
optional audit / migration impact / debug comparison 도구
production score 전환의 필수 prerequisite 아님
```

허용 가능한 역할:

```text
release 전 영향 분석
fixed score와 range-aware candidate의 차이 샘플 확인
마이그레이션이 필요한 앱에서 old/new score impact audit
debug/report-only comparison
```

계속 금지:

```text
scoreDeltaPreview numeric formula test design을 다음 필수 단계로 진행
comparison.scoreDeltaPreview production number 구현
record / goalSnapshot / backup 저장
Recent aggregation
Today / Records / Recent UI 표시
DailyCoach action / reason 사용
production score readiness 판단의 blocker로 사용
```

## 5. old fixed Records 정책 재정의

기존 no-mutation guard가 보호한 것:

```text
saved adherencePercent
saved adherenceScoringVersion
goalSnapshot
backup schema
Recent fixed-basis aggregation
Records object mutation
DailyCoach production signal
```

이 보호선의 의미:

```text
production 전 안전 가드
조용한 의미 변경 방지
test-local / report-only preview가 저장값으로 새지 않게 하는 경계
```

이번 realignment에서 닫는 것:

```text
old fixed Records no-mutation은 영구 제품 요구가 아니다.
실사용 데이터 보존 요구가 약한 앱에서는 old fixed 비교/보존보다 새 scoring version 전환이 우선될 수 있다.
old fixed Records를 계속 비교 대상으로 끌고 가기 위해 scoreDeltaPreview를 필수화하지 않는다.
```

향후 production 전환에서 선택할 수 있는 경로:

```text
1. 새 Records부터 range-aware scoring 적용
2. 기존 Records를 fixed basis로 남기되 UI에서 구분
3. 실사용 데이터가 없거나 보존 부담이 낮으면 기존 샘플/로컬 기록을 재생성 또는 재계산
4. migration audit가 필요할 때만 scoreDeltaPreview류 comparison을 별도 debug 도구로 사용
```
