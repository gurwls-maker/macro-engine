# v8.3 continuous scoring test design

작성일: 2026-07-07
변경 성격: docs-only test design
구현 여부: code implementation 없음
점수 숫자 확정 여부: 없음
용도: v8.3 anchor-based continuous scoring 구현 전 테스트 설계

## 1. 문서 목적

이 문서는 구현 지시서가 아니다.
이 문서는 exact score 표가 아니다.
이 문서는 `docs/v8.3_fixture_direction_table_2026-07-07.md`의 fixture 방향을 실제 자동 테스트로 옮길 때 무엇을 검증해야 하는지 닫는 문서다.

닫는 것:
- future suite 후보와 test group.
- fixture별 assertion type.
- 기존 v8.2 exact contribution test가 어떤 방식으로 supersede되어야 하는지.
- alcohol kcal와 physiology risk가 둘 다 반영되는지 검증하는 방법.
- exercise를 final score 보너스가 아니라 carb context로 검증하는 방법.
- near-zero tolerance, monotonicity, no plateau, no hard cap, null gate 검증 방식.
- implementation branch에서 같이 갱신해야 할 기존 suite 목록.

닫지 않는 것:
- final penalty curve coefficient.
- alcohol 56g의 exact point.
- fat upper 2.0x의 exact point.
- carb 8~10g/kg의 exact point.
- `index.html` formula implementation.
- `ADHERENCE_SCORING_VERSION` 변경.
- storage/schema/backup/Recent/UI/DailyCoach 확장.
- scoreDeltaPreview는 열지 않음.

내 판단:
- 이 단계에서 test-only suite를 먼저 구현해 현재 v8.2 code에 통과시키는 방식은 부적절하다.
- 이유는 현재 `index.html`의 production score가 fixed contribution table을 실제로 사용하고 있고, v8.3 테스트는 그 구조를 supersede해야 하기 때문이다.
- 따라서 다음 구현 브랜치는 "테스트 기대값 갱신 + v8.3 formula implementation"을 같은 implementation gate에서 다루는 편이 맞다.
- 다만 이 문서는 그 implementation gate 전에 어떤 테스트가 필요한지 상세히 고정한다.

## 2. 실제 repo 테스트 구조 확인

확인한 현재 구조:

  - `test:daily-coach`: `runDailyCoachTestCases` 단일 suite를 실행한다.

  - `core` profile 안에 `runAdherenceScoringTests`, `runMacroRangeProductionScoreTransitionDecisionTests`, `runMacroRangeProductionScoreImplementationDecisionTests`, 기존 macro range guard suite들이 들어 있다.
  - runner는 `index.html?codexInternalTests=1`을 로드하고 `window.run*Tests` 함수를 수집해 실행한다.
  - `runDailyCoachTestCases`는 `run*Tests` 이름 규칙과 별도로 수집된다.
- `index.html`
  - `ADHERENCE_SCORING_VERSION`은 현재 `MACRO_RANGE_PRODUCTION_SCORING_VERSION`을 가리킨다.
  - `MACRO_RANGE_PRODUCTION_SCORING_VERSION`은 현재 `"v8.2_macro_range_score_v1"`이다.
  - `getMacroRangeProductionScoreSummary(balance)`는 `kcal/protein/fat/carbExchange` fixed contribution을 더해 `100 + sum(points)`로 percent를 만든다.
  - `getDailyAdherenceScore(...)`는 위 summary를 production path로 사용하고, blocked state는 `percent: null`, `scoringVersion: null`로 반환한다.
  - `runMacroRangeProductionScoreImplementationDecisionTests`는 현재 v8.2 exact fixture와 component contribution 값을 직접 검증한다.

중요한 결론:
- v8.3 implementation은 새 suite만 추가하면 충분하지 않다.
- 기존 `runMacroRangeProductionScoreImplementationDecisionTests`의 exact v8.2 fixture는 v8.3에서 direction/curve assertion으로 대체되어야 한다.
- `runAdherenceScoringTests`와 `runDailyCoachTestCases`도 user-facing score/reason behavior가 바뀌는지 확인해야 한다.
- `test:macro-policy`, `test:core`, `test:daily-coach`는 구현 브랜치의 최소 검증 범위다.

## 4. Assertion type catalog

### T01 monotonicity

같은 조건과 같은 축에서 risk가 커지면 점수는 같거나 낮아져야 한다.
fixture direction table에서는 대부분 "더 낮아야 한다"로 둔다.

검증 예:
- `alcohol 80g`은 `alcohol 56g`보다 높으면 실패.
- `fat upper 3.0x`는 `fat upper 2.0x`보다 높으면 실패.
- `totalKcal/TDEE 2.0+`는 `totalKcal/TDEE 1.5`보다 높으면 실패.

테스트 구현 메모:
- 같은 맥락을 유지해야 한다.
- training context가 바뀐 carb fixture는 단순 monotonicity로만 비교하지 않는다.

### T02 near-zero tolerance

작은 이탈은 작은 변화여야 한다.

검증 예:
- target +1kcal은 큰 감점이면 실패.
- fat upper +1g은 큰 감점이면 실패.

테스트 구현 메모:
- exact 점수는 닫지 않는다.
- 대신 baseline과의 차이가 "작다"는 tolerance를 둔다.
- tolerance 숫자는 implementation decision에서 정하되, v8.2 bucket급 큰 jump는 실패로 둔다.

### T03 no bucket plateau

같은 status label 안에서 점수가 멈추면 실패다.

검증 예:
- alcohol 56g, 80g, 112g, 140g이 같은 점수면 실패.
- fat upper 1.2x, 2.0x, 3.0x가 같은 점수면 실패.

폐기:
- high = -10 고정 감점표는 v8.3 production body로 쓰지 않는다.
- severe = -16 고정 감점표는 v8.3 production body로 쓰지 않는다.

### T04 no intermediate cap

중간 score cap은 금지한다.
마지막 0~100 clamp만 표시 범위로 허용한다.

검증 예:
- "alcohol penalty 이후 최대 60점" 같은 중간 cap으로 더 큰 입력 차이가 사라지면 실패.
- "fat penalty 이후 최대 70점" 같은 중간 cap으로 TDEE overload가 가려지면 실패.

### T05 curve-mediated collapse

hard zero threshold primary policy는 금지한다.
하지만 여러 연속 penalty가 커져 자연스럽게 final score가 0점 근처로 내려가는 것은 허용한다.

검증 예:
- fat 1000g + extreme TDEE overload는 매우 낮아질 수 있다.
- alcohol 140g + large TDEE overload는 매우 낮아질 수 있다.
- 특정 단일 threshold를 넘었다는 이유만으로 즉시 0점이면 실패.

### T06 null gate

계산 불가와 낮은 점수는 다르다.

검증 예:
- no meals: `percent === null`
- one meal pending: `percent === null`
- snapshotless/range missing: `percent === null`
- sentinel 0은 실패.

### T07 axis separation

서로 다른 생리학적 risk 축은 분리해서 본다.

검증 예:
- alcohol kcal는 totalKcal/TDEE overload에 반영된다.
- alcoholG/standard drinks/intensity는 `alcoholPhysiologyPenalty`에 반영된다.
- 이 둘 중 하나가 완전히 꺼지면 실패.

### T08 context modifier, not bonus

exercise bonus는 금지한다.
운동은 finalScore에 더하는 보너스가 아니라 carb curve context다.

검증 예:
- 같은 carb 450g에서 REST와 HIGH_VOLUME_OR_LONG_SESSION은 carb curve 해석이 달라질 수 있다.
- 하지만 운동했다는 이유만으로 final score에 fixed plus를 더하면 실패.
- high-volume이어도 totalKcal/TDEE extreme overload가 있으면 낮아질 수 있다.

### T09 no preview/storage leak

v8.3 scoring implementation은 preview field 저장/노출을 열지 않는다.

검증 예:
- `candidateScorePreview`, `pointsPreview`, `scoreDeltaPreview`, `rangeScoreCandidateSummary`가 record/backup/Recent/UI/DailyCoach payload로 새면 실패.
- `scoreDeltaPreview`는 optional audit-only이며 scoring mainline이 아니다.

## 9. 최종 판단

수용:
- fixture direction table을 test design의 선행 입력으로 사용한다.
- v8.3 score는 anchor-based continuous penalty로 간다.
- alcohol은 kcal axis와 physiology risk axis를 둘 다 반영한다.
- 운동은 score bonus가 아니라 carb curve context다.
- protein shortage는 강하게, protein excess는 맥락적으로 완만하게 본다.
- fat/carb는 exchange와 TDEE overload를 함께 본다.
- null gate와 no sentinel 0은 유지한다.
- scoreDeltaPreview는 optional audit-only로 유지한다.

폐기:
- fixed penalty table.
- exact v8.2 contribution table.
- candidate fixed sum/clamp formula.
- 중간 score cap.
- hard zero threshold primary policy.
- exercise bonus.
- v6.1 alcoholImpactPenalty 후처리 부활.
- scoreDeltaPreview mainline 금지.

통합:
- v8.2의 range-aware 문제의식은 유지한다.
- v8.2의 exact contribution implementation은 v8.3 continuous scoring으로 supersede한다.
- existing production score path는 `getDailyAdherenceScore` 중심으로 유지하되, 내부 formula를 continuous penalty path로 바꿀 후보로 본다.
- existing Records contract는 `adherencePercent`/`adherenceScoringVersion` 중심으로 유지하되, preview field를 저장하지 않는다.

보류:
- implementation entry decision.
- actual formula implementation.
- exact coefficient decision.
- storage/schema/UI/Recent/DailyCoach 확장.
- old records migration/recompute/reset.
- v8.2 archive 이동.

내 판단 다음 단계:
- 추천: v8.3 implementation entry decision docs-only 또는 사용자가 명시 승인하면 implementation branch로 진입.

- 현재 repo runner에 pending/red suite 개념이 없으므로, 실패하는 future suite만 먼저 등록하는 방식은 피한다.
