## 비개발자용 설명

### 무엇을 바꿨는지

앱 화면과 점수 계산은 바꾸지 않았다. 대신 이전에 연구 후보로 남겼던 “항목점수를 모두 곱하는 방식”이 정말 괜찮은지 다시 반증했고, 같은 검사를 다른 컴퓨터에서도 반복할 수 있는 전용 도구를 저장소에 남겼다.

도구는 다음을 한 번에 확인한다.

- 탄수화물, 단백질, 지방, 총칼로리 점수가 모두 비슷할 때 최종점수가 어떻게 되는지
- 한 항목만 매우 나쁠 때 심각도가 보존되는지
- 탄수·지방 조합 감점이 임계점에서 갑자기 생기거나 사라지는지
- 앱이 만든 목표를 그대로 섭취한 54개 조건의 현행 점수
- 개인정보를 제거한 실제 54일 기록에서 후보별 점수 분포

### 왜 바꿨는지

이전 곱셈형 후보는 한 항목이 매우 나쁜 경우를 잘 잡았지만, 모든 항목이 조금씩만 나쁜 경우를 지나치게 무너뜨렸다.

```text
95 / 95 / 95 / 95 -> 81.451점
90 / 90 / 90 / 90 -> 65.610점
80 / 80 / 80 / 80 -> 40.960점
```

각 항목의 90점이 “대체로 잘 맞음”을 뜻한다면 최종 66점은 같은 점수 의미를 지키지 못한다. 따라서 raw product를 유일한 실용 후보로 남겼던 이전 결론은 반증됐다.

다른 후보도 바로 채택할 수 없었다. 실제 매우 부족한 날은 현행 0점, raw product 약 20점이었지만 soft-min은 약 69점, minimum residual은 약 58점이었다. 중간점수는 자연스럽게 보존했지만 여러 실제 부족을 너무 약하게 합칠 가능성이 드러났다.

### 실제 사용자 화면이나 계산 결과가 어떻게 달라지는지

이번 작업으로 달라지는 것은 없다.

- 오늘의 식단 점수는 계속 현행 9개 연속 감점을 100점에서 빼서 계산한다.
- 탄수화물·단백질·지방·총칼로리 타일도 계속 감점값을 표시한다.
- 탄수·지방 조합 감점도 아직 현행 방식으로 계산된다.
- 저장된 과거 점수와 오늘 점수를 다시 계산하지 않는다.

다만 다음 점수 관련 구현 순서는 바뀐다. 항목점수 UI를 먼저 만드는 것이 아니라, 앱 목표를 그대로 먹어도 감점되는 현재 정합성 문제를 먼저 고쳐야 한다.

### 정책, 산식, 데이터 해석이 바뀌었는지

production 정책과 산식은 바뀌지 않았다. 이번에 바뀐 것은 후속 판단이다.

1. raw product는 폐기한다.
2. soft-min과 minimum residual은 연구 후보로만 남기고 선택하지 않는다.
3. 탄수·지방 조합을 유지한다면 개별 탄수·지방·칼로리가 이미 설명한 문제를 다시 처벌하지 않고, 공동배분 예산에서 설명되지 않는 잔여 이탈만 보도록 한다.
4. 현재 21/54 target mismatch는 미래 항목점수 구현의 부속 문제가 아니라 현행 점수의 독립 release blocker로 다룬다.
5. 숫자가 극단적이라는 이유만으로 영양 점수를 하나 더 만들지 않는다. 데이터 확인 상태와 영양 평가는 분리한다.

### 바꾸지 않은 범위와 보류한 내용

- `index.html` production 경로를 수정하지 않았다.
- `ADHERENCE_SCORING_VERSION`을 바꾸지 않았다.
- Today UI와 DailyCoach를 바꾸지 않았다.
- storage, schema, backup, Records를 바꾸지 않았다.
- 과거 기록을 새 산식으로 재계산하지 않았다.
- soft-min의 지수와 minimum residual의 계수는 production 정책으로 승인하지 않았다.
- joint residual의 정확한 곡선과 반올림도 구현하지 않았다.

이번 단계에서 사용자가 새 점수 공식을 고를 필요는 없다. 큰 방향은 다음처럼 정리됐다.
