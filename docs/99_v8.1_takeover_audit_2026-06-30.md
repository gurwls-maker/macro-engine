# v8.1 인수인계 감사

작성일: 2026-06-30

이 문서는 v8.0 final 이후 다음 개발자가 앱을 어디까지 믿어도 되는지, 어디를 조심해야 하는지, 무엇부터 열어야 하는지 파악하기 위한 인수인계 감사 문서다.

이번 작업은 기능 개발이 아니다. `index.html` 동작, 산식, 문구, UI, 저장 schema는 변경하지 않는다.

이 문서의 위치:

- 실제 동작 판단의 source of truth는 `v8.0` tag가 가리키는 HEAD 코드와 통과한 테스트다.
- 다음 작업 착수 전 위험 분류, 금지선, 우선순위 판단은 이 감사 문서를 우선 참고한다.

## 1. 기준점

| 항목 | 현재 기준 |
| --- | --- |
| 감사 브랜치 | `audit/v8.1-takeover` |
| 기준 커밋 | `3492bae v8.0 final 릴리스` |
| 기준 태그 | `v8.0` |
| canonical source | Git remote + `v8.0` tag |
| zip | canonical source가 아니다. 전달용 복사본일 수 있으나 기준 판단은 Git tag/remote로 한다. |
| working tree | 감사 시작 시 clean |
| `git diff --check` | 감사 시작 시 통과 |

확인 명령:

```powershell
git status -sb
git log --oneline --decorate -8
git tag --list "v8.0*"
git diff --check
```

주의:

- `v8.0` tag는 안정 기준점이다. 수정하거나 덮어쓰지 않는다.
- 감사 문서 커밋은 release cut이 아니므로 새 release tag를 찍지 않는다.
- `git reset --hard`, `git clean -fd`, `git clean -fdx`는 기본 절차가 아니다.
- destructive command는 disposable zip 검증 디렉터리에서만, 별도 허가 후 제한적으로 쓴다.
- 실제 작업 repo에서는 remote/tag checkout과 clean 상태 확인을 우선한다.

## 2. 개발환경 재현성

현재 repo에는 `package.json`, `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`가 없다.

테스트 도구는 `tools/render_audit/*.cjs`에 있으며 Node.js와 Playwright를 사용한다. 현재 스크립트는 repo `node_modules`, 실행 Node 주변 `node_modules`, Codex runtime cache 후보를 찾아 `NODE_PATH`를 구성한다. 이 구조 덕분에 현재 Codex 환경에서는 테스트가 돌지만, 새 컴퓨터에서 동일하게 재현된다는 보장은 약하다.

감사 중 확인한 명령:

```powershell
node tools/render_audit/run_internal_tests.cjs --list-profiles
node tools/render_audit/run_internal_tests.cjs --profile=smoke
node tools/render_audit/run_internal_tests.cjs --profile=mobile --mobile
node tools/render_audit/run_internal_tests.cjs --suite=runTargetMacroProductionPolicyTests,runAuthoritativeTargetCalTests,runTargetCalSemanticGuardTests,runDualBasisProductionTests,runModeFreeCalculationContextTests
node tools/render_audit/analyze_render_audit.cjs
```

감사 중 결과:

| 명령 | 결과 |
| --- | --- |
| `--list-profiles` | smoke/core/ui/mobile/calibration/full 프로필 목록 출력 성공 |
| smoke | 19 suites, 180 cases, failed 0 |
| mobile 390px | 19 suites, 145 cases, failed 0 |
| targetCal/macro/mode-free targeted suites | 5 suites, 41 cases, failed 0 |
| render audit analyze | 70 captures, failed 0 |

새 컴퓨터에서 막힐 수 있는 지점:

- Playwright 설치 위치가 repo에 고정되어 있지 않다.
- 브라우저 channel은 `msedge`, `chrome`, bundled Playwright 순으로 시도하지만, 새 환경에서는 브라우저 설치 상태에 따라 실패할 수 있다.
- render audit capture는 오래 걸리고 생성물이 ignored 경로에 쌓인다.
- render audit analyze 결과는 같은 source 시점에서 생성된 manifest/captures일 때만 UI 증거로 인정한다. source 변경 후에는 capture부터 다시 실행해야 한다.
- full exported browser tests는 release gate 성격이다. 작은 문구 작업마다 돌릴 필요는 없지만, release 전에는 다시 돌려야 한다.

개선 후보:

- `package.json`과 npm scripts 추가 여부 검토.
- Node/Playwright/브라우저 channel 설치 절차 문서화.
- Windows PowerShell 기준 QA 명령을 README에 고정.
- `run_internal_tests.cjs --list-profiles`가 Playwright 없이도 완전히 안전하게 동작하는지 확인.
- render audit capture/analyze의 실행 시간, 생성물, 실패 대응 절차 문서화.

단, `package.json` 도입은 앱 동작 변경은 아니더라도 개발환경 기준을 바꾸는 작업이다. 감사 문서 커밋과 섞지 않고 별도 판단, 별도 커밋으로 진행한다.

## 4. production / report-only / candidate / legacy / test-only 분류

### Production으로 실제 연결된 것

- `calculate()`의 current runtime.
- 시간 소유권 TDEE production application.
- `targetCal`은 목표와 소비량 기준 authoritative 값.
- `buildDualBasisProduction()`은 guide/activity 내부 basis를 만들지만 `showToggle: false`이며, 사용자-facing mini toggle을 되살리지 않는다.
- Settings의 운동 여부는 `exerciseManagementMode`로 들어가고, 내부 basis 선택은 `getMacroBasisIdForExerciseMode()`가 소유한다.
- `profile_candidate_v2`는 이름은 candidate지만, guard가 통과하면 `buildProfileCandidateV2ProductionWiring()`을 통해 좁은 조건에서 production target/macro로 적용될 수 있다.
- external macro policy는 `buildExternalMacroActiveProductionApplication()`에서 selected authoritative target 안에서 protein/fat/carbs를 재배분한다.
- Records/Recent/Score는 저장 당시 `goalSnapshot`을 해석 기준으로 쓴다.
- full backup / records-only import / smart restore conflict / backup warning modal은 Settings 데이터 관리 흐름의 production 데이터 보호 영역이다.
- Records 상세 수정 식사 삭제는 confirm 후 draft에서만 제거되고, 실제 record는 수정 저장 시점에 반영된다.

### Report-only 감사 코드

아래 계열은 production을 설명하거나 검증하기 위한 감사/비교 레이어다. 존재 자체가 production 적용 승인이 아니다.

- calibration scan/report suites.
- physiology review, weekly budget review, cardio target policy review.
- macro cap candidate/philosophy review.
- formula benchmark, cardio convention reflection review.
- target macro joint policy/expanded scan.
- goal macro principle/dual-basis review.
- V8 scenario runner와 full Cartesian shard/campaign 도구.
- 문서 안의 historical/report-only 설계 기록.

### Candidate 정책/실험 코드

주의할 점은 "candidate"가 모두 같은 의미가 아니라는 것이다.

- `candidate-v8-profile-macro-v2-linked-target-v0`는 runtime proposal과 production wiring이 모두 있다.
- `PROFILE_CANDIDATE_V2_FORMULA_APPROVED_FOR_PRODUCTION = true`이므로, guard가 맞으면 일부 path는 production에 실제 반영된다.
- 반면 많은 older candidate/review block은 report-only 또는 candidate comparison이다.
- 다음 개발자는 candidate라는 단어만 보고 "미적용" 또는 "전부 적용"으로 단정하면 안 된다.

### Legacy compatibility 코드

- `state.mode`, `goalSnapshot.mode`, `MODE_CONFIGS`, `MODE_RUNTIME_PRESETS`는 storage/compatibility/history를 위해 남아 있다.
- `calculate({ runtime: "legacy" })`, `legacyA58`, legacy import/scoring alias는 비교와 호환용이다.
- 숨겨진 legacy Today form/id bank는 stable id와 테스트/렌더 호환용이다.
- legacy 값은 current calculation source로 되살리면 안 된다.

### Test-only/export 코드

- `run*Tests` suite 113개가 `window`로 export된다.
- test sandbox는 app state를 직접 조작한다.
- `window.*` export가 많지만, 모두 사용자-facing API는 아니다.
- raw source grep에서 forbidden copy가 보여도 테스트명/문서/코드 내부 용어일 수 있다. 사용자-facing 판단은 rendered text와 UI tests 기준으로 해야 한다.

## 5. 문서와 실제 구현의 일치/불일치

### 현재 문서와 구현이 맞는 부분

- `v8.0` top comment, title, app-version, backup appVersion은 final 기준이다.
- Today에 guide/activity mini toggle은 사용자 선택지로 노출되지 않는다.
- targetCal authoritative와 diet `targetCal <= totalBurn` guard는 targeted tests에서 통과했다.
- Records 상세 수정 체성분 2자리 정밀도와 unchanged 저장 guard가 smoke에 포함되어 통과했다.
- Records 상세 수정 식사 행 삭제 confirm/draft/save 흐름이 smoke/mobile에 포함되어 통과했다.
- backup warning 기본 노출과 smart restore conflict가 smoke/mobile에 포함되어 통과했다.
- Recent render audit seed와 mobile chart readability가 smoke/mobile에 포함되어 통과했다.
- 문구 기준은 내부 용어를 user-facing에 노출하지 않는 방향으로 정리되어 있다.

### 문서끼리 또는 문서-구현 사이에 헷갈릴 수 있는 부분

1. `docs/README.md`의 문서 우선순위
   - `02_대화의도_근거표_2026-06-16.txt` 설명에는 문서끼리 충돌하면 이 문서와 최신 대화 의도를 우선한다고 되어 있다.
   - 하지만 README의 "충돌 판단 순서"는 `00_현재작업기준`을 `02_대화의도`보다 위에 둔다.
   - 둘 다 최신 HEAD보다 아래에 있으므로 즉시 위험은 낮지만, 다음 개발자에게 문서 우선순위가 모호할 수 있다.

2. `docs/00_현재작업기준_2026-06-16.txt` 안의 과거 실패 서술
   - 문서 중간에는 "운동 안 함에서도 운동인 단백질 정책을 공유하는 문제가 확인됐다"는 과거 감사 문장이 남아 있다.
   - 같은 문서 후반에는 "운동함과 운동 안 함은 서로 다른 탄단지 정책을 사용한다"는 닫힌 기준이 있다.
   - 현재 targeted tests는 macro production policy가 통과하므로, 과거 실패 문장은 historical finding으로 더 명확히 표시하는 것이 좋다.

3. 주제별 참고 문서의 historical/report-only 밀도
   - `v8_운동프로필_수준별산식_통합실행설계_2026-06-04.txt`는 report-only, candidate, scoped production, historical note가 매우 많다.
   - 문서 자체가 위험을 경고하지만, 새 개발자가 중간 절만 읽으면 현재 production 기준으로 오해할 수 있다.

4. candidate-v2 명명
   - 실제 코드에서는 profile candidate v2가 guard 조건에서 production wiring으로 연결될 수 있다.
   - "candidate"라는 이름 때문에 미적용 후보로 오해하기 쉽다.
   - 감사 문서나 후속 문서에서는 "candidate-v2 이름이 남아 있지만 일부 production path에 연결됨"을 분명히 써야 한다.

5. render audit 산출물
   - `manifest.json`, screenshots, pages가 repo working directory에 있을 수 있지만 ignored/generated다.
   - source 변경 후 이전 manifest analyze를 현재 UI 증거로 읽으면 안 된다. capture와 analyze의 source 시점이 맞아야 한다.
   - render audit analyze 결과는 해당 manifest/captures가 같은 source 시점에서 생성되었을 때만 UI 증거로 인정한다. source 변경 후에는 capture부터 다시 실행해야 한다.

## 8. v8.0.1 후보

현재 확정된 v8.0.1 hotfix 항목은 없다.

v8.0.1로만 처리해야 하는 조건:

- 데이터 손실.
- 앱 기동 불능.
- full backup / records-only import / smart restore 파괴.
- Records 저장 또는 상세 수정 저장 오류.
- Today/Records/InBody 값 왕복 중 `goalSnapshot`이 손상되는 오류.
- targetCal이 diet에서 totalBurn을 초과하거나 macro kcal sum이 targetCal과 크게 어긋나는 산식 치명 오류.
- 모바일 390px에서 핵심 입력/저장/복원 경로가 사용 불능.

문서 정리, 도움말, UI polish, 개발환경 문서화는 v8.0.1이 아니라 v8.1 또는 운영 개선으로 둔다.

## 9. v8.1 후보

감사 이후 첫 v8.1 후보는 "설명문 전수조사 + 도움말 체계"가 가장 안전하다.

후보:

- 화면에 항상 보이는 설명 줄이기.
- 계산 근거, 탄단지 범위, basis/운동 여부 설명을 `?` 도움말, 접힘, 상세 영역으로 이동.
- Today / Records / Recent / InBody / Settings 문구 톤 정리.
- 운동함 / 운동 안 함 안내 정교화.
- `carbsBelowGuideMin` 또는 "목표 안에서 조정" 상태의 사용자 이해 보강.
- 온보딩에서 운동 여부만 간단히 묻는 흐름 설계는 v8.1 후반 또는 별도 UX 설계 후보로 둔다. 설명문/도움말 정리 이후 검토한다.
- 모바일 390px에서 도움말, modal, long copy overflow 확인.
- 문구 테스트가 낡은 copy를 고정하지 않도록 의미 중심으로 조정.

v8.1에서 주의할 점:

- 산식과 문구를 같은 커밋에 섞지 않는다.
- 도움말 체계와 targetCal 계산식을 동시에 바꾸지 않는다.
- Records/backup/schema는 문구 작업 중 건드리지 않는다.
