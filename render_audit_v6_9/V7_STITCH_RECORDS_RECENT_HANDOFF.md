# v7.0 Stitch Records / Recent Handoff

작성 기준: 2026-05-21, v6.9 마감 이후 v7.0 UI 재설계 준비용.

## 1. 기록 탭

### 1-1. Stitch가 실제로 작동한 방식

Stitch는 로컬 코드베이스를 읽고 기능 상태를 추적하는 Codex식 도구가 아니다. 웹 기반 디자인 생성 도구에 가까워서, 첨부한 화면과 프롬프트를 보고 화면 후보를 만든다. 그래서 `index.html`의 실제 DOM id, 이벤트 핸들러, 저장 구조, Records goalSnapshot, 식사 수정 모달, 체중/메모 수정 모달, 당시 계산 기준 수정 흐름을 스스로 보존하지 못한다.

또한 한 번에 뽑아낼 수 있는 결과가 최대 4장 수준이라, Records처럼 상태가 많은 탭을 한 번에 완성된 상태 머신으로 만들지 못했다. 사용자는 처음에 여러 상태를 묶어 보여주면 Stitch가 전체 흐름을 이해하고 그 안에서 핵심 4장을 재구성해 줄 수 있을지 기대했지만, 실제로는 contact sheet 안의 전체 상태를 하나의 앱 흐름으로 해석하기보다 그중 일부 타일만 골라 독립 화면처럼 다시 그리는 경향이 있었다.

그 결과 Records는 “목록 → 상세 → 식사 추가 → 식사 수정 → 체중/메모 수정 → 당시 계산 기준 확인/수정 → 직접 관리 fallback”이 이어지는 하나의 제품 흐름으로 나오지 않았다. 대신 서로 다른 앱 후보처럼 보이는 archive 화면, 입력 모달, 설정형 화면, 기록 방식 선택 화면이 섞여 나왔다.

### 1-2. 그로 인해 생긴 문제

Records 탭은 현재 앱에서 기능 책임이 매우 무겁다. 날짜별 기록 보관함이면서, 과거 기록 상세 보기, 식사 추가/수정/삭제, 체중/메모 수정, 당시 계산 기준 수정, Records 공복 체중과 Today 계산용 체중 구분, 기록만 내보내기 문맥까지 품고 있다.

Stitch 결과는 표면적으로는 예뻤지만, 이 많은 기능을 같은 흐름 안에 정리하지 못했다. 특히 아래 문제가 있었다.

- 일부 화면은 Records가 아니라 새 기록 시작 또는 Today 입력 화면처럼 보였다.
- 일부 화면은 modal만 잘라낸 것처럼 보여 앞뒤 맥락이 없었다.
- 일부 화면은 데이터 관리 또는 설정 화면처럼 보여 Records 보관함 역할과 어긋났다.
- 날짜별 기록 목록과 선택 날짜 상세가 같은 제품 리듬으로 연결되지 않았다.
- 식사 추가/수정, 체중/메모 수정, 당시 계산 기준 수정이 서로 분리된 책임으로 보이지 않았다.
- “기록 탭은 확정 기록 보관함/수정 영역”이라는 v7.0 기준보다 “오늘 기록을 새로 시작하는 화면”처럼 보이는 후보가 섞였다.

따라서 Records Stitch 산출물을 그대로 구현 기준으로 삼으면, 현재 앱이 지켜야 하는 기능과 데이터 책임이 흔들릴 가능성이 높다.

### 1-3. 우리가 생각한 의도

Records에서 원한 것은 “예쁜 기록 입력 화면”이 아니라 다음 구조다.

- Records는 Today의 주 입력 공간이 아니라 확정 기록 보관함이다.
- 날짜별 기록 목록이 먼저 보이고, 사용자는 날짜를 골라 상세를 확인한다.
- 상세에서는 식사, 공복 체중, 메모, 점수/목표 대비, 당시 계산 기준을 볼 수 있다.
- 수정은 목적별로 분리된다.
  - 식사 하나 수정은 식사 전용 모달.
  - 공복 체중/메모는 기록 정보 전용 모달.
  - 당시 계산 기준은 별도 영역 또는 별도 편집 흐름.
- Records 직접 관리 fallback은 남기되, 메인 주인공처럼 보이면 안 된다.
- 현재 앱의 저장/수정/백업/goalSnapshot 구조는 유지한다.

즉, Stitch가 만들어 준 그림은 visual shell 참고자료일 뿐, Records 책임 구조는 현재 앱 기준으로 다시 조립해야 한다.

### 1-4. 해결하려고 시도한 방법

처음에는 Records 탭에 필요한 많은 상태를 한 번에 설명하고, Stitch에게 4장씩 여러 번 요청해 샘플을 받아왔다. 의도는 “각 상태를 모두 하나의 디자인 시스템으로 다시 그려 달라”는 것이었다.

받아온 결과는 `_1`부터 `_20`까지 쌓였고, 그 뒤 Codex에서 contact sheet를 만들어 전체를 한 화면에서 비교했다.

확인 방식:

- `records_all_contact_sheet.png`로 전체 흐름을 먼저 훑었다.
- 후보별로 개별 이미지를 열어, 목록/상세/입력/수정/기준 확인에 쓸 수 있는지 분류했다.
- “Stitch처럼 보이는가”와 “현재 앱 기능을 잃지 않는가”를 동시에 봤다.
- 화면이 예뻐도 현재 앱의 Records 책임을 잘못 설명하면 폐기 또는 보조 참고로 낮췄다.

### 1-5. 그로 인한 결과

Records 산출물은 통째로 통과시키기 어렵다. 다만 완전히 버릴 필요는 없다. 몇몇 화면은 카드 톤, 목록 밀도, 날짜 레일, bottom sheet 감각, 계산 기준 표시 방식의 참고자료로 쓸 수 있다.

현재 판정:

- Records 전체 흐름 기준: 불합격.
- Records visual material 기준: 일부 채택 가능.
- 구현 기준: Stitch 산출물을 복붙하거나 전체 구조로 삼지 말고, 선별 이미지 조합으로 재구성해야 한다.

### 1-7. 다음 Codex 구현 지침

Records를 다시 구현하거나 v7.0 기준으로 다듬을 때는 다음 순서를 권장한다.

1. `_17`을 Records 메인 archive 기준으로 삼는다.
2. `_20`을 선택 날짜 상세 기준으로 삼는다.
3. `_15`와 `_18`을 식사 추가/수정 modal tone 기준으로 삼는다.
4. `_6`을 당시 계산 기준 보기 기준으로 삼는다.
5. `_19`는 당시 계산 기준 수정의 tone만 참고한다.
6. 직접 관리 fallback은 Stitch에 없더라도 없애지 말고, Records 메인보다 낮은 계층의 카드로 둔다.
7. 기록만 내보내기 / 전체 백업 문맥은 Settings/Data Management 기준과 충돌하지 않게 둔다.

구현할 때 반드시 보존할 현재 기능:

- 날짜별 기록 목록.
- 선택 날짜 상세.
- 식사 추가 / 수정 / 삭제.
- 공복 체중 / 메모 수정.
- 당시 계산 기준 보기 / 수정.
- 웨이트 세션, 운동시간, 운동강도, 주간 웨이트일수 round-trip.
- 유산소 기준 round-trip.
- Records 체중 → Today 계산용 체중 일회성 반영 prompt.
- Today/Records 체중 불일치 안내.
- 기록 직접 관리 fallback.
- records-only / full backup / smart restore 정책.

### 1-8. Records에서 절대 하면 안 되는 처리

- Stitch에 없다는 이유로 현재 기능을 삭제하지 않는다.
- 기능을 카드 하단의 작은 링크나 접힌 텍스트로 숨겨 접근성을 낮추지 않는다.
- Records를 Today 입력 주 장소처럼 다시 만들지 않는다.
- 식사 수정 모달에 공복 체중, 기록 방식, 당시 계산 기준을 섞지 않는다.
- 당시 계산 기준 수정 화면을 Settings 전체 입력폼처럼 키우지 않는다.
- 기록만 내보내기를 full backup처럼 설명하지 않는다.
- Records 화면 하나를 예쁘게 만들기 위해 저장 구조나 schema를 바꾸지 않는다.

## 2. 최근 흐름 탭

### 2-1. Stitch가 실제로 작동한 방식

최근 흐름도 Records와 마찬가지로 Stitch가 로컬 앱 기능을 직접 이해하고 만든 것은 아니다. 하지만 최근 흐름 탭은 Records보다 상태 구조가 단순하다. 핵심이 “기간 선택 → 요약 → 문제 포인트 → 최근 흐름 Coach → 관련 날짜”라서, Stitch가 만든 화면들이 Records보다 더 한 제품 안의 후보처럼 보였다.

다만 여기서도 한 번에 4장 제한이 있었고, Stitch는 1일/3일/7일/14일/28일 전체 상태를 하나의 완성된 탭으로 연결하지 않았다. 일부 결과는 `7일/1개월/3개월`, `7/14/30`, 영어 문구, Organic Wellness 같은 다른 앱 이름이 섞여 있었다. 그러므로 최근 흐름 산출물도 그대로 구현 기준으로 쓰면 안 되고, 선별해야 한다.

### 2-2. 그로 인해 생긴 문제

최근 흐름에서 생긴 문제는 Records와 다르다. Records는 흐름 자체가 분절된 것이 문제였고, 최근 흐름은 “좋은 화면 후보는 많지만 우리 앱 기준과 다른 기간/문구/데이터 가정이 섞인 것”이 문제였다.

구체적으로는 아래 문제가 있었다.

- 우리 앱의 window는 1/3/7/14/28인데, 일부 Stitch 화면은 7일/1개월/3개월 또는 7/14/30을 썼다.
- 일부 화면은 영어 또는 다른 앱명으로 되어 있어 문구 기준으로 그대로 쓸 수 없다.
- 일부 그래프/수치가 실제 recentContext와 무관한 샘플 데이터처럼 보였다.
- InBody나 Records 연결은 느낌은 좋지만 현재 앱의 책임 분리를 그대로 반영하지 않는다.
- 긴 기간 분석 화면은 좋아 보이지만, 현재 앱의 실제 계산/Coach 로직을 덮어쓰면 위험하다.

### 2-3. 우리가 생각한 의도

최근 흐름에서 원한 것은 “예쁜 그래프 화면”이 아니라 다음 구조다.

- 최근 1/3/7/14/28일 window 기준을 유지한다.
- 기록 부족 상태와 실제 식단 문제를 구분한다.
- 최근 문제는 Records의 근거 날짜와 연결된다.
- 점수/Coach는 기존 recentContext와 Coach 로직을 유지한다.
- 장문 보고서처럼 보이지 않고, 사용자가 최근 흐름과 다음 행동을 빨리 이해한다.
- 그래프는 의미가 있을 때만 쓴다. 예쁘지만 데이터와 맞지 않으면 쓰지 않는다.
- InBody나 체중 흐름은 중립 관찰과 연결하되, 목표별 행동 조언으로 섞지 않는다.

### 2-4. 해결하려고 시도한 방법

최근 흐름도 Stitch에 여러 상태를 나눠 요청했고, 결과를 `_ui_refs/003.스티치/01.모바일/03.최근흐름`에 모았다. 이후 Codex에서 contact sheet를 만들어 한 화면에 놓고, 각 후보가 다음 기준에 맞는지 봤다.

확인 기준:

- 우리 앱의 1/3/7/14/28 window와 맞는가.
- 기록 부족 상태를 과하게 진단하지 않는가.
- 최근 문제 포인트와 관련 날짜 연결에 쓸 수 있는가.
- 그래프가 현재 데이터와 자연스럽게 맞는가.
- recentContext / Coach 로직을 대체하지 않고 담는 shell로 쓸 수 있는가.
- 모바일에서 “보고서”가 아니라 “흐름 확인 화면”으로 보이는가.

### 2-5. 그로 인한 결과

최근 흐름 산출물은 Records보다 훨씬 쓸 만하다. 완성된 구현안으로 바로 통과는 아니지만, v7.0 최근 흐름 기준으로 삼을 후보가 여럿 있다.

현재 판정:

- 최근 흐름 전체 기준: 조건부 채택 가능.
- visual shell 기준: Records보다 더 강하게 참고 가능.
- 구현 기준: `_11`, `_6`, `_7`, `28_*`, `inbody`, `_4`를 조합한다.

### 2-7. 다음 Codex 구현 지침

최근 흐름을 v7.0에서 구현할 때는 다음 순서를 권장한다.

1. 기간 선택은 반드시 1/3/7/14/28을 유지한다.
2. `_11`로 기록 부족 상태의 tone을 잡는다.
3. `_7`을 기록 충분 상태의 핵심 구조 후보로 삼는다.
4. `_6`으로 “최근 흐름 → 오늘 행동” 연결 방식을 참고한다.
5. 28일 화면은 `28_1`~`28_4`를 조합하되, 우리 recentContext가 실제로 제공하는 값만 넣는다.
6. 관련 날짜는 Records 상세/날짜 선택으로 이어지게 한다.
7. InBody 관련 상태는 `inbody` 이미지를 참고하되, InBody 원본 기록과 최근 흐름 Coach 책임을 섞지 않는다.
8. 그래프는 현재 앱 데이터와 맞을 때만 쓴다.

구현할 때 반드시 보존할 현재 기능:

- 1/3/7/14/28 window.
- 기록 부족 상태.
- 최근 식사 기록 품질 판단.
- 단백질 부족, 술, 지방 초과, 회복/운동 관련 recentContext 판단.
- 관련 Records 날짜 연결.
- Weekly / Recent Coach.
- Daily Coach와 recent Coach 역할 분리.
- InBody gate / 중립 관찰 기준.

### 2-8. 최근 흐름에서 절대 하면 안 되는 처리

- Stitch의 7/30/3개월 또는 7/14/30 기간을 그대로 가져오지 않는다.
- 예쁜 그래프를 위해 없는 데이터를 만든다거나 산식/저장 구조를 바꾸지 않는다.
- 기록 부족 상태에서 식단 문제를 확정적으로 말하지 않는다.
- 최근 흐름 Coach를 Daily Coach와 같은 문구로 반복하지 않는다.
- InBody 오래됨을 목표별 행동 조언처럼 바꾸지 않는다.
- 관련 날짜 CTA를 숨기지 않는다.
- 장문 보고서형 텍스트로 다시 돌아가지 않는다.

### 2-9. Records와 다르게 판단해야 하는 이유

Records는 Stitch 산출물이 여러 독립 화면으로 흩어져서 “선별 참고”에 가깝다. 반면 최근 흐름은 Stitch 산출물 중 몇 장이 우리 앱 목표와 꽤 맞았다. 따라서 다음 Codex는 둘을 같은 강도로 다루면 안 된다.

- Records: `_17` / `_20` / `_15` / `_18` / `_6` / `_19`를 조합해서 새 흐름을 직접 설계해야 한다.
- 최근 흐름: `_11` / `_6` / `_7` / `28_*` / `inbody`를 중심으로 visual shell을 더 직접적으로 참고할 수 있다.

이 차이를 놓치면 Records는 Stitch 흐름에 끌려가 기능 책임이 흐려지고, 최근 흐름은 쓸 만한 후보를 과하게 버리게 된다.
