0. 문서 목적
────────────────────────────────────────

이 문서는 탄단지 다이어리 HTML 앱의 최신 개발 기준 문서다.

기존 캔버스_updated.txt는 v6.2.1 기준의 전체 개발 캔버스 역할을 했고, v6.3-A 이후 새 최상위 제품 성격이 확정되었다.

현재 닫힌 UX baseline은 v6.5-UX2-10이다. UX2는 master에 병합 완료된 기준 구조이며, v6.5-UX2-8은 Records / Settings 압축 polish까지 포함한 이전 안정 기준, v6.5-UX2-9는 Today Coach 출력 슬롯 중복 정리, v6.5-UX2-10은 P1 출력 중복 polish까지 포함한 UX 안정 기준으로 본다.

현재 master 기준은 v6.6-A5.11이다. v6.6-A5.11은 guide/activity dual basis를 production 경로에 적용한 산출식 / Today 표시 기준 변경 단계이며, master merge / tag / post-tag 실사용 QA까지 완료된 안정 기준이다.

따라서 이 문서는 기존 캔버스를 단순 보완하는 문서가 아니라, v6.5-UX2-10 이후 개발 판단과 v6.6-A 산출식 QA 판단의 기준이 되는 통합 캔버스다.

이 문서는 구현 프롬프트가 아니다.
이 문서는 앞으로 Codex 프롬프트, GPT 설계 검토, 코드 리뷰, 문구 QA, 로드맵 판단의 기준으로 사용한다.

혹여나 버전이 다르다면, 먼저 최신 기준 문서와 index.html의 실제 구현 차이를 확인한다. 단, 구현 결과가 제품 철학과 충돌하면 구현 결과를 합리화하지 않고 기준 문서와 제품 성격을 우선 검토한다.

다이어트에서 targetCal > totalBurn이 재발하면 P0 계산 의미 회귀로 본다.

[현재 판단]
Stitch UI는 단순 색감 참고가 아니라 강한 시각 목표다.
하지만 Stitch 기능을 앱에 새로 들여오는 것이 아니라, 우리 앱 기능을 Stitch 수준의 화면 밀도, 카드 리듬, 버튼 우선순위, 입력과 결과의 가까운 흐름으로 재배치한다.
책임 분리는 여전히 중요하지만, 그 분리 때문에 화면이 딱딱 끊기거나 관리폼처럼 보이면 사용자 흐름을 우선해 재배치한다.

[QA 기준]
UI 작업은 코드만 보고 확정하지 않는다.
스크린샷으로 실제 렌더 화면을 확인하고, Stitch reference와 비교해 여백, 색, 폰트 크기, 줄맞춤, 카드 높이, 줄바꿈, 버튼/칩 밀도, inline SVG 품질을 본다.
계산 작업은 테스트 통과만으로 확정하지 않고, 생리학 / 스포츠영양학 관점에서 개인 사용자가 보기에 이상한 값이 없는지 함께 본다.

필요하면 향후 구현 단계에서 아래를 허용하는 방향으로 설계한다.

- 기존 localStorage records 초기화
- import/export payload breaking change
- export payload version 변경
- recordMode 명시 저장

단, 사용자가 명시적으로 백업/기존 데이터 보존을 요청한 경우만 별도 고려한다.

4-1. 최상위 정의

[확정]
이 앱은 최근 값을 보여주는 앱이 아니다.
최근 흐름을 믿을 수 있는지 판단한 뒤,

- 오늘 목표 칼로리는 몇 kcal인가?
- 오늘 탄수화물, 단백질, 지방은 몇 g인가?
- 오늘 얼마나 먹었고 얼마나 남았는가?

최종 앱이 답해야 하는 질문:

- 오늘 남은 식사에서 무엇을 해야 하는가?
- 최근 흐름에서 반복되는 문제는 무엇인가?
- 이 흐름을 믿을 만큼 기록이 충분한가?
- 이 산출값은 강한 조언에 쓸 만큼 신뢰할 수 있는가?
- 목표별로 이 상태를 어떻게 해석해야 하는가?
- 사용자는 지금 무엇을 먹고, 무엇을 멈추고, 무엇을 보완해야 하는가?

4-2. 최상위 확정 원칙

[확정]
모든 조언은 오늘 행동으로 끝나야 한다.

[확정]
최근 흐름은 평균보다 반복 빈도, 연속성, 기록 신뢰도가 중요하다.

[확정]
산출값은 직접 기록값 / 계산값 / 모델 추정값 / 추세 추론값 / 코치 해석값으로 등급을 나눈다.
같은 무게로 쓰지 않는다.

[확정]
confidence가 낮으면 코치는 단정하지 않는다.
기록 보완을 먼저 말한다.

[확정]
기본 rolling window는 1일 / 3일 / 7일 / 14일 / 28일이다.

[확정]
v6.2까지의 기존 구현은 이 최상위 제품 성격을 충분히 반영하지 않은 과도기 구조다.

단, 현재 앱의 산식, 기록 신뢰성, 데이터 책임 분리, A5.11 dual basis와 충돌하는 구조는 그대로 가져오지 않는다.
C4.8-S0은 기준문서 정리 단계이며, 아래 방향이 현재 코드에 이미 구현되었다는 뜻은 아니다.

이 앱은 단순한 하루 매크로 계산기가 아니라, 오늘 입력 → 오늘 계산 → 오늘 기록 확정 → 최근 흐름 기반 코치로 이어지는 기록 기반 코치 앱이다.
Today는 단순 계산 탭이 아니라 사용자가 오늘 할 일을 마무리하는 중심 작업공간이다.

[재분류]
기존 "Settings는 계산 기준 source of truth / Today 빠른 수정은 shortcut" 기준은 과도기 기준으로 재분류한다.
최종 방향은 Today가 오늘 계산값을 직접 소유하고, Settings는 매일 Today 값을 채워주는 기본값 / 프로필 / 장기 기준 / 데이터 관리 위치가 되는 구조다.

새 IA 책임:
- Today: 오늘 식사 입력, 오늘 공복 체중 입력 후보, 오늘 운동 / 유산소 / 활동 수정, 오늘 계산값 확인, 오늘 기록 저장 / 갱신, Today Coach를 묶는 중심 작업공간
- Records: 확정된 날짜별 기록 보관함, 과거 기록 보기 / 수정, 당시 계산 기준 확인 / 수정
- 최근 흐름: 최근 3 / 7 / 14 / 28일 흐름과 Weekly / Recent Coach
- InBody: 측정 기록 원본과 중립 관찰, InBody 기록을 Today 몸 상태에 적용하는 후보
- Settings / 나의 기준: 기본 프로필, 기본 생활 패턴, 운동 기준, 유산소 프리셋, 데이터 관리 / 전체 백업
- Onboarding: 처음 앱을 쓸 수 있게 기본값과 첫 Today 값을 채우는 흐름

6-2. Today 책임

핵심 질문:
오늘 남은 식사에서 뭘 해야 하지?

Today는 아래를 먼저 보여준다.

- 오늘 행동
- 오늘 섭취 현황
- 오늘 식단 점수
- 권장 탄단지
- 기록 시작점

Today는 긴 전체 설정폼을 품지 않지만, 오늘 계산에 쓰는 실제 값은 직접 소유한다.
Today 값은 사용자가 오늘 결과를 보면서 바로 수정할 수 있어야 한다.
오늘값은 날짜 단위 working value 후보로 보며, 개발명 후보는 todayCalculationDraft다.
사용자-facing에는 draft라는 말을 쓰지 않는다.

Today 오늘값 후보:
- 계산용 체중
- 골격근량
- 체지방량
- 체지방률
- 오늘 활동량
- 오늘 업무 유형
- 오늘 웨이트 세션
- 오늘 웨이트 시간
- 오늘 유산소 종류 / 시간 / 속도 / 경사
- guide/activity basis 선택
- 목표 칼로리 / 탄단지 / 남은 양 / 식단 점수 기준

초기화 기준 후보:
- 새 날짜에는 Settings 기본값 / 최신 InBody / 기본 프리셋에서 Today 값을 채운다.
- 사용자가 Today에서 수정한 날은 그 날짜의 수정값을 유지한다.
- 다음 날짜에는 다시 Settings 기본값과 최신 측정값을 기준으로 새 Today 값을 채운다.
- schema/localStorage 반영 여부는 후속 구현 단계에서 별도 판단한다.

체중은 사용자-facing에서 최소 세 의미를 구분한다.
- 계산용 체중: Today 목표 칼로리와 탄단지 계산에 쓰는 오늘값이다.
- 오늘 공복 체중: Records에 남기는 기록값이며 최근 체중 흐름과 목표 진행 판단에 쓴다.
- InBody 측정 체중: InBody 측정 결과에 포함된 값이며 InBody 탭에서 중립 관찰로 다룬다.
이 세 값을 모두 "체중"으로만 표시하면 사용자가 지금 무엇을 바꾸는지 헷갈리므로, UI 라벨과 helper에서는 역할을 분리한다.

기록 시작 팝업의 오늘 공복 체중 기준:
- 오늘 공복 체중은 Records 기록값이다.
- 계산용 체중은 Today 계산값이다.
- 기록 시작 팝업은 Today 몸 상태의 계산용 체중을 기본값 후보로 가져올 수 있다.
- 사용자는 팝업에서 오늘 공복 체중을 수정할 수 있다.
- "오늘 계산에도 반영" 체크 ON이면 Records 저장 + Today 계산용 체중에도 반영한다.
- 체크 OFF이면 Records 기록값으로만 저장한다.
- 식단 건너뛰기는 식단 점수 / 탄단지 판단을 만들지 않는다.
- 상세 기록의 식사 입력용 "식사 메모"와 공통 기록용 "오늘 메모"를 구분한다.

6-3. Records 책임

핵심 질문:
이 날짜에 무엇을 기록했고, 코치 판단 근거가 무엇인가?

Records는 입력폼 탭이 아니라 날짜별 기록 보관함이다.
기본 주인공은 날짜 카드와 상세 보기다.
Today는 오늘 기록을 시작하고 갱신하는 작업공간이고, 저장된 결과는 Records의 날짜별 기록으로 남는다.
Records는 확정 기록을 확인하고 수정하는 보관함이다.

Records 기준 요소:

- 날짜 카드
- 자세히 보기 / 접기
- 상세 인라인 편집
- 문제 chip
- InBody 있음 chip
- 당시 목표
- 당시 계산 기준
- 기록 삭제

Records 상세 카드 안의 "수정하기"가 기본 수정 경로다.
하단 "기록 직접 관리 / 날짜 직접 입력 / 백업" 접힘 영역은 기본 수정 경로가 아니라 보조 영역이다.
이 영역은 날짜를 직접 고르거나, 식사를 직접 입력하거나, 백업 / 불러오기를 관리할 때 사용한다.

v6.6-C1 이후 백업 / 불러오기는 Records 보관함 하단에 숨어 있는 보조 기능보다 앱 전체 데이터 관리에 가까운 축으로 재검토한다.
Records는 날짜별 기록 보관 / 상세 보기 / 수정 책임을 유지하고, 백업 / 불러오기는 C4에서 Settings 하단 또는 별도 데이터 관리 섹션으로 옮기는 후보로 둔다.
Records 하단 직접 관리 영역은 장기적으로 날짜 직접 입력과 예외 수정 fallback 역할만 남기는 방향을 검토한다.

Today ↔ Records 기준:
- 현재 구조상 기록만 내보내기는 저장된 날짜별 기록 중심의 경량 내보내기다.
- 현재 Today 계산값이나 Settings 전체 상태를 이어 쓰는 용도는 아니다.
- InBody 기록 포함 여부는 실제 export 범위와 맞춰 문구를 유지한다.
- 전체 백업은 다른 기기 / 브라우저에서도 앱 상태를 이어 쓰기 위한 백업이며, Settings / Today 계산값 / Records / InBody / 유산소 프리셋 등 앱 상태를 포함하는 방향으로 본다.
- 다른 기기에서 앱 상태를 이어 쓰려는 경우에는 전체 백업을 우선 안내한다.
- 최종 구조에서는 사용자가 이 차이를 기억해야 하는 부담을 줄여야 한다.

후속 구현 후보:
- Today에서 오늘 기록 저장 / 갱신 흐름 제공
- Today와 오늘 Records 기록의 계산 기준이 다를 때 "오늘 기록의 계산 기준이 현재 Today 계산값과 다릅니다."처럼 안내
- 선택 action 후보: "오늘 기준으로 기록 갱신", "그대로 두기"
- 기록만 내보내기는 저장된 날짜별 기록 중심의 경량 내보내기임을 명확히 표시
- 전체 백업은 앱 전체 상태를 이어 쓰는 용도임을 명확히 표시

6-4. 최근 흐름 책임

핵심 질문:
최근 흐름에서 반복되는 문제는 무엇이고, 어느 날짜를 확인해야 하는가?

최근 흐름은 최근 1 / 3 / 7 / 14 / 28일 흐름을 확인한다.
문제를 말하는 데서 끝나지 않고 "문제 날짜 보기" 또는 "기록에서 확인"을 통해 Records의 해당 날짜로 연결한다.

detailed 기록만 단백질 부족, 탄수 부족, 고지방, 술 영향, 기타 칼로리 같은 탄단지 원인 날짜로 연결한다.
simple / weight_only는 식단 원인을 단정하지 않고 기록 보완 또는 기록 부족 중심으로 연결한다.

6-5. InBody 책임

핵심 질문:
체중과 체성분 수치가 어떤 흐름을 보이는가?

InBody 탭은 측정 기록 원본과 중립 관찰 탭이다.
아래 역할만 담당한다.

- InBody 기록 입력
- 최신 수치
- 변화 추세
- 중립 관찰
- Records와 연결

최신 InBody 값은 Today 몸 상태 기본값의 출처가 될 수 있다.
Today 값이 비어 있을 때만 최신 InBody 자동 채움 후보로 둔다.
사용자가 Today에서 직접 수정한 뒤에는 자동 덮어쓰기하지 않는다.
새 InBody를 저장했을 때는 "오늘 계산에도 반영할까요?" 선택 후보를 둔다.
Today 화면에서는 길게 설명하지 않고 "출처: 최신 InBody"처럼 짧게 표시하거나 ? / 접힘에서 설명한다.
InBody 측정 체중을 Records 공복 체중처럼 말하지 않는다.

InBody 탭에서는 목표별 행동 조언을 하지 않는다.
다이어트 / 린매스업 / 벌크업에 따른 행동 번역은 Today Coach 범위다.

6-6. Settings 책임

핵심 질문:
나의 기본값과 장기 기준이 맞게 관리되고 있는가?

Settings는 "나의 기준"에 가깝다.
Today 값을 매일 채워주는 기본값 / 프로필 / 장기 기준 / 데이터 관리를 담당한다.
Settings는 계산 입력폼 전체를 Today에서 대신하는 장소가 아니라, 오늘값의 출발점을 관리하는 장소다.

Settings / Onboarding 기본값 후보:
- 목표
- 모드
- 키
- 성별
- 나이
- 기본 활동량
- 기본 업무 유형
- 루틴 종류
- 주당 웨이트일수
- 기본 웨이트 시간 후보
- 기본 유산소 프리셋 후보
- 전문가 설정
- 데이터 관리
- 온보딩 다시 하기

Settings는 계산 기준을 충분히 관리할 수 있어야 하지만, 모바일 첫 화면에서 거대한 입력폼처럼 보이면 안 된다.
접힘 구조와 요약을 통해 필요한 때 펼쳐 쓰는 방향을 유지한다.

활동량 / 업무 유형 기준:
- Settings에는 기본 활동량과 기본 업무 유형을 둔다.
- Today에는 오늘 활동량과 오늘 업무 유형을 항상 표시하고 수정 가능하게 둔다.
- Today 오늘값은 기본적으로 Settings 값을 불러온다.
- 사용자가 Today에서 수정하면 그 날짜에는 수정값을 유지한다.
- 다음 날에는 다시 Settings 기본값을 불러온다.
- 이것은 예외 대응이 아니라 실제 사용 흐름의 기본 구조다.
- 설명문은 길게 반복하지 않고, 필요하면 ?에서 "기본값은 나의 기준에서 가져오고, 오늘만 다르게 바꿀 수 있습니다." 정도로 설명한다.

유산소 기준:
- 유산소는 Today의 오늘 운동값이다.
- Settings에는 기본 유산소 프리셋 후보를 둘 수 있다.
- 매일 같은 유산소를 하는 사람은 Settings 프리셋을 Today 기본값으로 가져온다.
- 매일 다른 사람은 Today에서 그날만 수정한다.
- 기본 유산소 프리셋 후보는 종류 / 시간 / 속도 / 경사다.
- Today 유산소 실제값은 오늘 종류 / 오늘 시간 / 오늘 속도 / 오늘 경사다.
- 프리셋이 있어도 Today 값이 계산 기준이다.
- 유산소가 Settings의 계산 입력처럼 보이면 안 된다.

A5.11 guide/activity basis는 Today 권장 탄단지, 남은 양, 식단 점수 기준을 함께 바꾸는 production 기준이며, mini toggle은 항상 표시 기준을 유지한다.

6-6-1. Today 값 적용 패턴

[확정 / v6.6-C4.8-S0]
프리셋 / 기록 적용 / 기준 유지 / 일회성 반영 / 기본값 시작을 모두 자동 반영처럼 쓰지 않는다.
각 패턴은 아래처럼 분리한다.

1. 프리셋 적용형
- 저장해 둔 프리셋 / 템플릿을 Today 값에 한 번 복사한다.
- 적용 후 바로 직접 수정할 수 있다.
- 프리셋 원본은 바뀌지 않는다.
- 예: 유산소 프리셋.

2. 기록 적용형
- 저장된 원본 기록을 Today 값에 한 번 복사한다.
- 원본 기록은 바뀌지 않는다.
- 최신 기록만 강제하지 않고 특정 기록 선택이 가능할 수 있다.
- 예: 최신 InBody 적용, InBody 기록 선택.

3. 기준 유지형
- 계속 변하는 자동 기준을 따를지, 직접 입력값을 유지할지 선택한다.
- 기준 유지 상태에서는 기준값 변경이 반영될 수 있다.
- 직접 입력 상태에서는 자동 기준이 바뀌어도 현재 입력값을 유지한다.
- 예: 최근 기록 기준 주간 웨이트일수 / 직접 입력.

4. 일회성 반영형
- 특정 순간에 한 값을 다른 값으로 한 번만 복사한다.
- 계속 따라가는 구조가 아니다.
- 예: Records 공복 체중 → Today 계산용 체중.

5. 기본값 시작형
- 새 날짜 / 첫 생성 시 Settings 기본값으로 시작한다.
- 이후 해당 날짜에서 사용자가 직접 수정하면 그 날짜 값은 유지한다.
- 예: 기본 생활 패턴, 기본 운동 기준 후보.

6-7. Onboarding 책임 후보

[후속]
Onboarding은 아직 구현하지 않지만 IA에는 포함한다.
역할은 앱 첫 실행 시 Today 계산이 바로 가능하도록 Settings 기본값과 첫 Today 값을 채우는 것이다.

필수 후보:
- 목표
- 모드
- 키
- 성별
- 나이
- 기본 활동량
- 기본 업무 유형

운동 사용자 후보:
- 루틴 종류
- 주당 웨이트일수
- 기본 웨이트 시간
- 기본 유산소 프리셋

체성분 후보:
- 체중
- 골격근량
- 체지방량
- 체지방률

체성분은 모르면 건너뛸 수 있어야 한다.
InBody 값 있음 / 없음 / 모르겠음 흐름을 후보로 둔다.
온보딩은 구현 후순위지만, Today 중심 IA의 입력 후보에는 포함한다.

6-8. 설명문 최소화 / 전수감사 기준

[원칙]
기본 화면에는 설명문을 기본으로 깔지 않는다.
기본 화면은 라벨 + 값 + 버튼 중심으로 구성한다.
설명은 기본 숨김이 원칙이다.
헷갈릴 가능성이 큰 곳만 ? / 접힘 / 짧은 보조문구로 둔다.
실수하면 큰 영향이 있는 곳만 짧은 경고를 기본 노출한다.

설명문 정리는 바로 삭제 구현이 아니다.
프로세스:
1. Codex가 설명문 전수조사를 한다.
2. 사용자와 GPT가 삭제 / 축약 / ? 이동 / 접힘 이동 / 유지 여부를 판단한다.
3. 그 판단을 바탕으로 한 번에 문구 다이어트를 구현한다.

전수조사 대상:
- helper
- hint
- mini-note
- desc-box
- field-note
- notice
- warn
- details
- tooltip
- empty state
- card subtitle
- button 주변 설명문
- 기록 시작 팝업 설명
- Settings 설명
- Today 계산값 설명
- InBody 설명
- Records 하단 설명

전수조사 표 항목:
- 위치
- 현재 문구
- UI 위치
- 기본 노출 여부
- 문구 길이
- 없으면 오해할 위험
- 사용자가 직접 행동하는 데 필요한지
- 추천 분류

추천 분류:
- 유지
- 삭제 후보
- 축약 후보
- ? 이동 후보
- 접힘 이동 후보
- 경고로 유지
- 라벨로 대체

삭제 또는 축약 후보 예:
- 오늘 결과에 바로 반영됩니다
- 수정할 수 있습니다
- 자세한 설정에서 관리합니다
- 최근 InBody 기준으로 채웠습니다
- 불필요하게 긴 카드 설명

유지 또는 ? 이동 후보:
- 계산용 체중과 오늘 공복 체중 차이
- 오늘 계산에도 반영 체크박스
- guide/activity 기준 차이
- 삭제 / 초기화 / 불러오기
- 식단 건너뛰기 결과
- InBody 측정값과 Today 입력값 차이

────────────────────────────────────────
7. 기본 rolling window
────────────────────────────────────────

7-1. 확정 window

[확정]
기본 rolling window는 1일 / 3일 / 7일 / 14일 / 28일이다.

1일:
오늘 행동.
지금 뭘 해야 하나?
오늘 단백질, 술, 초과, 남은 식사 판단.

3일:
단기 흔들림.
최근 며칠 흔들렸나?
술, 회복, 연속 부족/초과 판단.

7일:
반복 패턴.
이번 주 반복되는 문제가 있나?
단백질 반복 부족, 지방 high/low 반복, 술 빈도, 기록 품질 판단.

14일:
진행 추세.
목표 방향으로 가고 있나?
체중 추세, 목표 진행 속도, targetCal 보정 후보 판단.

28일:
장기 후보.
설정/체성분/장기 패턴을 다시 볼 필요가 있나?
체성분 흐름, 유지칼로리 보정 후보, 장기 식단 패턴 판단.

7-2. 5일 window

[보류]
5일 window는 기본 rolling window에서 제외한다.

사유:
- 3일과 7일 사이에서 역할이 겹친다.
- 코치 문구가 중복될 가능성이 크다.
- 생리학적으로 5일만의 독립 의미가 약하다.
- 반복 식단 패턴은 7일이 더 자연스럽다.
- 단기 회복/술/연속성은 3일이 더 자연스럽다.

후속 후보:
- 평일 식단 패턴
- 주중/주말 분리 분석
- 월~금 루틴 분석

7-3. Window별 금지 판단

[확정]
1일 fat low만 보고 지방 보완을 강하게 말하지 않는다.

[확정]
3일 체중 변화로 감량 실패나 벌크 실패를 말하지 않는다.

[확정]
7일 InBody 변화로 근육 증가/감소를 단정하지 않는다.

[확정]
14일/28일 흐름은 자동 보정이 아니라 보정 후보로만 말한다.

────────────────────────────────────────
8. Goal change 처리 원칙
────────────────────────────────────────

8-1. 목표 변경 감지

[확정]
최근 window 안에서 목표 변경이 감지되면, 최근 흐름 결론보다 목표 변경 감지를 먼저 띄운다.

예:
최근 기록 안에서 목표가 바뀌었습니다.
과거 기록은 당시 목표 기준으로 보고, 오늘 행동은 현재 목표 기준으로 안내합니다.

8-2. 이후 처리

목표가 바뀐 window는 아래 원칙을 따른다.

1. 과거 기록 해석은 해당 날짜의 goalSnapshot 기준으로 한다.
2. 오늘 행동 조언은 현재 목표 기준으로 번역한다.
3. window 안에서 목표가 섞여 있으면 pattern confidence를 낮춘다.
4. 같은 목표로 충분한 기록이 쌓이기 전까지 장기 추세 문구를 약하게 한다.
5. 목표 변경 직후에는 “최근 흐름 판단”보다 “현재 목표에 맞춘 오늘 행동”을 우선한다.

8-3. 금지

[확정]
과거 다이어트 기록을 오늘 린매스업 기준으로 무조건 재평가하지 않는다.

[확정]
과거 벌크업 기록을 오늘 다이어트 기준으로 실패처럼 말하지 않는다.

────────────────────────────────────────
9. 데이터 적합성 감사
────────────────────────────────────────

9-1. 데이터 사용 원칙

[확정]
값이 있다는 것과 코치 판단에 쓸 수 있다는 것은 다르다.

각 데이터는 아래 질문으로 평가한다.

- 이 값은 직접 기록값인가, 계산값인가, 추정값인가?
- 누락될 수 있는가?
- 왜곡될 수 있는가?
- 최근 window로 집계해도 의미가 유지되는가?
- 목표별 해석이 달라지는가?
- 강한 행동 조언에 쓸 수 있는가?
- confidence가 낮을 때 어떤 문구로 낮춰야 하는가?

9-2. 핵심 데이터별 사용 기준

protein g:
사용 가능성 높음.
평균보다 부족 빈도 중요.
1일/7일 중심.

carbs g:
조건부.
운동일/휴식일 의미가 다름.
웨이트 기록 있는 날 기준.

fat g:
조건부.
하루 low 과해석 위험.
high는 즉시, low는 반복 조건.

alcoholKcal:
사용 가능성 높음.
술자리 전체 kcal 아님.
양과 빈도 분리.

otherKcal:
사용 가능성 높음.
실패가 아니라 기록 품질 문제.
macro confidence 낮춤.

mealCount:
사용 가능성 높음.
실제 1끼인지 기록 누락인지 불명.
confidence 조절.

adherence score:
조심해서 사용.
평균 점수가 행동과 안 맞을 수 있음.
component 빈도 우선.

weight:
조건부.
수분/염분/술 영향.
14일 이상 중심.

InBody:
제한적.
단일 측정 오차 큼.
28일 흐름 + 중립 관찰.

goalSnapshot:
필수.
없으면 과거 비교 왜곡.
snapshot coverage 필요.

9-3. 기록 해상도 / recordMode 기준

[확정]
상/중/하는 삭제하지 않고 "간단 식단 평가" 또는 "직접 식단 평가"로 재정의한다.
역할은 바쁜 날 기록 공백을 막는 낮은 해상도 식단 기록이다.
단백질 부족 반복, 운동일 탄수 부족, 지방 과다, 목표별 강한 Coach 행동의 단독 근거로 쓰지 않는다.

[확정]
식단 점수 자동 계산 토글은 recordMode 3단계와 역할이 겹치므로 제거한다.
앞으로 식단 기록 권한, 점수 계산 방식, 분석 권한은 recordMode가 결정한다.
detailed + 자동 점수 OFF 같은 중간 상태는 최종 구조에서 사용하지 않는다.

recordMode는 아래 3단계다.

simple:
- 간단 식단 평가 상/중/하
- 식사 상세 입력 불가
- 탄단지 세부 분석 불가
- 기록 지속성 / 낮은 해상도 참고
- Coach의 강한 탄단지 행동 근거로 단독 사용 금지

detailed:
- 식사 상세 입력 가능
- 간단 상/중/하 평가 불가
- 탄수화물 / 단백질 / 지방 / alcoholKcal / otherKcal 분석 가능
- 앱이 식단 점수 자동 계산
- goalSnapshot 저장
- recentContext macro pattern 판단 가능
- Coach / 최근 흐름 / InBody permission의 강한 판단 근거 가능

weight_only:
- 체중만 기록
- 식사 상세 입력 불가
- 상/중/하 평가 불가
- 식단 점수 없음
- 식단 판단 금지
- 체중 흐름에는 사용 가능
- 식단 원인 단정 금지

[확정]
한 날짜에서 simple과 detailed를 동시에 핵심 근거로 쓰지 않는다.
simple 기록일에는 식사 상세 입력을 막고, detailed 기록일에는 상/중/하 직접 평가를 막는 방향을 기본으로 한다.
weight_only는 식단 기록이 없는 체중 중심 기록이다.

전환 기준:
- simple → detailed: 식사 상세 입력을 시작하는 전환이며, 이후 detailed 기준이 우선한다.
- detailed → simple: 식사 상세 기록과 자동 계산 점수 제거 확인이 필요하다.
- detailed → weight_only: 식사 기록과 식단 판단 정보를 제거하는 전환이므로 확인이 필요하다.
- 전환 후 같은 날짜의 simple 평가와 detailed 식사 판단이 동시에 핵심 근거가 되지 않아야 한다.

[폐기]
자동 평가 ON/OFF, 식단 점수 자동 계산 토글, detailed + 자동 점수 OFF 상태는 더 이상 최종 기준으로 보지 않는다.
"자동 점수 계산 OFF는 분석 OFF가 아니다"라는 설명 구조는 이전 과도기 기준이었다.
recordMode 3단계 확정 후에는 토글 자체를 제거하므로 해당 설명도 사용자-facing / 구현 기준에서 사용하지 않는다.

goalSnapshot 기준:
- detailed 기록일: 항상 앱이 식단 점수를 자동 계산하고 goalSnapshot을 저장한다.
- simple 기록일: 당시 목표 맥락은 macro pattern 분석 권한으로 쓰지 않는다.
- weight_only 기록일: 식단 판단용 goalSnapshot은 기본적으로 저장하지 않는다.
- goalSnapshot 있음 = 상세 탄단지 분석 가능으로 단정하지 않는다.
- detailedGoalSnapshotCoverage와 simpleGoalContextCoverage는 분리 검토한다.

manual 90/75/60 권한:
- 상/중/하 환산값 90/75/60은 유지할 수 있다.
- 하지만 자동 adherencePercent와 같은 평균처럼 섞지 않는다.
- manual 90/75/60은 사용자 직접 평가 또는 간단 평가의 낮은 해상도 지표다.
- 자동 adherencePercent는 상세 식사 기록 기반 계산값이다.
- 최근 흐름에서는 상세 기록 평균과 간단 평가 분포를 분리한다.

좋은 예:
- 상세 기록 평균 82점 · 간단 평가 상 1일 / 중 2일
- 최근 7일 중 상세 기록 4일, 간단 기록 2일입니다.
- 단백질 부족이 반복되는지 보려면 상세 기록이 필요합니다.

나쁜 예:
- 최근 식단 점수 82점입니다.
- 자동 점수와 상/중/하를 한 평균처럼 표시
- 간단 평가를 목표 달성률처럼 단정

A2에서 구현된 기록 분류 기준:
- detailedDietDays
- simpleDietDays
- weightOnlyDays
- detailedAutoScoreDays
- simpleManualScoreDays
- detailedGoalSnapshotDays
- macroAnalysisDays
- simpleSummaryDays
- detailedCoverage
- simpleCoverage
- macroPatternAllowed
- simpleSummaryAllowed
- dietCauseAllowed
- Today Coach / InBody 판단 권한과 기록 해상도 연결
- recentContext에서 simple / weight_only가 탄단지 원인 count에 들어가지 않음

권한 기준:
- macro component counts는 detailed 기록 기준으로 만든다.
- proteinLowDays / carbLowTrainingDays / fatHighDays는 simple-only day에서 만들지 않는다.
- simple 기록은 기록 지속성과 낮은 해상도 흐름에는 반영할 수 있다.
- weight_only 기록은 체중 흐름에는 반영하되 식단 원인 판단에는 쓰지 않는다.
- usableDietDays는 detailed / simple / weight_only 기록 해상도를 분리해서 해석한다.

자동 점수 토글 제거 기준:
- 사용자-facing 식단 점수 자동 계산 토글 제거
- autoDietAnalysis 의존 UI 제거
- detailed 기록 저장 시 항상 goalSnapshot 저장
- detailed 기록 저장 시 항상 자동 식단 점수 계산
- simple은 상/중/하 직접 평가만 저장
- weight_only는 식단 평가 관련 필드 제거
- 기존 개발 중 데이터 보존을 이유로 복잡한 legacy migration 추가 금지
- 필요하면 기존 localStorage 초기화 또는 import/export breaking change 허용

후속 확장 후보:
- simpleGoalContextCoverage 고도화
- analysisPermission 세분화
- recordMode schema 정식화
- import/export payload version 고도화

최근 흐름 문구 기준:
- 최근 7일 중 상세 기록 4일, 간단 기록 2일입니다.
- 단백질 부족이 반복되는지 보려면 상세 기록이 필요합니다.
- 탄단지 기록이 부족해 식단 원인은 아직 말하기 어렵습니다.
- 상세 기록이 부족해서 식단 원인은 아직 말하기 어렵습니다.
- 탄단지 기록이 부족해요. 오늘 먹은 주요 식사부터 상세 기록으로 남겨주세요.

Today Coach 기준:
- detailed 기록이 충분하면 기존처럼 단백질 / 탄수 / 지방 / 술 / 기타 칼로리 반복 판단이 가능하다.
- simple 기록만 많으면 강한 탄단지 반복 조언을 하지 않는다.
- weight_only 기록만 많으면 식단 원인을 단정하지 않는다.
- 오늘 상세 식사 기록이 있다면 오늘 입력 기준 행동은 말할 수 있다.

좋은 예:
- 이유: 탄단지 기록이 부족해 식단 원인은 아직 말하기 어렵습니다.
- 행동: 오늘 먹은 주요 식사부터 상세 기록으로 남겨주세요.
- 단백질 부족이 반복되는지 보려면 상세 기록이 필요합니다.
- 체중 기록은 있지만 식사 기록이 부족합니다. 오늘 먹은 주요 식사부터 상세 기록으로 남겨주세요.

schema / import-export phase gate:
- 최종 구조에는 recordMode 명시 저장이 필요할 가능성이 높다.
- recordMode 추가 시 export payload version 변경을 허용한다.
- 기존 백업 파일 호환은 현재 1인 개발 단계에서는 우선순위가 낮다.
- 필요하면 기존 records / localStorage 초기화와 import/export breaking change를 허용한다.
- 정식 사용 단계에서만 migration 정책을 별도 설계한다.
- legacy 보조 처리 / mixed legacy / backward compatibility를 이유로 구조를 꼬지 않는다.

금지:
- mixed legacy를 처리하기 위한 죽은 코드 만들기
- medium-only 기록 구분을 위해 복잡한 legacy_unknown 보조 처리 만들기
- 기존 개발 중 임시 데이터 보존을 위해 recordMode 설계를 흐리기
- schema 변경을 피하려고 추론 helper를 영구 구조처럼 쓰기

9-4. UX2 탭 책임 재정의 / 기록 플로우 기준

[확정]
v6.5-UX2-10은 UX2 탭 책임 재정의와 P1 출력 중복 polish까지 포함한 닫힌 UX 구조 baseline이다.
이 기준은 UI 시안을 따라가는 결과물이 아니라, 기록 기반 코치 앱이라는 제품 성격을 화면 흐름으로 정리한 것이다.

Today:
- 오늘 행동
- 오늘 섭취 현황
- 식단 점수
- 권장 탄단지
- 기록 시작점
- 긴 전체 설정폼을 품지 않음
- 오늘 계산에 쓰는 실제 값은 Today가 직접 소유
- C1 / C2의 빠른 수정 shortcut 기준은 C4에서 과도기 기준으로 재분류
- 최종 방향은 계산용 체중, 오늘 활동량, 오늘 업무 유형, 오늘 웨이트 세션 / 시간, 오늘 유산소 종류 / 시간 / 속도 / 경사처럼 오늘값을 Today에서 항상 확인하고 수정하는 구조
- 새 날짜에는 Settings 기본값 / 최신 InBody / 기본 프리셋에서 Today 값을 불러오고, 사용자가 수정한 날은 그 날짜 수정값을 유지
- 목표 kcal / 탄단지 / 남은 양 / 식단 점수 기준은 Today 선택 basis와 오늘값을 따른다.
- Today Coach 상단 summary는 "오늘 먼저 할 일" 역할을 유지
- 첫 번째 Coach block은 summary와 같은 action을 반복하지 않음
- 첫 번째 Coach block의 상태 / 이유는 유지
- action-only 첫 번째 Coach item은 빈 block 없이 summary에만 표시

Today 기록 시작 기준:
- 버튼: 기록 시작하기
- 제목: 오늘은 어떻게 기록할까요?
- 상세 기록: 탄수화물·단백질·지방까지 입력
- 간단 기록: 오늘 식단을 상·중·하로 남김
- 식단 건너뛰기: 식단 판단 없이 오늘 기록을 남김
- 오늘 공복 체중 / 오늘 메모를 선택적으로 함께 남김
- 상세 기록에서는 식사 입력용 "식사 메모"와 공통 기록용 "오늘 메모"를 구분
- 오늘 공복 체중은 Records 기록값
- 계산용 체중은 Today 계산값
- "오늘 계산에도 반영" 체크 ON이면 Records 저장 + Today 계산용 체중 반영
- 체크 OFF이면 Records 기록값으로만 저장
- 활동량, 업무 유형, 운동 설정 같은 계산 입력은 기록 시작 팝업에 섞지 않음

Records:
- 날짜별 기록 보관함
- 날짜 카드
- 자세히 보기 / 접기
- 상세 보기
- 상세 인라인 편집
- 문제 chip
- InBody 있음 chip
- 당시 목표
- 당시 계산 기준

Today는 오늘 기록을 시작하고 갱신하는 작업공간이고, 저장된 결과는 Records의 날짜별 기록으로 남는다.
Records는 확정 기록을 확인하고 수정하는 보관함이다.
Records 상세 카드 안의 "수정하기"가 기본 수정 경로다.
Records 하단의 "기록 직접 관리 / 날짜 직접 입력 / 백업" 접힘 영역은 메인 수정 경로가 아니다.
하단 영역은 날짜 직접 입력, 식사 직접 입력, 선택 날짜 식사 확인, 백업 / 불러오기 보조 용도다.
백업 / 불러오기는 앱 전체 데이터 관리에 가까우므로 C4에서 Settings 하단 또는 별도 데이터 관리 섹션으로 옮기는 후보로 둔다.

전체 백업 / 기록만 내보내기 기준:
- 전체 백업은 다른 기기 / 브라우저에서도 앱 상태를 이어 쓰기 위한 백업이다.
- 전체 백업은 Settings / Today 계산값 / Records / InBody / 유산소 프리셋 등 앱 상태를 포함하는 방향으로 본다.
- 기록만 내보내기는 저장된 날짜별 기록 중심의 경량 내보내기다.
- 기록만 내보내기는 현재 Today 계산값이나 Settings 전체 상태를 이어 쓰는 용도가 아니다.
- InBody 기록 포함 여부는 실제 export 범위와 맞춰 문구를 유지한다.
- 다른 기기에서 앱 상태를 이어 쓰려면 전체 백업을 우선 안내한다.

Records 날짜별 요약 카드 chip 기준:
- 상세 기록
- 간단 기록
- 식단 판단 없음
- 식사 n건
- 점수
- 체중 있음
- InBody 있음
- 단백질 부족
- 탄수 부족
- 고지방
- 술 영향
- 기타 칼로리
- 기록 부족
- 목표 변경

simple / weight_only 기록에서는 단백질 부족, 탄수 부족, 고지방 같은 탄단지 원인 chip을 만들지 않는다.
이 경우에는 기록 보완 필요, 기록 부족, 식단 판단 없음처럼 기록 해상도에 맞는 표현을 쓴다.

최근 흐름:
- 최근 1 / 3 / 7 / 14 / 28일 흐름을 확인한다.
- 문제를 발견하면 "문제 날짜 보기", "관련 날짜 보기", "기록에서 확인"으로 Records에 연결한다.
- 해당 날짜 카드가 선택 / 하이라이트되고 가능하면 상세가 열린다.
- detailed 기록만 탄단지 원인 날짜로 연결한다.
- simple / weight_only는 식단 원인 단정을 피하고 기록 보완 중심으로 연결한다.

InBody:
- InBody 기록 입력
- 최신 수치
- 변화 추세
- 중립 관찰
- Records에서 전체 기록 확인
- Records에서 InBody 기록하기로 이동

InBody 탭은 계속 중립 관찰 탭이다.
목표별 식단 행동 조언은 InBody 탭에서 하지 않고 Today Coach 범위로 유지한다.

Settings:
- 목표 및 모드
- 나의 기준
- 기본값 / 프로필 / 장기 기준
- 기본 활동량 / 기본 업무 유형
- 루틴 종류 / 주당 웨이트일수
- 기본 웨이트 시간 후보
- 기본 유산소 프리셋 후보
- 전문가 설정
- 데이터 관리
- 온보딩 다시 하기 후보

Settings는 Today 값을 매일 채워주는 기본값과 장기 기준의 관리 장소다.
모바일 첫 화면에서 거대한 입력폼처럼 보이지 않도록 접힘 구조와 요약을 유지한다.
기존 source of truth 표현은 C4 기준에서 "Settings 기본값 / Today 오늘값" 구조로 재분류한다.
계산용 체중 / 오늘 공복 체중 / InBody 측정 체중은 Settings, Records, InBody의 역할에 맞게 분리해서 말한다.

UX2 완료 기준:
- v6.5-UX2-1: Today 기록 시작 팝업
- v6.5-UX2-2: 기본 계산 입력 Settings 분리
- v6.5-UX2-3: Records 기록 보관함 레이아웃

- v6.5-UX2-5: InBody ↔ Records 연결
- v6.5-UX2-6: 최근 흐름 문제 날짜 → Records 연결
- v6.5-UX2-7: 모바일 / 회귀 QA

[재분류]
UX2 이후 작업은 책임 분리를 유지하되, C4부터 입력값 소유권을 더 명확히 한다.
Today / Settings / Records / InBody가 같은 "체중", "활동", "유산소"를 다룰 때는 값의 역할을 먼저 분리한다.
Today 빠른 수정 shortcut은 결과 확인과 핵심 입력 조정의 왕복을 줄인 과도기 구조였다.
C4 최종 방향은 Today가 오늘 계산값을 직접 소유하고, Settings가 기본값 / 프로필 / 데이터 관리로 재정렬되는 것이다.
기존 기준은 폐기하지 않고 UX2 안정 baseline과 C1/C2 과도기 기준으로 남긴다.

10. 산출값 등급
────────────────────────────────────────

[확정]
산출값은 같은 무게로 쓰지 않는다.

A. 직접 기록값
사용자가 직접 넣은 원자료.
예: 식사 탄단지, alcoholKcal, 체중.
코치 사용 권한: 비교적 직접 사용 가능.

B. 단순 계산값
직접 기록 기반의 합산/차감.
예: macroKcal, totalKcal, 남은 양.
코치 사용 권한: 사용 가능.

C. 모델 추정값
공식/계수/보정이 섞인 추정.
예: BMR, totalBurn, targetCal, macro target.
코치 사용 권한: 보수적으로 사용.

D. 추세 추론값
여러 기록을 묶어 얻는 흐름.
예: 14일 체중 추세, 28일 체성분 흐름.
코치 사용 권한: confidence 필요.

E. 코치 해석값
패턴을 행동 언어로 번역.
예: 반복 부족, 회복 부담, 보정 후보.
코치 사용 권한: 조건 충족 시만 강하게 사용.

11-1. targetCal / totalBurn

등급: C
역할: 오늘 목표 칼로리와 소비 칼로리 추정

사용 가능 window:
1일: 오늘 행동 기준
7일: 평균 섭취 비교
14일: 체중 반응과 비교해 보정 후보
28일: 보정 후보 강화

강한 조언 가능:
- 오늘 목표 초과/부족에 따른 행동 안내

금지:
- 3일 기록만 보고 유지칼로리 오류 단정
- 하루 초과로 목표 실패 단정
- 7일 체중 변화만으로 자동 칼로리 수정

후속:
- targetCalibrationCandidate
- 산출값 대량 QA
- 개인화 보정 후보는 사용자 확인 후 적용

v6.6-A 연결:
targetCal / totalBurn / macro target은 모델 추정값이므로 v6.6-A에서 다양한 체형, 활동, 운동 조건으로 대량 검증한다.
이 QA는 식단 달성률 점수 산식 QA가 아니라 목표 칼로리와 탄단지 산출값의 현실성 검증이다.

11-2. protein target

등급: B~C
역할: 목표별 단백질 g 산출

사용 가능 window:
1일: 오늘 단백질 보완
3일: 연속 부족
7일: 반복 부족
14/28일: 장기 습관 패턴

강한 조언 가능:
- 오늘 severe low
- 최근 7일 usableDietDay >= 4 and proteinLowDays >= 3

금지:
- 평균만 보고 안정적이라고 말하기

필요 산출:
- proteinAvgRatio
- proteinLowDays7
- proteinSevereLowDays7
- proteinLowStreak3
- proteinActionAllowed

문구 방향:
단백질이 자주 부족합니다. 다음 식사에 단백질부터 챙기세요.

11-3. carbs target

등급: C
역할: 단백질/지방 배정 후 남은 칼로리 기반 탄수화물 목표

사용 가능 window:
1일: 운동 전후 보완
3일: 웨이트일 연속 부족
7일: 웨이트일 탄수 패턴
14/28일: 목표 반응 참고 또는 보정 후보

강한 조언 가능:
- 최근 7일 실제 웨이트 기록일 >= 2
- carbLowTrainingDays >= 2

웨이트일 판정:

[확정]
웨이트 날 판정은 사용자가 기록한 실제 웨이트만 기준으로 한다.

즉, 계획상 주당 웨이트일수나 현재 목표 설정만으로 과거 웨이트일을 추정하지 않는다.
기록된 웨이트가 없으면 training-day carb 판단 confidence를 낮춘다.

금지:
- 다이어트 휴식일 carb low를 강하게 보완 지시
- 전체 탄수 평균만 보고 운동 수행 문제 단정

필요 산출:
- carbLowTrainingDays7
- carbLowRestDays7
- trainingDayCarbCoverage
- carbTimingAdviceEligible

문구 방향:
운동한 날에 탄수화물이 자주 부족합니다. 운동 전후로 탄수화물 챙기세요.

11-4. fat target

등급: C
역할: 목표별 지방 g 산출

현재 문제:
fatPct / fatFloor / okMin / cap의 의미가 섞일 수 있다.

사용 가능 window:
1일: high 제한
3일: high 반복
7일: low/high 반복
14일: 장기 low 확인
28일: profile 재검토

강한 조언 가능:

fat high:
- 오늘 high 또는 최근 7일 fatHighDays >= 3

fat low:
- 최근 7일 usableDietDay >= 4
- fatLowDays >= 3~4
- otherKcalShare 낮음

금지:
- 하루 fat low만 보고 g 보완 강하게 말하기
- 현재 fat okMin/fatFloor를 생리학적 하한처럼 부르기

완료:
v6.3-C에서 FAT_TARGET_PROFILES를 도입하고 아래 4개로 분리했다.
- fatPhysioFloor
- fatActionFloor
- fatTargetRange
- fatPracticalCeiling

문구 방향:
지방이 며칠째 낮습니다. 달걀이나 견과류처럼 부담이 적은 지방을 조금 추가하세요.

단, 이 문구는 fat profile 재설계와 recentContext 연결 후에만 허용한다.

11-5. alcohol burden

등급: A~B
역할: 순수 에탄올 kcal 기반 술 영향 판단

사용 가능 window:
1일: 오늘 술 중단
3일: 회복 부담
7일: 술 빈도
14일: 습관 후보

강한 조언 가능:

11-10. v6.6-A 보정식 대량 QA 기준

[확정]
v6.6-A는 첫 실행 온보딩이 아니다.
v6.6-A는 보정식 대량 QA / 산출값 리포트 / 심각한 과보정 수정 후보 확인 단계다.

목적:
산출식이 다양한 사람 / 활동 / 운동 조건에서 현실적인 범위로 작동하는지 검증한다.
이 앱은 기록 기반 영양 코치 앱이므로 화면이 예뻐도 숫자가 과하면 신뢰가 깨진다.
따라서 기능 추가나 온보딩보다 보정식 QA를 먼저 한다.

[확정]
보정식 QA는 단순 PASS/FAIL 테스트가 아니다.
대량 산출값 리포트와 사람이 읽을 수 있는 이상값 리뷰표를 함께 만든다.

[확정]
보정식 QA는 목표 칼로리와 탄단지 산출값의 현실성 검증이다.
식단 달성률 점수 산식 QA가 아니다.

[확정]
보정식 QA는 산출값 리포트부터 시작한다.
하지만 대량 QA에서 명백한 과보정, 비현실값, 특정 조건의 반복 오류가 확인되면 산출식 수정까지 허용한다.

[확정]
QA 없이 산출식을 감으로 바꾸지 않는다.
하지만 문제가 확인됐는데도 "읽기 전용 단계"라는 이유로 수정하지 않는 것도 금지한다.

QA 축:
- 성별: 남자 / 여자
- 키: 남자 160~190, 여자 150~180
- 체중: 남자 50~120, 여자 40~100
- 골격근량: 남자 28~50, 여자 18~40
- 목표: 다이어트 / 린매스업 / 벌크업
- 활동: 낮음 / 보통 / 높음
- 업무강도: 사무 / 서서 일함 / 현장 / 고강도 현장
- 웨이트: 주 0~7회, 오늘 0 / 60 / 90 / 120분, 강도 여러 단계
- 유산소: 0 / 20 / 40 / 60분
- 유산소 경사: 0 / 5 / 10 / 15
- 유산소 속도: 3 / 5 / 7 / 10 / 15 / 20

특히 볼 지점:
- BMR 대비 totalBurn이 과하게 높지 않은가
- 유지칼로리 대비 targetCal이 과하게 높거나 낮지 않은가
- 주간 운동횟수와 오늘 웨이트 시간이 이중으로 과하게 반영되지 않는가
- 활동량과 업무강도가 중복되어 유지칼로리가 과하게 올라가지 않는가
- 유산소 속도 / 경사 / 시간 조합이 비현실적으로 높지 않은가
- EPOC 반영이 과하지 않은가
- 린매스업 targetCal이 사실상 벌크업처럼 나오지 않는가
- 벌크업 targetCal이 고지방 / 술 섭취를 합리화하는 값으로 보이지 않는가
- 다이어트 targetCal이 너무 낮아 단백질 / 훈련 회복을 해치지 않는가
- 남녀 / 키 / 체중 / 골격근량 극단값에서 튀는 결과가 없는가

보정식 QA 산출물:
- 대표 케이스 수동 리뷰표
- 대량 자동 스캔 요약
- 이상값 목록
- 케이스별 보정 분해표
- 변경 전후 비교가 가능한 대표 스냅샷
- 사람이 읽을 수 있는 요약 리포트

보정 분해표 항목:
- BMR
- 생활 활동 보정
- 업무 보정
- 주간 웨이트 보정
- 오늘 웨이트 소모
- 유산소 소모
- EPOC
- 목표 보정
- 최종 targetCal
- 탄수화물
- 단백질
- 지방

이상값 등급:
- 주의: 조금 높거나 낮아 보이지만 사람 리뷰가 필요한 값
- 경고: 특정 조건에서 반복적으로 과하거나 낮은 값
- 심각: 대표 케이스에서 바로 비현실적이거나, 사용자가 그대로 따르면 목표를 망칠 수 있는 값

[확정]
아래 경우에는 보정식 수정이 허용된다.

- 특정 체형 / 성별 / 목표에서 목표 칼로리가 명백히 비현실적인 경우
- 주간 운동횟수와 오늘 운동량이 겹쳐 과한 보정이 반복되는 경우
- 활동량과 업무강도가 겹쳐 과한 유지칼로리가 반복되는 경우
- 유산소 속도 / 경사 / 시간 조합에서 비정상적으로 높은 칼로리가 나오는 경우
- 린매스업 / 벌크업 목표 칼로리가 유지칼로리 대비 과도하게 벌어지는 경우
- 다이어트 목표가 지나치게 낮아 단백질 / 훈련 회복을 해칠 수 있는 경우
- 대표 케이스에서 사람이 봐도 바로 이상하다고 느낄 정도의 값이 반복되는 경우

- 오늘 웨이트 칼로리 상한

- A5.10: 가이드 기준 / 활동량 기준 dual basis 산식표와 카드 안 mini toggle UI를 report-only로 설계했다.
- A5.11: guide/activity dualBasis를 production 경로에 적용했다.

A5.11 production 변경:
- calculate()는 guide basis와 activity basis를 함께 계산한다.
- 기본 선택은 가이드 기준이다.
- 카드 안 mini toggle은 항상 표시한다.
- selected basis에 따라 active targetCal / protein / carbs / fat이 바뀐다.
- Today remaining과 식단 점수에 들어가는 target도 selected basis를 따른다.
- score 산식 자체는 변경하지 않는다.
- Coach decision matrix / recentContext / InBody gate / 저장 schema / import-export는 변경하지 않는다.
- cardioKcal / totalBurn / BMR / activity / work / weight / weeklyTrainingBaselineKcal 산식은 변경하지 않는다.
- selected basis는 uiState / session 범위만 사용하며 저장 schema에는 넣지 않는다.
- ADHERENCE_SCORING_VERSION과 RECENT_CONTEXT_WINDOWS는 유지한다.

[Archive / C4.8-CAL1에서 재해석]
위 A5.11의 "selected basis에 따라 active targetCal이 바뀐다" 기준은 C4.8-CAL1에서 폐기된 과거 구조로 본다.
최신 기준에서는 targetCal은 목표/소비량 기준 authoritative 값이며, guide/activity basis는 targetCal을 덮지 않고 같은 목표 안에서 탄단지 배분과 참고 기준을 보여준다.
Today remaining과 식단 점수는 최신 authoritative targetCal과 그 안에서 조정된 macro 기준을 따른다.

[확정 / C4.8-CAL2 반영]
C4.8-CAL2에서는 dual basis guide의 high / very-high 탄수 범위를 목표별 production soft range와 맞췄다.
린매스업 guide는 8.5g/kg, 벌크업 guide는 10g/kg을 practical max로 보며, 목표 칼로리와 현실적인 탄수 / 지방 범위가 충돌하면 targetCal을 끌어올리거나 탄수를 무리하게 올리지 않는다.
남는 kcal 차이는 A5.1 macro allocation 설명층으로 사용자에게 보여주는 것이 최신 기준이다.
diet targetCal <= totalBurn, CAL1 semantic guard, mode-free current calculation, score / Coach / recentContext / InBody gate / backup schema는 변경하지 않는다.

[후속 / v6.6-A5.11.1 반영]
A5.11 이후 dual basis 도움말은 조금씩 붙이지 않는다.
가이드 기준 / 활동량 기준 mini toggle은 production으로 들어갔지만, 계산 근거와 탄단지 범위 도움말은 후속 A5.12 또는 별도 UX 단계에서 전체 도움말 체계로 설계한다.

dual basis 도움말 후속 후보:
- 가이드 기준이 무엇인지
- 활동량 기준이 무엇인지
- 두 기준이 왜 다를 수 있는지
- 목표 칼로리가 어떻게 계산되는지
- 단백질 / 탄수화물 / 지방 g/kg 범위가 어떻게 잡히는지
- 가이드 기준의 권장 범위
- 활동량 기준에서 권장 범위를 넘는 경우의 설명
- 활동량 기준이 높다고 해서 입력 오류로 단정하지 않는다는 설명
- 탄수화물 / 지방이 권장 범위를 넘을 때 막는 것이 아니라 설명하는 기준
- 사용자가 어느 기준을 참고하면 좋은지

12. bodyCompReliability vs bodyCompTrendConfidence
────────────────────────────────────────

12-1. bodyCompReliability

bodyCompReliability = 체성분 값을 오늘 계산식에 얼마나 반영할지에 대한 신뢰도

역할:
- v6.3-A runtime config 축
- BMR / LBM / 계산 반영 비율에 가까움
- general / athlete / expert preset, homeInbody, expertLbmAlpha와 연결

12-2. bodyCompTrendConfidence

bodyCompTrendConfidence = 최근 체성분 기록 흐름을 코치 판단 근거로 믿어도 되는지에 대한 신뢰도

역할:
- rolling context / recent flow 축
- InBody 기록 수, 측정 간격, 측정 조건 일관성, suspicious flag와 연결
- 체성분 탭의 중립 해석과 오늘 코치의 보조 근거에 사용

12-3. 관계

[확정]
bodyCompReliability와 bodyCompTrendConfidence는 직접 합치지 않는다.

구분:

bodyCompReliability:
체성분 값을 계산에 얼마나 반영할지.
사용 위치: macro calculation / runtime config.

bodyCompTrendConfidence:
체성분 흐름을 해석해도 되는지.
사용 위치: recentContext / 체성분 탭 / Coach 보조 근거.

후속:
두 값은 contextFit에서 약하게 연결할 수 있다.
하지만 하나를 다른 하나의 대체값으로 쓰지 않는다.

────────────────────────────────────────
13. coachConfidence 합성 구조
────────────────────────────────────────

13-1. 구성 요소

recordConfidence:
기록 자체가 판단에 충분한가.

formulaConfidence:
산출식이 그 판단에 적합한가.

windowFit:
선택한 기간이 그 판단에 맞는가.

contextFit:
goalSnapshot, 실제 웨이트 기록, otherKcal, alcohol, volatility 같은 맥락이 맞는가.

13-2. 합성 원칙

[확정]
coachConfidence = min(recordConfidence, formulaConfidence, windowFit, contextFit)

하나라도 낮으면 강한 조언을 하지 않는다.

13-3. Confidence별 문구 강도

High:
강한 행동 조언 가능.
단정적 행동 문구.

Medium:
현재 기록 기준 조언 가능.
“현재 기록 기준으로” 삽입.

Low:
흐름 판단 이르다.
기록 보완 우선.

Blocked:
판단 금지.
기록 입력 요청.

예시:

High:
단백질이 자주 부족합니다. 다음 식사에 단백질부터 챙기세요.

Medium:
현재 기록 기준으로 단백질이 자주 부족합니다. 다음 식사에 단백질부터 챙기세요.

Low:
식사 기록이 더 필요해요. 오늘 먹은 주요 식사부터 상세 기록으로 남겨주세요.

13-4. 오늘 코치에서 confidence 노출

[확정]
오늘 코치에는 최근 흐름 근거와 confidence 상태를 항상 노출한다.

단, 사용자-facing 화면에는 confidence라는 단어를 그대로 노출하지 않는다.

기본 화면에서는 아래처럼 쉬운 표현으로 번역한다.

- 최근 기록 기준
- 기록이 아직 부족합니다
- 식사 기록이 더 필요해요
- 현재 기록 기준으로

개발 문서와 테스트에서는 coachConfidence 용어를 사용한다.

노출 내용은 confidence에 따라 달라진다.

High / Medium:
최근 흐름 근거를 행동 문구와 함께 보여준다.

Low / Blocked:
최근 흐름 근거 대신 기록 보완 안내를 보여준다.

예:

최근 7일 중 4일 단백질이 부족했습니다.

또는:

식사 기록이 더 필요해요. 오늘 먹은 주요 식사부터 상세 기록으로 남겨주세요.

14. 강한 조언 허가 조건
────────────────────────────────────────

protein low 반복:
recent7 usableDietDay >= 4, proteinLowDays >= 3

carb low 웨이트일:
recent7 실제 웨이트 기록일 >= 2, carbLowTrainingDays >= 2

fat low:
recent7 usableDietDay >= 4, fatLowDays >= 3~4, otherKcalShare 낮음

fat high:
오늘 high 또는 recent7 fatHighDays >= 3

alcohol high:
오늘 high 이상이면 즉시 허가

alcohol frequency:
recent7 alcoholDays >= 2~3

weight trend:
recent14 weightLoggedDays >= 8

InBody trend:
recent28 내 비교 가능한 기록 >= 2, suspicious flag 없음

target recalibration:
14일 이상 체중/섭취 흐름 + 충분한 기록

record quality warning:
mealLoggedDays 부족, singleMealDays 많음, otherKcalHeavyDays 존재

16-17. v6.5-UX1 / UX2

[확정]
v6.5-UX1은 공통 카드형 UI 시스템과 Today 화면 리디자인 실험이다.
UX2는 그 다음 단계로, 화면을 더 예쁘게 만드는 것이 아니라 탭 책임을 다시 나누는 작업이었다.

UX2에서 확정한 구조:
- Today는 결과 / 행동 / 빠른 기록 시작점으로 남긴다.
- 기본 계산 입력은 Settings의 계산 기준 설정으로 분리한다.
- Records는 입력 탭이 아니라 날짜별 기록 보관함이다.
- Records 상세 안의 "수정하기"가 기본 수정 경로다.
- Records 하단 "기록 직접 관리 / 날짜 직접 입력 / 백업"은 보조 영역이다.
- InBody 탭은 입력 / 최신 수치 / 변화 추세 / 중립 관찰 중심이다.
- InBody 기록 목록과 날짜별 확인은 Records에서 자연스럽게 본다.
- 최근 흐름은 문제를 말하고 끝나지 않고 Records의 문제 날짜로 연결한다.
- simple / weight_only 기록에서는 탄단지 원인을 단정하지 않는다.

v6.5-UX2-8은 위 구조를 모바일 390px과 회귀 QA 기준으로 닫은 이전 안정 기준이다.
v6.5-UX2-9는 그 위에 Today Coach summary와 첫 번째 Coach block의 동일 action 반복을 제거한 기준이다.
v6.5-UX2-10은 Recent Flow / InBody hint / otherKcal / 기록 부족의 P1 출력 중복 후보를 정리한 닫힌 UX 구조 baseline이다.

17. 테스트 기준
────────────────────────────────────────

17-1. P0

1. 식사 0끼: macro low/under 판단 차단, 기록 보완 문구 출력
2. 식사 1끼 + alcohol high: alcohol stop은 허용, low kcal/protein 단정은 차단
3. 식사 2끼: 강한 차단 금지, “현재 기록 기준” 문구로 downgrade
4. 오늘 severe protein low + 최근 7일 안정: 오늘 protein 행동 우선
5. 오늘 fat low + 최근 7일 안정: 강한 fat low 행동 금지
6. 최근 7일 protein 반복 부족: action escalate
7. recent7 usableDietDay < 4: 반복 패턴 판단 보류
8. goal change within 7/14 days: 목표 변경 감지 먼저 표시
9. goal change 이후: 과거는 snapshot 기준, 오늘은 today goal 기준
10. snapshot coverage 낮음: historical comparison 강도 down
11. otherKcal heavy day 존재: macro 해석보다 기록 품질 안내 우선
12. weight 7일은 있으나 14일 부족: 진행 추세 단정 금지
13. InBody 1회 기록: trend 단정 금지
14. InBody 2회라도 suspicious flag 존재: 강한 번역 금지
15. alcoholDays7 높음 + 오늘 alcohol high: 오늘 술 중단 우선
16. carb low on diet rest day: 강한 탄수 보완 금지
17. carb low on actual weight training days lean/bulk: 운동 전후 보완 허용
18. bodyCompReliability와 bodyCompTrendConfidence 혼동 방지
19. Weekly recent7 helper가 recentContext engine을 대체하지 않도록 분리
20. simple record는 macro component count를 만들지 않음
21. detailed record는 항상 자동 식단 점수 계산 + goalSnapshot 저장
22. Today Coach는 simple-only 최근 흐름에서 단백질/탄수/지방 반복을 단정하지 않음
23. InBody high + simple-only 최근 식단은 강한 식단 action으로 번역하지 않음

17-2. P1

1. manual adherence day는 상태 결론에 약하게만 반영
2. component frequency 강한 근거는 auto/snapshot 기반 day 우선
3. scoringVersion legacy day가 섞여도 component summary는 안전하게 보조 처리
4. Today Coach에 최근 기록 상태가 항상 보이는지 확인
5. Records quality summary와 coachConfidence label이 충돌하지 않는지 확인
6. 5일 window가 기본 UI와 기본 판단에 노출되지 않는지 확인
7. simple / detailed / weight_only count가 recentContext에서 분리되는지 확인
8. 최근 흐름 UI가 상세 기록일과 간단 기록일을 한 평균처럼 말하지 않는지 확인
9. simple → detailed, detailed → simple 전환에서 데이터 손실 확인이 필요한 경로를 테스트로 잠글 것

17-3. P2

1. Low confidence copy가 회피형/GPT체 없이 동작하는지 확인
2. 목표별 문구 차이가 유지되는지 확인
3. 내부 용어(signal, penalty, decision, confidence 등)가 사용자 화면에 노출되지 않는지 확인
4. 탭명 변경이 껍데기 변경으로 끝나지 않는지 확인
5. "기록이 있습니다", "현재 기록 기준으로 볼 수 있습니다" 같은 빈 문구가 다시 들어오지 않는지 확인

Stitch UI 흡수 / 주요 화면 UI 정리:
[확정]
Stitch UI 작업은 기존 UX2 책임 구조를 대체하지 않는다.
Today / Records / 최근 흐름 / InBody / Settings의 책임 분리는 유지한다.

[확정 / v6.6-C4.8-S0]
Stitch의 화면 밀도, 배치 감각, 버튼 우선순위, 결과와 입력의 가까운 흐름은 강하게 흡수한다.
단, 현재 앱의 산식, 기록 신뢰성, 데이터 책임 분리, A5.11 dual basis와 충돌하는 구조는 그대로 가져오지 않는다.
Stitch는 단순 참고보다 강한 목표 UI 감각에 가깝게 보되, 과거 실험 UI이므로 현재 앱의 기능과 책임 구조를 대체하지 않는다.
계산 구조, 기록 구조, Coach 판단 구조를 바꾸는 근거로 사용하지 않는다.
신규 주요 action과 navigation은 SVG icon + 풀네임 라벨을 우선한다.
기존 emoji / 장식 요소는 후속 UI 감사에서 정리한다.
중요한 action은 icon 단독으로 전달하지 않는다.
badge / chip / status / action의 역할을 구분하고 장식성 badge 남발을 피한다.
사용자가 결정을 내려야 하는 흐름은 inline 버튼보다 modal / bottom sheet를 우선 검토하고, 상태 안내는 inline을 유지한다.
모바일 390px에서 overflow 없는 구조를 우선한다.

[확정 / v6.6-C1]
C1 핵심 사용 루프 감사 결론은 "visual polish"가 아니라 입력 → 결과 확인 → 기록 → 수정 → 백업/불러오기 → 모드/목표 설정 흐름의 재정리다.
현재 UI 문제는 탭 수 자체보다 사용자가 지금 계산 기준, 기록값, 측정값 중 무엇을 바꾸는지 헷갈리는 지점이 큰 것으로 본다.
가장 먼저 줄일 왕복은 계산 기준 수정 → 결과 확인 루프다.
Stitch에서 흡수할 점은 결과와 입력이 가까운 흐름, 빠른 설정 접근성, 첫 화면에서 오늘 할 일 / 기록 시작 / 결과가 한 덩어리로 보이는 느낌, 카드 밀도와 버튼 우선순위다.
Stitch에서 폐기할 점은 Today에 긴 계산 입력폼을 되돌리는 구조, Records를 입력폼 중심으로 되돌리는 구조, A5.11 dual basis를 무시하는 macro 카드, 최근 흐름 / Records / InBody 책임을 섞는 구조다.

22. 최종 한 줄 기준
────────────────────────────────────────

이 기능은 최근 흐름을 믿을 수 있는지 판단한 뒤,
오늘 행동으로 번역하는 데 도움이 되는가?

도움이 되지 않는다면, 단순 수치 추가나 평균 표시 기능은 우선순위가 낮다.

최종 구조는 아래 순서를 따른다.

1. 기록한다.
2. 기록을 믿을 수 있는지 본다.
3. 산출값의 등급과 산출식 신뢰도를 본다.
4. window가 그 판단에 맞는지 본다.
5. 목표와 운동 맥락이 맞는지 본다.
6. 최근 흐름을 해석한다.
7. 오늘 행동으로 번역한다.
8. confidence가 낮으면 기록 보완을 먼저 말한다.
