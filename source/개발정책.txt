개발정책.txt

이 문서는 탄단지 다이어리 HTML 앱의 개발 운영 기준이다.

────────────────────────
1. 1인 개발 단계 원칙
────────────────────────

이 프로젝트는 1인 개발 프로젝트이며, 현재 실질 사용자는 개발자 본인이다.

따라서 개발 단계에서는 존재하지 않는 사용자의 반발이나 불편을 이유로 기준 변경, 구조 개선, UI 리팩토링을 피하지 않는다.
UI/UX에서 사용 편의성을 고려하는 것은 당연하지만, 실제 업데이트로 인해 불편함을 겪거나 기준이 바뀌었다고 이상함을 느낄 외부 사용자는 없다는 전제를 둔다.

개발 방향을 잡을 때는 최종 구조가 편하게 잡히는 것을 우선한다.
가능성을 충분히 열어두되, 무의미한 단계를 늘리지 않는다.

개발 중 임시 기록 데이터 보존은 최우선 기준이 아니다.
현재 records / manual / localStorage / import-export 데이터는 개발 과정의 임시 기록이며, 최종 완성 앱에서 사용자가 처음부터 다시 입력할 수 있다는 전제를 둔다.
기존 개발 중 데이터를 보호하려고 schema, recordMode, import/export 구조를 복잡하게 만들지 않는다.

legacy migration, mixed legacy, backward compatibility는 사용자가 명시적으로 요청하지 않는 한 우선순위가 낮다.
기존 데이터가 깨지더라도 최종 구조가 명확해지는 방향을 우선하며, 죽은 코드와 compatibility alias를 누적하지 않는다.

다만 변경 범위가 큰 작업은 한 번에 무작정 밀어붙이지 않는다.
점수 산식, 저장 구조, Weekly, InBody, goalSnapshot, import/export처럼 영향이 큰 축은 범위를 분리해서 다룬다.

────────────────────────
2. 계획 우선 / Codex 결과 검토 원칙
────────────────────────

이 프로젝트에서 현재 소스나 Codex 결과가 최종 기준은 아니다.

기준 우선순위는 다음과 같다.

1. 사용자가 만들고 싶은 앱의 성격
2. 시간순으로 확정된 설계 의도
3. 최신 기준 문서
4. 최신 index.html
5. 테스트 결과
6. Codex 구현 결과

Codex가 그럴듯하게 구현했거나 테스트를 통과했더라도, 원래 계획과 제품 방향에 맞지 않으면 그대로 따르지 않는다.

Codex 결과 추종 방지 기준:
- Codex 결과가 좋아 보여도 기존 설계 기준과 충돌하면 수정하거나 보류한다.
- UI 시안은 참고 자료이지 제품 철학을 대체하지 않는다.
- 구현 결과를 합리화하지 말고, 사전에 합의한 기준과 맞는지 먼저 본다.
- 테스트 통과만으로 승인하지 않고 사용자 체감 흐름과 앱 성격을 확인한다.
- Codex에게 아이디어 정리와 우선순위 판단을 맡기지 않는다.
- 사용자와 GPT가 목적, 범위, 금지 범위, 우선순위를 먼저 확정한 뒤 Codex는 문서 반영과 구현을 맡는다.

특히 아래 경우에는 Codex 결과를 기준으로 삼지 않는다.

- 문구만 늘리고 판단 구조는 그대로인 경우
- goal-aware라고 말하지만 실제 우선순위는 공통 로직인 경우
- 카드 override가 목표별 판단을 덮어버리는 경우
- 점수 / 저장 / Weekly / InBody를 불필요하게 건드린 경우
- 테스트 통과를 이유로 사용자 체감 가치가 약한 변경을 합리화하는 경우
- 현재 소스 구조를 보존하기 위해 최종 구조 정합성을 낮추는 경우
- 프롬프트 결과를 뒤쫓느라 원래 설계 순서를 흐리는 경우

작업 후 검토 순서는 다음을 기본으로 한다.

1. 원래 계획과 맞는가
2. 사용자가 원하는 앱 성격과 맞는가
3. 목표별 판단 구조가 실제로 들어갔는가
4. 범위를 넘지 않았는가
5. 문구가 앱 기준에 맞는가
6. 다음 단계로 가도 되는 상태인가

2-2. 기준 문서 충돌 정리 원칙

기존 기준 문서와 새 판단이 충돌할 때, 충돌을 피하는 것이 목적이 아니다.
새 판단이 제품 완성도, 사용자 신뢰도, 장기 구조에 더 적합하면 기존 문서를 수정한다.

단, 기존 기준을 바꿀 때는 이유와 범위를 문서에 남긴다.
무엇을 폐기했는지, 무엇을 보류했는지, 무엇을 후속으로 재분류했는지 구분한다.

문서화 작업에서도 기존 후보를 조용히 삭제하지 않는다.
삭제 / 유지 / 보류 / 후속 / 재분류가 필요한 이유를 남기고, 다른 문서와 충돌하지 않게 정리한다.

C4.8-S0처럼 사용자 작업 흐름과 구조 문제가 새로 확인된 경우에는 구현보다 기준문서 갱신을 먼저 한다.
로드맵은 고정된 의식 절차가 아니라, 사용자가 실제로 앱을 쓰는 흐름과 기록 신뢰성을 기준으로 재정렬할 수 있다.
다만 재정렬할 때도 구현 완료처럼 쓰지 않고, 이전 흐름 / 새 후보 / 보류 항목을 구분한다.

2-3. C4.8 이후 최신 판단 기준

2026-05-18 현재 기준으로 C4.8-M / CAL / UI 흐름에서 아래 판단을 우선한다.

- 사용자-facing 일반인 / 운동인 / 전문가 전역 모드는 제거된 방향으로 본다.
- 내부 legacy mode는 당장 삭제 대상이 아니라 남은 호환 코드일 수 있지만, 현재 계산의 주도권은 실제 입력 기반 calculation context가 가진다.
- 목표 칼로리는 목표와 소비량 기준의 authoritative 값이며, guide/activity basis는 목표 칼로리를 덮어쓰는 기능이 아니라 같은 목표 칼로리 안에서 탄단지 배분과 참고 기준을 보여주는 기능이다.
- 다이어트에서 targetCal이 totalBurn을 초과하는 회귀는 제품 의미상 P0로 본다.
- 산출식 변경은 테스트 통과만으로 승인하지 않고, 생리학 / 스포츠영양학 관점에서 개인 사용자가 보기에 이상하지 않은지 확인한다.
- Stitch UI는 단순 참고보다 강한 시각 목표다. 다만 Stitch의 기능을 가져오는 것이 아니라, 우리 앱 기능을 Stitch 수준의 화면 밀도와 리듬으로 배치한다.
- UI 작업은 스크린샷 QA를 기본으로 한다. 구현 여부만 보지 말고 색, 폰트, 폰트 크기, 카드 줄맞춤, 여백, 버튼/칩 밀도, 줄바꿈, inline SVG 품질을 확인한다.
- 책임 분리는 목적이 아니라 수단이다. 분리 때문에 사용자 흐름이 딱딱 끊기거나 화면이 관리폼처럼 보이면, Stitch식 연결감을 우선해 재배치할 수 있다.
- source 문서가 이전 설계 순서를 고정하고 있으면, 최신 사용자 의도와 실제 구현 판단에 맞춰 갱신한다.

3. 계획 수정 원칙
────────────────────────

계획은 지켜야 하지만, 계획 자체가 틀렸다고 판단되면 계획 수정도 허용한다.

단, 계획을 수정할 때는 이유를 분명히 해야 한다.
예를 들어 코드 꼬임 방지, 회귀 원인 추적, 점수 산식 영향, 저장 구조 영향처럼 실제 이유가 있어야 한다.

“최소 변경이 편해서”, “기존 구조를 보존하고 싶어서”, “없는 사용자가 혼란스러워할까 봐” 같은 이유만으로 계획을 약하게 만들지 않는다.

────────────────────────
4. 작업 범위 관리 원칙
────────────────────────

작업은 목표별로 범위를 분리한다.

예:
- Coach 판단 구조 작업에서 점수 산식을 같이 바꾸지 않는다.
- 입력 UI 작업에서 저장 schema를 불필요하게 바꾸지 않는다.
- 문구 polish 작업에서 decision / gate / card policy를 바꾸지 않는다.
- version/header 마감 작업에서 ADHERENCE_SCORING_VERSION을 바꾸지 않는다.

큰 구조 개선이 필요하면 피하지 않는다.
다만 서로 다른 축을 한 프롬프트에 섞어서 회귀 원인 추적이 어려워지는 상황은 피한다.

산출식처럼 앱 신뢰도에 직접 영향을 주는 작업은 기능 추가보다 QA를 우선한다.
목표 칼로리, totalBurn, 탄단지 목표처럼 사용자가 그대로 믿고 행동할 수 있는 값은 다양한 체형 / 활동 / 운동 조건에서 먼저 검증한다.
보정식 QA에서 심각한 과보정이나 비현실값이 확인되면 산출식 수정까지 허용하되, 식단 달성률 점수 산식, Coach decision matrix, recentContext, InBody gate, 저장 schema와 한 번에 섞지 않는다.

targetCal 또는 탄단지 산출식이 production 경로에서 바뀌는 경우에는 대표 케이스 변경 전후값과 full scan 결과를 함께 남긴다.
대표 케이스에는 목표, 활동, 운동, 유산소 조건, targetCal 변화, 탄수화물 / 단백질 / 지방 변화, 남는 차이 변화가 포함되어야 한다.
full scan에는 target conflict, high-cardio conflict, 탄수화물 / 지방 최대치, 기존 정상 케이스 영향 여부를 확인한다.

산출식 수정은 산출식 축으로 닫는다.
targetCal / macro 변경을 이유로 점수 산식, Coach decision matrix, recentContext, InBody gate, 저장 schema, import/export를 함께 바꾸지 않는다.

source 문서와 index.html version notes가 서로 다른 사실을 말하면 merge / tag 전에 충돌로 본다.
산식 변경 단계에서는 문서의 완료 이력, index.html header notes, 테스트 로그가 같은 변경 범위와 금지 범위를 가리키는지 확인한다.

도움말 / 툴팁은 부분 문구를 조금씩 누적하지 않는다.
산식 신뢰도와 관련된 도움말은 도움말 위치, 접힘 구조, ? 팝업, 계산 근거 상세, 모바일 표시, 문구 길이, 테스트 기준을 함께 설계한 뒤 적용한다.
특히 dual basis처럼 targetCal / macro / remaining / 식단 점수 기준과 연결되는 설명은 index.html notes와 source 문서가 같은 사실을 말해야 한다.

온보딩처럼 앱 전체 진입 흐름에 영향을 주는 기능은 주요 UI 안정화 이후 진행한다.
화면 구조와 주요 사용 흐름이 계속 바뀌는 단계에서 온보딩을 먼저 만들면 재작업 가능성이 높으므로, 첫 실행 온보딩은 장기 후순위 후보로 둔다.

────────────────────────
5. UX 묶음 병합 기준
────────────────────────

UX 묶음은 중간 단계가 아니라 최종 회귀 QA까지 닫힌 기준으로 본다.

- 중간 브랜치는 무조건 master에 합치지 않는다.
- UX 묶음은 마지막 회귀 QA 후 master 병합 / tag한다.
- 중간 단계에서 tag를 잘못 찍으면 push 전 복구한다.
- master 병합 전에는 source 문서 기준과 구현 결과가 충돌하지 않는지 확인한다.

master merge 전 확인:
- working tree clean
- 브라우저 QA 완료
- 전체 테스트 통과
- 금지 범위 유지
- source 문서 기준과 충돌 없음

────────────────────────
6. Git 기준
────────────────────────

- 기능 브랜치에서 작업한다.
- QA 후 commit / push한다.
- master 병합은 최종 기준에서만 진행한다.
- tag는 master 병합 후 HEAD에 찍는다.
- 커밋 메시지는 한글 중심으로 쓴다.

────────────────────────
7. UX2 확정 구조 보호 / C4 입력 책임 재분류
────────────────────────

v6.5-UX2-10 기준으로 아래 탭 책임을 흐리지 않는다.

- Today는 오늘 행동과 기록 시작점이다.
- Records는 날짜별 기록 보관함이다.
- 최근 흐름은 반복 문제를 Records 근거 날짜와 연결한다.
- InBody 탭은 입력 / 최신 수치 / 변화 추세 / 중립 관찰을 유지한다.
- Settings는 계산 기준 관리 장소였던 UX2 baseline을 가진다.

InBody 탭은 목표별 행동 조언을 하지 않는다.
Records 하단 기록 직접 관리 영역은 기본 수정 경로가 아니라 날짜 직접 입력 / 보조 관리 / 백업 fallback 영역이다.
Records 상세 안의 "수정하기"가 기본 수정 경로다.

UX 개선은 탭 수를 줄이는 것보다 사용자가 지금 바꾸는 값의 역할과 사용 루프를 명확히 하는 것을 우선한다.
계산 기준, 기록값, 측정값이 같은 단어로 보이면 사용자가 흐름을 오해하므로 라벨과 IA에서 역할을 분리한다.

v6.6-C2 / C3 이후에는 기존 "Settings source of truth / Today shortcut" 전제를 과도기 기준으로 재분류한다.
사용자는 Today에서 값을 넣으면 Today 결과가 바로 계산되기를 기대하므로, 최종 제품 구조에서는 Today가 오늘 계산값을 직접 소유한다.
Settings는 기본값 / 프로필 / 장기 기준 / 데이터 관리 위치로 재정의한다.
기존 기준을 조용히 삭제하지 말고, UX2 안정 baseline 또는 C1/C2 과도기 기준으로 남긴다.

입력값 소유권 원칙:
- Today 값: 계산용 체중, 오늘 활동량, 오늘 업무 유형, 오늘 웨이트 세션 / 시간, 오늘 유산소 종류 / 시간 / 속도 / 경사, guide/activity basis, 목표 칼로리 / 탄단지 / 남은 양 / 점수 기준
- Settings 기본값: 목표, 모드, 키, 성별, 나이, 기본 활동량, 기본 업무 유형, 루틴 종류, 주당 웨이트일수, 기본 웨이트 시간 후보, 기본 유산소 프리셋 후보, 전문가 설정, 데이터 관리
- Records 기록값: 오늘 공복 체중, 오늘 메모, 식사 기록, 기록 방식, 날짜별 기록
- InBody 측정값: 측정일, 측정 체중, 골격근량, 체지방량, 체지방률

Today 계산값은 날짜 단위 working value 후보로 둔다.
개발명 후보로 todayCalculationDraft를 쓸 수 있지만 사용자-facing에는 draft를 쓰지 않는다.
schema/localStorage 변경은 후속 구현 단계에서 별도 판단한다.
개발 중 임시 데이터 보존보다 최종 구조 명확성이 우선이라는 기존 원칙을 유지한다.
Codex는 기존 source 문서의 낡은 기준에 맞춰 보수적으로 구현하기보다, 최신 제품 구조가 명시된 경우 그 구조를 우선한다.

Records는 날짜별 기록 보관함과 상세 수정 책임을 유지한다.
Records 하단 직접 관리 영역을 메인 입력 / 수정 경로처럼 강화하지 않는다.
백업 / 불러오기는 Records 보관함의 일부라기보다 앱 전체 데이터 관리 축으로 재검토할 수 있으며, Settings 하단 또는 별도 데이터 관리 섹션 후보로 분리한다.

C4.8 이후 Stitch UI 흡수 우선순위:
1. 기능 보존
2. 기록 신뢰성
3. 데이터 책임 분리
4. 사용자 작업 흐름 명확성
5. Stitch UI 강한 흡수
6. 기존 코드 구조 편의성

Stitch의 화면 밀도, 배치 감각, 버튼 우선순위, 결과와 입력의 가까운 흐름은 강하게 흡수한다.
단, 현재 앱의 산식, 기록 신뢰성, 데이터 책임 분리, A5.11 dual basis와 충돌하는 구조는 그대로 가져오지 않는다.
Stitch 구조를 그대로 복원하거나, A5.11 dual basis / UX2 탭 책임 분리를 약화하는 근거로 쓰지 않는다.

Today는 오늘 기록을 시작하고 갱신하는 작업공간이고, 저장된 결과는 Records의 날짜별 기록으로 남는다.
Records는 확정 기록을 확인하고 수정하는 보관함이다.

전체 백업 / 기록만 내보내기 기준:
- 전체 백업은 다른 기기 / 브라우저에서도 앱 상태를 이어 쓰기 위한 백업이다.
- 전체 백업은 Settings / Today 계산값 / Records / InBody / 유산소 프리셋 등 앱 상태를 포함하는 방향으로 본다.
- 기록만 내보내기는 저장된 날짜별 기록 중심의 경량 내보내기다.
- 기록만 내보내기는 현재 Today 계산값이나 Settings 전체 상태를 이어 쓰는 용도가 아니다.
- InBody 기록 포함 여부는 실제 export 범위와 맞춰 문구를 유지한다.
- 다른 기기에서 앱 상태를 이어 쓰려면 전체 백업을 우선 안내한다.

Today 값 적용 패턴은 프리셋 적용형, 기록 적용형, 기준 유지형, 일회성 반영형, 기본값 시작형으로 나눠 문서화한다.
프리셋 / 기록 적용 / 기준 유지 / 일회성 반영을 모두 자동 반영처럼 구현하거나 설명하지 않는다.

설명문 / 도움말 원칙:
- 기본 화면은 라벨 + 값 + 버튼 중심으로 구성한다.
- 설명은 기본 숨김을 원칙으로 하고, 헷갈릴 가능성이 큰 곳만 ? / 접힘 / 짧은 보조문구로 둔다.
- 실수하면 큰 영향이 있는 곳만 짧은 경고를 기본 노출한다.
- 설명문은 Codex가 독단적으로 삭제하지 않는다.
- 먼저 helper / hint / mini-note / desc-box / field-note / notice / warn / details / tooltip / empty state / card subtitle / button 주변 설명문을 전수감사한다.
- 사용자와 GPT가 삭제 / 축약 / ? 이동 / 접힘 이동 / 유지 여부를 판단한 뒤 일괄 수정한다.
