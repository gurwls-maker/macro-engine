개발정책.txt

이 문서는 탄단지 다이어리 HTML 앱의 개발 운영 기준이다.

────────────────────────
1. 1인 개발 단계 원칙
────────────────────────

이 프로젝트는 1인 개발 프로젝트이며, 현재 실질 사용자는 개발자 본인이다.

따라서 개발 단계에서는 존재하지 않는 사용자의 반발이나 불편을 이유로 기준 변경, 구조 개선, UI 리팩토링을 피하지 않는다.
UI/UX에서 사용 편의성을 고려하는 것은 당연하지만, 실제 업데이트로 인해 불편함을 겪거나 기준이 바뀌었다고 이상함을 느낄 외부 사용자는 없다는 전제를 둔다.

개발 방향을 잡을 때는 최종 구조가 편하게 잡히는 것을 우선한다.
가능성을 충분히 열어두되, 무의미한 단계를 늘리지 않는다.

다만 변경 범위가 큰 작업은 한 번에 무작정 밀어붙이지 않는다.
점수 산식, 저장 구조, Weekly, InBody, goalSnapshot, import/export처럼 영향이 큰 축은 범위를 분리해서 다룬다.

────────────────────────
2. 계획 우선 / Codex 결과 검토 원칙
────────────────────────

이 프로젝트에서 현재 소스나 Codex 결과가 최종 기준은 아니다.

기준 우선순위는 다음과 같다.

1. 사용자가 만들고 싶은 앱의 성격
2. 시간순으로 확정된 설계 의도
3. 최신 기준 문서
4. 최신 index.html
5. 테스트 결과
6. Codex 구현 결과

Codex가 그럴듯하게 구현했거나 테스트를 통과했더라도, 원래 계획과 제품 방향에 맞지 않으면 그대로 따르지 않는다.

특히 아래 경우에는 Codex 결과를 기준으로 삼지 않는다.

- 문구만 늘리고 판단 구조는 그대로인 경우
- goal-aware라고 말하지만 실제 우선순위는 공통 로직인 경우
- 카드 override가 목표별 판단을 덮어버리는 경우
- 점수 / 저장 / Weekly / InBody를 불필요하게 건드린 경우
- 테스트 통과를 이유로 사용자 체감 가치가 약한 변경을 합리화하는 경우
- 현재 소스 구조를 보존하기 위해 최종 구조 정합성을 낮추는 경우
- 프롬프트 결과를 뒤쫓느라 원래 설계 순서를 흐리는 경우

작업 후 검토 순서는 다음을 기본으로 한다.

1. 원래 계획과 맞는가
2. 사용자가 원하는 앱 성격과 맞는가
3. 목표별 판단 구조가 실제로 들어갔는가
4. 범위를 넘지 않았는가
5. 문구가 앱 기준에 맞는가
6. 다음 단계로 가도 되는 상태인가

────────────────────────
3. 계획 수정 원칙
────────────────────────

계획은 지켜야 하지만, 계획 자체가 틀렸다고 판단되면 계획 수정도 허용한다.

단, 계획을 수정할 때는 이유를 분명히 해야 한다.
예를 들어 코드 꼬임 방지, 회귀 원인 추적, 점수 산식 영향, 저장 구조 영향처럼 실제 이유가 있어야 한다.

“최소 변경이 편해서”, “기존 구조를 보존하고 싶어서”, “없는 사용자가 혼란스러워할까 봐” 같은 이유만으로 계획을 약하게 만들지 않는다.

────────────────────────
4. 작업 범위 관리 원칙
────────────────────────

작업은 목표별로 범위를 분리한다.

예:
- Coach 판단 구조 작업에서 점수 산식을 같이 바꾸지 않는다.
- 입력 UI 작업에서 저장 schema를 불필요하게 바꾸지 않는다.
- 문구 polish 작업에서 decision / gate / card policy를 바꾸지 않는다.
- version/header 마감 작업에서 ADHERENCE_SCORING_VERSION을 바꾸지 않는다.

큰 구조 개선이 필요하면 피하지 않는다.
다만 서로 다른 축을 한 프롬프트에 섞어서 회귀 원인 추적이 어려워지는 상황은 피한다.
